<td class="answercell">
    <div class="post-text" itemprop="text">
<p>The number of map tasks for a given job is driven by the number of input splits and not by the mapred.map.tasks parameter. For each input split a map task is spawned. So, over the lifetime of a mapreduce job the number of map tasks is equal to the number of input splits. mapred.map.tasks is just a hint to the InputFormat for the number of maps.</p>

<p>In your example Hadoop has determined there are 24 input splits and will spawn 24 map jobs in total. But, you can control how many map tasks can be executed in parallel by each of the task tracker.</p>

<p>Also, removing a space after -D might solve the problem for reduce.</p>

<p>For more information on the number of map and reduce tasks, please look at the below url</p>

<p><a href="http://wiki.apache.org/hadoop/HowManyMapsAndReduces">http://wiki.apache.org/hadoop/HowManyMapsAndReduces</a></p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>As Praveen mentions above, when using the basic <code>FileInputFormat</code> classes is just the number of input splits that constitute the data. The number of reducers <em>is</em> controlled by <code>mapred.reduce.tasks</code> specified in the way you have it: <code>-D mapred.reduce.tasks=10</code> would specify 10 reducers. Note that the space after <code>-D</code> is <em>required;</em> if you omit the space, the configuration property is passed along to the relevant JVM, not to Hadoop.</p>

<p>Are you specifying <code>0</code> because there is no reduce work to do? In that case, if you're having trouble with the run-time parameter, you can also set the value directly in code. Given a <code>JobConf</code> instance <code>job</code>, call</p>

<pre><code>job.setNumReduceTasks(0);
</code></pre>

<p>inside, say, your implementation of <code>Tool.run</code>. That should produce output directly from the mappers. If your job actually produces no output whatsoever (because you're using the framework just for side-effects like network calls or image processing, or if the results are entirely accounted for in Counter values), you can disable output by also calling</p>

<pre><code>job.setOutputFormat(NullOutputFormat.class);
</code></pre>

<p>It's important to keep in mind that the MapReduce framework in Hadoop allows us only to </p>

<blockquote>
  <p><strong>suggest</strong> the number of Map tasks for a job</p>

<blockquote>
  <p><strong>demand</strong> that it provide n reducers.</p>
</blockquote>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>To explain it with a example:</p>

<p>Assume your hadoop input file size is 2 GB and you set block size as 64 MB so 32 Mappers tasks are set to run while each mapper will process 64 MB block to complete the Mapper Job of your Hadoop Job.</p>

<p>==> Number of mappers set to run are completely dependent on 1) File Size and 2) Block Size</p>

<p>Assume you have running hadoop on a cluster size of 4:
Assume you set mapred.map.tasks and mapred.reduce.tasks parameters in your conf file to the nodes as follows: </p>

<pre><code>Node 1: mapred.map.tasks = 4 and mapred.reduce.tasks = 4
Node 2: mapred.map.tasks = 2 and mapred.reduce.tasks = 2
Node 3: mapred.map.tasks = 4 and mapred.reduce.tasks = 4
Node 4: mapred.map.tasks = 1 and mapred.reduce.tasks = 1
</code></pre>

<p>Assume you set the above paramters for 4 of your nodes in this cluster. If you notice Node 2 has set only 2 and 2 respectively because the processing resources of the Node 2 might be less e.g(2 Processors, 2 Cores) and Node 4 is even set lower to just 1 and 1 respectively might be due to processing resources on that node is 1 processor, 2 cores so can't run more than 1 mapper and 1 reducer task.</p>

<p>So when you run the job Node 1, Node 2, Node 3, Node 4 are configured to run a max. total of (4+2+4+1)11 mapper tasks simultaneously out of 42 mapper tasks that needs to be completed by the Job. After each Node completes its map tasks it will take the remaining mapper tasks left in 42 mapper tasks.</p>

<p>Now comming to reducers, as you set mapred.reduce.tasks = 0 so we only get mapper output in to 42 files(1 file for each mapper task) and no reducer output.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>In your example, the <code>-D</code> parts are not picked up:</p>

<pre><code>hadoop jar Test_Parallel_for.jar Test_Parallel_for Matrix/test4.txt Result 3 \ -D mapred.map.tasks = 20 \ -D mapred.reduce.tasks =0
</code></pre>

<p>They should come after the classname part like this:</p>

<pre><code>hadoop jar Test_Parallel_for.jar Test_Parallel_for -Dmapred.map.tasks=20 -Dmapred.reduce.tasks=0 Matrix/test4.txt Result 3
</code></pre>

<p>A space after <code>-D</code> is allowed though.</p>

<p>Also note that changing the number of mappers is probably a bad idea as other people have mentioned here.</p>

<p>From what I understand reading above, it depends on the input files. If Input Files are 100 means - Hadoop will create 100 map tasks.
However, it depends on the Node configuration on How Many can be run at one point of time.
If a node is configured to run 10 map tasks - only 10 map tasks will run in parallel by picking 10 different input files out of the 100 available.
Map tasks will continue to fetch more files as and when it completes processing of a file.</p>
