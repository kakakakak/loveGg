<p><strong>Solution:</strong></p>

<p>This can be solved by configuring Hive to add a jar via Hive CLI to its class path as follows:</p>

<pre><code>hive&gt; ADD JAR [path to JSON SerDe jar file];
</code></pre>

<p>For example:</p>

<pre><code>hive&gt; ADD JAR /usr/lib/hive/lib/json-serde-1.1.4-jar-with-dependencies.jar;
</code></pre>

<p>Hive will confirm the addition by returning the following statement:</p>

<pre><code>Added /usr/lib/hive/lib/json-serde-1.1.4-jar-with-dependencies.jar to class path
Added resource: /usr/lib/hive/lib/json-serde-1.1.4-jar-with-dependencies.jar
</code></pre>

<p>The above has to be executed at the start of every Hive session.</p>

<p><strong>Explanation:</strong></p>

<p>The query presented by the original question produces errors because of the select-from clause. If the following query was submitted to Hive CLI, the same error will be experienced:</p>

<pre><code>SELECT
   id
FROM tweets_sample;
</code></pre>

<p>The source table tweets_sample has its rows stored in JSON SerDe format. This can be seen from the query that generated tweets_sample at the end of the question:</p>

<pre><code>CREATE EXTERNAL TABLE tweets_sample (
   id BIGINT,
   ...
   hour int
)
ROW FORMAT SERDE 'org.openx.data.jsonserde.JsonSerDe'
LOCATION '/data/oct25_tweets';
</code></pre>

<p>By default, Hive does not know how to parse or extract columns in this format. One would notice that the following query will still actually work even before adding the JSON SerDe jar file:</p>

<pre><code>SELECT *
FROM tweets_sample;
</code></pre>

<p>This query works because Hive does not need to extract elements from a particular column within a row and, thus, does not need to know what the format of a row is.</p>

<p>By specifying the JSON SerDe jar file before executing any JSON SerDe-format-dependent queries as presented in the solution above, Hive will know how to execute such queries.</p>
