<td class="answercell">
    <div class="post-text" itemprop="text">
<p>In my day job, I've just spent some time helping a C++ person to pick up enough Java to use some Java libraries via JNI (Java Native Interface) and then shared memory into their primarily C++ application. Here are some of the key things I noticed:</p>

<ol>
<li>You cannot manage for anything beyond a toy project without an IDE. The very first thing you should do is download a popular Java IDE (Eclipse is a fine choice, but there are also alternatives including Netbeans and IntelliJ). Do not be tempted to try and manage with vi / emacs and javac / make. <strong>You will be living in a cave and not realising it</strong>. Once you're up to speed with even basic IDE functions you will be literally dozens of times more poductive than without an IDE.</li>
<li>Learn how to layout a simple project structure and packages. There will be simple walkthroughs of how to do this on the Eclipse site or elsewhere. Never put anything into the default package.</li>
<li>Java has a type system whereby the reference and primitive types are relatively separate for historic / performance reasons.</li>
<li>Java's generics are <strong>not</strong> the same as C++ templates. Read up on "type erasure".</li>
<li>You may wish to understand how Java's GC works. Just google "mark and sweep" - at first, you can just settle for the naivest mental model and then learn the details of how a modern production GC would do it later.</li>
<li>The core of the Collections API should be learned without delay. Map / HashMap, List / ArrayList &amp; LinkedList and Set should be enough to get going.</li>
<li>Learn modern Java concurrency. Thread is an assembly-language level primitive compared to some of the cool stuff in java.util.concurrent. Learn ConcurrentHashMap, Atomic*, Lock, Condition, CountDownLatch, BlockingQueue and the threadpools from Executors. Good books here are those by Brian Goetz and Doug Lea.</li>
<li>As soon as you want to use 3rd party libraries, you'll need to learn how the classpath works. It's not rocket science, but it is a bit verbose.</li>
</ol>

<p>If you're a low-level C++ guy, then you may find some of this interesting also:</p>

<ol>
<li>Java has virtual dispatch by default. The keyword static on a Java method is used to indicate a class method. private Java methods use invokespecial dispatch, which is a dispatch onto the exact type in use.</li>
<li>On an Oracle VM at least, objects comprise two machine words of header (the mark word and the class word). The mark word is a bunch of flags the VM uses - notably for thread synchronization. The class word you can think of as a pointer to the VM's representation of the Class object (which is where the vtables for methods live). Following the class word are the member fields of the instance of the object.</li>
<li>Java .class files are an intermediate language, and not really that similar to x86 object code. In particular there are lots more useful tools for .class files (including the javap disassembler which ships with the JVM)</li>
<li>The Java equivalent of the symbol table is called the Constant Pool. It's typed and it has a lot of information in it - arguably more than the x86 object code equivalent.</li>
<li>Java virtual method dispatch consists of looking up the correct method to be called in the Constant Pool and then converting that to an offset into a vtable. Then walking up the class hierarchy until a not-null value is found at that vtable offset.</li>
<li>Java starts off interpreted and then goes compiled (for Oracle and some other VMs anyway). The switch to compiled mode is done method-by-method on a as-need basis. When benchmarking and perf tuning you need to make sure that you've warmed the system up before you start, and that you should typically profile at the method level to start with. The optimizations that are made can be quite aggressive / optimistic (with a check and a fallback if the assumptions are violated) - so perf tuning is a bit of an art.</li>
</ol>

<p>Hopefully there's some useful stuff in there to be going on with - please comment / ask followup questions.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Learning "just enough" Java is learning Java.  Either you learn all the core principles and language design decisions, or you suffer along making easily avoidable mistakes.  Considering that you already know how to program, a lot of the information can be skimmed (with an eye for where it differs from other languages you are intimately familiar).</p>

<p>so you need to learn:</p>

<ol>
<li>How to <a href="http://download.oracle.com/javase/tutorial/getStarted/index.html">get started</a></li>
<li><a href="http://download.oracle.com/javase/tutorial/java/index.html">The language itself</a></li>
<li>The core, <a href="http://download.oracle.com/javase/tutorial/essential/index.html">essential classes</a></li>
<li>The <a href="http://download.oracle.com/javase/tutorial/collections/index.html">major Collections</a></li>
</ol>

<p>And if you don't have a build framework in place, how to <a href="http://download.oracle.com/javase/tutorial/deployment/index.html">package your compiled code</a>.</p>

<p>Beyond that, nearly every other item you might need to learn depends heavily on what you intend to do.  Don't discount <a href="http://download.oracle.com/javase/tutorial/">the on-line tutorials from Oracle/Sun</a>, they are quite good (compared to other online tutorials).</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I don't know how familiar are you with other higher level programming languages. <strong>Garbage collection</strong> is an important function in Java. It would be important to read a bit about the GC in your VM of choice.</p>

<p>Besides the obvious packages, check out the <a href="http://download.oracle.com/javase/6/docs/api/java/util/package-summary.html" rel="nofollow">java.util</a> packages for the <a href="http://download.oracle.com/javase/6/docs/technotes/guides/collections/index.html" rel="nofollow">collection framework</a>. You might want to check out the source of some classes. I suggest <code>HashMap</code> to get the idea of the computing/memory cost of these operations.</p>

<p>Java likes to use <strong>streams</strong> instead of buffers when processing large amounts of data. That may take some time getting used to.</p>

<p>Java has no unsigned types. Depending on the packets of data you need to process at once you can either use larger variables and streight arythetics (if we're talking about relatively small packets), or you have to <code>(b[i] &amp; 0xff)</code> every time you read for example unsigned bytes. Also note that Java uses network byte order (msbf) when serializing multibyte numbers.</p>

<p>The most beloved design patterns by the API are Singleton, Decorator and Factory. Check the source of JFC itself for best practices, how these patterns are achieved in the language.</p>

<p>... and you can still post more concrete questions on SO :)</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p><strong>Answer 1 :</strong></p>

<ul>
<li>It is very desirable to know Java. Hadoop is written in Java. Its popular Sequence File format is dependent on Java. </li>
<li>Even if you use Hive or Pig, you'll probably need to write your own UDF someday. Some people still try to write them in other languages, but I guess that Java has more robust and primary support for them.</li>
<li>Most Hadoop tools are not mature enough (like Sqoop, HCatalog and so on), so you'll see many Java error stack traces and probably you'll want to hack the source code someday</li>
</ul>

<p><strong>Answer 2</strong></p>

<ul>
<li>It is not required for you to know Java.</li>
<li>As the others said, it would be very helpful depending on how complex your processing may be. However, there is an incredible amount you can do with just Pig and say Hive. </li>
<li>I would agree that it is fairly likely you will eventually need to write a user defined function (UDF), however, I've written those in Python, and it is very easy to write UDFs in Python. </li>
<li>Granted, if you have very stringent performance requirements, then a Java based MapReduce program would be the way to go. However, great advancements in performance are being made all of the time in both Pig and Hive. </li>
<li>So, the short answer to your question is, "No", it is not required for you to know Java in order to perform Hadoop development.</li>
</ul>
