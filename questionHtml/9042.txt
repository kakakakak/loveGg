    <div class="post-text" itemprop="text">

<p>I just gave <a href="http://www.tutorialspoint.com/hadoop/hadoop_introduction.htm" rel="nofollow">this Hadoop tuorial</a> a read which state that Hadoop has an Achilles' tendon (a single point of failure) in <code>JobTracker</code>:</p>

<blockquote>
  <p>The JobTracker is a single point of failure for the Hadoop MapReduce service which means if JobTracker goes down, all running jobs are halted.</p>
</blockquote>

<p>And also <a href="https://developer.yahoo.com/hadoop/tutorial/module7.html" rel="nofollow">this article</a> which states that the <code>NameNode</code> is a SPOF:</p>

<blockquote>
  <p>The single point of failure in a Hadoop cluster is the NameNode.</p>
</blockquote>

<p>Single points of failure are bad, mkay? <strong>What strategies/techniques/tooling/etc. can be done to circumvent these SPOFs and make Hadoop redundant, faul-tolerant and resilient (buzzword alert!)?</strong></p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>HDFS and Mapreduce are the core components in Hadoop,  In the earlier Apache Hadoop releases, Namenode and Jobtracker were SPOF (Only one instance can be configured). This problem is fixed from Hadoop 2.X. <br>
<br>
<strong>Jobtracker HA.</strong>
<br><br> Jobtracker HA can be achived by configuring 2 <code>Jobtracker(JT)</code> instance in <code>Active - Standby</code> mode on two nodes. If one JT goes down, second Jobtracker will be available to serve the request. Only one jobtracker(Active) will be available for serving request at a time, second JT(Standby ) will be running in Read only mode. Jobtracker HA requires zookeeper instance, Failure over(switching) can be configured as either Manaul or Automcatic. Automatic failover  requires another process called <code>Failover Controller (FC)</code>.  In the current release,  if active  JT  fails, all running jobs will be halted, However new job will automatically  be submitted to new JT.  This functionality is not available in the current release. </p>

<p><code>MR2</code> is the second Generation of mapreduce which uses YARN, <code>Resource Manager(RM)</code> is the master service in YARN, RM can also be configured in Active-Standby mode. RM failure will not impact running Jobs/Application.</p>

<p><strong>Namenode HA</strong>
<br><br>    Namenode HA is something important. Namenode HA can also be configured in Active-Standby mode(Maximum 2 namenode instances). <code>Quorum based Journaling</code> is the Widely accepted method, which internally uses zookeeper. Only one namenode will be active at a time.</p>

<p><code>Secondary Namenode(SNN)</code> is not a <code>Standby Namenode(SN)</code> and vice versa, SNN has a different functionaly in Non HA configuration,  Namenode HA set up doesn't require SNN, as SN namenode performs checkpointing (Functionality of SNN)</p>

<p><code>Processes Namenode HA</code></p>

<ul>
<li>Active namenode  </li>
<li>Standby namenode </li>
<li>Failover controller : For Fencing to avoid split-brain scenario. </li>
<li>Jounalnodes ( Min 3 instances required) : Namespace modfication will
be logged to Journal nodes and Standby namenode reads from there. Only one namenode will be allowed to write at a time inorder to avoide split-brain issue. </li>
</ul>

<p>There are <code>High availability</code> mechanisms built into Hadoop for a while.
'Secondary NameNode', 'Backup JobTracker' will serve as a hot backup of their respective counterparts.</p>

<p>Most of the past 'SPOF' has been eliminated with recent hadoop releases.</p>

<p>It is explained in the following docs in depth.</p>

<ul>
<li><a href="http://docs.hortonworks.com/HDPDocuments/HDP2/HDP-2.2.0/Hadoop_HA_v22/index.html#Item1.1" rel="nofollow">Hortonworks HA Doc</a></li>
