<p>What are the benefits of using either <strong>Hadoop</strong> or <strong>HBase</strong> or <strong>Hive</strong> ?</p>

<p>From my understanding, <strong>HBase</strong> avoids using map-reduce and has a column oriented storage on top of HDFS. <strong>Hive</strong> is a sql-like interface for <strong>Hadoop</strong> and <strong>HBase</strong>. </p>

<p>I would also like to know how <strong>Hive</strong> compares with <strong>Pig</strong>.</p>

<p>MapReduce is just a <strong>computing framework</strong>. HBase has nothing to do with it. That said, you can efficiently put or fetch data to/from HBase by writing MapReduce jobs. Alternatively you can write sequential programs using other HBase APIs, such as Java, to put or fetch the data. But we use Hadoop, HBase etc to deal with gigantic amounts of data, so that doesn't make much sense. Using normal sequential programs would be highly inefficient when your data is too huge.</p>

<p>Coming back to the first part of your question, Hadoop is basically 2 things: a <strong>Distributed FileSystem (HDFS)</strong> + a <strong>Computation or Processing framework (MapReduce)</strong>. Like all other FS, HDFS also provides us storage, but in a fault tolerant manner with high throughput and lower risk of data loss (because of the replication). But, being a FS, HDFS lacks <strong>random read and write access</strong>. This is where HBase comes into picture. It's a <strong>distributed, scalable, big data store</strong>, modelled after Google's BigTable. It stores data as key/value pairs.</p>

<p>Coming to Hive. It provides us data <strong>warehousing</strong> facilities on top of an existing Hadoop cluster. Along with that it provides an <strong>SQL like</strong> interface which makes your work easier, in case you are coming from an SQL background. You can create tables in Hive and store data there. Along with that you can even map your existing HBase tables to Hive and operate on them.</p>

<p>While Pig is basically a <strong>dataflow language</strong> that allows us to process enormous amounts of data very easily and quickly. Pig basically has 2 parts: the Pig <strong>Interpreter</strong> and the language, <strong>PigLatin</strong>. You write Pig script in PigLatin and using Pig interpreter process them. Pig makes our life a lot easier, otherwise writing MapReduce is always not easy. In fact in some cases it can really become a pain.</p>

<p>I had written an <a href="http://smartdatacollective.com/mtariq/120791/hadoop-toolbox-when-use-what" rel="nofollow">article on a short comparison of different tools of the Hadoop ecosystem</a> some time ago. It's not an in depth comparison, but a short intro to each of these tools which can help you to get started.
(Just to add on to my answer. No self promotion intended)</p>

<p><strong>Both Hive and Pig queries get converted into MapReduce jobs under the hood.</strong></p>

<p>HTH</p>

<p>I implemented a Hive Data platform recently in my firm and can speak to it in first person since I was a one man team. </p>

<h1>Objective</h1>

<ol>
<li>To have the daily web log files collected from 350+ servers daily queryable thru some SQL like language</li>
<li>To replace daily aggregation data generated thru MySQL with Hive</li>
<li>Build Custom reports thru queries in Hive </li>
</ol>

<h1>Architecture Options</h1>

<p>I benchmarked the following options
1. Hive+HDFS
2. Hive+HBase - queries were too slow so dumped this option</p>

<h1>Design</h1>

<ol>
<li>Daily log Files were transported to HDFS</li>
<li>MR jobs parsed these log files and output files in HDFS</li>
<li>Create Hive tables with partitions and locations pointing to HDFS locations</li>
<li>Create Hive query scripts (call it HQL if u like as diff from SQL) that in turn ran MR jobs in the background and generated aggregation data</li>
<li>Put all these steps into an Oozie workflow - scheduled with Daily Oozie Coordinator</li>
</ol>

<h1>Summary</h1>

<p>HBase is like a Map. If u know the key, u can instantly get the value. But if u want to know how many integer keys in Hbase are between 1000000 and 2000000 that is not suitable for Hbase alone.
If you have data to be aggregated, rolled up, analyzed across rows then consider Hive</p>

<p>hopefully  this helps.
Good luck. Hive actually rocks really well...I know I have lived it for 12 months now...so does Hbase....  </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>First of all we should get clear that <strong>Hadoop</strong> was created as a faster alternative to <strong>RDBMS</strong>. To process large amount of data at a very fast rate which earlier took a lot of time in RDBMS.</p>

<p>Now one should know the two terms : </p>

<ol>
<li><p><strong>Structured Data</strong> : This is the data that we used in traditional RDBMS and is divided into well defined structures.</p></li>
<li><p><strong>Unstructured Data</strong> : This is important to understand, about 80% of the world data is unstructured or semi structured. These are the data which are on its raw form and cannot be processed using RDMS. Example : facebook, twitter data. (<a href="http://www.dummies.com/how-to/content/unstructured-data-in-a-big-data-environment.html" rel="nofollow">http://www.dummies.com/how-to/content/unstructured-data-in-a-big-data-environment.html</a>).</p></li>
</ol>

<p><em>So, large amount of data was being generated in the last few years and the data was mostly unstructured, that gave birth to HADOOP. It was mainly used for very large amount of data that takes unfeasible amount of time using RDBMS. It had many drawbacks, that it could not be used for comparatively small data in real time but they have managed to remove its drawbacks in the newer version.</em></p>

<p><em>Before going further I would like to tell that a new Big Data tool is created when they see a fault on the previous tools. So, whichever tool you will see that is created has been done to overcome the problem of the previous tools.</em></p>

<p><strong>Hadoop</strong> can be simply said as two things : <strong>Mapreduce</strong> and <strong>HDFS</strong>. Mapreduce is where the processing takes place and HDFS is the DataBase where data is stored. This structure followed <strong>WORM</strong> principal i.e. write once read multiple times. So, once we have stored data in HDFS, we cannot make changes. This led to the creation of <strong>HBASE</strong>, a NOSQL product where we can make changes in the data also after writing it once.</p>

<p><em>But with time we saw that Hadoop had many faults and for that we created different environment over the Hadoop structure. PIG and HIVE are two popular examples.</em></p>

<p><strong>HIVE</strong> was created for people with <strong>SQL</strong> background. The queries written is similar to SQL named as <strong>HIVEQL</strong>. HIVE was developed to process completely <strong>structured data</strong>. It is not used for ustructured data.</p>

<p><strong>PIG</strong> on the other hand has its own query language i.e. <strong>PIG LATIN</strong>. It can be used for both <strong>structured</strong> as well as <strong>unstructured data</strong>.</p>

<p>Moving to the difference as when to use HIVE and when to use PIG, I don't think anyone other than the architect of PIG could say. Follow the link :
<a href="https://developer.yahoo.com/blogs/hadoop/comparing-pig-latin-sql-constructing-data-processing-pipelines-444.html" rel="nofollow">https://developer.yahoo.com/blogs/hadoop/comparing-pig-latin-sql-constructing-data-processing-pipelines-444.html</a></p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p><a href="https://hadoop.apache.org/" rel="nofollow"><strong>Hadoop</strong></a> is a a framework that allows for the distributed processing of large data sets across clusters of computers using simple programming models.</p>

<p>There are four main modules in Hadoop.</p>

<ol>
<li><p><strong>Hadoop Common</strong>: The common utilities that support the other Hadoop modules.</p></li>
<li><p>Hadoop Distributed File System (<strong>HDFS™</strong>): A distributed file system that provides high-throughput access to application data.</p></li>
<li><p>Hadoop <strong>YARN</strong>: A framework for job scheduling and cluster resource management.</p></li>
<li><p>Hadoop <strong>MapReduce</strong>: A YARN-based system for parallel processing of large data sets.</p></li>
</ol>

<p>Before going further, Let's note that we have three different types of data.</p>

<ul>
<li><p><strong>Structured</strong>: Structured data has strong schema and schema will be checked during write &amp; read operation. e.g. Data in RDBMS systems like Oracle, MySQL Server etc.</p></li>
<li><p><strong>Unstructured</strong>: Data does not have any structure and it can be any form - Web server logs, E-Mail, Images etc.</p></li>
<li><p><strong>Semi-structured</strong>: Data is not strictly structured but have some structure. e.g. XML files.</p></li>
</ul>

<p><strong>Depending on type of data to be processed, we have to choose right technology.</strong> </p>

<p>Some more projects, which are part of Hadoop:</p>

<ul>
<li><p><strong>HBase™</strong>: A scalable, distributed database that supports structured data storage for large tables.</p></li>
<li><p><strong>Hive</strong>™: A data warehouse infrastructure that provides data summarization and ad-hoc querying.</p></li>
<li><p><strong>Pig™</strong>: A high-level data-flow language and execution framework for parallel computation.</p></li>
</ul>

<p><strong>Hive Vs PIG</strong> comparison can be found at my other post in this <a href="http://stackoverflow.com/questions/3356259/difference-between-pig-and-hive-why-have-both/33353322#33353322">question</a>.</p>

<p><strong>HBASE</strong> won't replace Map Reduce. <strong>HBase</strong> is scalable distributed database &amp; <strong>Map Reduce</strong> is programming model for distributed processing of data. Map Reduce may act on data in HBASE in processing. </p>

<p>You can use <strong>HIVE/HBASE</strong> for structured/semi-structured data and process it with Hadoop Map Reduce</p>

<p>You can use <strong>SQOOP</strong> to import structured data from traditional RDBMS database Oracle, SQL Server etc and process it with Hadoop Map Reduce</p>

<p>You can use <strong>FLUME</strong> for processing Un-structured data and process with Hadoop Map Reduce</p>

<p>Have a look at: <a href="http://hadoopilluminated.com/hadoop_illuminated/Hadoop_Use_Cases.html" rel="nofollow">Hadoop Use Cases</a>.</p>

<p><strong>Hive</strong> should be used for analytical querying of data collected over a period of time. e.g Calculate trends, summarize website logs but it can't be used for real time queries.</p>

<p><strong>HBase</strong> fits for real-time querying of Big Data. Facebook use it for messaging and real-time analytics.</p>

<p><strong>PIG</strong> can be used to construct dataflows, run a scheduled jobs, crunch big volumes of data, aggregate/summarize it and store into relation database systems. Good for ad-hoc analysis. </p>

<p><strong>Hive</strong> can be used for ad-hoc data analysis but it can't support all un-structured data formats unlike PIG.</p>

<p>I am a beginner in Hadoop, currently i am working on a project with Hadoop    and Pig Latin. with a little experience i have i can comment on Pig vs   Hive.     </p>

<p>Here is the comparision:   </p>

<ul>
<li>Pig Latin is procedural, where Hive is declarative.</li>
<li>Pig Latin allows pipeline developers to decide where to checkpoint data   in the pipeline.</li>
<li>Pig Latin allows the developer to select specific operator    implementations directly rather than relying on the optimizer.</li>
<li>Pig Latin supports splits in the pipeline.   </li>
<li>Pig Latin allows developers to insert their own code almost anywhere in    the data pipeline.</li>
</ul>
