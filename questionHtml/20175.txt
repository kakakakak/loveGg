<p>The interesting line is this:</p>

<pre><code>&lt;caller organisation="org.apache.hadoop" name="hadoop-hdfs" conf="default, compile, runtime, master" rev="6.1.26" rev-constraint-default="6.1.26" rev-constraint-dynamic="6.1.26" callerrev="2.2.0"/&gt;
</code></pre>

<p>I'm not familiar with Ivy, but I'm guessing it means the <code>hadoop-hdfs</code> dependency wants jetty 6.1.26, when everything else wants <code>6.1.26.cloudera.4</code> or <code>6.1.26.cloudera.2</code></p>

<p>I'm now trying to find the best way to make this work.</p>

<p>update: Found a solution</p>

<p>I don't know if this is a <em>good</em> solution - As I said, I don't know much about Apache Ivy - but it worked for me, and maybe it will work around this issue for you...</p>

<p>I added the following line to sqoop's <code>ivy.xml</code> file, just after the other <code>exclude</code> elements:</p>

<p><code>
&lt;exclude org="org.mortbay.jetty" module="jetty" type="zip" /&gt;
</code></p>

<p>I hope somebody will correct me if I'm wrong, but this should prevent Ivy from trying to use the <code>zip</code> type artifact of jetty, and so it will fall-back to something else. In this case, it looks to me that it's using the <code>jar</code> type. Since the jar type is available at the repos ivy is already trying to use, it works.</p>

<p>At least, that's what it looks like to me :)</p>
