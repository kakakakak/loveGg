<p>I'm suffering an issue similar to <a href="http://stackoverflow.com/questions/6575742/maven-2-different-dependency-versions-in-test-and-compile">Maven 2 - different dependency versions in test and compile</a> but the specified answer there does not work.</p>

<p>In my project I need to depend on a Cloudera distribution of Hadoop and a 'vanilla' version for JUnit testing, as the former only works on *nix.</p>

<p>When I try and execute my application, I get <code>Exception in thread "main" java.lang.NoClassDefFoundError: org/apache/hadoop/conf/Configuration</code>. When I run JUnit tests from Maven or Eclipse, everything works fine. If I comment out the <code>test</code> dependencies, the application runs successfully.</p>

<p>Why is the <code>compile</code> dependency getting ignored when the <code>test</code> dependency is uncommented?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Looks like a maven bug - or if not, something that is fishy enough I wouldn't trust the documentation if there were any.</p>

<p>In your case, I would probably:</p>

<ol>
<li>upgrade maven and see if the issue is fixed in the latest m3</li>
<li>try repackaging one of the hadoop versions with another artifact id or group name. maven-shade-plugin, don't bother changing the package name, but just introduce something maven won't know is no different from org.apache.hadoop:hadoop-core</li>
<li>try moving the tests to a separate project downstream</li>
<li>start debugging maven and understand what's going on exactly (and then fall back on 1 or 2...)</li>
</ol>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>What you desire, is no longer valid for Maven 3 (this used to be valid for Maven 2). Maven 3 will attempt to obtain the nearest dependency, effectively ensuring that only one of the compile or test scoped dependency is used for both the compile and test phases.</p>

<p>In your case <code>org.apache.hadoop:hadoop-core:1.0.0:test</code> overrides <code>org.apache.hadoop:hadoop-core:0.20.2-cdh3u2:compile</code> and hence becomes the nearest dependency. You may see the following warning displayed by Maven when running the <code>mvn dependency:list</code> goal, that alludes to this being a problem in your project model: </p>

<blockquote>
  <p>[WARNING] 'dependencies.dependency.(groupId:artifactId:type:classifier)' must be
  unique: org.apache.hadoop:hadoop-core:jar -> version 0.20.2-cdh3u2 vs 1.0.0 @ line xyz, column xyz</p>
</blockquote>

<p>To 'fix' this, it would be better to split your tests into a separate project whose project model can define a separate set of test dependencies.</p>
