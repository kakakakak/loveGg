    <div class="post-text" itemprop="text">

<p>I am trying to append to a file on an hdfs on a single node cluster. I also tried on a 2 node cluster but get the same exceptions.</p>

<p>In hdfs-site, I have <code>dfs.replication</code> set to 1. If I set <code>dfs.client.block.write.replace-datanode-on-failure.policy</code> to <code>DEFAULT</code> I get the following exception</p>

<pre><code>java.io.IOException: Failed to replace a bad datanode on the existing pipeline due to no more good datanodes being available to try. (Nodes: current=[10.10.37.16:50010], original=[10.10.37.16:50010]). The current failed datanode replacement policy is DEFAULT, and a client may configure this via 'dfs.client.block.write.replace-datanode-on-failure.policy' in its configuration.
</code></pre>

<p>If I follow the recommendation in the comment for the <a href="http://hadoop.apache.org/docs/r2.3.0/hadoop-project-dist/hadoop-hdfs/hdfs-default.xml" rel="nofollow">configuration in hdfs-default.xml</a> for extremely small clusters (3 nodes or less) and set <code>dfs.client.block.write.replace-datanode-on-failure.policy</code> to <code>NEVER</code> I get the following exception:</p>

<pre><code>org.apache.hadoop.ipc.RemoteException(org.apache.hadoop.hdfs.server.namenode.SafeModeException): Cannot append to file/user/hadoop/test. Name node is in safe mode.
The reported blocks 1277 has reached the threshold 1.0000 of total blocks 1277. The number of live datanodes 1 has reached the minimum number 0. In safe mode extension. Safe mode will be turned off automatically in 3 seconds.
</code></pre>

<p>Here's how I try to append:</p>

<pre><code>Configuration conf = new Configuration();
conf.set("fs.defaultFS", "hdfs://MY-MACHINE:8020/user/hadoop");
conf.set("hadoop.job.ugi", "hadoop");

FileSystem fs = FileSystem.get(conf);
OutputStream out = fs.append(new Path("/user/hadoop/test"));

PrintWriter writer = new PrintWriter(out);
writer.print("hello world");
writer.close();
</code></pre>

<p>Is there something I am doing wrong in the code? 
maybe, there is something missing in the configuration?
Any help will be appreciated!</p>

<p><strong>EDIT</strong></p>

<p>Even though that <code>dfs.replication</code> is set to <code>1</code>, when I check the status of the file through</p>

<pre><code>FileStatus[] status = fs.listStatus(new Path("/user/hadoop"));
</code></pre>

<p>I find that <code>status[i].block_replication</code> is set to <code>3</code>. I don't think that this the problem because when I changed the value of <code>dfs.replication</code> to <code>0</code> I got a relevant exception. So apparently it does indeed obey the value of <code>dfs.replication</code> but to be on the safe side, is there a way to change the <code>block_replication</code> value per file?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>As I mentioned in the edit. Even though the <code>dfs.replication</code> is set to <code>1</code>, <code>fileStatus.block_replication</code> is set to 3.</p>

<p>A possible solution is to run </p>

<pre><code>hadoop fs -setrep -w 1 -R /user/hadoop/
</code></pre>

<p>Which will change the replication factor for each file recursively in the given directory. Documentation for the command can be found <a href="http://hadoop.apache.org/docs/r0.19.0/hdfs_shell.html#setrep">here</a>.  </p>

<p>What to be done now is to look why the value in hdfs-site.xml is ignored. And how to force the value <code>1</code> to be the default. </p>

<p><strong>EDIT</strong></p>

<p>It turns out that the <code>dfs.replication</code> property has to be set in the <code>Configuration</code> instance too, otherwise it requests that the replication factor for the file be the default which is 3 regardless of the value set in hdfs-site.xml</p>

<p>Adding to the code the following statement will solve it.</p>

<pre><code>conf.set("dfs.replication", "1");
</code></pre>
