<p>I'm running neo4j version 2.2.5. I love all the CYPHER language, Python integration, ease of use, and very responsive user community. </p>

<p>I've developed a prototype of an application and am encountering some very poor performance times. I've read a lot of links related to performance tuning. I will attempt to outline my entire database here so that someone can provide guidance to me. </p>

<p>My machine is a MacBook Pro, 16GB of RAM, and 500GB SSD. It's very fast for everything else I do in Spark + Python + Hadoop. It's fast for Neo4j too, BUT when I get to like 2-4M nodes then it's insanely slow. </p>

<p>I've used both of these commands to start up neo4j, thinking they will help, and neither is that helpful:</p>

<pre><code>./neo4j-community-2.2.5/bin/neo4j start -Xms512m -Xmx3g -XX:+UseConcMarkSweepGC
./neo4j-community-2.2.5/bin/neo4j start -Xms512m -Xmx3g -XX:+UseG1GC
</code></pre>

<p>My neo4j.properties file is as follows:</p>

<pre><code>################################################################
# Neo4j
#
# neo4j.properties - database tuning parameters
#
################################################################

# Enable this to be able to upgrade a store from an older version.
#allow_store_upgrade=true

# The amount of memory to use for mapping the store files, in bytes (or
# kilobytes with the 'k' suffix, megabytes with 'm' and gigabytes with 'g').
# If Neo4j is running on a dedicated server, then it is generally recommended
# to leave about 2-4 gigabytes for the operating system, give the JVM enough
# heap to hold all your transaction state and query context, and then leave the
# rest for the page cache.
# The default page cache memory assumes the machine is dedicated to running
# Neo4j, and is heuristically set to 75% of RAM minus the max Java heap size.
dbms.pagecache.memory=6g

# Enable this to specify a parser other than the default one.
#cypher_parser_version=2.0

# Keep logical logs, helps debugging but uses more disk space, enabled for
# legacy reasons To limit space needed to store historical logs use values such
# as: "7 days" or "100M size" instead of "true".
#keep_logical_logs=7 days

# Enable shell server so that remote clients can connect via Neo4j shell.
#remote_shell_enabled=true
# The network interface IP the shell will listen on (use 0.0.0 for all interfaces).
#remote_shell_host=127.0.0.1
# The port the shell will listen on, default is 1337.
#remote_shell_port=1337

# The type of cache to use for nodes and relationships.
#cache_type=soft
</code></pre>

<p>To create my database from a fresh start, I first create these indexes, they are on all of my node types, and edges that I'm using.</p>

<pre><code>CREATE CONSTRAINT ON (id:KnownIDType) ASSERT id.id_type_value IS UNIQUE; 
CREATE CONSTRAINT ON (p:PerspectiveKey) ASSERT p.perspective_key IS UNIQUE;
CREATE INDEX ON :KnownIDType(id_type);
CREATE INDEX ON :KnownIDType(id_value);
CREATE INDEX ON :KNOWN_BY(StartDT);
CREATE INDEX ON :KNOWN_BY(EndDT);
CREATE INDEX ON :HAS_PERSPECTIVE(Country);
</code></pre>

<p>I have 8,601,880 nodes. </p>

<p>I run this query, and it takes 9 minutes. </p>

<pre><code>MATCH (l:KnownIDType { id_type:'CodeType1' })&lt;-[e1:KNOWN_BY]-(m:KnownIDType { id_type:'CodeType2' })-[e2:KNOWN_BY]-&gt;(n:KnownIDType)&lt;-[e3:KNOWN_BY]-(o:KnownIDType { id_type:'CodeType3' })-[e4:KNOWN_BY]-&gt;(p:KnownIDType { id_type:'CodeType4' }), (n)-[e5:HAS_PERSPECTIVE]-&gt;(q:PerspectiveKey {perspective_key:100})
WHERE 1=1
AND l.id_type IN ['CodeType1']
AND m.id_type IN ['CodeType2']
AND n.id_type IN ['CodeTypeA', 'CodeTypeB', 'CodeTypeC']
AND o.id_type IN ['CodeType3']
AND p.id_type IN ['CodeType4']
AND 20131231 &gt;= e1.StartDT and 20131231 &lt; e1.EndDT
AND 20131231 &gt;= e2.StartDT and 20131231 &lt; e2.EndDT
AND 20131231 &gt;= e3.StartDT and 20131231 &lt; e3.EndDT
AND 20131231 &gt;= e4.StartDT and 20131231 &lt; e4.EndDT
WITH o, o.id_value as KnownIDValue, e5.Country as Country, count(distinct p.id_value) as ACount
WHERE AmbiguousCount &gt; 1
RETURN 20131231 as AsOfDate, 'CodeType' as KnownIDType, 'ACount' as MetricName, count(ACount) as MetricValue
;
</code></pre>

<p>I'm looking for more like 15s or less response time. Like I do with &lt; 1M nodes. </p>

<p>What would you suggest? I am happy to provide more information if you tell me what you need. </p>

<p>Thanks a bunch in advance.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Here are a couple of ideas how to speed up your query:</p>

<ul>
<li>Don't use <code>IN</code> if there is only one element. Use <code>=</code></li>
<li>With a growing number of nodes, the index lookup will obviously take longer. Instead of having a single label with an indexed property, you could use the <code>id_type</code> property as label. Something like <code>(l:KnownIDTypeCode1)&lt;-[e1:KNOWN_BY]-(m:KnownIDTypeCode2)</code>.</li>
<li>Split up the query in two parts. First <code>MATCH</code> your <code>KNOWN_BY</code> path, then collect what you need using <code>WITH</code> and <code>MATCH</code> the <code>HAS_PERSPECTIVE</code> part.</li>
<li>The range queries on the <code>StartDT</code> and <code>EndDT</code> property could be slow. Try to remove them to test if this slows down the query.</li>
<li>Also, it looks like you could replace the <code>&gt;=</code> and <code>&lt;</code> with <code>=</code>, sind you use the same date everywhere.</li>
<li>If you really have to filter date ranges a lot, it might help to implement it in your graph model. One option would be to use <code>Knownby</code> nodes instead of <code>KNOWN_BY</code> relationships and connect them to <code>Date</code> nodes.</li>
</ul>
