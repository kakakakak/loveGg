    <div class="post-text" itemprop="text">

<p>As a seasoned Spring user I was assuming that Spring Integration would make the most sense in a recent project requiring some (JMS) messaging capabilities (<a href="http://stackoverflow.com/questions/3012802/how-to-leverage-spring-integration-in-a-real-world-jms-distributed-architecture">more details</a>). After some days working with Spring Integration it still feels like a lot of configuration overhead given the amount of channels you have to configure to bring some request-response (listening on different JMS queues) communications in place. </p>

<p>Therefore I was looking for some background information how Camel is different from Spring Integration, but it seems like information out there are pretty spare, I found:</p>

<ul>
<li><a href="http://java.dzone.com/articles/spring-integration-and-apache">http://java.dzone.com/articles/spring-integration-and-apache</a> (Very neutral comparison between implementing a real-world integration scenario in Spring Integration vs. Camel, from December 2009)</li>
<li><a href="http://hillert.blogspot.com/2009/10/apache-camel-alternatives.html">http://hillert.blogspot.com/2009/10/apache-camel-alternatives.html</a> (Comparing Camel with other solutions, October 2009)</li>
<li><a href="http://raibledesigns.com/rd/entry/taking_apache_camel_for_a">http://raibledesigns.com/rd/entry/taking_apache_camel_for_a</a> (Matt Raible, October 2008)</li>
</ul>

<p>Question is: what experiences did you make on using the one stack over the other? In which scenarios would you recommend Camel were Spring Integration lacks support? Where do you see pros and cons of each? Any advise from real-world projects are highly appreciated.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>We choose Camel over Spring-Integration because the fluent API is really nice. We actually use it in Spring projects and use Spring to configure part of it. The programming API's are clear and there is a large set of sensible components. </p>

<p>We did a small scale shootout and basically at that time for our requirement Camel won. We use it mainly to transfer internal datafiles to/from external parties which usually requires format conversions sending it using ftp/sftp/... or attaching it to an email and sending it out.</p>

<p>We found the edit-compile-debug cycle reduced. Using groovy to experiment setting up routes are added bonuses.</p>

<p>Spring-Integration is a great product too, and I am quite sure it would satisfy our needs too. </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I only recommend Spring Integration if you already have got a Spring project and you have just to add some "basic" integration using File, FTP, JMS, JDBC, and so on. </p>

<p>Apache Camel has two main advantages: </p>

<ol>
<li>Many, many more technologies are supported.</li>
<li>Besides, a (good) XML DSL, there are fluent APIs for Java, Groovy and Scala.</li>
</ol>

<p>Because Apache Camel has very good integration with Spring, I would even use it instead if Spring Integration in most Spring projects.</p>

<p>If you need more details, you can read my experiences in my blog post: <a href="http://www.kai-waehner.de/blog/2012/01/10/spoilt-for-choice-which-integration-framework-to-use-spring-integration-mule-esb-or-apache-camel/">Spoilt for Choice: Which Integration Framework to use – Spring Integration, Mule ESB or Apache Camel?</a></p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Most comparisons of Camel and SI that I've seen don't take the following into account:</p>

<p>1.) The effect that Spring Boot has had on developer productivity for Spring Integration</p>

<p>2.) The effect of Spring XD has had on making Spring Integration applications available with no code compilation - also Spring XD sources and sinks are simply Spring Integration channel adapters, when you're looking to extend Spring XD.</p>

<p>3.) The effect of Spring XD has had on making unifying Spring Integration, Spring Batch, Spring Data (+Hadoop!) in one stack, effectively bringing batch and stream processing, HDFS/Apache Hadoop support, and much more to Spring Integration.</p>

<p>4.) The effect of the soon-to-be-released Spring Integration 4.0 Java DSL <a href="https://github.com/spring-projects/spring-integration-extensions/wiki/Spring-Integration-Java-DSL-Reference">https://github.com/spring-projects/spring-integration-extensions/wiki/Spring-Integration-Java-DSL-Reference</a></p>

<p>For your consideration,</p>

<p>/Pieter (disclaimer I work at Pivotal)</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I have recently conducted a Camel vs Spring Integration shoot-out with the aim to integrate <a href="https://kafka.apache.org/" rel="nofollow">Apache Kafka</a>. Despite being an avid Spring developer, I sadly found my suspicion with Spring's ever-growing Project stack confirmed: <strong>Spring is awesome as IOC-Container to serve as glue for other framework, but it fails at providing viable alternatives <em>to</em> those frameworks</strong>. There might be exceptions to this, namely everything to do with MVC, where Spring came from and where it does a great job, but other attempts to provide new functionality on top of container features fall short for <strong>three reasons</strong> and the <em>SI Kafka use case</em> confirms all of them:</p>

<ul>
<li>Introduction of a long-winded difficult to use DSL for XML-configuration.</li>
<li>Pages of xml-configuration code to get all framework components wired-up.</li>
<li>Missing resources to provide functionality on par with dedicated frameworks.</li>
</ul>

<p>Now, back to the results of my shoot-out: most importantly I am impressed by Camels overall concept of <strong>routes between endpoints</strong>. Kafka seamlessly integrates with this concept and three lines of configuration are enough to get everything up-and-running. Problems encountered during the process are neatly addressed by <a href="http://camel.apache.org/user-guide.html" rel="nofollow">ample documentation from the project team</a> as well as a lot of questions on Stackoverflow. Last but not least, there is a <strong>comprehensive integration into Spring</strong> that leaves no wishes unfulfilled.</p>

<p>With SI on the contrary, the documentation for the Kafka integration is <a href="https://github.com/spring-projects/spring-integration-kafka" rel="nofollow">merely one page long</a> and ambitiously yet insufficiently tries to explain how to integrate Kafka. The integration of Kafka is <em>pressed</em> into the SI-way of doing things, which adds extra complexity. Other documentation, e.g. on Stackoverflow is also less plentiful and less helpful than for Camel.</p>

<p><strong>My conclusion: cobbler stick to your trade - use Spring as a container and Camel as system integration framework.</strong></p>

<p>One reason to use Camel over Spring Integration is when you need a more featureful EIP set.  Spring Integration doesn't provide abstractions over things such as ThreadPool.</p>

<p>Camel does provide additional constructs for this simplifying some of the aspects of working with concurrent code:</p>

<p><a href="http://camel.apache.org/camel-23-threadpool-configuration.html" rel="nofollow">http://camel.apache.org/camel-23-threadpool-configuration.html</a></p>

<p>If you have no need for this sort of thing and just want to connect file, JMS, FTP endpoints etc... then just use Spring Integration.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Camel has the nice fluent interface but debugging Camel routes is really next to impossible. Having worked on some large projects that used Camel, I would recommend Spring Integration. It just became a chore to debug routes and led to a large project turning into a pile of spaghetti. Granted, Camel has taken pains to integrate with Spring but there is a slight impedance between the terminology used in the Holpe book and the names Camel chose to give to some components. </p>

<p>In short, having worked on a large real-world project with Camel, I would not choose that technology again. I highly recommend that you choose Spring Integration. I personally trust the Spring Framework. </p>
