<p><strong>Grouping Comparator</strong></p>

<p>Once the data reaches a reducer, all data is grouped by key. Since we have a composite key, we need to make sure records are grouped solely by the natural key. This is accomplished by writing a custom GroupPartitioner. We have a Comparator object only considering the yearMonth field of the TemperaturePair class for the purposes of grouping the records together.</p>

<pre><code>public class YearMonthGroupingComparator extends WritableComparator {

    public YearMonthGroupingComparator() {
        super(TemperaturePair.class, true);
    }

    @Override
    public int compare(WritableComparable tp1, WritableComparable tp2) {
        TemperaturePair temperaturePair = (TemperaturePair) tp1;
        TemperaturePair temperaturePair2 = (TemperaturePair) tp2;
        return temperaturePair.getYearMonth().compareTo(temperaturePair2.getYearMonth());
    }
}
</code></pre>

<p>I find it easy to understand certain concepts with help of diagrams and this is certainly one of them.</p>

<p>Lets assume that our secondary sorting is on a composite key made out of Last Name and First Name.</p>

<p><img src="http://i.stack.imgur.com/T58wv.png" alt="Composite Key"></p>

<p>With the composite key out of the way, now lets look at the secondary sorting mechanism</p>

<p><img src="http://i.stack.imgur.com/l6IEl.png" alt="Secondary Sorting Steps"></p>

<p>The partitioner and the group comparator use only <strong>natural key</strong>, the partitioner uses it to channel all records with the same natural key to a single reducer. <em>This partitioning happens in the Map Phase, data from various Map tasks are received by reducers where they are <strong>grouped</strong> and then sent to the reduce method</em>. This grouping is where the group comparator comes into picture, if we would <strong>not</strong> have specified a custom group comparator then Hadoop would have used the default implementation which would have considered the entire composite key, which would have lead to incorrect results.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Here is an example for grouping. Consider a composite key <code>(a, b)</code> and its value <code>v</code>. And let's assume that after sorting you end up, among others, with the following group of (key, value) pairs:</p>

<pre><code>(a1, b11) -&gt; v1
(a1, b12) -&gt; v2
(a1, b13) -&gt; v3
</code></pre>

<p>With the default group comparator the framework will call the <code>reduce</code> function 3 times with respective (key, value) pairs, since all keys are different. However, if you provide your own custom group comparator, and define it so that it depends only on <code>a</code>, ignoring <code>b</code>, then the framework concludes that all keys in this group are equal and calls the reduce function only once using the following key and the list of values:</p>

<pre><code>(a1, b11) -&gt; &lt;v1, v2, v3&gt; 
</code></pre>

<p>Note that only the first composite key is used, and that b12 and b13 are "lost", i.e., not passed to the reducer.</p>

<p>In the well known example from the "Hadoop" book computing the max temperature by year, <code>a</code> is the year, and <code>b</code>'s are temperatures sorted in descending order, thus b11 is the desired max temperature and you don't care about other <code>b</code>'s. The reduce function just writes the received (a1, b11) as the solution for that year.</p>

<p>In your example from "bigdataspeak.com" all <code>b</code>'s are required in the reducer, but they are available as parts of respective values (objects) <code>v</code>.</p>

<p>In this way, by including your value or its part in the key, you can use Hadoop to sort not only your keys, but also your values.</p>

<p>Hope this helps.</p>

<ol>
<li>Mapout should be (Key+Value, Value)</li>
<li>When we have joined Key&amp;Value. Still we need to have mechanism to sort on original Key as well as on value.So we would add a custom comparator.</li>
<li>Now data is sorted on original Key but if we send this data to reducer, it will not guarantee to send all value of a given key to one reducer as we are using Key+Value as key. To make sure it we would add group comparator.</li>
