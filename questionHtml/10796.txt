<p>i'm trying to find the best solution to create scalable storage for big files. File size can vary from 1-2 megabytes and up to 500-600 gigabytes. </p>

<p>I have found some information about Hadoop and it's HDFS, but it looks a little bit complicated, because i don't need any Map/Reduce jobs and many other features. Now i'm thinking to use MongoDB and it's GridFS as file storage solution.</p>

<p>And now the questions:</p>

<ol>
<li>What will happen with gridfs when i try to write few files
    concurrently. Will there be any lock for read/write operations? (I will use it only as file storage)</li>
<li>Will files from gridfs be cached in ram and how it will affect read-write perfomance?</li>
<li>Maybe there are some other solutions that can solve my problem more efficiently?</li>
</ol>

<p>Thanks.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I can only answer for MongoDB here, I will not pretend I know much about HDFS and other such technologies.</p>

<p>The GridFs implementation is totally client side within the driver itself. This means there is no special loading or understanding of the context of file serving within MongoDB itself, effectively MongoDB itself does not even understand they are files ( <a href="http://docs.mongodb.org/manual/applications/gridfs/" rel="nofollow">http://docs.mongodb.org/manual/applications/gridfs/</a> ).</p>

<p>This means that querying for any part of the <code>files</code> or <code>chunks</code> collection will result in the same process as it would for any other query, whereby it loads the data it needs into your working set ( <a href="http://en.wikipedia.org/wiki/Working_set" rel="nofollow">http://en.wikipedia.org/wiki/Working_set</a> ) which represents a set of data (or all loaded data at that time) required by MongoDB within a given time frame to maintain optimal performance. It does this by paging it into RAM (well technically the OS does).</p>

<p>Another point to take into consideration is that this is driver implemented. This means that the specification can vary, however, I don't think it does. All drivers will allow you to query for a set of documents from the <code>files</code> collection which only houses the files meta data allowing you to later serve the file itself from the <code>chunks</code> collection with a single query.</p>

<p>However that is not the important thing, you want to serve the file itself, including its data; this means that you will be loading the <code>files</code> collection and its subsequent <code>chunks</code> collection into your working set.</p>

<p>With that in mind we have already hit the first snag:</p>

<blockquote>
  <p>Will files from gridfs be cached in ram and how it will affect read-write perfomance?</p>
</blockquote>

<p>The read performance of small files could be awesome, directly from RAM; the writes would be just as good.</p>

<p>For larger files, not so. Most computers will not have 600 GB of RAM and it is likely, quite normal in fact, to house a 600 GB partition of a single file on a single <code>mongod</code> instance. This creates a problem since that file, in order to be served, needs to fit into your working set however it is impossibly bigger than your RAM; at this point you could have page thrashing ( <a href="http://en.wikipedia.org/wiki/Thrashing_%28computer_science%29" rel="nofollow">http://en.wikipedia.org/wiki/Thrashing_%28computer_science%29</a> ) whereby the server is just page faulting 24/7 trying to load the file. The writes here are no better as well.</p>

<p>The only way around this is to starting putting a single file across many shards <code>:\</code>.</p>

<p>Note: one more thing to consider is that the default average size of a <code>chunks</code> "chunk" is 256KB, so that's a lot of documents for a 600GB file. This setting is manipulatable in most drivers. </p>

<blockquote>
  <p>What will happen with gridfs when i try to write few files concurrently. Will there be any lock for read/write operations? (I will use it only as file storage)</p>
</blockquote>

<p>GridFS, being only a specification uses the same locks as on any other collection, both read and write locks on a database level (2.2+) or on a global level (pre-2.2). The two do interfere with each other as well, i.e. how can you ensure a consistent read of a document that is being written to?</p>

<p>That being said the possibility for contention exists based on your scenario specifics, traffic, number of concurrent writes/reads and many other things we have no idea about.</p>

<blockquote>
  <p>Maybe there are some other solutions that can solve my problem more efficiently?</p>
</blockquote>

<p>I personally have found that S3 (as @mluggy said) in reduced redundancy format works best storing a mere portion of meta data about the file within MongoDB, much like using GridFS but without the chunks collection, let S3 handle all that distribution, backup and other stuff for you.</p>

<p>Hopefully I have been clear, hope it helps.</p>

<p>Edit: Unlike what I accidently said, MongoDB does not have a collection level lock, it is a database level lock.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I'll start by answering the first two:</p>

<ol>
<li>There is a write lock when writing in to GridFS, yes. No lock for reads.</li>
<li>The files wont be cached in memory when you query them, but their metadata will.</li>
</ol>

<p>GridFS may not be the best solution for your problem. Write locks can become something of a pain when you're dealing with this type of situation, particularly for huge files. There are other databases out there that may solve this problem for you. HDFS is a good choice, but as you say, it is very complicated. I would recommend considering a storage mechanism like Riak or Amazon's S3. They're more oriented around being storage for files, and don't end up with major drawbacks. S3 and Riak both have excellent admin facilities, and can handle huge files. Though with Riak, last I knew, you had to do some file chunking to store files over 100mb. Despite that, it generally is a best practice to do some level of chunking for huge file sizes. There are a lot of bad things that can happen when transferring files in to DBs- From network time outs, to buffer overflows, etc. Either way, your solution is going to require a fair amount of tuning for massive file sizes.</p>
