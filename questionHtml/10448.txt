    <div class="post-text" itemprop="text">

<p>In <a href="http://www.usenix.org/event/osdi04/tech/full_papers/dean/dean.pdf" rel="nofollow">MapReduce papers</a> is described that input files are partitioned in M input splits. I know the HDFS in Hadoop makes partitioning automatically to blocks of 64 MB (default) and then replicate these blocks to few other nodes in cluster for providing fault tolerance. I'd like to know if this partitioning of files in HDFS means the input splitting described in mentioned MapReduce papers. Is fault tolerance single reason of this splitting or are there more important reasons? </p>

<p>And what if I have MapReduce over cluster of nodes without distributed file system (data only on local disks with common file sytem)? Do I need to split input files on local disk before map phase?</p>

<p>Thank you for your answers.</p>

<hr>

<p><strong>HDFS</strong> </p>

<p>A file is stored as blocks(<em>Fault/Node Torrance</em>) .The block size (64MB-128MB) 64MB . So a file is divided in blocks , blocks are stored on different Nodes on cluster . A block is being replicated by a replication factor(default =3).</p>

<p><strong>Map-Reduce</strong></p>

<p>The file which is already stored in HDFS is logically divided into <strong>INPUT-SPLITS</strong> .
The splits size can be set by the user </p>

<pre><code>Property name           Type   Default value
</code></pre>

<hr>

<pre><code>mapred.min.split.size   int     1
mapred.max.split.sizea  long    Long.MAX_VALUE.
</code></pre>

<p>And then the split size is calculated by the formula:</p>

<p><strong>max(minimumSize, min(maximumSize, blockSize))</strong></p>

<p><strong>NOTE:</strong>: <em>The Split are logical</em> </p>

<hr>

<hr>

<p>Hope to ans your questions now </p>

<pre><code> I'd like to know if this partitioning of files in HDFS means the input splitting described in mentioned MapReduce papers. 
</code></pre>

<p>NO , Not at all HDFS blocks and Map-Reduce splits are not at all same thing </p>

<pre><code>Is fault tolerance single reason of this splitting or are there more important reasons?
</code></pre>

<p>No , Distributed computing will be the reason .</p>

<pre><code>And what if I have MapReduce over cluster of nodes without distributed file system (data only on local disks with common file sytem)? Do I need to split input files on local disk before map phase?
</code></pre>

<p>In your case ,I Guess ,Yes you will have to split the input-file for Map Phase , and also you will have to split the intermediate output(from Mapper) for Reduce Phase.
Other prob : consistency of Data,Fault tolerance,Data Loss(in hadoop its =1%).</p>

<p><em>Map-Reduce is made for Distributed Computing , so using Map-Reduce in Non-Distributed environment is not useful.</em></p>

    <div class="post-text" itemprop="text">
<pre><code>I'd like to know if this partitioning of files in HDFS means the input splitting described in mentioned MapReduce papers.
</code></pre>

<p>No, the input splitting in MapReduce is to take advantage of the computing capacity of multiple processors during the reduce phase. The mapper takes in a large amount of data and splits the data into logical partitions (most of the times as specified by the custom implementation of the mapper by the programmer). This data then goes to individual nodes where  independent processes called reducers perform the data crunching and then, the result gets collated in the end.</p>

<pre><code>Is fault tolerance single reason of this splitting or are there more important reasons?
</code></pre>

<p>No, it is not the single reason for doing so. You can compare it to the file-system level block size for ensuring transfer of data into chunks, compression of data on a per-block basis and allocation of I/O buffers. </p>
