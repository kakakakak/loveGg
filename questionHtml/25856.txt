    <div class="post-text" itemprop="text">

<p>I have setup a Hadoop cluster containing 5 nodes on Amazon EC2. Now, when i login into the Master node and submit the following command</p>

<pre><code>bin/hadoop jar &lt;program&gt;.jar &lt;arg1&gt; &lt;arg2&gt; &lt;path/to/input/file/on/S3&gt;
</code></pre>

<p>It throws the following errors (not at the same time.) The first error is thrown when i don't replace the slashes with '%2F' and the second is thrown when i replace them with '%2F':</p>

<pre><code>1) Java.lang.IllegalArgumentException: Invalid hostname in URI S3://&lt;ID&gt;:&lt;SECRETKEY&gt;@&lt;BUCKET&gt;/&lt;path-to-inputfile&gt;
2) org.apache.hadoop.fs.S3.S3Exception: org.jets3t.service.S3ServiceException: S3 PUT failed for '/' XML Error Message: The request signature we calculated does not match the signature you provided. check your key and signing method.
</code></pre>

<p>Note: </p>

<p>1)when i submitted jps to see what tasks were running on the Master, it just showed</p>

<pre><code>1116 NameNode
1699 Jps
1180 JobTracker
</code></pre>

<p>leaving DataNode and TaskTracker.</p>

<p>2)My Secret key contains two '/' (forward slashes). And i replace them with '%2F' in the S3 URI.</p>

<p>PS: The program runs fine on EC2 when run on a single node. Its only when i launch a cluster, i run into issues related to copying data to/from S3 from/to HDFS. And, what does distcp do? Do i need to distribute the data even after i copy the data from S3 to HDFS?(I thought, HDFS took care of that internally)</p>

<p>IF you could direct me to a link that explains running Map/reduce programs on a hadoop cluster using Amazon EC2/S3. That would be great.</p>

<p>Regards,</p>

<p>Deepak.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>You probably want to use s3n:// urls, not s3:// urls. s3n:// means "A regular file, readable from the outside world, at this S3 url".  s3:// refers to an HDFS file system mapped into an S3 bucket.</p>

<p>To avoid the URL escaping issue for the access key (and to make life much easier), put them into the <code>/etc/hadoop/conf/core-site.xml</code> file:</p>

<pre><code>&lt;property&gt;
  &lt;name&gt;fs.s3.awsAccessKeyId&lt;/name&gt;
  &lt;value&gt;0123458712355&lt;/value&gt;
&lt;/property&gt;
&lt;property&gt;
  &lt;name&gt;fs.s3.awsSecretAccessKey&lt;/name&gt;
  &lt;value&gt;hi/momasgasfglskfghaslkfjg&lt;/value&gt;
&lt;/property&gt;
&lt;property&gt;
  &lt;name&gt;fs.s3n.awsAccessKeyId&lt;/name&gt;
  &lt;value&gt;0123458712355&lt;/value&gt;
&lt;/property&gt;
&lt;property&gt;
  &lt;name&gt;fs.s3n.awsSecretAccessKey&lt;/name&gt;
  &lt;value&gt;hi/momasgasfglskfghaslkfjg&lt;/value&gt;
&lt;/property&gt;
</code></pre>

<p>There was at one point an outstanding issue with secret keys that had a slash -- the URL was decoded in some contexts but not in others. I don't know if it's been fixed, but I do know that with the keys in the .conf this goes away.</p>

<p>Other quickies:</p>

<ul>
<li>You can most quickly debug your problem using the hadoop filesystem commands, which work just fine on s3n:// (and s3://) urls. Try <code>hadoop fs -cp s3n://myhappybucket/</code> or <code>hadoop fs -cp s3n://myhappybucket/happyfile.txt /tmp/dest1</code> and even <code>hadoop fs -cp /tmp/some_hdfs_file s3n://myhappybucket/will_be_put_into_s3</code></li>
<li>The <code>distcp</code> command runs a mapper-only command to copy a tree from there to here. Use it if you want to copy a very large number of files to the HDFS. (For everyday use, <code>hadoop fs -cp src dest</code> works just fine).</li>
<li>You don't have to move the data to the HDFS if you don't want.  You can pull all the source data straight from s3, do all further manipulations targeting either the HDFS or S3 as you see fit. </li>
<li>Hadoop can become confused if there is a file s3n://myhappybucket/foo/bar and a "directory" (many files with keys s3n://myhappybucket/foo/bar/something). Some old versions of the s3sync command would leave just such 38-byte turds in the S3 tree.</li>
<li>If you start seeing <code>SocketTimeoutException</code>'s, <a href="https://issues.apache.org/jira/browse/HADOOP-6254">apply the patch for HADOOP-6254</a>. We were, and we did, and they went away.</li>
</ul>

<td class="answercell">

<p>Use</p>

<pre><code>-Dfs.s3n.awsAccessKeyId=&lt;your-key&gt; -Dfs.s3n.awsSecretAccessKey=&lt;your-secret-key&gt;
</code></pre>

<p>e.g.</p>

<pre><code>hadoop distcp -Dfs.s3n.awsAccessKeyId=&lt;your-key&gt; -Dfs.s3n.awsSecretAccessKey=&lt;your-secret-key&gt; -&lt;subsubcommand&gt; &lt;args&gt;
</code></pre>

<p>or</p>

<pre><code>hadoop fs -Dfs.s3n.awsAccessKeyId=&lt;your-key&gt; -Dfs.s3n.awsSecretAccessKey=&lt;your-secret-key&gt; -&lt;subsubcommand&gt; &lt;args&gt;
