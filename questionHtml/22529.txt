    <div class="post-text" itemprop="text">

<p>In the spark, I understand how to use <code>wholeTextFiles</code> and <code>textFiles</code>, but I'm not sure which to use when.  Here is what I know so far:</p>

<ul>
<li>When dealing with files that are not split by line, one should use <code>wholeTextFiles</code>, otherwise use <code>textFiles</code>.  </li>
</ul>

<p>I would think that by default, <code>wholeTextFiles</code> and <code>textFiles</code> partition by file content, and by lines, respectively.  But, both of them allow you to change the parameter <code>minPartitions</code>.  </p>

<p>So, how does changing the partitions affect how these are processed?  </p>

<p>For example, say I have one very large file with 100 lines.  What would be the difference between processing it as <code>wholeTextFiles</code> with 100 partiions, and processing it as <code>textFile</code> (which partitions it line by line) using the default of parition 100.  </p>

<p>What is the difference between these? </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>For reference, <code>wholeTextFiles</code> uses <code>WholeTextFileInputFormat</code> which extends <a href="https://hadoop.apache.org/docs/r2.6.1/api/org/apache/hadoop/mapreduce/lib/input/CombineFileInputFormat.html" rel="nofollow">CombineFileInputFormat</a>.  </p>

<p>A couple of notes on <code>wholeTextFiles</code>.</p>

<ul>
<li>Each record in the RDD returned by <code>wholeTextFiles</code> has the file name and the entire contents of the file.  This means that a file cannot be split (at all).</li>
<li>Because it extends <code>CombineFileInputFormat</code>, it will try to combine groups of smaller files into one partition.</li>
</ul>

<p>If I have two small files in a directory, it is possible that both files will end up in a single partition.  If I set <code>minPartitions=2</code>, then I will likely get two partitions back instead.  </p>

<p>Now if I were to set <code>minPartitions=3</code>, I will still get back two partitions because the contract for <code>wholeTextFiles</code> is that each record in the RDD contain an entire file.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I am assuming you meant <code>textFile</code> object and not <code>textFiles</code>.<br>
The use case is more from what you described. <code>wholeTextFiles</code> is used to read multiple small files in bulk, while <code>textFile</code> is used for any parallelized processing on a single file.</p>

<p>From the Pyspark documentation at <a href="http://spark.apache.org/docs/latest/api/python/pyspark.html" rel="nofollow">http://spark.apache.org/docs/latest/api/python/pyspark.html</a>:</p>

<p><code>textFile</code></p>

<blockquote>
  <p>Read a text file from HDFS, a local file system (available on all nodes), or any Hadoop-supported file system URI, and return it as an RDD of Strings.</p>
</blockquote>

<p><code>wholeTextFiles</code></p>

<blockquote>
  <p>Read a directory of text files from HDFS, a local file system (available on all nodes), or any Hadoop-supported file system URI. Each file is read as a single record and returned in a key-value pair, where the key is the path of each file, the value is the content of each file.</p>
</blockquote>

<p>To answer your question, for 1 file with 100 lines, <code>textFile</code> will be faster since it will work in parallel on a RDD, while <code>wholeTextFiles</code> won't do that since it is not a RDD.</p>
