    <div class="post-text" itemprop="text">

<p>I know this question has been answered many times, but I tried everything and I do not come to a solution. I have the following code which raises a NotSerializableException</p>

<pre><code>val ids : Seq[Long] = ...
ids.foreach{ id =&gt;
 sc.sequenceFile("file", classOf[LongWritable], classOf[MyWritable]).lookup(new LongWritable(id))
}
</code></pre>

<p>With the following exception</p>

<pre><code>Caused by: java.io.NotSerializableException: org.apache.hadoop.io.LongWritable
Serialization stack:
...
org.apache.spark.serializer.JavaSerializationStream.writeObject(JavaSerializer.scala:47)
at org.apache.spark.serializer.JavaSerializerInstance.serialize(JavaSerializer.scala:84)
at org.apache.spark.util.ClosureCleaner$.ensureSerializable(ClosureCleaner.scala:301)
</code></pre>

<p>When creating the SparkContext, I do</p>

<pre><code>val sparkConfig = new SparkConf().setAppName("...").setMaster("...")
sparkConfig.set("spark.serializer", "org.apache.spark.serializer.KryoSerializer")
sparkConfig.registerKryoClasses(Array(classOf[BitString[_]], classOf[MinimalBitString], classOf[org.apache.hadoop.io.LongWritable]))
sparkConfig.set("spark.kryoserializer.classesToRegister", "org.apache.hadoop.io.LongWritable,org.apache.hadoop.io.Text,org.apache.hadoop.io.LongWritable")
</code></pre>

<p>and looking at the environment tab, I can see these entries. However, I do not understand why </p>

<ol>
<li>the Kryo serializer does not seem to be used (the stack does not mention Kryo)</li>
<li>LongWritable is not serialized.</li>
</ol>

<p>I'm using Apache Spark v. 1.5.1</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<ol>
<li><p>Loading repeatedly the same data inside a loop is extremely inefficient. If you perform actions against the same data load it once and cache:</p>

<pre><code>val rdd = sc
  .sequenceFile("file", classOf[LongWritable], classOf[MyWritable])

rdd.cache()
</code></pre></li>
<li><p><strong>Spark doesn't consider Hadoop Writables to be serializable. There is an open JIRA (<a href="https://issues.apache.org/jira/browse/SPARK-2421" rel="nofollow">SPARK-2421</a>) for this. To handle <code>LongWritables</code> simple <code>get</code> should be enough</strong>:</p>

<pre><code>rdd.map{case (k, v) =&gt; k.get()}
</code></pre>

<p>Regarding your custom class it is your responsibility to deal with this problem.</p></li>
<li><p>Effective lookup requires a partitoned RDD. Otherwise it has to search every partition in your RDD.</p>

<pre><code>import org.apache.spark.HashPartitioner

val numPartitions: Int = ???
val partitioned = rdd.partitionBy(new HashPartitioner(numPartitions))
</code></pre></li>
<li><p>Generally speaking RDDs are not designed for random access. Even with defined partitioner <code>lookup</code> has to linearly search candidate partition. With 5000 uniformly distributed keys and 10M objects in an RDD it most likely means a repeated search over a whole RDD. You have few options to avoid that:</p>

<ul>
<li><p>filter </p>

<pre><code>val idsSet = sc.broadcast(ids.toSet)
rdd.filter{case (k, v) =&gt; idsSet.value.contains(k)}
</code></pre></li>
<li><p>join</p>

<pre><code>val idsRdd = sc.parallelize(ids).map((_, null)) 
idsRdd.join(rdd).map{case (k, (_, v)) =&gt; (k, v)}
</code></pre></li>
<li><p><a href="https://github.com/amplab/spark-indexedrdd" rel="nofollow"><code>IndexedRDD</code></a> - it doesn't like a particularly active project though</p></li>
</ul></li>
<li><p>With 10M entries you'll probably be better with searching locally in memory than using Spark. For a larger data you should consider using a proper key-value store.</p></li>
</ol>
