    <div class="post-text" itemprop="text">

<p>Suppose, hypothetically that I have a star schema in a data warehouse setting.
There is one VERY, VERY long fact table (think billions to trillions of rows) and several low cardinality dimension tables (think 100 dimension tables). Each fact table foreign key 
that points to a dimension table primary key is bitmap indexed. Each dimension table primary key is also bitmap indexed. This is all for fast joins. All pretty standard.</p>

<p>Suppose that the data warehouse is starting to show performance degradation. The time it 
takes to return results from a bitmapped join is getting worse the longer the fact table gets. The business requirement is that the fact table keeps growing (we can not move data older than a year off to archival storage) </p>

<p>I'm thinking of the following solutions:</p>

<ol>
<li>Hash partition the fact table, but this just holds off the inevitable growth issue temporarily.</li>
<li>Database partition the physical star schema database as multiple schemas/databases. 1..N fact tables and their dimension copies, each holding data assigned to them via a hash(1..N) function that is executed in a separate ETL staging database to determine which database/schema the fact row (resulting from the ETL process) will go into. If any dimension changes, replicate the change to the other databases corresponding dimension. Again, this will not work as a permanent solution.</li>
<li>Collapse the dimensions and store all dimension values directly in the fact table.
Then, import the fact table to HBASE on Hadoop. You get a massive HBASE table, key value store that has no dimension tables. I would do this because joins are cost prohibitative in HBASE (so no fact to dimension joins, just enforce dimension values on the dimension columns).</li>
</ol>

<p>Has anyone ever done this before?</p>

<p>Does anyone have any tips for solution #3? </p>

<p>Is the HBASE solution optimal as far as scaling up with fast reads?</p>

<p>As far as writes, I don't care for fast writes as they would be done off hours as batch processes.</p>

<p>If anyone has chosen solution 1 or 2, has anyone used a <a href="http://en.wikipedia.org/wiki/Consistent_hashing" rel="nofollow">consistent hashing algorithm</a> (to avoid remapping as in a plain old hash if more partitions, hash keys are created dynamically)? Dynamic growth in the number of partitions without a full remap is probably not an option (I haven't seen it done in practice as far as partitioned tables are concerned) so it just seems to me that any partition solution would lead to scaling problems.</p>

<p>Any thoughts, advice and experience with moving a giant fact table with many dimensions (a traditional DW star schema) to an HBASE giant dimensionless table?</p>

<p><strong>Related question:</strong></p>

<p>How do aggregate data collections that traditionally reside in materialized views (or alternately as separate fact tables linked to the same dimensions as the most granular fact table -- i.e. hourly/daily/weekly/monthly where the base fact table is hourly) in a data warehouse map across to HBASE?</p>

<p><img src="http://i.stack.imgur.com/7qCvI.png" alt="Aggregate fact tables partial star schema"></p>

<p>My thoughts are that since there are no materialized views in HBASE, the aggregate data collections are stored as HBASE tables that get updated/inserted into any time there are changes to the most granular, lowest level fact table.</p>

<p>Any thoughts on aggregate tables in HBASE?
Has anyone used Hive scripts to essentially mimic the behavior of materialized views in updating aggregate column data in secondary HBASE tables that have aggregate data stored in them (i.e. daily_aggregates_fact_table, weekly_aggregates_fact_table, monthly_aggregates_fact_table) on a change to the most granular fact table?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>HBase is not a good choice for a general purpose data warehouse (with real-timish query times)
Any single table will only allow you to drill down along one dimension or along one path through dimensions (if you design the right composite key right).
It isn't undoable (e.g. <a href="http://www.infoq.com/news/2011/11/eBay_new_search" rel="nofollow">ebay built their new search engine on HBase</a>) but it isn't out of the box </p>

<p>There are several efforts to provide high-performance SQL over Hadoop (e.g. <a href="http://www.hadapt.com/" rel="nofollow">Hadapt</a> or <a href="http://rainstor.com/products/query/" rel="nofollow">Rainstor</a> ) but they won't give you the performance of a good massively parallel  databases like <a href="http://www.vertica.com/" rel="nofollow">Vertica</a>, <a href="http://www.greenplum.com/products" rel="nofollow">Greenplum</a>, <a href="http://www.asterdata.com/product/alwaysparallel.php" rel="nofollow">Asterdata</a>, <a href="http://www-01.ibm.com/software/data/netezza/" rel="nofollow">Netezza</a> and the like </p>
