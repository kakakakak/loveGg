<p>I'm currently creating a website for a social project in switzerland.</p>

<p>And before there is an overflow of user, I want to prepare the application to scale.</p>

<p>I answered by myself many questions but some are left.</p>

<p>I explain what I want to do.</p>

<hr>

<p><strong>First</strong></p>

<p>at the beginnning, the Application will have only one server (short time) with DNS, PHP, Mysql, Data, and memcache.</p>

<hr>

<p><strong>Second</strong></p>

<p>Then I will split them in two</p>

<ol>
<li>DNS, Mysql, memcache</li>
<li>Data, PHP</li>
</ol>

<hr>

<p><strong>Third</strong></p>

<p>Here is the problem, I don't know how to do it exactly here to keep the application running well.</p>

<p>I could do :</p>

<ol>
<li>Front : Load Balancer, memcache, DNS</li>
<li>Web 1 : PHP, DATA</li>
<li>Web 2 : PHP, DATA</li>
<li>Mysql</li>
</ol>

<p>This would be the scheme, all PHP sessions are kept in the DB.</p>

<p>BUT, how do I sync the data?
do I run a Rsync to keep them up to date.
do I put them on a separate disk (network disk) to be sure ?
but in this case, how can I do in case of user uploads ?</p>

<p>and if the website gets more success and we have to go on greater structures, would'nt it create some latency on updates ?</p>

<p>or would it be a good thing to go directly to amazon's web services ?</p>

<p><strong>some infos</strong>
I use codeigniter as Framework.
I use linux as webserver (distribution not chosen now, but should be Debian)</p>

<p>Thanks in advance for your answers.</p>

<p>According to <a href="http://en.wikipedia.org/wiki/Languages_of_Switzerland">Wikipedia</a>, Switzerland has 4.6 million German speakers, 1.5 million French speakers, and .5 million speakers of Italian, Romansch and other languages. <p>So, I don't think you need to worry about scaling yet!  Bonus:  The time you don't spend worrying about this problem, you can use to solve other problems for your users.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>There are a few common paths to scaling web services up, in order of what sites like Flickr and Facebook seem to use:</p>

<ul>
<li>Split servers based on concepts (API, login, media files, ads, static pages, dynamic pages)</li>
<li>Split databases based on concepts that don't need to be JOINed (logins, long term reporting, page data, etc.)</li>
<li>Compile/optimize your PHP and other resources (sprites, compiled css, zend)</li>
<li>Add caching (front end, back end)</li>
<li>Add delegation (round robin, etc.)</li>
</ul>

<p>But, before scaling, measure.  Set of tests, calculate your capacity, and don't optimize before you need to.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I see some questionable things:</p>

<ul>
<li><p>You have one SQL server, and you are storing sessions in a database on a site where you expect extremely high volume. How many queries does that take to produce a single page if someone is logged in and what is the expected slow down when you eventually employ MySQL replication?</p></li>
<li><p>If using a cluster FS, everything is 'just kept' in sync. You won't end up with build A on webserver 1 while build B on webserver 2 breaks. If you are really expecting that much traffic, in the time it takes to upload a change, then sync all nodes, you just pissed off a thousand people.</p></li>
</ul>

<p>I've deployed apps running on clusters using OCFS2 with over 40 nodes without issue, and OCFS2 is not exactly the 'best' cluster FS available. Check out <a href="http://wiki.lustre.org/index.php/Main_Page" rel="nofollow">Lustre</a> and consider keeping sessions on disk.</p>

<p>Remember you can mount/share folders. </p>

<p>What data would you be syncing?</p>

<p>You might consider putting data on the database machine or other machine. The db machine is usually a good idea at first since it is likely to have greater IO than a regular web server.</p>

<p>It is probably a good idea to setup a SAN or similar so your data stays in one place. Multiple copies of data is a pain to deal with. Going this route means you can put the db files there too.</p>
