<pre><code>&lt;property&gt;

&lt;/property&gt;

&lt;/property&gt;
&lt;property&gt;

&lt;value&gt;192.168.1.8:8031&lt;/value&gt;
&lt;/property&gt;
</code></pre>

<p>Restart the hadoop processes.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>From the <code>netstat</code> output you can see the process is listening on address <code>127.0.0.1</code></p>

<pre><code>tcp        0      0 127.0.0.1:9000          0.0.0.0:*  ...
</code></pre>

<p>from the exception message you can see that it tries to connect to address <code>127.0.1.1</code></p>

<pre><code>java.net.ConnectException: Call From marta-komputer/127.0.1.1 to localhost:9000 failed ...
</code></pre>

<p>further in the exception it's mentionend</p>

<pre><code>For more details see:  http://wiki.apache.org/hadoop/ConnectionRefused
</code></pre>

<p>on this page you find</p>

<blockquote>
  <p>Check that there isn't an entry for your hostname mapped to 127.0.0.1 or 127.0.1.1 in /etc/hosts (Ubuntu is notorious for this)</p>
</blockquote>

<p>so the conclusion is to remove this line in your <code>/etc/hosts</code></p>

<pre><code>127.0.1.1       marta-komputer
</code></pre>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I had the similar prolem with OP. As the terminal output suggested, I went to 
<a href="http://wiki.apache.org/hadoop/ConnectionRefused" rel="nofollow">http://wiki.apache.org/hadoop/ConnectionRefused</a>  </p>

<p>I tried to change my /etc/hosts file as suggested here, i.e. remove 127.0.1.1 as OP suggested it will create another error. </p>

<p>So in the end, I leave it as is. The following is my /etc/hosts </p>

<pre><code>127.0.0.1       localhost.localdomain   localhost
127.0.1.1       linux
# The following lines are desirable for IPv6 capable hosts
::1     ip6-localhost ip6-loopback
fe00::0 ip6-localnet
ff00::0 ip6-mcastprefix
ff02::1 ip6-allnodes
ff02::2 ip6-allrouters
</code></pre>

<p>In the end, I found that my namenode <strong>did not</strong> started correctly, i.e.
When you type <code>sudo netstat -lpten | grep java</code> in the terminal, there will not be any JVM process running(listening) on port 9000.</p>

<p>So I made two directories for namenode and datanode respectively(if you have not done so). You don't have to put where I put it, please replace it based on your hadoop directory.
 i.e.</p>

<pre><code>mkdir -p /home/hadoopuser/hadoop-2.6.2/hdfs/namenode
mkdir -p /home/hadoopuser/hadoop-2.6.2/hdfs/datanode
</code></pre>

<p>I reconfigured my <strong>hdfs-site.xml</strong>.</p>

<pre><code>&lt;configuration&gt;
    &lt;property&gt;
        &lt;name&gt;dfs.replication&lt;/name&gt;
        &lt;value&gt;1&lt;/value&gt;
    &lt;/property&gt;
   &lt;property&gt;
        &lt;name&gt;dfs.namenode.name.dir&lt;/name&gt;
        &lt;value&gt;file:/home/hadoopuser/hadoop-2.6.2/hdfs/namenode&lt;/value&gt;
    &lt;/property&gt;
    &lt;property&gt;
        &lt;name&gt;dfs.datanode.data.dir&lt;/name&gt;
        &lt;value&gt;file:/home/hadoopuser/hadoop-2.6.2/hdfs/datanode&lt;/value&gt;
    &lt;/property&gt;
&lt;/configuration&gt;
</code></pre>

<p>In terminal, stop your hdfs and yarn with script <code>stop-dfs.sh</code> and <code>stop-yarn.sh</code>. They are located in your hadoop directory/sbin. In my case, it's /home/hadoopuser/hadoop-2.6.2/sbin/.</p>

<p>Then start your hdfs and yarn with script <code>start-dfs.sh</code> and <code>start-yarn.sh</code>
After it is started, type <code>jps</code> in your terminal to see if your JVM processes are running correctly. It should show the following.</p>

<pre><code>15678 NodeManager
14982 NameNode
15347 SecondaryNameNode
23814 Jps
15119 DataNode
15548 ResourceManager
</code></pre>

<p>Then try to use netstat again to see if your namenode is listening to port 9000</p>

<pre><code>sudo netstat -lpten | grep java
</code></pre>

<p>If you successfully set up the namenode, you should see the following in your terminal output.</p>

<p><code>tcp        0      0 127.0.0.1:9000          0.0.0.0:*               LISTEN      1001       175157      14982/java</code></p>

<p>Then try to type the command <code>hdfs dfs -mkdir /user/hadoopuser</code>
If this command executes sucessfully, now you can list your directory in the HDFS user directory by <code>hdfs dfs -ls /user</code></p>

<p>Your issue is a very interesting one. Hadoop setup could be frustrating some time due to the complexity of the system and many moving parts involved. I think the issue you faced is definitely a firewall one.
My hadoop cluster has similar setup. With a firewall rule added with command:</p>

<pre><code> sudo iptables -A INPUT -p tcp --dport 9000 -j REJECT
</code></pre>

<p>I'm able to see the exact issue:</p>

<pre><code>15/03/02 23:46:10 INFO client.RMProxy: Connecting to ResourceManager at  /0.0.0.0:8032
java.net.ConnectException: Call From mybox/127.0.1.1 to localhost:9000 failed on connection exception: java.net.ConnectException: Connection refused; For more details see:  http://wiki.apache.org/hadoop/ConnectionRefused
     at sun.reflect.NativeConstructorAccessorImpl.newInstance0(Native Method)
</code></pre>

<p>You can verify your firewall settings with command:</p>

<pre><code>/usr/local/hadoop/etc$ sudo iptables -L
Chain INPUT (policy ACCEPT)
target     prot opt source               destination         
REJECT     tcp  --  anywhere             anywhere             tcp dpt:9000 reject-with icmp-port-unreachable

Chain FORWARD (policy ACCEPT)
target     prot opt source               destination         

Chain OUTPUT (policy ACCEPT)
target     prot opt source               destination   
</code></pre>

<p>Once the suspicious rule is identified, it could be deleted with a command like:</p>

<pre><code> sudo iptables -D INPUT -p tcp --dport 9000 -j REJECT 
</code></pre>

<p>Now, the connection should go through. </p>
