    <div class="post-text" itemprop="text">

<p>We are building cv scoring service, and we are using Postgres for making complex queries to find cv's that match vacancy best.
The problem is, that we use really complex set of heuristics to score cv to vacancy, and the average number of cvs to be scored per query is growing.
I want to put this kind of load outside of database, and looking for existing solutions for horizontal scaling such load.
Query should be executed in fraction of a second, there can be hundreds of concurrent queries. Each query scores on average 10k cvs. Each cv is like about 50 records in maybe 10 tables in its current relational form.</p>

<p>I want a clustered system to run each query in multiple parallel processes (on many servers) and return aggregated result. It should be fast and fault tolerant.
I was looking to Hadoop, but it looks like it is designed for batch processing, and not for realtime low latency load. There is Apache Storm, but it is designed for continous stream processing. So I am not shure :) </p>

<p>What kind of tool could will suit my needs?
Thank you!</p>

    <div class="post-text" itemprop="text">
<ol>
<li><p>Make sure you are not redoing work, if a cv has been scored tag it as scored and don't reprocess unless it's necessary.</p></li>
<li><p>Unless you are partitioning the data in postgres you might want to do that. Usually not all rows need to be accessed regularly.</p></li>
<li><p>Sounds like you want to primarily scale reads, in that case a postgres read-only cluster could be an option.</p></li>
<li><p>Take a look at Elasticsearch, it is designed to do weighted scoring, faceting, etc. It should also scale, haven't tried that myself though.</p></li>
</ol>

<p>I would definitely start with 1 though, don't do work unless you have to.</p>
