    <div class="post-text" itemprop="text">
<p>There are a few options for backup and recovery.  As s.singh points out, data replication is not DR.  </p>

<p>HDFS supports snapshotting.  This can be used to prevent user errors, recover files, etc.  That being said, this isn't DR in the event of a total failure of the Hadoop cluster. (<a href="http://hadoop.apache.org/docs/current/hadoop-project-dist/hadoop-hdfs/HdfsSnapshots.html" rel="nofollow">http://hadoop.apache.org/docs/current/hadoop-project-dist/hadoop-hdfs/HdfsSnapshots.html</a>)  </p>

<p>Your best bet is keeping off-site backups.  This can be to another Hadoop cluster, S3, etc and can be performed using distcp. (<a href="http://hadoop.apache.org/docs/stable1/distcp2.html" rel="nofollow">http://hadoop.apache.org/docs/stable1/distcp2.html</a>), (<a href="https://wiki.apache.org/hadoop/AmazonS3" rel="nofollow">https://wiki.apache.org/hadoop/AmazonS3</a>)</p>

<p>Here is a Slideshare by Cloudera discussing DR (<a href="http://www.slideshare.net/cloudera/hadoop-backup-and-disaster-recovery" rel="nofollow">http://www.slideshare.net/cloudera/hadoop-backup-and-disaster-recovery</a>)</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>In hadoop, no need to take backup for the data. Hadoop is designed to work on the big cluster with 1000's of nodes. Data loss is possibly less. You can increase the replication factor to replicate the data into many nodes across the cluster.</p>

<p>Refer <a href="http://hadoop.apache.org/docs/r1.2.1/hdfs_design.html#Data+Replication" rel="nofollow">Data Replication</a></p>

<p>For Namenode log backup, Either you can use the <a href="http://hadoop.apache.org/docs/stable1/hdfs_user_guide.html#Secondary+NameNode" rel="nofollow">secondary namenode</a> or <a href="https://hadoop.apache.org/docs/r2.6.0/hadoop-project-dist/hadoop-hdfs/HDFSHighAvailabilityWithQJM.html" rel="nofollow">Hadoop High Availability</a></p>

<p><strong>Secondary Namenode</strong></p>

<p>Secondary namenode will take backup for the namnode logs. If namenode fails then you can recover the namenode logs (which holds the data block information) from the secondary namenode.</p>

<p><strong>High Availability</strong></p>

<p>High Availability is a new feature to run more than one namenode in the cluster. One namenode will be active and the other one will be in standby. Log saves in both namenode. If one namenode fails then the other one becomes active and it will handle the operation. </p>
