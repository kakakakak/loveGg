    <div class="post-text" itemprop="text">

<p>Is there a very fast compression library for Java?  The standard gzip library is slower than I would like.  I'm looking for something similar to <a href="http://www.oberhumer.com/opensource/lzo/">http://www.oberhumer.com/opensource/lzo/</a> that's native Java code that provides fast compression and decompression.  Thanks!</p>

<p>A few other fast compression libraries for future reference:</p>

<p>QuickLZ - 
C/C#/Java - GPL or commercial
<a href="http://www.quicklz.com/">http://www.quicklz.com/</a> </p>

<p>libLZF - 
C - BSD style license
<a href="http://oldhome.schmorp.de/marc/liblzf.html">http://oldhome.schmorp.de/marc/liblzf.html</a></p>

<p>FastLZ - 
C - MIT style license
<a href="http://fastlz.org/">http://fastlz.org/</a></p>

<p>LZO
C - GPL or commercial
<a href="http://www.oberhumer.com/opensource/lzo/">http://www.oberhumer.com/opensource/lzo/</a></p>

<p>zlib - 
C / Java (GZIP and deflate) - Commercial friendly license
<a href="http://zlib.net/">http://zlib.net/</a></p>

<p>Hadoop-LZO integration (JNI):
<a href="http://github.com/kevinweil/hadoop-lzo">http://github.com/kevinweil/hadoop-lzo</a></p>

<p>Snappy-Java (JNI):
<a href="https://github.com/xerial/snappy-java">https://github.com/xerial/snappy-java</a></p>

<p>Benchmarks from the QuickLZ folks:
<a href="http://www.quicklz.com/bench.html">http://www.quicklz.com/bench.html</a></p>

    <div class="post-text" itemprop="text">
<p>You could use the DeflatorOutputStream and InflatorInputStream. These both use LZW compression. You could just use the library they provide.</p>

<p>EDIT: Real time performance is usually measured in terms of latency, however you quote numbers in terms of throughtput.  Could you clarify what you mean by real-time.</p>

<p>For latency, using the BEST_SPEED, each call took 220 ns + 13 ns/byte on average.</p>

<p>Note: in low latency situations you often get many times the latency you might expect when the CPU is running "hot". You have perform the timing in a realistic situation.</p>

<p>EDIT: This is the compression rates I got with Java 6 update 21; </p>

<pre><code>Raw OutputStream.write() - 2485 MB/sec

Deflator.NO_COMPRESSION - 99 MB/s

Deflator.BEST_SPEED - 85 MB/s.

Deflator.FILTERED - 77 MB/s

Deflator.HUFFMAN_ONLY - 79 MB/s

Deflator.DEFAULT_COMPRESSION - 30 MB/s

Deflator.BEST_COMPRESSION - 14 MB/s
</code></pre>

<p>Note: I am not sure why the default setting is faster than the "best speed" setting.  I can only assume the former has been optimised.</p>

<p>The output buffer size was 4KB, you might find a different size is best for you.</p>

<p>EDIT: The following code prints for a large CSV file. The latency is for a 5KB block.</p>

<pre><code>Average latency 48532 ns. Bandwidth 91.0 MB/s.
Average latency 52560 ns. Bandwidth 83.0 MB/s.
Average latency 47602 ns. Bandwidth 93.0 MB/s.
Average latency 51099 ns. Bandwidth 86.0 MB/s.
Average latency 47695 ns. Bandwidth 93.0 MB/s.
</code></pre>

<p>.</p>

<pre><code>public class Main {
    public static void main(String... args) throws IOException {
        final String filename = args[0];
        final File file = new File(filename);
        DataInputStream dis = new DataInputStream(new FileInputStream(file));
        byte[] bytes = new byte[(int) file.length()];
        dis.readFully(bytes);
        test(bytes, false);
        for (int i = 0; i &lt; 5; i++)
            test(bytes, true);
    }

    private static void test(byte[] bytes, boolean print) throws IOException {
        OutputStream out = new ByteOutputStream(bytes.length);
        Deflater def = new Deflater(Deflator.BEST_SPEED);
        DeflaterOutputStream dos = new DeflaterOutputStream(out, def, 4 * 1024);
        long start = System.nanoTime();
        int count = 0;
        int size = 5 * 1024;
        for (int i = 0; i &lt; bytes.length - size; i += size, count++) {
            dos.write(bytes, i, size);
            dos.flush();
        }
        dos.close();
        long time = System.nanoTime() - start;
        long latency = time / count;
        // 1 byte per ns = 1000 MB/s.
        long bandwidth = (count * size * 1000L) / time;
        if (print)
            System.out.println("Average latency " + latency + " ns. Bandwidth " + bandwidth + " MB/s.");    
    }
}
</code></pre>

<p>More the merrier: there is a fast <a href="https://github.com/dain/snappy">pure Java version of Snappy</a> at github.
It is currently the fastest pure java codec, and possibly the fastest codec from Java even including C codecs (as it is close in speed to native Snappy codec accessed via JNI).</p>

<p>So there are 3 fast Java decompressors: Snappy, <a href="https://github.com/ning/compress">LZF</a> and <a href="https://github.com/decster/jnicompressions">LZ4</a> (see <a href="https://github.com/ning/jvm-compressor-benchmark">https://github.com/ning/jvm-compressor-benchmark</a>) for details</p>

<p>EDIT (aug 2013): LZ4 is currently the fastest codec; Snappy and LZF share the 2nd slot. All have progressed nicely over past year or so, and are significantly faster than GZIP.</p>

<p>One more; a very efficient (Java optimized, benchmarked) LZF implementation at <a href="https://github.com/ning/compress">https://github.com/ning/compress</a>.
