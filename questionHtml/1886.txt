    <div class="post-text" itemprop="text">

<p>According to the <a href="http://spark.apache.org/docs/latest/running-on-mesos.html#uploading-spark-package">Spark on Mesos docs</a> one needs to set the <code>spark.executor.uri</code> pointing to a Spark distribution:</p>

<pre><code>val conf = new SparkConf()
  .setMaster("mesos://HOST:5050")
  .setAppName("My app")
  .set("spark.executor.uri", "&lt;path to spark-1.4.1.tar.gz uploaded above&gt;")
</code></pre>

<p>The docs also note that one can build a custom version of the Spark distribution. </p>

<p>My question now is whether it is possible/desirable to pre-package external libraries such as </p>

<ul>
<li>spark-streaming-kafka</li>
<li>elasticsearch-spark</li>
<li>spark-csv</li>
</ul>

<p>which will be used in mostly all of the job-jars I'll submit via <code>spark-submit</code> to</p>

<ul>
<li>reduce the time <code>sbt assembly</code> need to package the fat jars</li>
<li>reduce the size of the fat jars which need to be submitted</li>
</ul>

<p>If so, how can this be achieved? Generally speaking, are there some hints on how the fat jar generation on job submitting process can be speed up? </p>

<p>Background is that I want to run some code-generation for Spark jobs, and submit these right away and show the results in a browser frontend asynchronously. The frontend part shouldn't be too complicated, but I wonder how the backend part can be achieved.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>When you say pre-package do you really mean distribute to all the slaves and set up the jobs to use those packages so that you don't need to download those every time? That might be an option, however it sounds a bit cumbersome because distributing everything to the slaves and keeping all the packages up to date is not an easy task.</p>

<p>How about breaking your .tar.gz into smaller pieces, so that instead of a single fat file your jobs fetch several smaller files? In this case it should be possible to leverage the <a href="http://mesos.apache.org/documentation/latest/fetcher/" rel="nofollow">Mesos Fetcher Cache</a>. So you'll see <em>bad</em> performance when the agent cache is cold, but once it warms up (i.e. once one job runs and downloads the common files locally) consecutive jobs will complete faster. </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Yeah, you can copy the dependencies out to the workers and put them in the system-wide jvm lib directory in order to get them on the classpath.  </p>

<p>Then you can mark those dependencies as provided in your sbt build, and they won't be included in the assembly. This does speed up assembly and transfer time.</p>

<p>I haven't tried this on mesos specifically, but have used it on spark standalone for things that are in every job and rarely change.</p>
