    <div class="post-text" itemprop="text">

<p>I am in the middle of a Java project which will be using a 'big dictionary' of words. By 'dictionary' I mean certain numbers (int) assigned to Strings. And by 'big' I mean a file of the order of 100 MB. The first solution that I came up with is probably the simplest possible. At initialization I read in the whole file and create a large HashMap which will be later used to look strings up.</p>

<p>Is there an efficient way to do it without the need of reading the whole file at initialization? Perhaps not, but what if the file is really large, let's say in the order of the RAM available? So basically I'm looking for a way to look things up efficiently in a large dictionary stored in memory.</p>

<p>Thanks for the answers so far, as a result I've realised I could be more specific in my question. As you've probably guessed the application is to do with text mining, in particular representing text in a form of a sparse vector (although some had other inventive ideas :)). So what is critical for usage is to be able to look strings up in the dictionary, obtain their keys as fast as possible. Initial overhead of 'reading' the dictionary file or indexing it into a database is not as important as long as the string look-up time is optimized. Again, let's assume that the dictionary size is big, comparable to the size of RAM available. </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Consider <code>ChronicleMap</code> (<a href="https://github.com/OpenHFT/Chronicle-Map" rel="nofollow">https://github.com/OpenHFT/Chronicle-Map</a>) in a non-replicated mode. It is an off-heap Java <code>Map</code> implementation, or, from another point of view, a superlightweight NoSQL key-value store.</p>

<p>What it does useful for your task out of the box:</p>

<ul>
<li>Persistance to disk via memory mapped files (see comment by Michał Kosmulski)</li>
<li>Lazy load (disk pages are loaded only on demand) -> fast startup</li>
<li>If your data volume is larger than available memory, operating system will unmap rarely used pages automatically.</li>
<li>Several JVMs can use the same map, because off-heap memory is shared on OS level. Useful if you does the processing within a map-reduce-like framework, e. g. Hadoop.</li>
<li>Strings are stored in UTF-8 form, -> ~50% memory savings if strings are mostly ASCII (as maaartinus noted)</li>
<li><code>int</code> or <code>long</code> values takes just 4(8) bytes, like if you have primitive-specialized map implementation.</li>
<li>Very little per-entry memory overhead, much less than in standard <code>HashMap</code> and <code>ConcurrentHashMap</code></li>
<li>Good configurable concurrency via lock striping, if you already need, or are going to parallelize text processing in future.</li>
</ul>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>At the point your data structure is a few hundred MB to orders of RAM, you're better off not initializing a data structure at run-time, but rather using a database which supports <a href="http://en.wikipedia.org/wiki/Database_index" rel="nofollow"><code>indexing</code></a>(which most do these days). Indexing is going to be one of the only ways you can ensure the fastest retrieval of text once you're file gets so large and you're running up against the -<a href="http://stackoverflow.com/questions/5374455/what-does-java-option-xmx-stand-for"><code>Xmx</code></a> settings of your JVM. This is because if your file is as large, or much larger than your maximum size settings, you're inevitably going to <a href="http://www.slideshare.net/gengmao/inside-the-jvm-memory-management-and-troubleshooting" rel="nofollow">crash your JVM</a>.</p>

<p>As for having to read the whole file at initialization. You're going to have to do this eventually so that you can efficiently search and analyze the text in your code. If you know that you're only going to be searching a certain portion of your file at a time, you can implement <a href="http://en.wikipedia.org/wiki/Lazy_loading" rel="nofollow">lazy loading</a>. If not, you might as well bite the bullet and load your entire file into the DB in the beggenning. You can implement <a href="http://docs.oracle.com/javase/tutorial/collections/streams/parallelism.html" rel="nofollow">parallelism</a> in this process, if there are other parts of your code execution that doesn't depend on this.</p>

<p>Please let me know if you have any questions!</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>As stated in a comment, a <a href="https://en.wikipedia.org/wiki/Trie" rel="nofollow"><code>Trie</code></a> will save you a lot of memory.</p>

<p>You should also consider using <code>byte</code>s instead of <code>char</code>s as this saves you a factor of 2 for plain ASCII text or when using your national charset  as long as it has no more than 256 different letters.</p>

<p>At the first glance, combining this low-level optimization with tries makes no sense, as with them the node size is dominated by the pointers. But there's a way if you want to go low level.</p>

<blockquote>
  <p>So what is critical for usage is to be able to look strings up in the dictionary, obtain their keys as fast as possible.</p>
</blockquote>

<p>Then forget any database, as they're damn slow when compared to <code>HashMap</code>s.</p>

<p>If it doesn't fit into memory, the cheapest solution is usually to get more of it. Otherwise, consider loading only the most common words and doing something slower for the others (e.g., a memory mapped file).</p>

<hr>

<p>I was asked to point to a good tries implementation, especially off-heap. I'm not aware of any.</p>

<p>Assuming the OP needs no mutability, especially no mutability of keys, it all looks very simple.</p>

<p>I guess, the whole dictionary could be easily packed into a single <code>ByteBuffer</code>. Assuming mostly ASCII and with some bit hacking, an arrow would need 1 byte per arrow label character and 1-5 bytes for the child pointer. The child pointer would be relative (i.e., difference between the current node and the child), which would make most of them fit into a single byte when stored in a <a href="http://en.wikipedia.org/wiki/LEB128" rel="nofollow">base 128 encoding</a>.</p>

<p>I can only guess the total memory consumption, but I'd say, something like &lt;4 bytes per word. The above compression would slow the lookup down, but still nowhere near what a single disk access needs.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>It sounds too big to store in memory. Either store it in a relational database (easy, and with an index on the hash, fast), or a NoSQL solution, like Solr (small learning curve, very fast).</p>

<p>Although NoSQL is very fast, if you really want to tweak performance, and there are entries that are far more frequently looked up than others, consider using a limited size cache to hold the most recently used (say) 10000 lookups.</p>
