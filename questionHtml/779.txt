    <div class="post-text" itemprop="text">

<p>We have big dataset (about 191 million of records, will be grow), every record contains the values of filters (11 filters - datetime and integer values), and some additional data (cost). For example:</p>

<pre><code>Depature City = 1
Arrival City = 5
Country Id = 7
Check In Date = 2013-05-05
    ... etc

Cost 1250
    ... etc
</code></pre>

<p>We have a search interface with 11 filters. In every filter user can choose: one value, a set of values, all values. </p>

<p>Every filter have the different set of possible values, it can vary from 4 to 5000 values.</p>

<p>The result of search must be sorted by ascending cost, there are paging (50 result per page)</p>

<p>Every search query must be completed in 100 mS, usually expected 50-70 requests/sec (200 as maximum). </p>

<p>The data will be changes often, but the speed of data changing has the lower priority, than search this process can be slow.</p>

<p>What is the best way to organise such search engine? Data in memory (we tried some tree algoritms), Map-Reduce (Hadoop?), OLAP?</p>

<p><strong>UPDATE</strong>. What do you think about some in memory solution? The records can be loaded to the operation memory in some good for search and sort structure. What structure is the best?</p>

<p>In production environment, the client will be able to supply appropriate hardware for good solution.</p>

<p>In general, we have a .NET solution - so, this module must be compatible with it.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>In-memory solution may be feasible. Since you need to store 12 values x 200M records, you'll need about 20GB of RAM net (assuming 8 bytes per value). You'll need to optimize (stoing 1/2/4 bytes values where possible and disable memory alignments). Practically, you'll probably need a 64GB or stronger machine.</p>

<p>One think you can't afford is using data structures that requires tons of small memory allocations. Even if you'll store the data in a single huge buffer, you'll probably need many small allocations for tree-structure indexes.</p>

<p>There is another reason why trees are not so good for your problem: Since the user may select a set of values for each filter, you'll need to traverse the tree in search for any combination. This can be a huge number of tree traversals.</p>

<p>How about a simpler solution? Select the 2 filters that divide the data-set to the maximal number of groups (this would probably be the filters with the ~5000 values). Use a 2D array. In each cell, if it is not empty, store an array of struct of all remaining 10 values (9 filters + cost). You can sort these arrays by the 3rd most dominant filter.</p>

<p>Upon a user query, determine the relevant cells in the 2D array and check your input against each of the values in the relevant cell (which is sorted by he 3rd most dominant filter). For most cells, you'll have much less than 1000 values to check against.</p>

<p>Depending on your data distribution, you can save some memory by using a sparse matrix instead of a 2D array. Some .NET sparse matrix implementations are available online.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>[TrollModeOn] I had a problem.... tried to solve it with no-sql solution, now i have 2 problems [/TrollModeOff]. </p>

<p>As it seems to me, no-sql solution isn't good for handling so much filter's stuff. I'd start from sql-based solution. E.g. if we have ms sql server we can use user defined table types for filters, some kind of:</p>

<pre><code>CREATE TYPE [FilterTable] AS TABLE(
    [id] [int] NOT NULL   --or any datatype needed
)
</code></pre>

<p>After that you can pass table type as a parameter to filtering stored procedure (or do it with sql query), like:</p>

<pre><code>CREATE PROCEDURE [SomeFilterProcedureName]
    @Filter1 FilterTable READONLY,
    @Filter2 FilterTable READONLY
    ....
</code></pre>

<p>And your query would be kind of:</p>

<pre><code>SELECT
    field1,
    field2,
    field3
FROM MyTable t
WHERE
    (@Filter1 IS NULL OR t.field1 IN (SELECT id FROM @Filter1))
    AND (@Filter2 IS NULL OR t.field2 IN (SELECT id FROM @Filter2))
    ....
ORDER BY
    whatever
</code></pre>

<p>So basically you check if your parameters contains some values, if yes - you filter out column values according to filter-parameter data.</p>

<p>RDBMS do exellent work storing, finding, filtering and sorting huge amounts of data, but you'l need to tune it right way to make it work faster, e.g. you will need to set up your indexes correct. Also you can cache data for some period, but make sure you build cache key correct depending on varying parameters.</p>

<p>If your db server isn't good enough to handle 200 queries per second you might want to make a cluster or keep several db servers with same data up and use some kind of db balancer.</p>

<p><strong>upd: it's too big to place it in comments</strong></p>

<p><code>It the worst case he can select "All" for every 11 filter and we have to sort 192 million records to find 20-100 with the lowest cost</code></p>

<p>All filter, lowest cost? Isn't it same as:
<code>Select top(20) * from someTableName order by cost</code>.</p>

<ol>
<li><code>Db Locks</code>. Work better on your indexes and queries</li>
<li><code>Sorting</code>. Ok, you got 100million records that fit filters. How are you going to sort em? QSort, MergeSort, BubbleSort? Or maybe stackoverflowSort? Do you know which algorythm you have to choose? But first - DBMS knows, and it choses best algorythm for case, cause it has statistics, and second - of course data is stored preordered in indexes. So every 100m record sort operation will kill no-sql solution but will work perfect on rdbms</li>
<li><code>High load</code>. Isn't it what we're talking about? In your case it's not real highload there. There are companies that have 100-150 million monthly active users with hella big databases, thousand's of queries per second, and yes, they use rdbms. Dozens of servers, sharding, balancing, and it works perfect.</li>
</ol>

<td class="answercell">
    <div class="post-text" itemprop="text">
<h1><strong>This is exactly the scenario SQL is designed for</strong></h1>

<p>SQL Server on a modern system (e.g. Quad Core CPU with 8 GB RAM) can easily handle all the filters, or no filters at all, in the time span you require, provided you create an INDEX on every field you're filtering on.</p>

<p>You could use Sergio's stored procedure to implement the filters; but it's prob. just as easy to just generate the correct SQL statement directly in C# (or VB.NET).</p>

<h1>Profile, profile, profile</h1>

<p>Before looking for Map-Reduce or other (b)leading edge technology, try SQL.  Creating the table and indexes can be done in about 15 minutes, and you can time the query.  If it's close to your requirements, then you can begin writing code to generate the correct SQL SELECT based on the filter.  If the SQL query is slower than your requirements, you can decide if you want to optimize it, or look elsewhere.  <strong>But until you've profiled, there's absolutely no reason to try anything else.</strong></p>
