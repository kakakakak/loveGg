    <div class="post-text" itemprop="text">

<p>Ok, I'll try and keep this short, sweet and to-the-point.</p>

<p>We do massive GeoIP updates to our system by uploading a MASSIVE CSV file to our PHP-based CMS. This thing usually has more than 100k records of IP address information. Now, doing a simple import of this data isn't an issue at all, but we have to run checks against our current regional IP address mappings.</p>

<p>This means that we must validate the data, compare and split overlapping IP address, etc.. And these checks must be made for each and every record. </p>

<p>Not only that, but I've just created a field mapping solution that would allow other vendors to implement their GeoIP updates in different formats. This is done by applying rules to IPs records within the CSV update. </p>

<p>For instance a rule might look like:</p>

<p>if 'countryName' == 'Australia' then send to the 'Australian IP Pool'</p>

<p>There might be multiple rules that have to be run and each IP record must apply them all. For instance, 100k records to check against 10 rules would be 1 million iterations; not fun.</p>

<p>We're finding 2 rules for 100k records takes up to 10 minutes to process. I'm fully aware of the bottleneck here which is the shear amount of iterations that must occur for a successful import; just not fully aware of any other options we may have to speed things up a bit.</p>

<p>Someone recommended splitting the file into chunks, server-side. I don't think this is a viable solution as it adds yet another layer of complexity to an already complex system. The file would have to be opened, parsed and split. Then the script would have to iterate over the chunks as well. </p>

<p>So, question is, considering what I just wrote, what would the BEST method be to speed this process up a bit? Upgrading the server's hardware JUST for this tool isn't an option unfortunately, but they're pretty high-end boxes to begin with.</p>

<p>Not as short as I thought, but yeah. Halps? :(</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>The simple key to this is keeping as much work out of the inner loop as possible.</p>

<p>Simply put, anything you do in the inner loop is done "100K times", so doing nothing is best (but certainly not practical), so doing as little possible is the next best bet.</p>

<p>If you have the memory, for example, and it's practical for the application, defer any "output" until after the main processing. Cache any input data if practical as well. This works best for summary data or occasional data.</p>

<p>Ideally, save for the reading of the CSV file, do as little I/O as possible during the main processing.</p>

<p>Does PHP offer any access to the Unix mmap facility, that is typically the fastest way to read files, particularly large files.</p>

<p>Another consideration is to batch your inserts. For example, it's straightforward to build up your INSERT statements as simple strings, and ship them to the server in blocks of 10, 50, or 100 rows. Most databases have some hard limit on the size of the SQL statement (like 64K, or something), so you'll need to keep that in mind. This will dramatically reduce your round trips to the DB.</p>

<p>If you're creating primary keys through simple increments, do that en masses (blocks of 1000, 10000, whatever). This is another thing you can remove from your inner loop.</p>

<p>And, for sure, you should be processing all of the rules at once for each row, and not run the records through for each rule.</p>

<div class="vote">

        <a class="vote-up-off" title="This answer is useful">up vote</a>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>100k records isn't a large number. 10 minutes isn't a bad job processing time for a single thread. The amount of raw work to be done in a straight line is probably about 10 minutes, regardless if you're using PHP or C. If you want it to be faster, you're going to need a more complex solution than a while loop. </p>

<p>Here's how I would tackle it:</p>

<ol>
<li>Use a map/reduce solution to run the process in parallel. Hadoop is probably overkill. Pig Latin may do the job. You really just want the map part of the map/reduce problem. IE: you're forking of a chunk of the file to be processed by a sub process. Your reducer is probably <code>cat</code>. A simple version could be having PHP fork processes for each 10K record chunk, wait for the children, then re-assemble their output.</li>
<li>Use a queue/grid processing model. Queue up chunks of the file, then have a cluster of machines checking in, grabbing jobs and sending the data somewhere. This is very similar to the map/reduce model, just using different technologies, plus you could scale by adding more machines to the grid.</li>
<li>If you can write your logic as SQL, do it in a database. I would avoid this because most web programmers can't work with SQL on this level. Also, SQL is sort of limited for doing things like RBL checks or ARIN lookups.</li>
</ol>
