    <div class="post-text" itemprop="text">

<p>I'm trying to model a fairly simple relationship in CouchDB and I'm having trouble determining the best way to accomplish this. I'd like users to be able to create lists of video game objects. I have the video game documents stored in the DB already with <code>"type":"game"</code>. I'd like to be able to query a list object's ID (via a view) and get back the list's metadata (title, creation date, etc.) and portions of the game document (such as title and release date). Furthermore, I'd like to be able to add/removes games to/from lists without downloading the entire list document and posting it back (so this means I can't simply store the game's information in the list document) as I'd eventually like to support multiple users contributing to the same list, and I don't want to introduce conflicts.</p>

<p>After reading the CouchDB wiki on <a href="http://wiki.apache.org/couchdb/EntityRelationship" rel="nofollow">EntityRelationships</a>, I've determined that setting up relationship documents might be the best solution.</p>

<h3>Game:</h3>

<pre><code>{
    "_id": "2600emu",
    "type": "game"
}
</code></pre>

<h3>List:</h3>

<pre><code>{
    "_id": 123,
    "title": "Emulators",
    "user_id": "dstaley",
    "type": "list"
}
</code></pre>

<h3>Game-List Relationship:</h3>

<pre><code>{
    "_id": "98765456789876543",
    "type": "relationship",
    "list_id": 123,
    "game_id": "2600emu"
}
</code></pre>

<p>But, from what I understand, this wouldn't allow me to get the list's metadata and the game's metadata in <strong>one request</strong>. Any advice?</p>

    <div class="post-text" itemprop="text">
<p>Great question.  You identify a few very important reasons that using a "normalized" data model (different document types with links) is an optimal model:</p>

<ol>
<li>You have a many-to-many relationship between users &lt;==> lists &lt;==> games.</li>
<li>One-to-many relationships are easy to represent in a single document that uses a container for the 'many' portion, but they get large and you can have concurrency conflicts.</li>
<li>Extending the single-doc model to store a many-to-many relationship is untenable.</li>
<li>In general, document immutability is a great fit for concurrent systems.  In CouchDB you do that exactly as you noted, by storing 'write-once' documents that represent an edge in your graph, then using secondary indexes to rebuild the portions of the links that you want and to get the information you want in a single API query call.</li>
</ol>

<p>You're also right that the solution here is a 'map-side-join' (to borrow from the hadoop community).  Basically you want to use different rows in the map output to represent different pieces of information.  You can then use a range query (startkey/endkey) to query just the portion of the map result that you need, and, voila, your materialized view of the 'join' table.  However, the one piece of the puzzle you didn't find in the documentation is this:</p>

<p><a href="http://wiki.apache.org/couchdb/Introduction_to_CouchDB_views#Linked_documents">http://wiki.apache.org/couchdb/Introduction_to_CouchDB_views#Linked_documents</a></p>

<p>The first line there:</p>

<blockquote>
  <p>"If you emit an object value which has {'_id': XXX} then include_docs=true will fetch the document with id XXX rather than the document which was processed to emit the key/value pair."</p>
</blockquote>

<p>says it all.   That's how you dereference the pointer to the linked document that you have stored by a foreign key.  You then combine that with the use of compound keys (keys that are JS arrays) and the view coallation rules:</p>

<p><a href="http://wiki.apache.org/couchdb/View_collation?action=show&amp;redirect=ViewCollation#Collation_Specification">http://wiki.apache.org/couchdb/View_collation?action=show&amp;redirect=ViewCollation#Collation_Specification</a></p>

<p>So that your view rows we be sorted like:</p>

<pre><code>["list_1"], null
["list_1", "game"], {"_id":"game_1234"}
["list_1", "game"], {"_id":"game_5678"}
["list_2"], null
["list_2","game"], {"_id":"game1234"}
["list_3"], null
...
</code></pre>

<p>Putting this together with your existing data model, here is some (untested) pseudocode that should do the trick:</p>

<pre><code>function(doc) {
    if (doc.type=="list") {
        //this is the one in the one-to-many
        emit( [doc._id]),);
    }
    else if (doc.type=="relationship") {
        //this is the many in the one-to-many
        //doc.list_id is our foreign key to the list.  We use that as the key
        //doc.game_id is the foreign key to the game.  We use that as the value
        emit( [doc.list_id,'game'],  {'_id': doc.game_id});
    }
}   
</code></pre>

<p>Finally, you would query that with a startkey/endkey so that you get all rows that begin with the list_id that you're interested in.  It would look something like:</p>

<pre><code>curl -g 'https://usr:pwd@usr.cloudant.com/db/_design/design_doc_name/_view/view_name?startkey=["123"]&amp;endkey=["123",{}]&amp;include_docs=true'
</code></pre>

<p>The <code>-g</code> option tells curl not to glob, which means you don't have to dereference your square brackets, etc, and the <code>include_docs=true</code> option will follow the pointer to the foreign key that you specified with <code>game_id</code> in the <code>relationship</code> document.</p>

<p>Analysis:</p>

<ol>
<li>You are using essentially immutable documents to store state changes and you let the database compute the aggregate state for you.  That is a lovely model at scale, and one of our most successful patterns.</li>
<li>Very efficient for additions or deletions to lists.</li>
<li>Excellent scaling properties under high-concurrency</li>
<li>At Cloudant (and CouchDB v2.0) we don't yet have 'read-your-writes consistency for secondary indexes.  It is high in the priority list, but there are potential corner cases where, in failure scenarios or high-load, you may not see immediate consistency between the primary and secondary indexes.  Long story short, quorum is used for primary indexes, but quorum isn't a viable model for secondary indexes, so another consistency strategy is being developed.</li>
</ol>
