    <div class="post-text" itemprop="text">

<p>I use a hadoop (version 1.2.0) cluster of 16 nodes, one with a public IP (the master) and 15 connected through a private network (the slaves). </p>

<p><strong>Is it possible to use a remote server (in addition to these 16 nodes) for storing the output of the mappers?</strong> The problem is that the nodes are running out of disk space during the map phase and I cannot compress map output any more.</p>

<p>I know that <code>mapred.local.dir</code>in <code>mapred-site.xml</code> is used to set a comma-separated list of dirs where the tmp files are stored. Ideally, I would like to have one local dir (the default one) and one directory on the remote server. When the local disk fills, then I would like to use the remote disk.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I am not very sure about about this but as per the link (<a href="http://hadoop.apache.org/docs/current/hadoop-mapreduce-client/hadoop-mapreduce-client-core/mapred-default.xml" rel="nofollow">http://hadoop.apache.org/docs/current/hadoop-mapreduce-client/hadoop-mapreduce-client-core/mapred-default.xml</a>)  it says that:</p>

<blockquote>
  <p>The local directory is a directory where MapReduce stores intermediate data files.
  May be a comma-separated <strong>list of directories on different devices in
  order to spread disk i/o</strong>. Directories that do not exist are ignored.</p>
</blockquote>

<p>Also there are some other properties which you should check out. These might be of help:</p>

<blockquote>
  <ol>
  <li><p><strong>mapreduce.tasktracker.local.dir.minspacestart</strong>: If the space in mapreduce.cluster.local.dir drops under this, do not ask for more tasks. Value in bytes</p></li>
  <li><p><strong>mapreduce.tasktracker.local.dir.minspacekill</strong>: If the space in mapreduce.cluster.local.dir drops under this, do not ask more tasks until all the current ones have finished and cleaned up. Also, to save the rest of the tasks we have running, kill one of them, to clean up some space. Start with the reduce tasks, then go with the ones that have finished the least. Value in bytes.</p></li>
  </ol>
</blockquote>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>The solution was to use the <a href="http://forum.synology.com/wiki/index.php/How_to_use_iSCSI_Targets_on_Linux" rel="nofollow">iSCSI technology</a>. A technician helped us out to achieve that, so unfortunately I am not able to provide more details on that. </p>

<p>We mounted the remote disk to a local path (<code>/mnt/disk</code>) of each slave node, and created a <code>tmp</code> file there, with <code>rwx</code> priviledges for all users. </p>

<p>Then, we changed the <code>$HADOOP_HOME/conf/mapred-site.xml</code> file and added the property:</p>

<pre><code>&lt;property&gt;
    &lt;name&gt;mapred.local.dir&lt;/name&gt;
    &lt;value&gt;/mnt/disk/tmp&lt;/value&gt;
&lt;/property&gt;
</code></pre>

<p>Initially, we had two, comma-separated values for that property, with the first being the default value, but it still didn't work as expected (we still got some "No space left on device" errors). So we left only one value there. </p>
