    <div class="post-text" itemprop="text">

<p>I've setup a simple local PHD 3.0 Hadoop cluster and followed the steps described in the <a href="https://spring.io/guides/gs/yarn-basic-single/" rel="nofollow">Spring Yarn Basic Getting Started guide</a></p>

<p>Running the app against my Hadoop cluster gives</p>

<pre><code>org.apache.hadoop.security.AccessControlException: SIMPLE authentication is not enabled.  Available:[TOKEN]
</code></pre>

<p>and the following stack trace in the YARN ResourceManager:</p>

<pre><code>org.apache.hadoop.security.AccessControlException: SIMPLE authentication is not enabled.  Available:[TOKEN]
    at org.apache.hadoop.ipc.Server$Connection.initializeAuthContext(Server.java:1554)
    at org.apache.hadoop.ipc.Server$Connection.readAndProcess(Server.java:1510)
    at org.apache.hadoop.ipc.Server$Listener.doRead(Server.java:762)
    at org.apache.hadoop.ipc.Server$Listener$Reader.doRunLoop(Server.java:636)
    at org.apache.hadoop.ipc.Server$Listener$Reader.run(Server.java:607)
</code></pre>

<p>This is probably a very basic question.
I'd like simply to run a YARN app test without setting up any authentication.</p>

<p>As I understand, YARN does not allow SIMPLE client authentication:
<a href="https://issues.apache.org/jira/browse/YARN-2156" rel="nofollow">https://issues.apache.org/jira/browse/YARN-2156</a></p>

<p>According to this question
<a href="http://stackoverflow.com/questions/30651452/how-can-i-pass-a-kerberos-ticket-to-spring-yarn-application">How can I pass a Kerberos ticket to Spring Yarn application</a></p>

<p>I might end up setting up a Kerberos authentication.</p>

<p>Is there a way to run Spring YARN example without elaborate authentication setup?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>My mistake was simple.
I had to add </p>

<pre><code>spring:
    hadoop:
        resourceManagerAddress: myyarnhost:8050
        resourceManagerSchedulerAddress: myyarnhost:8030
</code></pre>

<p>to the <code>application.yml</code> too, but mixed up the port numbers (8030 for manager and 8050 for ManagerScheduler). 
And that typo has caused such effect.</p>

<p>Maybe adding these two configuration properties to the getting started guide could save some time to the next readers.</p>

<p>Also, to run the example against a freshly installed PHD3.0 I had to modify the HDFS client user name by exporting the default <code>HADOOP_USER_NAME</code>:</p>

<pre><code>export HADOOP_USER_NAME=hdfs
</code></pre>

    <div class="post-text" itemprop="text">
<p>I just tried that with 5 node phd30 cluster and everything was ok:</p>

<p>In <code>build.gradle</code> I used phd30 packages instead of vanilla(which depends on hadoop 2.6.0). Versions in this case should not matter afaik.</p>

<pre><code>compile("org.springframework.data:spring-yarn-boot:2.2.0.RELEASE-phd30")
testCompile("org.springframework.data:spring-yarn-boot-test:2.2.0.RELEASE-phd30")
</code></pre>

<p>In <code>src/main/resources/application.yml</code> I changed hdfs, rm and scheduler addresses to match cluster settings:</p>

<pre><code>spring:
    hadoop:
        fsUri: hdfs://ambari-2.localdomain:8020
        resourceManagerAddress: ambari-3.localdomain:8050
        resourceManagerSchedulerAddress: ambari-3.localdomain:8030
</code></pre>

<p>Then I just ran it externally from my own computer:</p>

<pre><code>$ java -jar target/gs-yarn-basic-single-0.1.0.jar
</code></pre>

<p>There's one <code>appmaster</code> and one <code>container</code> executed and app should succeed.</p>

<p>If it still doesn't work then there's something else. I didn't deploy <code>hawk</code> if that makes a difference.</p>
