<p>So, using something like:</p>

<pre><code>for (int i = 0; i &lt; files.length; i++) {
            if (!files[i].isDirectory() &amp;&amp; files[i].canRead()) {
                try {
                    Scanner scan = new Scanner(files[i]);
                System.out.println("Generating Categories for " + files[i].toPath());
                while (scan.hasNextLine()) {
                    count++;
                    String line = scan.nextLine();
                    System.out.println("  -&gt;" + line);
                    line = line.split("\t", 2)[1];
                    System.out.println("!- " + line);
                    JsonParser parser = new JsonParser();
                    JsonObject object = parser.parse(line).getAsJsonObject();
                    Set&lt;Entry&lt;String, JsonElement&gt;&gt; entrySet = object.entrySet();
                    exploreSet(entrySet);
                }
                scan.close();
                // System.out.println(keyset);
            } catch (FileNotFoundException e) {
                e.printStackTrace();
            }

        }
    }
</code></pre>

<p>as one goes over a Hadoop output file, one of the JSON objects in the middle is breaking... because scan.nextLine() is not fetching the whole line before it brings it to split. ie, the output is:</p>

<pre><code>  -&gt;0   {"Flags":"0","transactions":{"totalTransactionAmount":"0","totalQuantitySold":"0"},"listingStatus":"NULL","conditionRollupId":"0","photoDisplayType":"0","title":"NULL","quantityAvailable":"0","viewItemCount":"0","visitCount":"0","itemCountryId":"0","itemAspects":{   ... !- {"Flags":"0","transactions":{"totalTransactionAmount":"0","totalQuantitySold":"0"},"listingStatus":"NULL","conditionRollupId":"0","photoDisplayType":"0","title":"NULL","quantityAvailable":"0","viewItemCount":"0","visitCount":"0","itemCountryId":"0","itemAspects":{   ... </code></pre>

<p>Most of the above data has been sanitized (not the URL (for the most part) however... )</p>

<p>and the URL continues as:
    $(KGrHqZHJCgFBsO4dC3MBQdC2)Y4Tg~~60_1.JPG?set_id=8800005007
 in the file.... </p>

<p>So its slightly miffing.</p>

<p>This also is entry #112, and I have had other files parse without errors... but this one is screwing with my mind, mostly because I dont see how scan.nextLine() isnt working...</p>

<p>By debug output, the JSON error is caused by the string not being split properly.</p>

<p>And almost forgot, it also works JUST FINE if I attempt to put the offending line in its own file and parse just that.</p>

Also blows up if I remove the offending line in about the same place.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Based on your code, the best explanation I can come up with is that the line really does end after the <code>"~~"</code> according to the criteria used by <code>Scanner.nextLine()</code>.</p>

<p>The criteria for an end-of-line are:</p>

<ul>
<li>Something that matches this regex: <code>"\r\n|[\n\r\u2028\u2029\u0085]"</code> or</li>
<li>The end of the input stream</li>
</ul>

<p>You say that the file continues after the <code>"~~"</code>, so lets put EOF aside, and look at the regex.  That will match any of the following:</p>

<p>The usual line separators:</p>

<ul>
<li><code>&lt;CR&gt;</code></li>
<li><code>&lt;NL&gt;</code></li>
<li><code>&lt;CR&gt;&lt;NL&gt;</code> </li>
</ul>

<p>... and three unusual forms of line separator that Scanner also recognizes.</p>

<ul>
<li>0x0085 is the <code>&lt;NEL&gt;</code> or "next line" control code in the "ISO C1 Control" group</li>
<li>0x2028 is the Unicode "line separator" character</li>
<li>0x2029 is the Unicode "paragraph separator" character</li>
</ul>

<p>My theory is that you've got one of the "unusual" forms in your input file, and this is not showing up in .... whatever tool it is that you are using to examine the files.</p>

<hr>

<p>I suggest that you examine the input file using a tool that can show you the actual bytes of the file; e.g. the <code>od</code> utility on a Linux / Unix system.  Also, check that this isn't caused by some kind of character encoding mismatch ... or trying to read or write binary data as text.</p>

<p>If these don't help, then the next step should be to run your application using your IDE's Java debugger, and single-step it through the <code>Scanner.hasNextLine()</code> and <code>nextLine()</code> calls to find out what the code is actually doing.</p>

<hr>

<blockquote>
  <p>And almost forgot, it also works JUST FINE if I attempt to put the offending line in its own file and parse just that.</p>
</blockquote>

<p>That's interesting.  But if the tool you are using to extract the line is the same one that is not showing the (hypothesized) unusual line separator, then this evidence is not reliable.  The process of extraction may be altering the "stuff" that is causing the problems.</p>
