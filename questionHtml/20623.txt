<p>My goal is to parse large csv files with C++ in a QT project in OSX environment.
( When i say csv i mean tsv and other variants 1GB ~ 5GB ). </p>

<p>It seems like a simple task , but things get complicated when file sizes get bigger . I dont want to write my own parser because of the many edge cases related to parsing csv files.</p>

<p>I have found various csv processing libraries to handle this job , but parsing 1GB file takes about 90 ~ 120 seconds on my machine wich is not acceptable . i am not doing anything with the data right now , i just process and discard the data for testing purposes .</p>

<p><a href="http://sourceforge.net/projects/cccsvparser/">cccsvparser</a> is one of the libraries i have tried . But the the only fast enough library was <a href="https://code.google.com/p/fast-cpp-csv-parser/">fast-cpp-csv-parser</a> wich gives acceptable results : 15 secs on my machine , but it works only when the file structure is known .</p>

<p>Example using : <a href="https://code.google.com/p/fast-cpp-csv-parser/">fast-cpp-csv-parser</a></p>

<pre><code>#include "csv.h"

int main(){
    io::CSVReader&lt;3&gt; in("ram.csv");
    in.read_header(io::ignore_extra_column, "vendor", "size", "speed");
    std::string vendor; int size; double speed;
    while(in.read_row(vendor, size, speed)){
    // do stuff with the data
    }
}
</code></pre>

<p>As you can see i cannot load arbitrary files and i must specifically define variables to match my file structure . i'm not aware of any method that allows me to create those variables dynamically in runtime . </p>

<p>The other approach i have tried is to read csv file line by line with <a href="https://code.google.com/p/fast-cpp-csv-parser/">fast-cpp-csv-parser</a> LineReader class wich is really fast ( about 7 secs to read whole file) , and then parse each line with <a href="http://sourceforge.net/projects/cccsvparser/">cccsvparser</a> lib that can process strings . but this takes about 40 seconds until done, it is an improvement compared to the first attempts but still unacceptable . </p>

<p>i have seen various stackoverflow questions related to csv file parsing none of them takes large file processing in to account . </p>

<p>Also i spent a lot of time googling to find a solution to this problem , and i really miss the freedom that package managers like <strong>npm</strong> or <strong>pip</strong> offer when searching for out of the box solutions .</p>

<p>I will appreciate any suggestion about how to handle this problem . </p>

<p><strong>Edit:</strong></p>

<p>When using <a href="http://stackoverflow.com/users/1917249/fbucek">@fbucek</a>'s approach, processing time reduced to 25 seconds , which is a great improvement.  </p>

<p><strong>can we optimize this even more ?</strong></p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I am assuming you are using only one thread.</p>

<p>Multithreading can speedup your process.</p>

<p>Best accomplishment so far is <strong>40 sec</strong>. Let's stick to that.</p>

<p>I have assumed that first you read then you process -> <em>( about 7 secs to read whole file)</em> </p>

<p><strong>7 sec</strong> for reading
<strong>33 sec</strong> for processing</p>

<p>First of all <strong>you can divide your file into chunks</strong>, let's say 50MB.
That means that you can start processing after reading 50MB of file. You do not need to wait till whole file is finished. 
That's 0.35 sec for reading ( now it is 0.35 + 33 second for processing = cca 34sec )</p>

<p>When you use Multithreading, <strong>you can process multiple chunks at a time</strong>. That can speedup process theoretically up to number of your cores. Let's say you have 4 cores.
That's 33/4 = 8.25 sec.</p>

<p>I think you can speed up you processing with 4 cores up to <strong>9 s.</strong> in total.</p>

<p>Look at <a href="http://qt-project.org/doc/qt-5/qthreadpool.html">QThreadPool</a> and <a href="http://qt-project.org/doc/qt-5/qrunnable.html">QRunnable</a> or <a href="http://qt-project.org/doc/qt-5/qtconcurrent.html">QtConcurrent</a> 
I would prefer QThreadPool</p>

<p>Divide task into parts:</p>

<ol>
<li>First try to loop over file and divide it into chunks. And do nothing with it.</li>
<li>Then create "ChunkProcessor" class which can process that chunk</li>
<li>Make "ChunkProcessor" a subclass of <a href="http://qt-project.org/doc/qt-5/qrunnable.html">QRunnable</a> and in reimplemented run() function execute your process</li>
<li>When you have chunks, you have class which can process them and that class is QThreadPool compatible, you can pass it into </li>
</ol>

<p>It could look like this</p>

<pre><code>loopoverfile {
  whenever chunk is ready {
     ChunkProcessor *chunkprocessor = new ChunkProcessor(chunk);
     QThreadPool::globalInstance()-&gt;start(chunkprocessor);
     connect(chunkprocessor, SIGNAL(finished(std::shared_ptr&lt;ProcessedData&gt;)), this, SLOT(readingFinished(std::shared_ptr&lt;ProcessedData&gt;)));
  }   
}
</code></pre>

<p>You can use std::share_ptr to pass processed data in order not to use QMutex or something else and avoid serialization problems with multiple thread access to some resource.</p>

<p>Note: in order to use custom signal you have to register it before use</p>

<pre><code>qRegisterMetaType&lt;std::shared_ptr&lt;ProcessedData&gt;&gt;("std::shared_ptr&lt;ProcessedData&gt;");
</code></pre>

<p><strong>Edit: (based on discussion, my answer was not clear about that)</strong>
It does not matter what disk you use or how fast is it. Reading is single thread operation.
This solution was suggested only because it took 7 sec to read and again does not matter what disk it is. <strong>7 sec</strong> is what's count. And only purpose is to start processing as soon as possible and not to wait till reading is finished.</p>

<p>You can use:</p>

<pre><code>QByteArray data = file.readAll();
</code></pre>

<p>Or you can use principal idea: ( I do not know why it take 7 sec to read, what is behind it )</p>

<pre><code> QFile file("in.txt");
 if (!file.open(QIODevice::ReadOnly | QIODevice::Text))
   return;

 QByteArray* data = new QByteArray;    
 int count = 0;
 while (!file.atEnd()) {
   ++count;
   data-&gt;append(file.readLine());
   if ( count &gt; 10000 ) {
     ChunkProcessor *chunkprocessor = new ChunkProcessor(data);
     QThreadPool::globalInstance()-&gt;start(chunkprocessor);
     connect(chunkprocessor, SIGNAL(finished(std::shared_ptr&lt;ProcessedData&gt;)), this, SLOT(readingFinished(std::shared_ptr&lt;ProcessedData&gt;)));
     data = new QByteArray; 
     count = 0;
   }
 }
</code></pre>

<p>One file, one thread, read almost as fast as read by line "without" interruption. 
What you do with data is another problem, but has nothing to do with I/O. It is already in memory.
So only concern would be <strong>5GB file and ammout of RAM</strong> on the machine.</p>

<p>It is very simple solution all you need is subclass QRunnable, reimplement run function, emit signal when it is finished, pass processed data using shared pointer and in main thread joint that data into one structure or whatever. Simple thread safe solution.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I would propose a multi-thread suggestion with a slight variation is that one thread is dedicated to reading file in predefined (configurable) size of chunks and keeps on feeding data to a set of threads (more than one based cpu cores). Let us say that the configuration looks like this:</p>

<p>chunk size = 50 MB <br>
Disk Thread = 1 <br>
Process Threads = 5 <br></p>

<ol>
<li>Create a class for reading data from file. In this class, it holds a data structure which is used to communicate with process threads. For example this structure would contain starting offset, ending offset of the read buffer for each process thread. For reading file data, reader class holds 2 buffers each of chunk size (50 MB in this case)</li>
<li>Create a process class which holds a pointers (shared) for the read buffers and offsets data structure. </li>
<li>Now create driver (probably main thread), creates all the threads and waiting on their completion and handles the signals.</li>
<li>Reader thread is invoked with reader class, reads 50 MB of the data and based on number of threads creates offsets data structure object. In this case t1 handles 0 - 10 MB, t2 handles 10 - 20 MB and so on. Once ready, it notifies processor threads. It then immediately reads next chunk from disk and waits for processor thread to completion notification from process threads.</li>
<li>Processor threads on the notification, reads data from buffer and processes it. Once done, it notifies reader thread about completion and waits for next chunk.</li>
<li>This process completes till the whole data is read and processed. Then reader thread notifies back to the main thread about completion which sends PROCESS_COMPLETION, upon all threads exits. or main thread chooses to process next file in the queue.</li>
</ol>

<p>Note that offsets are taken for easy explanation, offsets to line delimiter mapping needs to be handled programmatically.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>If the parser you have used is not distributed obviously the approach is not scalable.</p>

<p>I would vote for a technique like this below</p>

<ul>
<li>chunk up the file into a size that can be handled by a machine / time constraint</li>
<li>distribute the chunks to a cluster of machines (1..*) that can meet your time/space requirements </li>
<li>consider dealing at block sizes for a given chunk </li>
<li>Avoid threads on same resource (i.e given block) to save yourself from all thread related issues. </li>
<li>Use threads to achieve non competing (on a resource) operations - such as reading on one thread and writing on a different thread to a different file.</li>
<li>do your parsing (now for this small chunk you can invoke your parser).</li>
<li>do your operations.</li>
<li>merge the results back / if can distribute them as they are..</li>
</ul>

<p>Now, having said that, why can't you use Hadoop like frameworks?</p>
