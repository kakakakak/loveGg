    <div class="post-text" itemprop="text">

<p><em>(I'm sure a similar question exists, but I haven't found the answer I'm looking for yet.)</em> </p>

<p>I'm using <strong>Hadoop</strong> and <strong>Hive</strong> (for our developers with SQL familiarity) to batch process multiple terabytes of data nightly. From an input of a few hundred massive CSV files, I'm outputting four or five fairly large CSV files. Obviously, Hive stores these in HDFS. Originally these input files were extracted from a giant SQL data warehouse.</p>

<p>Hadoop is extremely valuable for what it does. But what's the industry standard for <strong>dealing with the output?</strong> Right now I'm using a shell script to copy these back to a local folder and upload them to another data warehouse.</p>

<p>This question: ( <a href="http://stackoverflow.com/questions/4800994/hadoop-and-mysql-integration">Hadoop and MySQL Integration</a> ) calls the practice of re-importing Hadoop exports non-standard. <strong>How do I explore my data with a BI tool, or integrate the results into my ASP.NET app?</strong> Thrift? Protobuf? Hive ODBC API Driver? There must be a better way.....</p>

<p>Enlighten me.</p>

    <div class="post-text" itemprop="text">
<p>At foursquare I'm using Hive's Thrift driver to put the data into databases/spreadsheets as needed.</p>

<p>I maintain a job server that executes jobs via the Hive driver and then moves the output wherever it is needed. Using thrift directly is very easy and allows you to use any programming language.</p>

<p>If you're dealing with hadoop directly (and can't use this) you should check out <a href="https://github.com/cloudera/sqoop/wiki/" rel="nofollow">Sqoop, built by Cloudera</a> </p>

<p>Sqoop is designed for moving data in batch (whereas Flume is designed for moving it in real-time, and seems more aligned with putting data into hdfs than taking it out).</p>

<p>Hope that helps.</p>
