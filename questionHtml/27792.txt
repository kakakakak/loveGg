<p>I have written a simple bash script. The exact code is here.<br/>
ideone.com/8XQCjH<br/></p>

<pre><code>#!/bin/bash
if ! bzip2 -t "$file"
then 
    printf '%s is corrupted\n' "$file"
    rm -f "$file"
    #echo "$file" "is corrupted" &gt;&gt; corrupted.log
else
    tar -xjvf "$file" -C ./uncompressed
    rm -f  "$file"
fi
</code></pre>

<p>Basically, it reads a compressed file, tests it and uncompresses it and moves it to another directory.</p>

<p>How do I modify this code so that it will be able to read files in a hdfs input directory instead and output to another hdfs output directory ?</p>

<p>I have seen some examples here which though involves reading the contents of the file. Though in my case, I am not interested in reading any contents.<br/>
<a href="http://www.oraclealchemist.com/news/tf-idf-hadoop-streaming-bash-part-1/" rel="nofollow">http://www.oraclealchemist.com/news/tf-idf-hadoop-streaming-bash-part-1/</a></p>

<p>If anyone could write a hadoop command which unzips files in a hdfs or a similar example, that'll greatly help me.</p>

<p><strong>Edit:</strong> <br/>
<strong>Try 1:</strong><br/>
    hadoop fs -get /input/temp.tar.bz2 | tar -xjv | hadoop fs -put - /output</p>

<p>Not good as it moves the file into the native filesystem, uncompresses it and puts it back into the output directory in hdfs.<br/></p>

<p><strong>Try 2</strong>:<br/>
wrote a script uncompress.sh with just one line of code</p>

<pre><code>uncompress.sh
tar -xjv


hadoop jar contrib/streaming/hadoop-streaming.jar \
-numReduceTasks 0 \
-file /home/hadoop/uncompress.sh \
-input /input/temp.tar.bz2 \
-output /output \
-mapper uncompress.sh \
-verbose
</code></pre>

<p>However this gave the below error.</p>

<pre><code>INFO mapreduce.Job: Task Id : attempt_1409019525368_0015_m_000002_0, Status : FAILED
Error: java.lang.RuntimeException: PipeMapRed.waitOutputThreads(): subprocess failed with code 2
</code></pre>

<p>Thanks</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>From the man page of <code>bzip2</code>:</p>

<blockquote>
  <p><strong>-t --test</strong></p>
  
  <p>Check integrity of the specified file(s), but don't decompress them. This really performs a trial decompression and throws away the result</p>
</blockquote>

<p>This means that there is no way to check the file without reading it. Also, if you are going to perform the decompression after that if the archive is deemed valid, you should probably decompress it directly.</p>

<p>That said, you can use</p>

<pre><code>hadoop fs -cat hdfs://my_file_name | bzip2 -ct
</code></pre>

<p>to test the file and </p>

<pre><code>tmpdir=`mktmp -d`
hadoop fs -cat hdfs://my_file_name | tar jxv -C $tmpdir
hadoop fs -copyFromLocal $tmpdir/ hdfs://dest_dir
</code></pre>

<p>to decompress it. There is no way to have <code>tar</code> write the files directly into hdfs. Hadoop streaming is thought as "download the stuff you need, perform the job in a temp directory, upload them back".</p>

<p>That said, you are using hadoop to perform decompression of a large number of files, or you want to parallelize the decompression of a big single giant file? In the second case you have to write an ad-hoc program to split the input into multiple parts, and decompress them. Hadoop will not automatically parallelize tasks for you. In the first case, you can use a script like this as mapper:</p>

<pre><code>#!/bin/bash
while IFS="\n" read filename ; do
tmpdir=`mktmp -d`
hadoop fs -cat "hdfs:/$filename" | tar jxv -C $tmpdir
hadoop fs -copyFromLocal $tmpdir/ "hdfs:/$filename".dir/
rm -rf $tmpdir
done
</code></pre>

<p>and as input you use instead a file with the list of the tar.bz2 files to decompress</p>

<pre><code>...
/path/my_file.tar.bz2
/path2/other_file.tar.bz2
....
</code></pre>
