<p>If you were to collect social data from a data service such as twitter.com what would your data strategy be?</p>

<p>Here's what I could think of ( I could be wrong).</p>

<p>Consider Twitter.com and full fire hose access, what are we talking? </p>

<ol>
<li>200+ Million tweets per day - avg of 2314tweets per sec =
5.6MB/s.</li>
<li>Current record TPS (tweets per second): 7000. Each tweet is about 2.5KB = 7000*2.5KB = 17MB/sec.</li>
</ol>

<p>What would I need to read and store this data (say temporarily) ?</p>

<ul>
<li><strong>High speed internet</strong>. At least 30MB/sec. Do data centers get access to these kind of internet speeds? One dedicated machine with a direct connection? </li>
<li><em>Reliability</em>: What would I do if this machine goes down? :-( - May be I'll need a multiple machine setup, but twitter stream cannot be split up in real time via different connections?</li>
<li><em>Scalability</em>: Twitter TPS can shoot up in near future? Need to plan for thicker pipes in the near future?</li>
<li><strong>Faster harddisks</strong>: WD/Seagate server class 7200RPM with 64MB cache can do up to 128MB/s 
<em>Reliability</em>: What happens if this hard drive fails? a bunch of disks should do? But what config? RAID? </li>
<li><em>Scalability</em>:For temporary storage this should work well, but if I need to archive 200 million tweets, I am probably going to need a more scalable solution. Hadoop HDFS is a good idea?</li>
<li><strong>Security</strong>: A corporate would want this machine to sit outside a DMZ? So, storing data temporarily (say 5minutes) of it on this server class machine and then pulling it into a HDFS would be a good idea? </li>
</ul>

<p>Let me know your thoughts, guys. If you think full twitter firehose is a pretty unrealistic scenario, lets assume that I'm looking for filtered feed which matches a bunch of keywords (but I can't miss any of the tweets). 
How would you go about architecting such a system? </p>
