<p>I am analyzing on the possibilities to use hadoop (HDFS) as data archival solution which is giving linear scalability and lower cost maintenance per tera byte.</p>

<p>Please let me know the your recommendations and set of the parameters like I/O, Memory, Disk which has to be analyzed to viz hadoop as data archival system.</p>

<p>On the related query, While trying to upload a 500MB sized file using hadoop shell as,</p>

<p>$ #We've 500MB file created using dd</p>

<p>$ dd if=/dev/zero of=500MBFile.txt bs=524288000 count=1</p>

<p>$ hadoop fs -Ddfs.block.size=67108864 -copyFromLocal 500MBFile.txt /user/cloudera/</p>

<p>Please let me know why the <strong>input file is not getting splitted</strong> based on the block size (64MB). This will be good to understand since as part of data archival if we're getting 1TB file, how this will be splitted and distributed across the cluster.</p>

<p>I've tried the exercise using single node cloudera hadoop setup and replication factor is 1.</p>

    <div class="post-text" itemprop="text">
<p>You can use HDFS as archiving/storage solution, while I am doubt it is optimal. Specifically it is not as high-available as let say OpenStack Swift and not suited for store small files<br>
In the same time if HDFS is Your choice I would suggest to build the cluster with storage oriented nodes. I would describe them as: <br>
a) Put large and slow SATA disks. Since data is not going to be read / written constantly - desktop grade disks might do - it will be a major saving. <br>
b) Put minimal memory - I would suggest 4 GB. It will not add much costs, but still enable ocaassional MR processing. <br>
c) Sinlge CPU will do. <br></p>

<p>Regarding copyFromLocal. Yep, file is getting split according to the defined block size.<br> 
<br> Distribution on cluster will be even across the cluster, taking to the account replication factor.  HDFS will also try to put each block on more then one rack<br></p>
