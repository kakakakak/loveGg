    <div class="post-text" itemprop="text">

<p>I have a hadoop streaming job whose output does not contain key/value pairs. You can think of it as value-only pairs or key-only pairs.</p>

<p>My streaming reducer (a php script) is outputting records separated by newlines. Hadoop streaming treats this as a key with no value, and inserts a tab before the newline. This extra tab is unwanted.</p>

<p>How do I remove it?</p>

<p>I am using hadoop 1.0.3 with AWS EMR. I downloaded the source of hadoop 1.0.3 and found this code in hadoop-1.0.3/src/contrib/streaming/src/java/org/apache/hadoop/streaming/PipeReducer.java :</p>

<pre><code>reduceOutFieldSeparator = job_.get("stream.reduce.output.field.separator", "\t").getBytes("UTF-8");
</code></pre>

<p>So I tried passing <code>-D stream.reduce.output.field.separator=</code> as an argument to the job with no luck. I also tried <code>-D mapred.textoutputformat.separator=</code> and <code>-D mapreduce.output.textoutputformat.separator=</code> with no luck.</p>

<p>I've searched google of course and nothing I found worked. One search result even stated there was no argument that could be passed to achieve the desired result (though, the hadoop version in that case was really really old).</p>

<p>Here is my code (with added line breaks for readability):</p>

<pre><code>hadoop jar streaming.jar -files s3n://path/to/a/file.json#file.json
    -D mapred.output.compress=true -D stream.reduce.output.field.separator=
    -input s3n://path/to/some/input/*/* -output hdfs:///path/to/output/dir
    -mapper 'php my_mapper.php' -reducer 'php my_reducer.php'
</code></pre>

                <span class="comment-copy">I used a workaround. My output did have several fields, but there was no key/value concept (as explained). The output is sent to AWS Redshift, which needs only one type of field delimiter. So, by choosing to use tabs (instead of what I was previously using), I &quot;tricked&quot; the hadoop streaming jar into not adding an extra trailing tab. (Ie, instead of outputting <code>a|b|c</code>, I output <code>a\tb\tc</code> and then just tell AWS Redshift to delimit on <code>\t</code> instead of on <code>|</code>).</span>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Looking at the org.apache.hadoop.mapreduce.lib.output.TextOutputFormat source, I see 2 things:</p>

<ol>
<li>The <code>write(key,value)</code> method writes a separator if key or value is non-null</li>
<li>The separator is always set, using the default (<code>\t</code>), when the <code>mapred.textoutputformat.separator</code> returns null (which I'm assuming happens with <code>-D stream.reduce.output.field.separator=</code></li>
</ol>

<p>Your only solution maybe to write your own OutputFormat that works around these 2 issues.</p>

<h3>My testing</h3>

<p>In a task I had, I wanted to reformat a line from</p>

<pre><code>id1|val1|val2|val3
id1|val1
</code></pre>

<p>into:</p>

<pre><code>id1|val1,val2,val3
id2|val1
</code></pre>

<p>I had a custom mapper (Perl script) to convert the lines. And for this task, I initially tried to do as a key-only (or value-only) input, but got the results with the trailing tab.</p>

<p>At first I just specified: </p>

<blockquote>
  <p>-D stream.map.input.field.separator='|' -D stream.map.output.field.separator='|'</p>
</blockquote>

<p>This gave the mapper a key, value pair, since my mapping wanted a key anyway. But this output now had the tab after the first field</p>

<p>I got the desired output when I added: </p>

<blockquote>
  <p>-D mapred.textoutputformat.separator='|'</p>
</blockquote>

<p>If I didn't set it or set to blank </p>

<blockquote>
  <p>-D mapred.textoutputformat.separator=</p>
</blockquote>

<p>then I would again get a tab after the first field.</p>

<p>It made sense once I looked at the source for TextOutputFormat</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>As helpful for others, using the tips above, I was able to do an implementation:  </p>

<pre><code>CustomOutputFormat&lt;K, V&gt; extends org.apache.hadoop.mapred.TextOutputFormat&lt;K, V&gt; {....}
</code></pre>

<p>with exactly one line of the built-in implementation of 'getRecordWriter' changed to: </p>

<pre><code>String keyValueSeparator = job.get("mapred.textoutputformat.separator", ""); 
</code></pre>

<p>instead of: </p>

<pre><code>String keyValueSeparator = job.get("mapred.textoutputformat.separator", "\t"); 
</code></pre>

<p>after compiling that into a Jar, and including it into my hadoop streaming call (via the instructions on hadoop streaming), the call looked like: </p>

<pre><code>hadoop   jar  /usr/lib/hadoop/contrib/streaming/hadoop-streaming-1.0.3.jar     \
-archives 'hdfs:///user/the/path/to/your/jar/onHDFS/theNameOfTheJar.jar' \
-libjars theNameOfTheJar.jar \
-outputformat com.yourcompanyHere.package.path.tojavafile.CustomOutputFormat  \
-file yourMapper.py    -mapper  yourMapper.py     \
-file yourReducer.py   -reducer yourReducer.py    \
-input $yourInputFile    \
-output $yourOutputDirectoryOnHDFS
</code></pre>

<p>I also included the jar in the folder I issued that call from. </p>

<p>It was working great for my needs (and it created no tabs at the end of the line after the reducer).</p>

<hr>

<p>update: based on a comment implying this is indeed helpful for others, here's the full source of my CustomOutputFormat.java file: </p>

<pre><code>import java.io.DataOutputStream;
import java.io.IOException;

import org.apache.hadoop.fs.FSDataOutputStream;
import org.apache.hadoop.fs.FileSystem;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.io.compress.CompressionCodec;
import org.apache.hadoop.io.compress.GzipCodec;
import org.apache.hadoop.mapred.FileOutputFormat;
import org.apache.hadoop.mapred.JobConf;
import org.apache.hadoop.mapred.RecordWriter;
import org.apache.hadoop.mapred.TextOutputFormat;
import org.apache.hadoop.util.Progressable;
import org.apache.hadoop.util.ReflectionUtils;

public class CustomOutputFormat&lt;K, V&gt; extends TextOutputFormat&lt;K, V&gt; {

    public RecordWriter&lt;K, V&gt; getRecordWriter(FileSystem ignored, JobConf job, String name,
        Progressable progress) throws IOException {
    boolean isCompressed = getCompressOutput(job);

    //Channging the default from '\t' to blank
    String keyValueSeparator = job.get("mapred.textoutputformat.separator", ""); // '\t'
    if (!isCompressed) {
        Path file = FileOutputFormat.getTaskOutputPath(job, name);
        FileSystem fs = file.getFileSystem(job);
        FSDataOutputStream fileOut = fs.create(file, progress);
        return new LineRecordWriter&lt;K, V&gt;(fileOut, keyValueSeparator);
    } else {
        Class&lt;? extends CompressionCodec&gt; codecClass = getOutputCompressorClass(job,
            GzipCodec.class);
        // create the named codec
        CompressionCodec codec = ReflectionUtils.newInstance(codecClass, job);
        // build the filename including the extension
        Path file = FileOutputFormat.getTaskOutputPath(job, name + codec.getDefaultExtension());
        FileSystem fs = file.getFileSystem(job);
        FSDataOutputStream fileOut = fs.create(file, progress);
        return new LineRecordWriter&lt;K, V&gt;(new DataOutputStream(
            codec.createOutputStream(fileOut)), keyValueSeparator);
    }
    }
}
</code></pre>

<hr>

<p>FYI: For your usage context, be sure to check this does not adversely affect hadoop-streaming managed interactions (in terms of separating key vs. value) between your mapper and reducer. To clarify: </p>

<ul>
<li><p>From my testing -- if you have a 'tab' in every line of your data (with something on each side of it), you can leave the built in defaults as they are: streaming will interpret the first thing before the first tab as your 'key', and all on that row after it as your 'value.' As such, it does not see a 'null value,' and won't append a tab that shows up after your reducer. (You'll see your final outputs sorted on the value of the 'key' that streaming interprets in each row as what it sees as occuring before each tab.) </p></li>
<li><p>Conversely, if you have no tabs in your data, and you don't override the defaults using the above trick(s), then you'll see the tabs after the run completes, for which the above override becomes a fix. </p></li>
</ul>
