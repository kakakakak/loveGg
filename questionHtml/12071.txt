<p>I use wordnet similarity java api to measure similarity between two synsets as such:</p>

<pre><code> public class WordNetSimalarity {
 private static ILexicalDatabase db = new NictWordNet();
 private static RelatednessCalculator[] rcs = {
                 new HirstStOnge(db), new LeacockChodorow(db), new Lesk(db),  new WuPalmer(db), 
                 new Resnik(db), new JiangConrath(db), new Lin(db), new Path(db)
                 };

 public static double computeSimilarity( String word1, String word2 ) {
         WS4JConfiguration.getInstance().setMFS(true);
         double s=0;
         for ( RelatednessCalculator rc : rcs ) {
                 s = rc.calcRelatednessOfWords(word1, word2);
                // System.out.println( rc.getClass().getName()+"\t"+s );
         }

        return s;
 } 
</code></pre>

<p>Main class</p>

<pre><code>      public static void main(String[] args) {
         long t0 = System.currentTimeMillis();

         File source = new File ("TagsFiltered.txt");

         try {

            Scanner scanner = new Scanner(source);
            PrintStream psStream= new PrintStream(target);
            while (scanner.hasNext()) {

            }
            for (int i = 0; i &lt; sList.size(); i++) {

                psStream.println(sList.get(i)+" "+sList.get(j)+" "+WordNetSimalarity.computeSimilarity(sList.get(i), sList.get(j)));

            psStream.close();
        } catch (Exception e) {e.printStackTrace();

         long t1 = System.currentTimeMillis();
         System.out.println( "Done in "+(t1-t0)+" msec." );
 }

<p>My database contain 595 synsets that's mean method <code>computeSimilarity</code> will be called  (595*594/2) time 
To compute Similarity between two words it spend more than <code>5000 ms</code>!
so to finalize my task I need at least one week !!</p>

<p>My question is how to reduce this period !</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Perl is different from a lot of other languages when it comes to threading/forking.</p>

<p>One of the key things that makes Perl threads different from other threads is that data is not shared by default. This makes threads much easier and safer to work with, you don't have to worry about thread safety of libraries or most of your code, just the threaded bit. However it can be a performance drag and memory hungry as Perl must put a copy of the interpreter and all loaded modules into each thread.</p>

<p>When it comes to forking I will only be talking about Unix. Perl emulates fork on Windows using threads, it works but it can be slow and buggy.</p>

<p>Forking Advantages</p>

<ul>
<li>Very fast to create a fork</li>
<li>Very robust</li>
</ul>

<p>Forking Disadvantages</p>

<ul>
<li>Communicating between the processes can be slow and awkward</li>
</ul>

<p>Thread Advantages</p>

<ul>
<li>Thread coordination and data interchange is fairly easy</li>
<li>Threads are fairly easy to use</li>
</ul>

<p>Thread Disadvantages</p>

<ul>
<li>Each thread takes a lot of memory</li>
<li>Threads can be slow to start</li>
<li>Threads can be buggy (better the more recent your perl)</li>
<li>Database connections are not shared across threads</li>
</ul>

<p>In general, to get good performance out of Perl threads it's best to start a pool of threads and reuse them. Forks can more easily be created, used and discarded.</p>

<p>For either case, you're likely going to want something to manage your pool of workers. For forking you're going to want to use Parallel::ForkManager or Child. Child is particularly nice as it has built in inter-process communication.</p>

<p>For threads you're going to want to use threads::shared, Thread::Queue and read perlthrtut.
Also, the number of threads is going to be dependent on the number of cores that your computer has.  If you have four cores, creating more than 3 threads isn't really going to be very helpful (3 + 1 for your main program).
<br></p>

<p>To be honest, though, threads/forking may not be the way to go.  In fact, in many situations they can even slow stuff down due to overhead.  If you really need the speed, the best way to get it is going to be through distributed computing.  I would suggest that you look into some sort of distributed computing platform to make your runtime better.  If you can reduce the dimensionality of your search/compareTo space to less than n^2, then map reduce or Hadoop may be a good option; otherwise, you'll just have a bunch of overhead and no use of the real scalability that Hadoop offers (@Thomas Jungblut).   </p>
