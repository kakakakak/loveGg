<td class="answercell">
    <div class="post-text" itemprop="text">
<p>What a mapper runs is controlled by the input split, and is completely up to you how you specify it. The HDFS block size has <em>nothing</em> to do with it (other than the fact that most splitters use the block size as a basic 'block' for creating the input splits in order to achieve good data locality). You can write your own splitter that takes an HDFS block and splits in 100 splits, if you so fancy. Aslo look at <a href="http://stackoverflow.com/questions/9678180/change-file-split-size-in-hadoop">Change File Split size in Hadoop</a>.</p>

<p>Now that being said, the wisdom of doing that ('many mappers with small splits') is highly questionable. Everybody else is trying to do the <em>opposite</em> (create few mappers with aggregated splits). See <a href="http://snowplowanalytics.com/blog/2013/05/30/dealing-with-hadoops-small-files-problem/" rel="nofollow">Dealing with Hadoop's small files problem</a>, <a href="http://blog.cloudera.com/blog/2009/02/the-small-files-problem/" rel="nofollow">The Small Files Problem</a>, <a href="http://www.slideshare.net/AmazonWebServices/amazon-elastic-mapreduce-deep-dive-and-best-practices-bdt404-aws-reinvent-2013" rel="nofollow">Amazon Elastic MapReduce Deep Dive and Best Practices</a> and so on.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>You dont really have to decrease the block size to have more mappers , that would process smaller amount of data.</p>

<p>You dont have to modify the HDFS block size ( <code>dfs.blocksize</code> ), let it be with th default global value as per your cluster configuration.</p>

<p>You may use <code>mapreduce.input.fileinputformat.split.maxsize</code> property in your job configuration with lower value than the block size.</p>

<p>The input splits will be calculated with this value and one mapper will be triggered for every input split calculated.</p>
