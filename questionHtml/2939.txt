    <div class="post-text" itemprop="text">

<p>On the <a href="https://hadoop.apache.org/docs/current/hadoop-project-dist/hadoop-hdfs/HDFSHighAvailabilityWithNFS.html" rel="nofollow">https://hadoop.apache.org/docs/current/hadoop-project-dist/hadoop-hdfs/HDFSHighAvailabilityWithNFS.html</a> there are words: </p>

<blockquote>
  <p>the DataNodes are configured with the location of both NameNodes, and send block location information and heartbeats to both.</p>
</blockquote>

<p>But why is this information sent to the namenode and its fallback brother? I thought that this information already contains in the namenode's fs image. The namenode should know where he put blocks. </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Name Node contains the meta data of the entire cluster. It contains the details of each folder, file, replication factor, block names etc. The Name Node also stores the information about the location of the blocks for each file (this information is constructed from the Block Reports sent by the Data Nodes) in memory.</p>

<p>Data Nodes store following information for each block:</p>

<ul>
<li>Actual data stored in the block </li>
<li>Meta data for the data stored in the block. Mainly contains checksums for the data stored in the block.</li>
</ul>

<p>They periodically send the heart beat and block reports to the Name Node.</p>

<p><strong>Heart Beat</strong>:</p>

<ul>
<li>Interval of heart beat reports is determined by configuration parameter <code>dfs.heartbeat.interval</code> (in hdfs-site.xml). By default this is set to 3 seconds.</li>
<li>Some of the information contained in the Heart beat is: 

<ul>
<li><strong>Registration</strong>: Data node registration information</li>
<li><strong>Capacity</strong>: Total storage capacity available at Data Node</li>
<li><strong>dfsUsed</strong>: Storage used by HDFS</li>
<li><strong>remaining</strong>: Remaining storage available for HDFS</li>
<li><strong>blockPoolUsed</strong>: Storage used by the block pool</li>
<li><strong>xmitsInProgress</strong>: Number of transfers from this Data Node to others</li>
<li><strong>xceiverCount</strong>: Number of active transceiver threads</li>
<li><strong>xmitsInProgress</strong>: Number of transfers from this Data Node to others</li>
<li><strong>cacheCapacity</strong>: Total cache capacity available at Data Node</li>
<li><strong>cacheUsed</strong>: Amount of cache used</li>
</ul></li>
<li>This information is used by the Name Node in the following ways:

<ul>
<li><strong>Health of the Data Node</strong>: Should this data node be marked as dead or alive?</li>
<li><strong>Registration of new Data Node</strong>: If this is a newly added Data Node, its information is registered</li>
<li><strong>Update the metrics of the Data Node</strong>: The information sent in the heart beat is used for updating the metrics of the node</li>
<li><strong>Issue commands to the Data Node</strong>: The Name Node can issue following commands to the Data Node, based on the information received in the heart beat: <code>BlockRecoveryCommand</code> (to recover specified blocks), <code>BlockCommand</code> (for transferring blocks to another Data Node, for invalidating certain blocks), <code>Cache/Uncache</code> (commands for caching / uncaching the blocks)</li>
</ul></li>
</ul>

<p><strong>Block Reports:</strong></p>

<ul>
<li>Interval of block reports is determined by configuration <code>dfs.blockreport.intervalMsec</code> (in hdfs-site.xml). By default this is set to 21600000 milliseconds.</li>
<li>Some of the information contained in the block report is:

<ul>
<li><strong>Registration</strong>: Data node registration information</li>
<li><strong>blocks</strong>: Information about the blocks, which contains: block ID, block length, block generation timestamp, state of the block replica (For e.g. replica is finalized or waiting to be recovered etc.) </li>
</ul></li>
<li>This information is used by the Name Node for:

<ul>
<li><strong>Process first block report</strong>: If it is a first time report for the newly registered Data Node, it just adds all the valid replicas. It ignores all the invalid blocks, till the next block report. </li>
<li><strong>For updating the information about blocks</strong>: The (Data Node -> Blocks) map is updated in the Name Node. The new block report is compared with the old report and information about successful blocks, corrupted blocks, invalidated blocks etc. is updated</li>
</ul></li>
</ul>
