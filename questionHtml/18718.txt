<p>You log in as <code>ec2-user</code> as Klaus suggested:</p>

<pre><code>ssh -i key.pem ec2-user@host
</code></pre>

<p>... and then you use <code>sudo</code> to run commands. E.g., to edit the <code>/etc/hosts</code> file which is owned by root and requires root privileges: <code>sudo nano /etc/hosts</code>.</p>

<p>Or you run <code>sudo su</code> to become the <code>root</code> user.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>By default <code>root</code> user is not allowed to login but you can use <code>ec2-user</code> as indicated by others.</p>

<p>Once you login with <code>ec2-user</code> you switch to <code>root</code> and change the SSH configuration.</p>

<p>To become the root user you run:</p>

<pre><code>sudo su -
</code></pre>

<p>Edit the SSH daemon configuration file <code>/etc/ssh/sshd_config</code>, e.g. by using vi, and replace the <code>PermitRootLogin</code> entry with the following:</p>

<pre><code>PermitRootLogin without-password
</code></pre>

<p>Reload the SSH daemon configuration by running:</p>

<pre><code>/etc/init.d/sshd reload
</code></pre>

<p>The message <code>Please login as the ec2-user user rather than root user.</code> is displayed because a command is executed when you login with the private key. To remove that command edit <code>~/.ssh/authorized_keys</code> file and remove the <code>command</code> option. The line should start with the key type (Eg. ssh-rsa).</p>

<p>(*) Do at your own risk. I recommend you to leave always a console open just in case you're not able to login after you make the configuration changes.</p>

<p>For reference you can read the man pages:</p>

<pre><code>man sshd_config
man sshd
</code></pre>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I have encountered a similar problem when setting up a hadoop cluster on Amazon ec2. </p>

<p>My head node needs to have root ssh access to each worker/slave nodes. I aliased the connects by adding each slave node's IP address, private address, and alias name to the <code>/etc/hosts/</code> file. (I get that data by running the command <code>echo -e "`hostname -i`\t`hostname -f`\talias-name"</code> where <code>alias-name</code> is what I call each node (<code>head</code> or <code>n1</code> for example). Then I put that output for each node in every node's <code>/etc/hosts</code> file.</p>

<p>The problem I have been encountering is that when I type ssh n1 while in my head node to ssh into my first slave node, I get that same error message: <code>Please login as the use "ec2-user" rather than the user "root".</code>
So after doing some research, I figured out how to fix it.</p>

<p><strong>First:</strong> </p>

<ul>
<li>ssh into your server. non-root (ec2-user) access is fine here.  </li>
<li>Then    <code>su -</code> your way into root. Now <code>vi /etc/ssh/sshd_config</code> and 
un-comment the line <code>PermitRootLogin yes</code>.</li>
<li>Exit vi editor.</li>
<li>Now    restart ssh daemon by typing <code>service sshd stop</code> then <code>service
sshd    start</code>.</li>
</ul>

<p><strong>Second:</strong> </p>

<ul>
<li>Now, here is the part I had to dig for, </li>
<li>run <code>vi /root/.ssh/authorized_keys</code></li>
<li>Comment out everything up to <code>ssh-rsa.</code> Just put a <code>#</code> at the beginning
of the file's content, before <code>no-port-forwarding</code>... and hit enter on <code>ssh-rsa</code> to move it to
the next line (this way you dont have to delete anything in case you
want to backtrack). </li>
<li>exit vi editor</li>
</ul>

<p>Now you should be able to login to root without that error message popping up. </p>

<p>Also, if you are using aliases for a cluster setup; Repeat the same steps on each node. First ssh in using ec2-user then follow the steps.
After adding the IP address, private address, and alias name info to your <code>/etc/hosts</code> file you should be able to ssh into each node's root using the alias name for example <code>ssh n1</code>.</p>

<p>The tutorial I followed is here: <a href="https://www.youtube.com/watch?v=xrxQXfE7t9A" rel="nofollow">https://www.youtube.com/watch?v=xrxQXfE7t9A</a></p>

<p>But it didnt discuss the problem with root login.</p>

<p>Hope that helps! It worked for me.</p>

<p>*Keep in mind that I havnt taken any security into concern. This is simply a practice/dev setup.</p>
