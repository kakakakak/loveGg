<p>In my Reducer code, I am using this code snippet to sum the values: </p>

<pre><code>for(IntWritable val : values) {
    sum += val.get();           
}
</code></pre>

<p>As the above mentioned gives me expected output, I tried changing the code to:</p>

<pre><code>for(IntWritable val : values) {
    sum += 1;
}
</code></pre>

<p>Can anyone please explain what is the difference it makes when I use <code>sum += 1</code> in the reducer rather than <code>sum += val.get()</code>? Why does it give me the same output? Does it have anything to do with Combiner, because when I used this same reducer code as Combiner, class the output was incorrect with all words showing a count of 1.</p>

<p>Mapper Code :</p>

<pre><code>public void map(LongWritable key, Text value, Context context) throws IOException, InterruptedException {

        String line = value.toString();
        StringTokenizer token = new StringTokenizer(line);

        while(token.hasMoreTokens()) {
            word.set(token.nextToken());
            context.write(word, new IntWritable(1));
        }
    }
</code></pre>

<p>Reducer Code :</p>

<pre><code>public void reduce(Text key, Iterable&lt;IntWritable&gt; values, Context context) throws IOException, InterruptedException {

        int sum = 0;

        for(IntWritable val : values) {
            //sum += val.get();
            sum += 1;
        }
        context.write(key, new IntWritable(sum));
    }
</code></pre>

<p>Driver Code:</p>

<pre><code>job.setJarByClass(WordCountWithCombiner.class);
        //job.setJobName("WordCount");

        job.setMapperClass(WordCountMapper.class);
        job.setCombinerClass(WordCountReducer.class);
        job.setReducerClass(WordCountReducer.class);

        FileInputFormat.addInputPath(job, new Path(args[0]));
        FileOutputFormat.setOutputPath(job, new Path(args[1]));

        job.setMapOutputKeyClass(Text.class);
        job.setMapOutputValueClass(IntWritable.class);

        job.setOutputKeyClass(Text.class);
        job.setOutputValueClass(IntWritable.class);
</code></pre>

<p>Input - "to be or not to be"</p>

<p>Expected Output - (be,2) , (to,2) , (or,1) , (not,1)</p>

<p>But Output i am getting is - (be,1) , (to,1) , (or,1) , (not,1)</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<blockquote>
  <p>Can anyone please explain what is the difference it makes when I use <code>sum += 1</code> in the reducer rather than <code>sum += val.get()</code>?</p>
</blockquote>

<p>Both the statements are performing addition operation. In the first you are counting how many times the <code>for-loop</code> has run. In the later, you are are actually doing a sum operation, on the <code>int</code> value returned by the each <code>val</code> object for a given <code>key</code>.</p>

<blockquote>
  <p>Why does it give me the same output? Does it have anything to do with Combiner</p>
</blockquote>

<p>The answer is <strong>Yes</strong>. It is because of the <code>Combiner</code>.</p>

<p>Now lets look at the input you are passing, this will instantiate only one <code>Mapper</code>. The output of the <code>Mapper</code> is:</p>

<pre><code>(to,1), (be,1), (or,1), (not,1), (to,1), (be,1) 
</code></pre>

<p>When this goes to the <code>Combiner</code>, which is essentially same logic as <code>Reducer</code>. The output will be:</p>

<pre><code>(be,2) , (to,2) , (or,1) , (not,1)
</code></pre>

<p>Now the above output of <code>Combiner</code> goes to the <code>Reducer</code> and it will perform the sum operation however you define it. So if your logic is <code>sum += 1</code> then output will be:</p>

<pre><code>(be,1) , (to,1) , (or,1) , (not,1)
</code></pre>

<p>But if your logic is <code>sum += val.get()</code> then your output will be:</p>

<pre><code>(be,2) , (to,2) , (or,1) , (not,1)
</code></pre>

<p>I hope you understand it now. The logic of the <code>Combiner</code> and <code>Reducer</code> is same, But the input which is coming to them for processing is <strong>different</strong>.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>All depends on the value of <code>sum += val.get();</code></p>

<p>If always <code>val.get()</code> return <strong>1</strong>, then <code>sum += val.get();</code> is the same than <code>sum += 1;</code> as it is happening in your reducer.</p>

<p>BUT</p>

<p>The <strong>combainer</strong> is used to do a pre-aggregation (similar than the reducer aggregation) in the mapper side, previous to send the key-values pairs to the recuder(s).</p>

<p>Hadoop framework doesn't warranty the times that the combiner is executed by Mapper, it will depend on the number of Mapper's outputs. Then, if only one time the combiner is executed, the aggregation in the mapper side will be ok but in the reducer instead to only receive <strong>1's</strong> you could receive other number (<code>val.get() &gt;= 1</code>). And if you use <code>sum += 1;</code> in your reducer, you will be dropping the aggregated numbers in the mapper, generating a wrong output.</p>

<p>If the combiner is executed more than one time in the Mapper side, then you could imagine that the problem could be even worst.</p>

<p>In summary, <code>sum += 1;</code> only works <strong>if and only if</strong> that statement is executed only <strong>one time</strong> for each key-value. Using the combiner, that is not warranted.</p>

<p><code>val.get();</code> return an <code>int</code> so basically both the codes are same. The reason we are using val.get() depends on the problem we are trying to solve. In your case we are sure that in the mapper each word is emitted as the key and the value as 1, so in the reducer you can be sure that val.get() will always return 1. Hence the hard coded integer value 1 gives the same result.</p>

<p>Also using the same reducer as the combiner function should not cause any problem. One of the scenario where the output would be with all words giving count as '1' would be when the number of reducers is set as 0 and the mapper output is written to the output path.</p>
