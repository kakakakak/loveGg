<p><em>1st of all this not a question asking for help to deploy below components step by step. What I'm asking is for an advice on how the architecture should be designed. What I'm planning to do is develop a reporting platform using existing data. Following is data I gathering by researching.</em></p>

<p>I have an existing RDBMS which has large number of records. So I'm using</p>

<ul>
<li>Scoop - Extract data from RDBMS to Hadoop</li>
<li>Hadoop - Storage platform</li>
<li>Hive - Datawarehouse</li>
<li>Spark - Since Hive is more like batch processing Spark on Hive will speed up things</li>
<li>JasperReports - To generate reports.</li>
</ul>

<p>What I have done up to know is deployed a Hadoop 2 cluster as follows</p>

<ul>
<li>192.168.X.A - Namenode</li>
<li>192.168.X.B - 2nd Namenode</li>
<li>192.168.X.C - Slave1</li>
<li>192.168.X.D - Slave2</li>
<li>192.168.X.E - Slave3</li>
</ul>

<p>My problems are</p>

<ul>
<li>In which node should I deploy Spark? A or B, Given that I want to support fail-over. That's why I have a separate namenode configured on B.</li>
<li>Should I deploy Spark on each and every instances? Who are the worker nodes should be?</li>
<li>In which node should I deploy Hive? Is there a better alternative to Hive?</li>
<li>How should I connect JasperReports? And to where? To Hive or Spark?</li>
</ul>

<p>Please tell me a suitable way to design the architecture? Please provide a elaborated answer.</p>

<p><em>Note that if you can provide any technical guides or case studies in similar nature it would be really helpful.</em> </p>

    <div class="post-text" itemprop="text">
<p>You've figured it out, already! All my answers are merely general opinions and might drastically change depending on data, flavors of operations to be performed. Also question implies data and results of such operations are mission critical, I assumed so.   </p>

<blockquote>
  <p>Spark on Hive will speed up things</p>
</blockquote>

<p>Not necessarily correct. Anecdotal evidence, <a href="http://blog.cloudera.com/blog/2014/09/new-benchmarks-for-sql-on-hadoop-impala-1-4-widens-the-performance-gap/" rel="nofollow">this post (by cloudera)</a>, proves the quite opposite. There is actually a move towards the vice-versa, i.e. <a href="https://cwiki.apache.org/confluence/display/Hive/Hive+on+Spark" rel="nofollow">Hive on Spark</a>.  </p>

<blockquote>
  <p>In which node should I deploy Spark? A or B, Given that I want to support fail-over. That's why I have a separate namenode configured on B.
  Should I deploy Spark on each and every instances? Who are the worker nodes should be?</p>
</blockquote>

<p>Definitely - in most cases anyway. Set A or B as master, all of the rest can be worker nodes. If you don't want to have SPOF in your architecture, see <a href="https://spark.apache.org/docs/1.5.1/spark-standalone.html" rel="nofollow">high availability section of spark documentation</a>, requires a bit of extra work.</p>

<blockquote>
  <p>Is there a better alternative to Hive?</p>
</blockquote>

<p>This one is both subjective and task-specific. If SQL querying feels natural and fits the task, there is also <a href="http://www.cloudera.com/content/www/en-us/products/apache-hadoop/impala.html" rel="nofollow">Impala</a> promoted by Cloudera, which claims to perform and order of magnitude faster than Hive. But is sort of a stranger in Apache Hadoop ecosystem. With Spark -and if you are fine typing a bit of python or scala- you can do SQL-like querying while still enjoying the expressive power these languages provide. </p>

<blockquote>
  <p>How should I connect JasperReports? And to where? To Hive or Spark?</p>
</blockquote>

<p>Don't know about this one.</p>
