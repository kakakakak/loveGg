    <div class="post-text" itemprop="text">

<p>I have a moderate-sized file (4GB CSV) on a computer that doesn't have sufficient RAM to read it in (8GB on 64-bit Windows).  In the past I would just have loaded it up on a cluster node and read it in, but my new cluster seems to arbitrarily limit processes to 4GB of RAM (despite the hardware having 16GB per machine),  so I need a short-term fix.</p>

<p>Is there a way to read in part of a CSV file into R to fit available memory limitations?  That way I could read in a third of the file at a time, subset it down to the rows and columns I need, and then read in the next third?</p>

<p>Thanks to commenters for pointing out that I can potentially read in the whole file using some big memory tricks:
<a href="http://stackoverflow.com/questions/1727772/quickly-reading-very-large-tables-as-dataframes-in-r">Quickly reading very large tables as dataframes in R</a></p>

<p>I can think of some other workarounds (e.g. open in a good text editor, lop off 2/3 of the observations, then load in R), but I'd rather avoid them if possible.</p>

<p>So reading it in pieces still seems like the best way to go for now.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>You could read it into a database using RSQLite, say, and then use an sql statement to get a portion.</p>

<p>If you only need a single portion then <code>read.csv.sql</code> in the sqldf package will read the data into an sqlite database.  It creates the database for you and the data does not go through R so limitations of R won't apply. After reading it into the database it reads the output of a specified sql statement into R and then destroys the database.  Depending on how fast it works with your data you might be able to just repeat the whole process for each portion if you have several.   </p>

<p>Since its only one line of code to do the whole thing for a single portion its a no-brainer to just try it.  <code>DF &lt;- read.csv.sql("myfile.csv", sql=..., ...other args...)</code>.   See <code>?read.csv.sql</code> and <code>?sqldf</code>  and also the <a href="http://sqldf.googlecode.com">sqldf home page</a>.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I know that this is a very old thread.  However I've come across it recently because I had a similar problem. After over-viewing this thread I noticed a conspicuous solution to this problem was not mentioned.  Use connections!</p>

<p>1) Open a connection to your file</p>

<pre><code>con = file("file.csv", "r")
</code></pre>

<p>2) Read in chunks of code with read.csv</p>

<pre><code>read.csv(con, nrows="CHUNK SIZE",...)
</code></pre>

<p>Side note: defining colClasses will greatly speed things up.  Make sure to define unwanted columns as NULL.</p>

<p>3) Do what ever you need to do</p>

<p>4) Repeat.</p>

<p>5) Close the connection</p>

<pre><code>close(con)
</code></pre>

<p>The advantage of this approach is connections.  If you omit this step, it will likely slow things down a bit.  By opening a connection manually, you essentially open the data set and do not close it until you call the close function.  This means that as you loop through the data set you will never lose your place.  Imagine that you have a data set with 1e7 rows.  Also imagine that you want to load a chunk of 1e5 rows at a time.  Since we open the connection we get the first 1e5 rows by running <code>read.csv(con, nrow=1e5,...)</code>, then to get the second chunk we run <code>read.csv(con, nrow=1e5,...)</code> as well, and so on.... </p>

<p>If we did not use the connections we would get the first chunk the same way, <code>read.csv("file.csv", nrow=1e5,...)</code>, however for the next chunk we would need to <code>read.csv("file.csv", skip = 1e5, nrow=2e5,...)</code>. Clearly this is inefficient.  We are have to find the 1e5+1 row all over again, despite the fact that we just read in the 1e5 row. </p>

<p>Notice that this approach is also embarrassingly parallel.  Simple database are not able to run in parallel.  Of course Hadoop, circumvents this problem.  Regardless, this approach can be done quickly in parallel.  Just remember that the cores on your machine share the same RAM.  Make sure not to make the chunks too big!</p>

<p>Finally, <code>data.table::fread</code> is great.  But you can not pass it connections.  So this approach does not work.</p>

<p>I hope this helps someone.</p>

<p><strong>UPDATE</strong></p>

<p>People keep upvoting this post so I thought I would add one more brief thought.  The new <code>readr::read_csv</code>, like <code>read.csv</code>, can be passed connections.  However, it is <a href="http://blog.rstudio.org/2015/04/09/readr-0-1-0/" rel="nofollow">advertised</a> as being roughly 10x faster.     </p>
