    <div class="post-text" itemprop="text">

<p>I am trying to design a table for storing time series data in HBase. Since all data that are written in a month will be read "all at once" for analysis, I was wondering whether <strong>using the month (as a string, like '201501') as column family</strong> was a good choice.</p>

<p>My application is going to save data as they arrive with timestamps then, at the end of the month, all data related to that month is going to be analyzed with a hadoop job.</p>

<p>According to the specs, all data in a column family will be stored together. Is this a good practice in terms of (1) design and (2) performance ?</p>

<p>I'm not sure, since this is that kind of choice that is far away from the relational world.</p>

    <div class="post-text" itemprop="text">
<p>In General, having many column families in HBase hurts performance. I think more than 2 or 3 column families is not recommended when using HBase. I would suggest browsing through this <a href="http://hbase.apache.org/book/number.of.cfs.html" rel="nofollow">link</a> for more details. </p>

<p>Also, the column family is part of the table schema when you create the table. </p>

<p>Example: <code>create 'hbase_table',{NAME=&gt;'columnFamily',VERSIONS=&gt;3}</code></p>

<p>So if you want a new column family for every month, you're going to have to alter the table schema to add column families whenever required, which is not ideal. </p>

<p>According to me, a good solution here would be to have the date (<a href="http://hbase.apache.org/book/rowkey.design.html" rel="nofollow">or reverse timestamp</a>) in your rowkey. 
Example: <code>XXX-YYY-20150106</code> or <code>XXX-YYY-9223370616269200807</code> (reverse timestamp)</p>

<p>Then, when you want to analyse data between 1 month, you can scan the table for the date range. 
Example: <code>scan 'hbase_table',{STARTROW=&gt;'XXX-YYY-20150101', STOPROW=&gt;'XXX-YYY-20150201'}</code></p>
