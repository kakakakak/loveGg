<p>I've basic understanding on what pig, hive abstractions are. But I don't have a clear idea on the scenarios that require HIVE , PIG or native map reduce.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Complex branching logic which has a lot of nested if .. else .. structures is easier and quicker to implement in Standard MapReduce, for processing structured data you could use <a href="http://pangool.net/introduction.html">Pangool</a>, it also simplifies things like JOIN. Also Standard MapReduce gives you full control to minimize the number of MapReduce jobs that your data processing flow requires, which translates into performance. But it requires more time to code and introduce changes.</p>

<p>Apache PIG is good for structured data too, but its advantage is the ability to work with BAGs of data (all rows that are grouped on a key), it is simpler to implement things like:</p>

<ol>
<li>Get top N elements for each group;</li>
<li>Calculate total per each group and than put that total against each row in the group;</li>
<li>Use Bloom filters for JOIN optimisations;</li>
<li>Multiquery support (it is when PIG tries to minimise the number on MapReduce Jobs by doing more stuff in a single Job)</li>
</ol>

<p>Hive is better suited for ad-hoc queries, but its main advantage is that it has engine that stores and partitions data. But its tables can be read from PIG or Standard MapReduce.</p>

<p>One more thing, Hive and PIG are not well suited to work with hierarchical data.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Short answer - We need MapReduce when we need very deep level and fine grained control on the way we want to process our data. Sometimes, it is not very convenient to express what we need exactly in terms of Pig and Hive queries.</p>

<p>It should not be totally impossible to do, what you can using MapReduce, through Pig or Hive. With the level of flexibility provided by Pig and Hive you can somehow manage to achieve your goal, but it might be not that smooth. You could write UDFs or do something and achieve that.</p>

<p>There is no clear distinction as such among the usage of these tools. It totally depends on your particular use-case. Based on your data and the kind of processing you need to decide which tool fits into your requirements better.</p>

<p><strong>Edit :</strong></p>

<p>Sometime ago I had a use case wherein I had to collect seismic data and run some analytics on it. The format of the files holding this data was somewhat weird. Some part of the data was <a href="http://en.wikipedia.org/wiki/EBCDIC">EBCDIC</a> encoded, while rest of the data was in binary format. It was basically a flat binary file with no delimiters like\n or something. I had a tough time finding some way to process these files using Pig or Hive. As a result I had to settle down with MR. Initially it took time, but gradually it became smoother as MR is really swift once you have the basic template ready with you.</p>

<p>So, like I said earlier it basically depends on your use case. For example, iterating over each record of your dataset is really easy in Pig(just a foreach), but what if you need <strong>foreach n</strong>?? So, when you need "that" level of control over the way you need to process your data, MR is more suitable.</p>

<p>Another situation might be when you data is hierarchical rather than row-based or if your data is highly unstructured.</p>

<p>Metapatterns problem involving job chaining and job merging are easier to solve using MR directly rather than using Pig/Hive.</p>

<p>And sometimes it is very very convenient to accomplish a particular task using some xyz tool as compared to do it using Pig/hive. IMHO, MR turns out to be better in such situations as well. For example if you need to do some statistical analyses on your BigData, R used with Hadoop streaming is probably the best option to go with.</p>

<p>HTH</p>

    <div class="post-text" itemprop="text">
<p><strong>Mapreduce:</strong></p>

<pre><code>Strengths:
      works both on structured and unstructured data.
      good for writing complex business logic.

Weakness:
     long development type
     hard to achieve join functionality
</code></pre>

<p><strong>Hive :</strong></p>

<pre><code>Strengths:
     less development time.
     suitable for adhoc analysis.
     easy for joins

Weakness :
     not easy for complex business logic.
     deals only structured data.
</code></pre>

<p><strong>Pig</strong></p>

<pre><code>Strengths :
      Structured and unstructured data.
      joins are easily written.

Weakness:
     new language to learn.
     converted into mapreduce.
</code></pre>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Hive</p>

<p>Pros:</p>

<p>Sql like
Data-base guys love that.
Good support for structured data.
Currently support database schema and views like structure 
Support concurrent multi users, multi session scenarios.
Bigger community support. Hive , Hiver server , Hiver Server2, Impala ,Centry  already </p>

<p>Cons:
Performance degrades as data grows bigger not much to do, memory over flow issues. cant do much with it.
Hierarchical data is a challenge.
Un-structured data requires udf like component
Combination of multiple techniques could be a nightmare dynamic portions with UTDF in case of big data  etc </p>

<p>Pig:
Pros:
Great script based data flow language. </p>

<p>Cons:</p>

<p>Un-structured data requires udf like component
Not a  big community support</p>

<p>MapReudce:
Pros:
Dont agree with "hard to achieve join functionality", if you understand what kind of join you want to implement you can implement with few lines of code.
Most of the times MR yields better performance.
MR support for hierarchical data is great especially implement tree like structures. 
Better control at partitioning / indexing the data.
Job chaining.</p>

<p>Cons:
Need to know api very well to get a better performance etc
Code / debug / maintain</p>

    <div class="post-text" itemprop="text">
<p><strong><em><a href="https://www.dezyre.com/article/-mapreduce-vs-pig-vs-hive/163" rel="nofollow">Scenarios</a> where <code>Hadoop Map Reduce</code> is preferred to <code>Hive</code> or <code>PIG</code></em></strong></p>

<ol>
<li><p>When you need definite driver program control</p></li>
<li><p>Whenever the job requires implementing a custom Partitioner</p></li>
<li><p>If there already exists pre-defined library of Java Mappers or Reducers for a job</p></li>
<li>If you require good amount of testability when combining lots of large data sets </li>
<li>If the application demands legacy code requirements that command physical structure</li>
<li>If the job requires optimization at a particular stage of processing by making the best use of tricks like in-mapper combining </li>
<li>If the job has some tricky usage of distributed cache (replicated join), cross products, groupings or joins</li>
</ol>

<p><a href="http://i.stack.imgur.com/nxzJs.png" rel="nofollow"><img src="http://i.stack.imgur.com/nxzJs.png" alt="Comparison between Map reduce/ Pig/ Hive "></a></p>

<p><strong><em>Pros of Pig/Hive :</em></strong></p>

<ol>
<li>Hadoop MapReduce requires more development effort than Pig and Hive.</li>
<li>Pig and Hive coding approaches are slower than a fully tuned Hadoop MapReduce program.</li>
<li>When using Pig and Hive for executing jobs, Hadoop developers need not worry about any version mismatch.</li>
<li>There is very limited possibility for the developer to write java level bugs when coding in Pig or Hive.</li>
</ol>

<p>Have a look at this post for <a href="http://stackoverflow.com/questions/3356259/difference-between-pig-and-hive-why-have-both/33353322#33353322"><code>Pig Vs Hive</code></a> comparison. </p>
