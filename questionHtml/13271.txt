<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Generally speaking, cloud computing will give you slower machines, and all your data will have to go over the internet. However you can buy machines by the hour, so that for bursts of compute power you can probably afford to get a lot of machines.</p>

<p>If you set up your own local grid, you'll be able to decide the spec of machines that you get, and set them up how you like. However it'll be expensive to buy a ton of them if you're not using them all the time.</p>

<p>If you want to do the cost analysis, you can take a look at our cost calculator here (<a href="https://secure.slicify.com/Calculator.aspx" rel="nofollow">https://secure.slicify.com/Calculator.aspx</a>) to get a rough idea how much on-demand compute power will cost from a variety of providers (full disclosure - this is a page on my site).</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I don't think that's ... the correct question to ask. For example, Google Compute Engine charges based on what hardware you're choosing.</p>

<p>In general, your code can run as fast in Cloud (does not equal "Grid") as in local LAN. You need figure out where (if there exists) the bottleneck is. All "Grid" implementations farms out calculation to "Nodes". But bottleneck on database tier.</p>

<p>Hadoop is one which get around this by "HDFS" (Hadoop Distributed File System).</p>

<p>Obvious alternative is, for example, database clustering.</p>

<p>Anyway, in general, there's no reason why your code run slower in Cloud (Please don't tell me link between database and your grid is slow however!)</p>
