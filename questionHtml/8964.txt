    <div class="post-text" itemprop="text">

<p>I know that MapReduce is a great framework for batch processing on Hadoop. But, Spark also can be used as batch framework on Hadoop that provides scalability, fault tolerance and high performance compared MapReduce. Cloudera, Hortonworks and MapR started supporting Spark on Hadoop with YARN as well. </p>

<p>But, a lot of companies are still using MapReduce Framework on Hadoop for batch processing instead of Spark.  </p>

<p>So, I am trying to understand what are the current challenges of Spark to be used as batch processing framework on Hadoop?  </p>

<p>Any thoughts?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Spark is an order of magnitude faster than mapreduce for iterative algorithms, since it gets a significant speedup from keeping intermediate data cached in the local JVM.</p>

<p>With Spark 1.1 which primarily includes a new shuffle implementation (<a href="https://issues.apache.org/jira/browse/SPARK-2045" rel="nofollow">sort-based shuffle</a> instead of hash-based shuffle), a <a href="https://issues.apache.org/jira/browse/SPARK-2468" rel="nofollow">new network module</a> (based on netty instead of using block manager for sending shuffle data), a new <a href="https://issues.apache.org/jira/browse/SPARK-3796" rel="nofollow">external shuffle service</a> made Spark perform the fastest PetaByte sort (on 190 nodes with 46TB RAM) and TeraByte sort breaking Hadoop's old record.</p>

<p>Spark can easily handle the dataset which are order of magnitude larger than the cluster's aggregate memory. So, my thought is that Spark is heading in the right direction and will eventually get even better.</p>

<p>For reference this <a href="http://databricks.com/blog/2014/10/10/spark-petabyte-sort.html" rel="nofollow">blog post</a> explains how databricks performed the petabyte sort.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I'm assuming when you say Hadoop you mean HDFS. </p>

<p>There are number of benefits of using Spark over Hadoop MR. </p>

<ol>
<li><p><strong>Performance:</strong> Spark is at least as fast as Hadoop MR. For iterative algorithms (that need to perform number of iterations of the same dataset) is can be a few orders of magnitude faster. Map-reduce writes the output of each stage to HDFS. </p>

<p>1.1. Spark can cache (depending on the available memory) this intermediate results and therefore reduce latency due to disk IO. </p>

<p>1.2. Spark operations are lazy. This means Spark can perform certain optimizing before it starts processing the data because it can reorder operations because they have executed yet. </p>

<p>1.3. Spark keeps a lineage of operations and recreates the partial failed state based on this lineage in case of failure.  </p></li>
<li><p><strong>Unified Ecosystem:</strong> Spark provides a unified programming model for various types of analysis - batch (spark-core), interactive (REPL), streaming (spark-streaming), machine learning (mllib), graph processing (graphx), SQL queries (SparkSQL)</p></li>
<li><p><strong>Richer and Simpler API:</strong> Spark's API is richer and simpler. Richer because it supports many more operations (e.g., groupBy, filter ...). Simpler because of the expressiveness of these functional constructs. Spark's API supports Java, Scala and Python (for most APIs). There is experimental support for R.</p></li>
<li><p><strong>Multiple Datastore Support:</strong> Spark supports many data stores out of the box. You can use Spark to analyze data in a normal or distributed file system, HDFS, Amazon S3, Apache Cassandra, Apache Hive and ElasticSearch to name a few. I'm sure support for many other popular data stores is comings soon. This essentially if you want to adopt Spark you don't have to move your data around. </p></li>
</ol>

<p>For example, here is what code for word count looks in Spark (Scala). </p>

<pre><code>val textFile = sc.textFile("some file on HDFS")
val wordCounts = textFile.flatMap(line =&gt; line.split(" ")).map(word =&gt; (word, 1)).reduceByKey((a, b) =&gt; a + b)
</code></pre>

<p>I'm sure you have to write a few more lines if you are using standard Hadoop MR. </p>

<p>Here are some common misconceptions about Spark.  </p>

<ol>
<li><p><em>Spark is just a in-memory cluster computing framework</em>. However, this is not true. Spark excels when your data can fit in memory because memory access latency is lower. But you can make it work even when your dataset doesn't completely fit in memory. </p></li>
<li><p><em>You need to learn Scala to use Spark</em>. Spark is written in Scala and runs on the JVM. But the Spark provides support for <em>most</em> of the common APIs in Java and Python as well. So you can easily get started with Spark without knowing Scala. </p></li>
<li><p><em>Spark does not scale</em>. Spark is for small datasets (GBs) only and doesn't scale to large number of machines or TBs of data. This is also not true. It has been used successfully to sort <a href="http://databricks.com/blog/2014/10/10/spark-petabyte-sort.html" rel="nofollow">PetaBytes of data</a> </p></li>
</ol>

<p>Finally, if you do not have a legacy codebase in Hadoop MR it makes perfect sense to adopt Spark, the simple reason being all major Hadoop vendors are moving towards Spark for good reason. </p>

<td class="answercell">

<p>Apache Spark runs in memory, making it much faster than mapreduce. 
Spark started as a research project at Berkeley. </p>

<p>Mapreduce use disk extensively (for external sort, shuffle,..). </p>

<p>As the input size for a hadoop job is in order of terabytes. Spark memory requirements will be more than traditional hadoop. </p>

<p>So basically, for smaller jobs and with huge memory in ur cluster, sparks wins. And this is not practically the case for most clusters.</p>

<p>Refer to spark.apache.org for more details on spark</p>
