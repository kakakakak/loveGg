    <div class="post-text" itemprop="text">

<p>Let's say I want to store million of blogs &amp; news articles to a mongodb server. </p>

<p>These articles will have some fields which I can use for aggregation such as: category, author, location,  domain, etc. I can store these articles in a mongo database but then those articles also have fields like text, summary which contain quite a lot of data and possibly make a document quite large (still &lt; 16mb)</p>

<p>My question is when mongodb runs aggregation, does it read the whole document into memory and do the aggregation from there? Apparently, all the data from disk cannot fit into memory. Does the size of document affect the aggregation performance? </p>

<p>What is the good design/alternative for storage &amp; "REALTIME" aggregation? </p>

<p>I don't want to use batch process like Hadoop for my project because Realtime aggregation is a must. I've seen a setup of 2 mongo dbs, 1 for storage which stores the original document, the other just store fields for aggregation in a star schema but I don't really like this approach because it would require to maintain the 2 versions of a document in 2 places.</p>

<p>Thanks.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<blockquote>
  <p>My question is when mongodb runs aggregation, does it read the whole document into memory and do the aggregation from there?</p>
</blockquote>

<p>No, since recent versions there has been a change in how projection works and now it is able to use covered queries or, rather, partial loading: <a href="http://docs.mongodb.org/manual/core/aggregation-pipeline-optimization/#projection-optimization" rel="nofollow">http://docs.mongodb.org/manual/core/aggregation-pipeline-optimization/#projection-optimization</a></p>

<blockquote>
  <p>The optimization phase applies the projection to the head of the pipeline such that only the _id and the amount fields return in the resulting documents from the $match stage as well.</p>
</blockquote>

<p>So you can load bits of the document without worrying about loading the entire text content etc as well.</p>

<blockquote>
  <p>Does the size of document affect the aggregation performance? </p>
</blockquote>

<p>It effects any operation. Bigger the document the more IO is required to load it, even if the allocation is contiguous on the hard drive.</p>

<p>It can also effect, as you noted, memory usage. Your working set could find problems with larger documents and you could face page thrashing.</p>

<blockquote>
  <p>What is the good design/alternative for storage &amp; "REALTIME" aggregation? </p>
</blockquote>

<p>Pre-aggregation with incremental map reduce is a good choice: <a href="http://docs.mongodb.org/ecosystem/use-cases/pre-aggregated-reports/" rel="nofollow">http://docs.mongodb.org/ecosystem/use-cases/pre-aggregated-reports/</a> I personally use it far more than the aggregation framework, with much greater success.</p>

<blockquote>
  <p>I don't want to use batch process like Hadoop for my project because Realtime aggregation is a must. </p>
</blockquote>

<p>"Realtime"? What exactly is "realtime"? Is it on the spot with the user waiting while your server crunches the data for about 30 mins or is it the users data being delayed by 2 minutes and the user not waiting at all for page load times etc?</p>

<p>If you need high granularity then you can make the wait closer to 5 seconds between updates.</p>

<p>Realtime is not always on the spot processing, just look to many other sites on this front.</p>
