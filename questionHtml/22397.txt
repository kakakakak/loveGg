    <div class="post-text" itemprop="text">

<p>Yarn differs in its infrastructure layer from the original map reduce architecture in the following way: </p>

<p>In YARN, the job tracker is split into two different daemons called <code>Resource Manager</code> and <code>Node Manager</code> (node specific). The resource manager only manages the allocation of resources to the different jobs apart from comprising a scheduler which just takes care of the scheduling jobs without worrying about any monitoring or status updates. Different resources such as memory, cpu time, network bandwidth etc. are put into one unit called the <code>Resource Container</code>. There are different <code>AppMasters</code> running on different nodes which talk to a number of these resource containers and accordingly update the Node Manager with the monitoring/status details.</p>

<p>I want to know that how does using this kind of an approach increase the performance from the map-reduce perspective? Also, if there is any definitive content on the motivation behind Yarn and its benefits over the existing implementation of Map-reduce, please point me to the same.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Here are some of the articles (<a href="http://developer.yahoo.com/blogs/hadoop/posts/2011/02/mapreduce-nextgen/">1</a>, <a href="http://www.cloudera.com/blog/2012/02/mapreduce-2-0-in-hadoop-0-23/">2</a>, <a href="http://www.cloudera.com/blog/2011/11/building-and-deploying-mr2/">3</a>) about YARN. These talk about the benefits of using YARN.</p>

<p>YARN is more general than MR and it should be possible to run other computing models like <a href="http://en.wikipedia.org/wiki/Bulk_synchronous_parallel">BSP</a> besides MR. Prior to YARN, it required a separate cluster for MR, BSP and others. Now they they can coexist in a single cluster, which leads to higher usage of the cluster. <a href="http://wiki.apache.org/hadoop/PoweredByYarn">Here</a> are some of the applications ported to YARN.</p>

<p>From a MapReduce perspective in legacy MR there are separate slots for Map and Reduce tasks, but in YARN their is no fixed purpose of a container. The same container can be used for a Map task, Reduce task, Hama BSP Task or something else. This leads to better utilization.</p>

<p>Also, it makes it possible to run different versions of Hadoop in the same cluster which is not possible with legacy MR, which makes is easy from a maintenance point.</p>

<p><a href="http://www.thecloudavenue.com/p/mrv2resources.html">Here</a> are some of the additional links for YARN. Also, <a href="http://shop.oreilly.com/product/0636920021773.do">Hadoop: The Definitive Guide, 3rd Edition</a> has an entire section dedicated to YARN.</p>

<p>FYI, it had been a bit <a href="http://mail-archives.apache.org/mod_mbox/hadoop-common-dev/201112.mbox/%3C71806AE9-0C01-4ED9-AA49-D9341422D899@open-mpi.org%3E">controversial</a> to develop YARN instead of using some of frameworks which had been doing something similar and had been running for ages successfully with bugs ironed out.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I do not think that Yarn will speedup the existing MR framework. Looking into architecture we can see that the system now is more modular - but modularity usually contradicts higher performance. <br>
It can be claimed that YARN has nothing to do with MapReduce. MapReduce just became one of the YARN applications. You can see it as moving from some embedded program to embeded OS with program within it<br>
At the same time Yarn opens the door for <strong>different</strong> MR implementations with different frameworks. For example , if we assume that our dataset is smaller then cluster memory we can get much better performance. I think <a href="http://www.spark-project.org/">http://www.spark-project.org/</a> is one such example <br>
To summarize it: Yarn does not improve the existing MR, but will enable other MR implementations to be better in all aspects. </p>
