<h2>reduce vs foldLeft</h2>

<p>A big big difference, not mentioned in any other stackoverflow answer relating to this topic clearly, is that <code>reduce</code> should be given a <em>commutative monoid</em>, i.e. an operation that is both commutative and associative.  This means the operation can be parallelized.</p>

<p>This distinction is very important for Big Data / MPP / distributed computing, and the entire reason why <code>reduce</code> even exists.  The collection can be chopped up and the <code>reduce</code> can operate on each chunk, then the <code>reduce</code> can operate on the results of each chunk - in fact the level of chunking need not stop one level deep.  We could chop up each chunk too.  This is why summing integers in a list is O(log N) if given an infinite number of CPUs.</p>

<p>If you just look at the signatures there is no reason for <code>reduce</code> to exist because you can achieve everything you can with <code>reduce</code> with a <code>foldLeft</code>.  The functionality of <code>foldLeft</code> is a greater than the functionality of <code>reduce</code>.</p>

<p><strong>But</strong> you cannot parallelize a <code>foldLeft</code>, so its runtime is always O(N) (even if you feed in a commutative monoid). This is because it's assumed the operation is <em>not</em> a commutative monoid and so the cumulated value will be computed by a series of sequential aggregations.  </p>

<p><code>foldLeft</code> does not assume commutativity nor associativity.  It's associativity that gives the ability to chop up the collection, and it's commutativity that makes cumulating easy because order is not important (so it doesn't matter which order to aggregate each of the results from each of the chunks). Strictly speaking commutativity is not necessary for parallelization, for example distributed sorting algorithms, it just makes the logic easier because you don't need to give your chunks an ordering.</p>

<p>If you have a look at the Spark documentation for <code>reduce</code> it specifically says "... commutative and associative binary operator"</p>

<p><a href="http://spark.apache.org/docs/1.0.0/api/scala/index.html#org.apache.spark.rdd.RDD">http://spark.apache.org/docs/1.0.0/api/scala/index.html#org.apache.spark.rdd.RDD</a></p>

<p>Here is proof that <code>reduce</code> is NOT just a special case of <code>foldLeft</code></p>

<pre><code>scala&gt; val intParList: ParSeq[Int] = (1 to 100000).map(_ =&gt; scala.util.Random.nextInt()).par

scala&gt; timeMany(1000, intParList.reduce(_ + _))
Took 462.395867 milli seconds

scala&gt; timeMany(1000, intParList.foldLeft(0)(_ + _))
Took 2589.363031 milli seconds
</code></pre>

<h2>reduce vs fold</h2>

<p>Now this is where it gets a little closer to the FP Category Theory roots, and a little trickier to explain.</p>

<p>There is no <code>fold</code> method in Scalding because under the (strict) Map Reduce programming model we cannot define <code>fold</code> because chunks do not have an ordering and <code>fold</code> only requires associativity, not commutativity.  Spark <em>does</em> have <code>fold</code> because its framework is a super-set of the Map Reduce programming model and can order its chunks. Well you can actually do this in Hadoop too, but Scalding doesn't seem to expose this functionality in the version I'm familiar with.  </p>

<p>Put simply, <code>reduce</code> works without an order of cumulation, <code>fold</code> requires an order of cumulation and it is that order of cumulation that necessitates a zero value NOT the existence of the zero value that distinguishes them.  Strictly speaking <code>reduce</code> <em>should</em> work on an empty collection, because its zero value can by deduced by taking an arbitrary value <code>x</code> and then solving <code>x op y = x</code>, but that doesn't work with a non-commutative operation as there can exist a left and right zero value that are distinct (i.e. <code>x op y != y op x</code>).  Of course Scala doesn't bother to work out what this zero value is as that would require doing some mathematics (which are probably uncomputable), so just throws an exception.</p>

<p>This difference between <code>reduce*</code> and <code>fold*</code> is a FP historical convention and has its roots in Category Theory.  I hope now this deep difference relating to parallelization will no longer go unnoticed :)</p>

<p>One other difference for Scalding is the use of combiners in Hadoop. </p>

<p>Imagine your operation is commutative monoid, with <em>reduce</em> it will be applied on the map side also instead of shuffling/sorting all data to reducers. With <em>foldLeft</em> this is not the case. </p>

<pre><code>pipe.groupBy('product) {
   _.reduce('price -&gt; 'total){ (sum: Double, price: Double) =&gt; sum + price }
   // reduce is .mapReduceMap in disguise
}

pipe.groupBy('product) {
   _.foldLeft('price -&gt; 'total)(0.0){ (sum: Double, price: Double) =&gt; sum + price }
}
</code></pre>

<p>It is always good practice to define your operations as monoid in Scalding.</p>

<p>If I am not mistaken, even though the Spark API does not require it, fold also requires for the f to be commutative. Because the order in which the partitions will be aggregated is not assured.
For example in the following code only the first print out is sorted:  </p>

<pre><code>import org.apache.spark.{SparkConf, SparkContext}

object FoldExample extends App{

  val conf = new SparkConf()
    .setMaster("local[*]")
    .setAppName("Simple Application")
  implicit val sc = new SparkContext(conf)

  val range = ('a' to 'z').map(_.toString)
  val rdd = sc.parallelize(range)

  println(range.reduce(_ + _))
  println(rdd.reduce(_ + _))
  println(rdd.fold("")(_ + _))
}  
</code></pre>

<p>abcdefghijklmnopqrstuvwxyz</p>
