    <div class="post-text" itemprop="text">

<p>I am trying NN metadata recovery.  I have taken backup of Namenode and Journal node meta data . It contains edit logs and fsimages.</p>

<p>There are two NNs in my system. I take backup of metadata on both NNs (hdfs metadata &amp; QJM metadata) at regular frequency. I want to test recovery procedure in a worst case scenario. Assume both the NNs and Journal node are down with the metadata completely deleted.</p>

<p>I want to recover NN metadata from backup and start NN. I know that there could be a data loss as the latest changes done after backup would be missing.</p>

<p><strong>Questions</strong>:</p>

<ol>
<li>Do you think such a scenario is possible/feasible ?</li>
<li>
I am facing some issues related to txn id mismatch, committed txn id. Please tell if there is a solution for the same.</li>
</ol>

<p><strong>Steps tried:</strong></p>

<ol>
<li>Take metadata backup of NN and QJM. Do some hdfs file operations
(create new files). </li>
<li>Stop NN and Journal node on both the machines.</li>
<li>Delete metadata from /data/hdfs and journal directories. </li>
<li>Restore Fsimages from backup (taken some time back).  </li>
<li>Start NN. It fails with below exception.</li>
</ol>

<p>Alternative approach: Restore all the edit logs and fsimage to both hdfs and qjm directories and start NN but still it fails.</p>

<p>Both the NNs are down and I can't bring up. I don't want to format hdfs as it will change Cluster ID and the backup won't be usable.</p>

<p><strong>Exceptions</strong>:</p>

<ol>
<li>There appears to be a gap in the edit log.  We expected txid 71453, but got txid 71466</li>
<li>Client trying to move committed txid backward from 71599 to 71453</li>
<li>recoverUnfinalizedSegments failed for required journal. Decided to synchronize log to startTxId: 71453 but logger 10.204.64.26:8485 had seen txid 71599 committed</li>
</ol>
