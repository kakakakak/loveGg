<p>Everyone</p>

<p>A few days ago I upgraded our 6 node EC2 cluster from cassandra 2.1.4 to 2.1.5.</p>

<p>Since then, all my nodes have "exploded" in their cpu usage - their at 100% cpu for much of the time and their load average is between 100-300 (!!!).</p>

<p>This did not start immediately after the upgrade. It started a few hours afterwards with one of the nodes, and slowly, more and more nodes began to exhibit the same behavior.
It seems to correlate with the compaction of our largest column family, and after the compaction is complete (~24 hours after it starts) it seems the node goes back to normal. It has only been 2 days or so, so I'm hoping it will not happen again, but I am still monitoring this.</p>

<p>Here are my questions</p>

<ol>
<li>Is this a bug or an expected behavior?</li>
</ol>

<p>If it is an expected behavior - </p>

<ol>
<li>What is the explanation for this issue?</li>
<li>Is it documented somewhere that I have missed?</li>
<li>Should I do upgrades differently? Maybe 1 or 2 nodes at a time every 24 hours or so? What is the best practice?</li>
</ol>

<p>If it is a bug - </p>

<ol>
<li>Is it known?</li>
<li>Where should I report this? What data should I add?</li>
<li>Will downgrading back to 2.1.4 work?</li>
</ol>

<p>Any feedback on this would be great</p>

<p>Thanks</p>

<p>Amir</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Answering my own question - </p>

<p>We are using a very specific thrift API - describe_splits_ex, and that seems to cause the issue.
It is obvious when looking at all the stack traces of all the different threads while the cpu usage goes to 100%.
For us, it was easy to fix, because we use this api as an optimization, not a must, so we just stopped using it, and the issue went away.</p>

<p>However, this api is also used by the cassandra-hadoop connector (at least it used to in earlier versions), so I would test before upgrading to 2.1.5 if you are using the connector.</p>

<p>Not sure what change in 2.1.5 caused the issue, but I know it did not happen in 2.1.4 and happened consistently in 2.1.5.</p>
