    <div class="post-text" itemprop="text">

<p>I've just set up a Hadoop cluster with Hadoop 0.20.205. I have a master (NameNode and JobTracker) and two other boxes (slaves).</p>

<p>I'm trying to understand, how to define the number of map and reduce tasks to use.</p>

<p>So far I understood that I can set the maximum number of map and reduce tasks that each TaskTracker is able to handle simultaneously with: <code>*mapred.tasktracker.map.tasks.maximum*</code> and <code>*mapred.tasktracker.reduce.tasks.maximum*</code>.</p>

<p>Also, I can define the maximum number of map tasks the whole cluster can run simultaneously with <code>*mapred.map.tasks*</code>. Is that right?</p>

<p>If so, how can I know what should be the value for <code>*mapred.tasktracker.map.tasks.maximum*</code>? I see that the default is 2. But why? What are the pros and cons of increasing or decreasing this value?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I don't think that there is a rule for that (like the rule for setting the number of reducers). </p>

<p>What I do is, set the number of mappers and reducers to the number of cores available minus 1 for each machine. Intuitively, this will leave each machine some memory for the other processes (like cluster communication). But I may be wrong. Anyway, this is the only thing I found from  "<a href="http://books.google.gr/books?id=x99mnGARqlYC&amp;pg=PA78&amp;dq=%22maximum%20concurrent%20map%20tasks%22&amp;hl=el&amp;sa=X&amp;ei=ypadUt7DBoThywOqpIGIBA&amp;ved=0CD4Q6AEwAA#v=onepage&amp;q=%22maximum%20concurrent%20map%20tasks%22&amp;f=false" rel="nofollow">Pro Hadoop</a>".  It suggests using as many mappers as the number of available cores and one or two reducers.
I hope it helps.</p>

            <div style="display: block;" class="comment-body">
                <span class="comment-copy">Yes, that&#39;s the correct number. With 0.95 factor, all reduce tasks are run at the same time, but each one takes typically longer. With 1.75 factor, each reduce task finsihses faster, but some reduce tasks are waiting in a queue to be executed. Prefer the 1.75 factor for better load balancing.  Intuitively, with 0.95, when a reducer finishes, the machine does nothing. With 1.75, when a reducer finishes faster than the others, it starts working on the reduce tasks waiting in the queue.</span>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Here is what I propose. Hope it helps!</p>

<ul>
<li><p>Run "hadoop fsck /" in the master node to find out the size and number of blocks. For e.g.: </p>

<pre><code>...
Total size: 21600037259 B
Total dirs: 78
Total files:    152
Total blocks (validated):   334 (avg. block size 64670770 B)
...
</code></pre></li>
<li><p>I set up reduce tasks as <code>num_of_blocks / 10</code>. <br>
set <code>mapred.map.tasks=33</code>; </p></li>
<li><p>I set up map tasks as <code>block_size (in MB) * 2</code>. <br>
<code>set mapred.reduce.tasks=124;</code></p></li>
</ul>

<p>So far that's the best configuration I've found. And you'll have to modify it according your cluster's configuration.</p>
