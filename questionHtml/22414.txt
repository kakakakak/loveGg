<td class="answercell">
    <div class="post-text" itemprop="text">
<blockquote>
  <p>it is taking 2 Hrs to process the whole file.</p>
</blockquote>

<p>50 GB / 2 hours equals approximately 7 MB/s. It's not a bad rate at all. A good (modern) hard disk should be capable of sustaining higher rate continuously, so maybe your bottleneck is not the I/O? You're already using BufferedReader, which, like the name says, is buffering (in memory) what it reads. You could experiment creating the reader with a bit bigger buffer than the default size (8192 bytes), like so:</p>

<pre><code>BufferedReader br = new BufferedReader(
    new InputStreamReader(new FileInputStream("FileName")), 100000);
</code></pre>

<p>Note that with the default 8192 bytes buffer and 7 MB/s throughput the BufferedReader is going to re-fill its buffer almost 1000 times per second, so lowering that number could really help cutting down some overhead. But if the processing that you're doing, instead of the I/O, is the bottleneck, then no I/O trick is going to help you much. You should maybe consider making it multi-threaded, but whether it's doable, and how, depends on what "processing" means here.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Your only hope is to parallelize the reading and processing of what's inside.  Your strategy should be to never require the entire file contents to be in memory at once.</p>

<p>Start by profiling the code you have to see where the time is being spent.  Rewrite the part that takes the most time and re-profile to see if it improved.  Keep repeating until you get an acceptable result.</p>

<p>I'd think about Hadoop and a distributed solution.  Data sets that are larger than yours are processed routinely now.  You might need to be a bit more creative in your thinking.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Without NIO you won't be able to break the throughput barrier. For example, try using <code>new Scanner(File)</code> instead of directly creating readers. Recently I took a look at that source code, it uses NIO's file channels.</p>

<p>But the first thing I would suggest is to run an empty loop with <code>BufferedReader</code> that does nothing but reading. Note the throughput -- and also keep an eye on the CPU. If the loop floors the CPU, then there's definitely an issue with the IO code.</p>

<ol>
<li><p>Disable the antivirus and any other program which adds to disk contention while reading the file.  </p></li>
<li><p>Defragment the disk. </p></li>
<li><p>Create a raw disk partition and read the file from there.  </p></li>
<li><p>Read the file from an SSD.</p></li>
<li><p>Create a 50GB Ramdisk and read the file from there.</p></li>
