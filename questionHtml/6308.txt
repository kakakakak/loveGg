    <div class="post-text" itemprop="text">

<p>I am new to <a href="http://docs.python.org/3.3/library/concurrent.futures.html">the futures module</a> and have a task that could benefit from parallelization; but I don't seem to be able to figure out exactly how to setup the function for a thread and the function for a process. I'd appreciate any help anyone can shed on the matter.</p>

<p>I'm running a <a href="http://en.wikipedia.org/wiki/Particle_swarm_optimization">particle swarm optimization (PSO)</a>. Without getting into too much detail about PSO itself, here's the basic layout of my code:</p>

<p>There is a <code>Particle</code> class, with a <code>getFitness(self)</code> method (which computes some metric and stores it in <code>self.fitness</code>). A PSO simulation has multiple particle instances (easily over 10; 100s or even 1000s for some simulations).<br>
Every so often, I have to compute the fitness of the particles. Currently, I do this in for-loop:</p>

<pre><code>for p in listOfParticles:
  p.getFitness(args)
</code></pre>

<p>However, I notice that the fitness of each particle can be computed independently of each other. This makes this fitness computation a prime candidate for parallelization. Indeed, I could do <code>map(lambda p: p.getFitness(args), listOfParticles)</code>.</p>

<p>Now, I can easily do this with <code>futures.ProcessPoolExecutor</code>:</p>

<pre><code>with futures.ProcessPoolExecutor() as e:
  e.map(lambda p: p.getFitness(args), listOfParticles)
</code></pre>

<p>Since the side-effects of calling <code>p.getFitness</code> are stored in each particle itself, I don't have to worry about getting a return from <code>futures.ProcessPoolExecutor()</code>.</p>

<p>So far, so good. But now I notice that <code>ProcessPoolExecutor</code> creates new processes, which means that it copies memory, which is slow. I'd like to be able to share memory - so I should be using threads. That's well and good, until I realize that running several processes with several threads inside each process will likely be faster, since multiple threads still run only on one processor of my sweet, 8-core machine.</p>

<p>Here's where I run into trouble:<br>
Based on the examples I've seen, <code>ThreadPoolExecutor</code> operates on a <code>list</code>. So does <code>ProcessPoolExecutor</code>. So I can't do anything iterative in <code>ProcessPoolExecutor</code> to farm out to <code>ThreadPoolExecutor</code> because then <code>ThreadPoolExecutor</code> is going to get a single object to work on (see my attempt, posted below).<br>
On the other hand, I cant slice <code>listOfParticles</code> myself, because I want <code>ThreadPoolExecutor</code> to do its own magic to figure out how many threads are required.</p>

<p><strong>So, the big question (at long last)</strong>:<br>
How should I structure my code so that I can effectively parallelize the following using both processes AND threads:</p>

<pre><code>for p in listOfParticles:
  p.getFitness()
</code></pre>

<p>This is what I've been trying, but I wouldn't dare try to run it, for I know it won't work:</p>

<pre><code>&gt;&gt;&gt; def threadize(func, L, mw):
...     with futures.ThreadpoolExecutor(max_workers=mw) as executor:
...             for i in L:
...                     executor.submit(func, i)
... 

&gt;&gt;&gt; def processize(func, L, mw):
...     with futures.ProcessPoolExecutor() as executor:
...             executor.map(lambda i: threadize(func, i, mw), L)
...
</code></pre>

<p>I'd appreciate any thoughts on how to fix this, or even on how to improve my approach</p>

<p>In case it matters, I'm on python3.3.2</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I'll give you working code that mixes processes with threads for solving the problem, but it's not what you're expecting ;-)  First thing is to make a mock program that doesn't endanger your real data.  Experiment with something harmless.  So here's the start:</p>

<pre><code>class Particle:
    def __init__(self, i):
        self.i = i
        self.fitness = None
    def getfitness(self):
        self.fitness = 2 * self.i
</code></pre>

<p>Now we have something to play with.  Next some constants:</p>

<pre><code>MAX_PROCESSES = 3
MAX_THREADS = 2 # per process
CHUNKSIZE = 100
</code></pre>

<p>Fiddle those to taste.  <code>CHUNKSIZE</code> will be explained later.</p>

<p>The first surprise for you is what my lowest-level worker function does.  That's because you're overly optimistic here:</p>

<blockquote>
  <p>Since the side-effects of calling p.getFitness are stored in
  each particle itself, I don't have to worry about getting a
  return from futures.ProcessPoolExecutor().</p>
</blockquote>

<p>Alas, <em>nothing</em> done in a worker process can have any effect on the <code>Particle</code> instances in your main program.  A worker process works on <em>copies</em> of <code>Particle</code> instances, whether via a copy-on-write implementation of <code>fork()</code> or because it's working on a copy made from unpickling a <code>Particle</code> pickle passed across processes.</p>

<p>So if you want your main program to <em>see</em> the fitness results, you need to arrange to send information back to the main program.  Because I don't know enough about your actual program, here I'm assuming that <code>Particle().i</code> is a unique integer, and that the main program can easily map integers back to <code>Particle</code> instances.  With that in mind, the lowest-level worker function here needs to return a pair:  the unique integer and the fitness result:</p>

<pre><code>def thread_worker(p):
    p.getfitness()
    return (p.i, p.fitness)
</code></pre>

<p>Given that, it's easy to spread a list of <code>Particle</code>s across threads, and return a list of <code>(particle_id, fitness)</code> results:</p>

<pre><code>def proc_worker(ps):
    import concurrent.futures as cf
    with cf.ThreadPoolExecutor(max_workers=MAX_THREADS) as e:
        result = list(e.map(thread_worker, ps))
    return result
</code></pre>

<p>Notes:</p>

<ol>
<li>That's the function each worker process will run.</li>
<li>I'm using Python 3, so use <code>list()</code> to force <code>e.map()</code> to materialize all the results in a list.</li>
<li>As mentioned in a comment, under CPython spreading CPU-bound tasks across threads is <em>slower</em> than doing them all in a single thread.</li>
</ol>

<p>It only remains to write code to spread a list of <code>Particle</code>s across processes, and retrieve the results.  This is dead easy to do with <code>multiprocessing</code>, so that's what I'm going to use.  I have no idea whether <code>concurrent.futures</code> can do it (given that we're also mixing in threads), but don't care.  But because I'm giving you working code, you can play with that and report back ;-)</p>

<pre><code>if __name__ == "__main__":
    import multiprocessing

    particles = [Particle(i) for i in range(100000)]
    # Note the code below relies on that particles[i].i == i
    assert all(particles[i].i == i for i in range(len(particles)))

    pool = multiprocessing.Pool(MAX_PROCESSES)
    for result_list in pool.imap_unordered(proc_worker,
                      (particles[i: i+CHUNKSIZE]
                       for i in range(0, len(particles), CHUNKSIZE))):
        for i, fitness in result_list:
            particles[i].fitness = fitness

    pool.close()
    pool.join()

    assert all(p.fitness == 2*p.i for p in particles)
</code></pre>

<p>Notes:</p>

<ol>
<li>I'm breaking the list of <code>Particle</code>s into chunks "by hand".  That's what <code>CHUNKSIZE</code> is for.  That's because a worker process wants <em>a list</em> of <code>Particle</code>s to work on, and in turn that's because that's what the <code>futures</code> <code>map()</code> function wants.  It's a Good Idea to chunk up work regardless, so you get some real bang for the buck in return for the per-invocation interprocess overheads.</li>
<li><code>imap_unordered()</code> makes no guarantees about the order in which results are returned.  That gives the implementation more freedom to arrange work as efficiently as possible.  And we don't care about the order here, so that's fine.</li>
<li>Note that the loop retrieves the <code>(particle_id, fitness)</code> results, and modifies the <code>Particle</code> instances accordingly.  Perhaps your real <code>.getfitness</code> makes other mutations to <code>Particle</code> instances - can't guess.  Regardless, the main program will never see any mutations made in workers "by magic" - you have to explicitly arrange for that.  In the limit, you could return <code>(particle_id, particle_instance)</code> pairs instead, and <em>replace</em> the <code>Particle</code> instances in the main program.  Then they'd reflect all mutations made in worker processes.</li>
</ol>

<p>Have fun :-)</p>

<h2>Futures all the way down</h2>

<p>Turns out it was very easy to replace <code>multiprocessing</code>.  Here are the changes.  This also (as mentioned earlier) <em>replaces</em> the original <code>Particle</code> instances, so as to capture all mutations.  There's a tradeoff here, though:  pickling an instance requires "a lot more" bytes than pickling a single "fitness" result.  More network traffic.  Pick your poison ;-)</p>

<p>Returning the mutated instance just requires replacing the last line of <code>thread_worker()</code>, like so:</p>

<pre><code>return (p.i, p)
</code></pre>

<p>Then replace all of the "<strong>main</strong>" block with this:</p>

<pre><code>def update_fitness():
    import concurrent.futures as cf
    with cf.ProcessPoolExecutor(max_workers=MAX_PROCESSES) as e:
        for result_list in e.map(proc_worker,
                      (particles[i: i+CHUNKSIZE]
                       for i in range(0, len(particles), CHUNKSIZE))):
            for i, p in result_list:
                particles[i] = p

if __name__ == "__main__":
    particles = [Particle(i) for i in range(500000)]
    assert all(particles[i].i == i for i in range(len(particles)))

    update_fitness()

    assert all(particles[i].i == i for i in range(len(particles)))
    assert all(p.fitness == 2*p.i for p in particles)
</code></pre>

<p>The code is very similar to the <code>multiprocessor</code> dance.  Personally, I'd use the <code>multiprocessing</code> version, because <code>imap_unordered</code> is valuable.  That's a problem with simplified interfaces:  they often buy simplicity at the cost of hiding useful possibilities.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>First, are you sure to leverage from running multiple thread while loading all your cores with processes? If it is cpu-bound, <a href="http://stackoverflow.com/a/67006/1265154">hardly yes</a>. At least some tests has to be made. </p>

<p>If adding threads leverage your performance, the next question is whether one can achive better performance with hand-made load balancing, or automatic. By hand-made I mean  careful workload partitioning into chunks of similar computational complexity and instatiating a new task processor per chunk, your orinal but doubted solution. By automatic, creation of pool of processes/threads and communication on work queue for new tasks, that one you strive for. In my view, first approach is one of Apache Hadoop paradigm, second is implemented by works queue processors, such as Celery. First approach may suffer from some tasks chunks being slower and running while others completed, second  adds commutication and waiting-on-task overheads, and this is second point of performance tests to be made.</p>

<p>Last, if you wish to have a static collection of processes with multithreads within, AFAIK, you can't achive it with <code>concurrent.futures</code> as is, and have to modify it a bit. I don't know, whether there are existing solutions for this task, but as <code>concurrent</code> is a pure python solution (with no C code), it can easely be done. Work processor is defined in <code>_adjust_process_count</code> <a href="http://hg.python.org/cpython/file/a75b88048339/Lib/concurrent/futures/process.py#l381" rel="nofollow">routine</a> of <code>ProcessPoolExecutor</code> class, and subclassing and overriding it with multi-threaded approach is rather straigtforward, you just have to supply your custom <code>_process_worker</code>, based on <a href="http://hg.python.org/cpython/file/3.3/Lib/concurrent/futures/thread.py" rel="nofollow"><code>concurrent.features.thread</code></a></p>

<p>Original <code>ProcessPoolExecutor._adjust_process_count</code> for reference:</p>

<pre><code>def _adjust_process_count(self):
    for _ in range(len(self._processes), self._max_workers):
        p = multiprocessing.Process(
                target=_process_worker,
                args=(self._call_queue,
                      self._result_queue))
        p.start()
        self._processes[p.pid] = p
</code></pre>
