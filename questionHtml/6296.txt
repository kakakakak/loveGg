    <div class="post-text" itemprop="text">

<p>I'm thinking about the possibility of teaching the use of Software Transactional Memory through 1 or 2 guided laboratories for a university course. I only know about Haskell's STM, but the students of the course probably never heard a word about it.</p>

<p>I already found some lists of such libraries online or in other questions (e.g., <a href="http://en.wikipedia.org/wiki/Software_transactional_memory#C.2FC.2B.2B" rel="nofollow">http://en.wikipedia.org/wiki/Software_transactional_memory#C.2FC.2B.2B</a>). I'm checking them out as you read this, but many of them do not seem to have a very nice documentation (most are research prototypes only vaguely described in papers, and I would rather teach about something more used and well documented).
Furthermore, many of the links provided by wikipedia are dangling.</p>

<p><strong>To sum it up</strong>, are there STM implementations aimed to industrial projects (or at least non-toy ones, to ensure a certain level of quality) and well documented (to give some good pointers to the students)?</p>

<p><strong>EDIT</strong>: I'm not the teacher of the course, I just help him with the laboratories. Of course the students will be taught basics of concurrency and distributed algorithms before. This was just an idea to propose something different towards the end of the course.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Production-quality STM-Libraries are not intended as a <strong>teaching tool</strong>, not even as "best practice". What is worth learning for any college/university-course is maybe <strong>1%</strong> of the code; the remaining 99% is <em>nitty-gritty platform-dependent intrinsic corner-cases</em>. The 1% that is interesting is not highlighted in any way so you have no way of finding it.</p>

<p>What I recommend for a college/university-course (no matter if introductory or advanced) is to implement STM-buildingblocks yourself (and only for 1 platform). </p>

<p>Start by introducing the problems: concurrency, cache...</p>

<p>Then introduce the atomic helpers we have: cas/cmpxchg, fence.</p>

<p>Then build examples together with your students, first easy, then harder and more complex.</p>

    <div class="post-text" itemprop="text">
<blockquote>
  <p>Start by introducing the problems: concurrency, cache...</p>
</blockquote>

<p>Leading on from <a href="http://stackoverflow.com/users/618598/eznme">eznme</a>, some good problems that I covered while at University for <code>concurrency</code>.</p>

<p><ul>
<li><a href="http://en.wikipedia.org/wiki/Dining_philosophers_problem" rel="nofollow">Dining philosophers problem</a></p>

<blockquote>
  <p>In computer science, the dining philosophers problem is an example problem often used in concurrent algorithm design to illustrate synchronization issues and techniques for resolving them.</p>
  
  <p><img src="http://upload.wikimedia.org/wikipedia/commons/thumb/6/6a/Dining_philosophers.png/200px-Dining_philosophers.png" alt="dining phil"></li>
  </ul>
  Using the same implementation from <a href="http://www.doc.ic.ac.uk/~jnm/book/book_applets/Diners.html" rel="nofollow">here</a>, by Je Magee and Je Kramer, and solving the problem using monitors.</p>
</blockquote>

<p>Most shared memory applications are more efficient with <code>Integers</code> than Strings (due to <code>AtomicInteger</code> class for Java). So the best way to demonstrate <code>shared memory</code> in my opinion is to get the students to write an application that uses a <code>threadpool</code> to calculate prime numbers, or to calculate some <code>integral</code>.</p>

<p>Or a good example of threads and shared memory is the <a href="http://en.wikipedia.org/wiki/Producer-consumer_problem" rel="nofollow">Producer-consumer problem</a>.</p>

<blockquote>
  <p>The producer-consumer problem (also known as the bounded-buffer problem) is a classic example of a multi-process synchronization problem. </p>
  
  <p><img src="http://cse.csusb.edu/tong/courses/cs460/images/producer-consumer.gif" alt="producer"></p>
</blockquote>

<p>Implementation found <a href="http://www.doc.ic.ac.uk/~jnm/book/book_applets/TimedProducerConsumer.html" rel="nofollow">here</a>, there is also an implementation from <a href="http://www.massey.ac.nz" rel="nofollow">Massey</a> from the professor in Software Eng <a href="https://sites.google.com/site/jensdietrich/" rel="nofollow">Jenz Dietrich</a>.</p>

<p>For distributed algorithms <a href="http://en.wikipedia.org/wiki/MapReduce" rel="nofollow">MapReduce</a> and <a href="http://hadoop.apache.org/" rel="nofollow">Hadoop</a> are highly documented distributed data structures. And for Distributed Programming Libraries look into <a href="http://parallel.vub.ac.be/documentation/mpi/index.html" rel="nofollow">MPI (Message Passing Interface)</a> and <a href="http://openmp.org/wp/" rel="nofollow">OpenMP</a> (or <a href="http://en.wikipedia.org/wiki/Preprocessor_directive" rel="nofollow">Pragma</a> for C++). There is also implementations of <a href="http://en.wikipedia.org/wiki/Dijkstra%27s_algorithm" rel="nofollow">Dijkstra shortest path algorithm</a> in <a href="http://www.mcs.anl.gov/~itf/dbpp/text/node35.html" rel="nofollow">parallel</a> too.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>There are three good ways to do STM today.</p>

<p>The first way is to use gcc and do TM in C or C++.  As of gcc 4.7, transactional memory is supported via the -fgnu-tm flag.  The gcc maintainers have done a lot of work, and as of the 4.9 (trunk) branch, you can even use hardware TM (e.g., Intel Haswell TSX).  There is a draft specification for the interface to the TM at <a href="http://justingottschlich.com/tm-specification-for-c-v-1-1/" rel="nofollow">http://justingottschlich.com/tm-specification-for-c-v-1-1/</a>, which is not too painful.  You can also find use cases of gcc's TM from the TM community (see, for example, the application track papers from transact 2014: <a href="http://transact2014.cse.lehigh.edu" rel="nofollow">http://transact2014.cse.lehigh.edu</a>).</p>

<p>The implementation itself is a bit complex, but that's what it takes to be correct.  There's a lot of literature on the things that can go wrong, especially in a type-unsafe language like C or C++.  GCC gets all of these things right.  Really.</p>

<p>The second way is to use Java.  You can either use DeuceSTM, which is very easy to extend (type safety makes TM implementation much easier!), or use Scala's Akka library for STM.  I prefer Deuce, because it's easier to extend and easier to use (you just annotate a method as @Atomic, and Deuce's java agents do the rest).</p>

<p>The third way is to use Scala.  I've not done much in this space, but researchers seem to love Akka.  If you're affiliated with a parallel/distributed class, you might even be using Scala already.</p>
