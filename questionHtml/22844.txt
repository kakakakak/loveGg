<p>I read lines from file, in one thread of course. Lines was sorted by key.</p>

<p>Then I collect lines with same key (15-20 lines), make parsing, big calculation, etc, and push resulting object to statistic class.</p>

<p>I want to paralell my programm to read in one thread, make parsing and calc in many threads, and join results in one thread to write to stat class.</p>

<p>Is any ready pattern or solution in java7 framework for this problem?</p>

<p>I realize it with executor for multithreading, pushing to blockingQueue, and reading queue in another thread, but i think my code sucks and will produce bugs</p>

<p>Many thanks</p>

<p>upd:</p>

<p>I can't map all file in memory - it's very big</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>You already have the main classes of approaches in mind. CountDownLatch, Thread.join, Executors, Fork/Join. Another option is the Akka framework, which has message passing overheads measured in 1-2 microseconds and is open source. However let me share another approach that often out performs the above approaches and is simpler, this approach is born from working on batch file loads in Java for a number of companies.</p>

<p>Assuming that your goal of splitting the work up is performance, rather than learning. Performance as measured by how long it takes from start to finish. Then it is often difficult to make it faster than memory mapping the file, and processing in a single thread that has been pinned to a single core. It is also gives much simpler code too. A double win.</p>

<p>This may be counter intuitive, however the speed of processing files is nearly always limited by how efficient the file loading is. Not how parallel the processing is. Hence memory mapping the file is a huge win. Once memory mapped we want the algorithm to have low contention with the hardware as it performs the file load. Modern hardware tend to have the IO controller and the memory controller on the same socket as the CPU; which when combined with the prefetchers within the CPU itself lead to a hell of a lot of efficiency when processing the file in a orderly fashion from a single thread. This can be so extreme that going parallel may actually be a lot slower. Pinning a thread to a core usually speeds up memory bound algorithms by a factor of 5. Which is why the memory mapping part is so important.</p>

<p>If you have not already, give it a try. </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Without facts and numbers it is hard to give you advices. So let's start from the beginning: </p>

<ol>
<li>You must identify the bottleneck. Do you really need to perform the computation in parallel or is your job IO bound ? Avoid concurrency if possible, it could be faster.</li>
<li>If computations must be done in parallel you must decide how fine or coarse grained your tasks must be. You need to measure your computations and tasks to be able to size them. Avoid to create too many tasks</li>
<li>You should have a IO thread, several workers, and a "data gatherer" thread. No mutable data.</li>
<li>Be sure to not slow down the IO thread because of task submission. Otherwise you should use more coarse grained tasks or use a better task dispatcher (who said <a href="http://lmax-exchange.github.com/disruptor/" rel="nofollow">disruptor</a> ?)</li>
<li>The "Data gatherer" thread should be the only one to mutate the final state</li>
<li>Avoid unnecessary data copy and object creation. Quite often, when iterating on large files the bottleneck is the GC. Last week, I achieved a 6x speedup replacing a standard scala object by a flyweight pattern. You should also try to pre-allocate everything and use large buffers (page sized).</li>
<li>Avoid disk seeks.</li>
</ol>

<p>Having that said you should be one the right track. You can start with an Executor using properly sized tasks. Tasks write into a data structure, like your blocking queue, shared between workers and the "data gatherer" thread. This threading model is really simple, efficient and hard to get wrong. It is usually efficient enough. If you still require better performances then you must profile your application and understand the bottleneck. Then you can decide the way to go: refine your task size, use faster tools like the disruptor/Akka, improve IO, create fewer objects, tune your code, buy a bigger machine or faster disks, move to Hadoop etc. Pinning each thread to a core (require platform specific code) could also provide a significant boost.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Here is what I would do if asked to split work as you are trying to:</p>

<pre><code>public class App {

    public static class Statistics {
    }

    public static class StatisticsCalculator implements Callable&lt;Statistics&gt; {

        private final List&lt;String&gt; lines;

        public StatisticsCalculator(List&lt;String&gt; lines) {
            this.lines = lines;
        }

        @Override
        public Statistics call() throws Exception {
            //do stuff with lines
            return new Statistics();
        }
    }

    public static void main(String[] args) {
        final File file = new File("path/to/my/file");
        final List&lt;List&lt;String&gt;&gt; partitionedWork = partitionWork(readLines(file), 10);
        final List&lt;Callable&lt;Statistics&gt;&gt; callables = new LinkedList&lt;&gt;();
        for (final List&lt;String&gt; work : partitionedWork) {
            callables.add(new StatisticsCalculator(work));
        }
        final ExecutorService executorService = Executors.newFixedThreadPool(Math.min(partitionedWork.size(), 10));
        final List&lt;Future&lt;Statistics&gt;&gt; futures;
        try {
            futures = executorService.invokeAll(callables);
        } catch (InterruptedException ex) {
            throw new RuntimeException(ex);
        }
        try {
            for (final Future&lt;Statistics&gt; future : futures) {
                final Statistics statistics = future.get();
                //do whatever to aggregate the individual
            }
        } catch (InterruptedException | ExecutionException ex) {
            throw new RuntimeException(ex);
        }
        executorService.shutdown();
        try {
            executorService.awaitTermination(1, TimeUnit.DAYS);
        } catch (InterruptedException ex) {
            throw new RuntimeException(ex);
        }
    }

    static List&lt;String&gt; readLines(final File file) {
        //read lines
        return new ArrayList&lt;&gt;();
    }

    static List&lt;List&lt;String&gt;&gt; partitionWork(final List&lt;String&gt; lines, final int blockSize) {
        //divide up the incoming list into a number of chunks
        final List&lt;List&lt;String&gt;&gt; partitionedWork = new LinkedList&lt;&gt;();
        for (int i = lines.size(); i &gt; 0; i -= blockSize) {
            int start = i &gt; blockSize ? i - blockSize : 0;
            partitionedWork.add(lines.subList(start, i));
        }
        return partitionedWork;
    }
}
</code></pre>

<p>I have create a <code>Statistics</code> object, this holds the result of the work done.</p>

<p>There is a <code>StatisticsCalculator</code> object which is a <code>Callable&lt;Statistics&gt;</code> - this does the calculation. It is given a <code>List&lt;String&gt;</code> and it processes the lines and creates the <code>Statistics</code>.</p>

<p>The <code>readLines</code> method I leave to you to implement.</p>

<p>The most important method in many ways is the <code>partitionWork</code> method, this divides the incoming <code>List&lt;String&gt;</code> which is all the lines in the file into a <code>List&lt;List&lt;String&gt;&gt;</code> using the <code>blockSize</code>.  This essentially decides how much work each thread should have, tuning of the <code>blockSize</code> parameter is very important. As if each work is only one line then the overheads would probably outweight the advantages whereas if each work of ten thousand lines then you only have one working <code>Thread</code>.</p>

<p>Finally the meat of the opertation is the <code>main</code> method. This calls the read and then partition methods. It spawns an <code>ExecutorService</code> with a number of threads equal to the number of bits of work but up to a maximum of 10. You may way to make this equal to the number of cores you have.</p>

<p>The <code>main</code> method then submits a <code>List</code> of all the <code>Callable</code>s, one for each chunk, to the <code>executorService</code>. The <code>invokeAll</code> method blocks until the work is done.</p>

<p>The method now loops over each returned <code>List&lt;Future&gt;</code> and gets the generated <code>Statistics</code> object for each; ready for aggregation.</p>

<p>Afterwards don't forget to shutdown the <code>executorService</code> as it will prevent your application form exiting.</p>

<p><strong>EDIT</strong></p>

<p>OP wants to read line by line so here is a revised <code>main</code></p>

<pre><code> public static void main(String[] args) throws IOException {
    final File file = new File("path/to/my/file");
    final ExecutorService executorService = Executors.newFixedThreadPool(10);
    final List&lt;Future&lt;Statistics&gt;&gt; futures = new LinkedList&lt;&gt;();
    try (final BufferedReader reader = new BufferedReader(new FileReader(file))) {
        List&lt;String&gt; tmp = new LinkedList&lt;&gt;();
        String line = null;
        while ((line = reader.readLine()) != null) {
            tmp.add(line);
            if (tmp.size() == 100) {
                futures.add(executorService.submit(new StatisticsCalculator(tmp)));
                tmp = new LinkedList&lt;&gt;();
            }
        }
        if (!tmp.isEmpty()) {
            futures.add(executorService.submit(new StatisticsCalculator(tmp)));
        }
    }
    try {
        for (final Future&lt;Statistics&gt; future : futures) {
            final Statistics statistics = future.get();
            //do whatever to aggregate the individual
        }
    } catch (InterruptedException | ExecutionException ex) {
        throw new RuntimeException(ex);
    }
    executorService.shutdown();
    try {
        executorService.awaitTermination(1, TimeUnit.DAYS);
    } catch (InterruptedException ex) {
        throw new RuntimeException(ex);
    }
}
</code></pre>

<p>This streams the file line by line and, after a given number of lines fires a new task to process the lines to the executor.</p>

<p>You would need to call <code>clear</code> on the <code>List&lt;String&gt;</code> in the <code>Callable</code> when you are done with it as the <code>Callable</code> instances are references by the <code>Future</code>s they return. If you clear the <code>List</code>s when you're done with them that should reduce the memory footprint considerably.</p>

<p>A further enhancement may well be to use the suggestion <a href="http://stackoverflow.com/questions/2001086/how-to-make-threadpoolexecutors-submit-method-block-if-it-is-saturated">here</a> for a <code>ExecutorService</code> that blocks until there is a spare thread - this will guranatee that there are never more than <code>threads*blocksize</code> lines in memory at a time if you clear the <code>List</code>s when the <code>Callable</code>s are done with them.</p>
