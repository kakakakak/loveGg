    <div class="post-text" itemprop="text">

<p>I have started working with a large dataset that is arriving in JSON format.  Unfortunately, the service providing the data feed delivers a non-trivial number of duplicate records.  On the up-side, each record has a unique Id number stored as a 64 bit positive integer (Java <strong>long</strong>).</p>

<p>The data arrives once a week and is about 10M records in each delivery.  I need to exclude duplicates from within the current delivery as well as records that were in previous batches.</p>

<p>The brute force approach to attacking the de-dup problem is push the Id number into a Java <strong>Set</strong>.  Since the <strong>Set</strong> interface requires uniqueness, a failure during the insert will indicate a duplicate.   </p>

<p>The question is:  Is there a better way to look for a duplicate <strong>long</strong> as I import records?</p>

<p>I am using Hadoop to mine the data, so if there is a good way to use Hadoop to de-dup the records that would be a bonus.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Could you create a MapReduce task where the map output has a key of the unique ID number?  That way, in your reduce task, you will be passed an iterator of all the values with that ID number.  Output only the first value and your reduced output will be free of duplicates.  </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Let me see. Each <code>java.lang.Long</code> takes 24 bytes. Each <code>HashMap$Entry</code> takes 24 bytes as well, and the array for the <code>HashMap</code> takes 4 bytes. So you have 52 * 10M = 512M of heap storage for the map. This is for the 10M records of one week, though.</p>

<p>If you are on a 64-bit system, you can just set the heap size to 5 GB and see how far you get.</p>

<p>There should be other implementations of a <code>java.util.Set</code> that only consume about 16 bytes per entry, so you can handle three times the data as with a <code>java.util.HashSet</code>. I've written one myself, but I cannot share it. You may try GNU Trove instead.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>You have to keep list of unique ids in HDFS and rebuild it after every batch load.</p>

<p>As since the cardinality in your case is quite large (you can expect > 1B unique records in one year) your unique id list needs to be split into multiple parts, say N. The partition algorithm is domain specific. The general approach is to convert ID into long hash string (16 bytes is OK) and creates 2^k buckets:</p>

<p>For k =8, for example:</p>

<p>bucket #1 contains all IDs whose hash value starts with 0
bucket #2 contains all IDs whose hash value starts with 1
...
bucket #256 contains all IDs whose hash value starts with 255 </p>

<p>On every new batch you receive run dedupe job first: Map reads records , takes record ID, hashes it and outputs Key=bucket# (0..255 in our case) and Value = ID. Each reducer receives all IDS for a given bucket. Reducer loads ALL unique IDs for a given bucket known in your system already into internal Set and checks ALL incoming record IDs with this this internal Set. If record has ID which s not known yet you update internal Set and output the record.</p>

<p>On reducer close you output internal Set of unique IDs back to HDFS.</p>

<p>By splitting the whole set of IDs into number of buckets you create solution which scales well. </p>
