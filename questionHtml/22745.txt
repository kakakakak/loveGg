<p>My hadoop job with default configurations -> Local mode on local file system was not using all the cores(16) in my system -> by using all cores I mean, I could see activity in all the cores at various instances.But, My CPU usage [from <code>top</code>] never went beyond 200%, hence I changed these configuration in my <code>conf</code>.</p>

<p><code>mapred.tasktracker.map.tasks.maximum</code>  to set this to 8 and <code>mapred.tasktracker.reduce.tasks.maximum</code> to set this to 8 as well. </p>

<p>Still the usage of my cores stays less than 300%. How do I get the maximum throughput from machine by using all the cores ?</p>

<p>Also, I my files are in the size of ~1TB.This is the sample log from job run. I see that it creates multiple(temp) splits while running the job. This cycle of creating a split  and processing it continues(repeatedly). My hunch is that the during each cycle, it creates a temp file and merges them finally. Is there a way we could increase some buffer size to process bigger chunks (for my large input files) ?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I assume you're using a Pseudo distribution, so your problem (at a guess) is that with so many process slots available to Hadoop, you're now limited by by disk IO - meaning that your Hard Disk (and again i'm assuming you only have a single HDD) can only read so much data in parallel - in this case your processes is said to be IO bound.</p>

<p>How can you use all your available CPU - well you could write a job that is not IO bound and for each input record performs some complex calculation (maybe some matrix multiplication, or the Pi Estimator example - but i'm not sure how CPU intensive that is).</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>From your description it seems to me that you are I/O bound. This simply means that your disks cannot provide enough data to keep all your cpus busy.</p>

<p>Essentially there's two ways to change this</p>

<ol>
<li>You put more io capacity in your system. I.e. put more separate disks in and ensure that the input files are evenly spread over those disks.</li>
<li>Male your CPUs do more with the same data.</li>
</ol>
