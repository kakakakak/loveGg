<p>First of all, thank you for reading my question!</p>

<p>I'm currently studying the replication model of Hadoop but I'm at a dead end. I study from the the book "<strong>Oreilly Hadoop The Definitive Guide 3rd Edition Jan 2012</strong>". To come to the question, I first need to to read the beneath text from the book.</p>

<p>On page 73, there is the following:</p>

<p>"The <strong>DistributedFileSystem</strong> returns an <strong>FSDataOutputStream</strong> for the client The Hadoop Distributed Filesystem to start writing data to. Just as in the read case, <strong>FSDataOutputStream</strong> wraps a <strong>DFSOutput Stream</strong>, which handles communication with the datanodes and namenode. As the client writes data (step 3),</p>

<p><strong>DFSOutputStream</strong> splits it into packets, which it writes to an internal queue, called the data queue. The data queue is consumed by the Data Streamer, whose responsibility it is to ask the namenode to allocate new blocks by picking a list of suitable datanodes to store the replicas."*</p>

<p>As you can see, the <strong>DFSOutputStream</strong> has a data queue of packets. The data queue is being consumed by the DataStreamer who asks the namenode to allocate new blocks. </p>

<p><strong>My questions:</strong>  </p>

<p><strong><em>How does this work?</em></strong></p>

<p><strong><em>How does the Namenode allocate new blocks?</em></strong> </p>

<p>Same question, ask differently: <strong>How does the Namenode create a list of suitable Datanodes?</strong></p>

<p>I can't find anything about this on the internet or in the book. The book explains the process from a high level.</p>

<p>I really appreciate your time helping me, I thank you!</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I might see 2 different questions, since you also mentioned <code>DataStreamer</code>.</p>

<p>You can find both answers in <em>"The Hadoop Distributed File System by Konstantin Shvachko, Hairong Kuang, Sanjay Radia, Robert Chansler"</em></p>

<p>1) How blocks are requested from client (DataStreamer) ?</p>

<blockquote>
  <p>When a client writes, it first asks the NameNode to choose DataNodes
  to host replicas of the first block of the file. The client organizes
  a pipeline from node-to-node and sends the data. When the first block
  is filled, the client requests new DataNodes to be chosen to host
  replicas of the next block. A new pipeline is organized, and the
  client sends the further bytes of the file. Each choice of DataNodes
  is likely to be different.</p>
</blockquote>

<p>2) How does NameNode create a list of suitable datanodes ?</p>

<p>As already answered by other user, Hadoop allows this policy to be configurable, but as default replica placement policy: </p>

<blockquote>
  <p>When a new block is created, HDFS places the first replica on
  the node where the writer is located, the second and the third
  replicas on two different nodes in a different rack, and the rest
  are placed on random nodes with restrictions that no more than
  one replica is placed at one node and no more than two replicas
  are placed in the same rack when the number of replicas is less
  than twice the number of racks.</p>
</blockquote>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Have a look at <a href="https://hadoop.apache.org/docs/r2.7.0/hadoop-project-dist/hadoop-hdfs/HdfsDesign.html" rel="nofollow">Apache HDFS Design</a><a href="http://i.stack.imgur.com/GSUMc.png" rel="nofollow"><img src="http://i.stack.imgur.com/GSUMc.png" alt="enter image description here"></a></p>

<p>For example, when the replication factor is three, HDFS’s placement policy is to put </p>

<pre><code>One replica on one node in the local rack, 

Another on a different node in the local rack,

And the last on a different node in a different rack.
</code></pre>

<p>This policy cuts the inter-rack write traffic which improves write performance. </p>

<p>The chance of rack failure is far less than that of node failure; This policy does not impact data reliability and availability guarantees.</p>

<p>With this policy, the replicas of a file do not evenly distribute across the racks. </p>

<blockquote>
  <p>One third of replicas are on one node</p>
  
  <p>two thirds of replicas are on one rack</p>
  
  <p>And the other third are evenly distributed across the remaining racks.</p>
</blockquote>

<p>This policy improves write performance without compromising data reliability or read performance.</p>
