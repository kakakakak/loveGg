<p>With SparkR, I'm trying for a PoC to collect an RDD that I created from text files which contains around 4M lines.</p>

<p>My Spark cluster is running in Google Cloud, is bdutil deployed and is composed with 1 master and 2 workers with 15gb of RAM and 4 cores each. My HDFS repository is based on Google Storage with gcs-connector 1.4.0.

<p>Here is the script I use :</p>

<pre><code>Sys.setenv("SPARK_MEM" = "1g")

test &lt;- collect(lines)
</code></pre>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>This does appear to be a simple combination of Java in-memory object representations being inefficient combined with some apparent long-lived object references which cause some collections to fail to be garbage-collected in time for the new collect() call to overwrite the old one in-place.</p>

<p>I experimented with some options, and for my sample 256MB file that contains ~4M lines, I indeed reproduce your behavior where collect is fine the first time, but OOMs the second time, when using <code>SPARK_MEM=1g</code>. I then set <code>SPARK_MEM=4g</code> instead, and then I'm able to ctrl+c and re-run <code>test &lt;- collect(lines)</code> as many times as I want.</p>

<p>For one thing, even if references didn't leak, note that after the first time you ran <code>test &lt;- collect(lines)</code>, the variable <code>test</code> is holding that gigantic array of lines, and the second time you call it, the <code>collect(lines)</code> executes <em>before</em> finally being assigned to the <code>test</code> variable and thus in any straightforward instruction-ordering, there's no way to garbage-collect the old contents of <code>test</code>. This means the second run will make the SparkRBackend process hold two copies of the entire collection at the same time, leading to the OOM you saw.</p>

<p>To diagnose, on the master I started SparkR and first ran</p>

<pre><code>dhuo@dhuo-sparkr-m:~$ jps | grep SparkRBackend
8709 SparkRBackend
</code></pre>

<p>I also checked <code>top</code> and it was using around 22MB of memory. I fetched a heap profile with <code>jmap</code>:</p>

<pre><code>jmap -heap:format=b 8709
mv heap.bin heap0.bin
</code></pre>

<p>Then I ran the first round of <code>test &lt;- collect(lines)</code> at which point running <code>top</code> showed it using ~1.7g of RES memory. I grabbed another heap dump. Finally, I also tried <code>test &lt;- {}</code> to get rid of references to allow garbage-collection. After doing this, and printing out <code>test</code> and showing it to be empty, I grabbed another heap dump and noticed RES still showed 1.7g. I used <code>jhat heap0.bin</code> to analyze the original heap dump, and got:</p>

<pre><code>Heap Histogram

All Classes (excluding platform)

Class   Instance Count  Total Size
class [B    25126   14174163
class [C    19183   1576884
class [&lt;other&gt;  11841   1067424
class [Lscala.concurrent.forkjoin.ForkJoinTask; 16  1048832
class [I    1524    769384
...
</code></pre>

<p>After running collect, I had:</p>

<pre><code>Heap Histogram

All Classes (excluding platform)

Class   Instance Count  Total Size
class [C    2784858 579458804
class [B    27768   70519801
class java.lang.String  2782732 44523712
class [Ljava.lang.Object;   2567    22380840
class [I    1538    8460152
class [Lscala.concurrent.forkjoin.ForkJoinTask; 27  1769904
</code></pre>

<p>Even after I nulled out <code>test</code>, it remained about the same. This shows us 2784858 instances of char[], for a total size of 579MB, and also 2782732 instances of String, presumably holding those char[]'s above it. I followed the reference graph all the way up, and got something like</p>

<p>char[] -> String -> String[] -> ... -> class scala.collection.mutable.DefaultEntry -> class [Lscala.collection.mutable.HashEntry; -> class scala.collection.mutable.HashMap -> class edu.berkeley.cs.amplab.sparkr.JVMObjectTracker$ -> java.util.Vector@0x785b48cd8 (36 bytes) -> sun.misc.Launcher$AppClassLoader@0x7855c31a8 (138 bytes)</p>

<p>And then AppClassLoader had something like thousands of inbound references. So somewhere along that chain something should've been removing their reference but failing to do so, causing the entire collected array to sit in memory while we try to fetch a second copy of it.</p>

<p>Finally, to answer your question about hanging after the <code>collect</code>, it appears it has to do with the data not fitting in the R process's memory; here's a thread related to that issue: <a href="https://www.mail-archive.com/user@spark.apache.org/msg29155.html" rel="nofollow">https://www.mail-archive.com/user@spark.apache.org/msg29155.html</a></p>

<p>I confirmed that using a smaller file with only a handful of lines, and then running <code>collect</code> indeed does not hang.</p>
