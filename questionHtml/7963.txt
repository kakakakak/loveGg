<td class="answercell">
    <div class="post-text" itemprop="text">
<p><a href="http://lucene.472066.n3.nabble.com/Does-Hadoop-compress-files-tp695145p695915.html">There</a> is no implicit compression in HDFS. In other words, if you want your data to be compressed, you have to write it that way. If you plan on writing map reduce jobs to process the compressed data, you'll want to use a splittable compression format.</p>

<p>Hadoop can process compressed files and here is a nice <a href="http://www.cloudera.com/blog/2009/11/hadoop-at-twitter-part-1-splittable-lzo-compression/">article</a> on it. Also, the intermediate and the final MR output can be <a href="http://hadoop.apache.org/common/docs/current/mapred_tutorial.html#Data+Compression">compressed</a>.</p>

<p>There is a <a href="https://issues.apache.org/jira/browse/HDFS-2115">JIRA</a> on 'Transparent compression in HDFS', but I don't see much progress on it.</p>

<p>I don't think there is a separate API for encryption, though you can you use a compression codec for encryption/decryption also. <a href="http://www.mail-archive.com/common-user@hadoop.apache.org/msg06222.html">Here</a> are more details about encryption and HDFS.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I very recently set compression up on a cluster.  The other posts have helpful links, but the actual code you will want to get LZO compression working is here: <a href="https://github.com/kevinweil/hadoop-lzo" rel="nofollow">https://github.com/kevinweil/hadoop-lzo</a>.</p>

<p>You can, out of the box, use GZIP compression, BZIP2 compression, and Unix Compress.  Just upload a file in one of those formats.  When using the file as an input to a job, you will need to specify that the file is compressed as well as the proper CODEC.  Here is an example for LZO compression.</p>

<pre><code>  -jobconf mapred.output.compress=true
  -jobconf mapred.output.compression.codec=com.hadoop.compression.lzo.LzopCodec
</code></pre>

<p>Why am I going on an on about LZO compression?  The <a href="http://www.cloudera.com/blog/2009/11/hadoop-at-twitter-part-1-splittable-lzo-compression/" rel="nofollow">cloudera article</a> reference by Praveen goes into this.  LZO compression is a splittable compression (unlike GZIP, for example).  This means that a single file can be split into chunks to be handed off to a mapper.  Without a splittable compressed file, a single mapper will receive the entire file.  This may cause you to have too few mappers and to move too much data around your network.</p>

<p>BZIP2 is also splittable.  It also has higher compression than LZO.  However, it is very slow.  LZO has a worse compression ratio than GZIP.  However it is optimized to be <strong><a href="http://aliver.wordpress.com/2010/06/22/huge-unix-file-compresser-shootout-with-tons-of-datagraphs/" rel="nofollow">extremely fast</a></strong>.  In fact, it can even increase the performance of your job by minimizing disk I/O.</p>

<p>It takes a bit of work to set up, and is a bit of a pain to use, but it is worth it (transparent encryption would be awesome).  Once again, the steps are:</p>

<ol>
<li>Install LZO and LZOP (command-line utility)</li>
<li>Install <a href="https://github.com/kevinweil/hadoop-lzo" rel="nofollow">hadoop-lzo</a></li>
<li>Upload a file compressed with LZOP.</li>
<li>Index the file as described by hadoop-lzo wiki (the index allows it to be split).</li>
<li>Run your job (with the proper parameters mapred.output.compress and mapred.output.compression.code)</li>
</ol>
