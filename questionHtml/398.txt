<td class="answercell">
    <div class="post-text" itemprop="text">
<p>There are many techniques that Google uses to achieve the amount of throughput it delivers. <a href="http://en.wikipedia.org/wiki/Map_reduce" rel="nofollow">MapReduce</a>, <a href="http://en.wikipedia.org/wiki/Google_filesystem" rel="nofollow">Google File System</a>, <a href="http://en.wikipedia.org/wiki/BigTable" rel="nofollow">BigTable</a> are a few of those.</p>

<p>There are a few very good Free &amp; Open Source alternatives to these, namely <a href="http://en.wikipedia.org/wiki/Hadoop" rel="nofollow">Apache Hadoop</a>, <a href="http://hadoop.apache.org/hbase/" rel="nofollow">Apache HBase</a> and <a href="http://www.hypertable.org/" rel="nofollow">Hypertable</a>. Yahoo! is using and promoting the Hadoop projects quite a lot and thus they are quite actively maintained.</p>

<td class="answercell">

<p>Awhile ago Google decided to put everything into RAM. </p>

<p><a href="http://googlesystem.blogspot.com/2009/02/machines-search-results-google-query.html" rel="nofollow">http://googlesystem.blogspot.com/2009/02/machines-search-results-google-query.html</a></p>

<p>If you never have to query the hard drive, your results will improve significantly. Caching helps because you don't query the hard drive as much, but you still do when there is a cache miss (Unless you mean caching with PHP, which means you only compile the PHP program when the source has been modified).</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>It really depends on what you are trying to do, but here are some examples:</p>

<ul>
<li><p>Analyze your queries with explain.  In your dev environment you can output your queries and execution time to the bottom of the page - reduce the number of queries and/or optimize those that are slow.</p></li>
<li><p>Use a caching layer.  Looks like Zend can be memcache enabled.  This can potentially greatly speed up your application by sending requests to the ultra-fast caching layer instead of the db.</p></li>
<li><p>Look at your front-end loading time.  Use Yahoo's YSlow add-on to Firebug.  Limit http requests, set far-future headers to cache js, css and images.  Etc.</p></li>
</ul>

<p>You can get lightning speeds on your web app, probably not as fast as google, if you optimize each layer of your application.  Your db connect times are probably not the slowest part of your app.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>PHP scripts by default are interpreted every time they are called by the http server, so every call initiates script parsing and probably compilation by the Zend Engine.
You can get rid of this bottleneck by using script caching, like APC. It keeps the once compiled PHP script in memory/on disk and uses it for all subsequent requests. Gains are often significant, especially in PHP apps created with sophisticated frameworks like ZF.</p>

<p>Every request by default opens up a connection to the database, so you should use some kind of database connection pooling or persistent connections (which don't always work, depending on http server/php configuration). I have never tried, but maybe there's a way to use memcache to keep database connection handles.</p>

<p>You could also use memcache for keeping session data, if they're used on every request. Their persistence is not that important and memcache helps make it very fast.</p>

<p>The actual "problem" is that PHP works a bit different than other frameworks, because it works in a SSI (server-side includes) way - every request is handled by http server and if it requires running a PHP script, its interpreter is initialized and scripts loaded, parsed, compiled and run. This can be compared to getting into the car, starting the engine and going for 10 meters.</p>

<p>The other way is, let's say, an application-server way, in which the web application itself is handling the requests in its own loop, always sharing database connections and not initializing the runtime over and over. This solution gives much lower latency. This on the other hand can be compared to already being in a running car and using it to drive the same 10 meters. ;)</p>

<p>The above caching/precompiling and pooling solutions are the best in reducing the init overhead. PHP/MySQL is still a RDBMS-based solution though, and there's a good reason why BigTable is, well, just a big, sharded, massively distributed hashtable (a bit of oversimplification, I know) - read up on <a href="http://highscalability.com/" rel="nofollow">High Scalability</a>.</p>
