    <div class="post-text" itemprop="text">

<p>We have started using a third party platform (GigaSpaces) that helps us with distributed computing. One of the major problems we are trying to solve now is how to manage our log files in this distributed environment. We have the following setup currently.</p>

<p>Our platform is distributed over 8 machines. On each machine we have 12-15 processes that log to separate log files using java.util.logging. On top of this platform we have our own applications that use log4j and log to separate files. We also redirect stdout to a separate file to catch thread dumps and similar.</p>

<p>This results in about 200 different log files.</p>

<p>As of now we have no tooling to assist in managing these files. In the following cases this causes us serious headaches.</p>

<ul>
<li><p>Troubleshooting when we do not beforehand know in which process the problem occurred. In this case we currently log into each machine using ssh and start using <code>grep</code>.</p></li>
<li><p>Trying to be proactive by regularly checking the logs for anything out of the ordinary. In this case we also currently log in to all machines and look at different logs using <code>less</code> and <code>tail</code>.</p></li>
<li><p>Setting up alerts. We are looking to setup alerts on events over a threshold. This is looking to be a pain with 200 log files to check.</p></li>
</ul>

<p>Today we have only about 5 log events per second, but that will increase as we migrate more and more code to the new platform.</p>

<p>I would like to ask the community the following questions.</p>

<ul>
<li>How have you handled similar cases with many log files distributed over several machines logged through different frameworks?</li>
<li>Why did you choose that particular solution?</li>
<li>How did your solutions work out? What did you find good and what did you find bad?</li>
</ul>

<p>Many thanks.</p>

<p><strong>Update</strong></p>

<p>We ended up evaluating a trial version of Splunk. We are very happy with how it works and have decided to purchase it. Easy to set up, fast searches and a ton of features for the technically inclined. I can recommend anyone in similar situations to check it out.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I would recommend to pipe all your java logging to <a href="http://www.slf4j.org/" rel="nofollow">Simple Logging Facade for Java</a> (SLF4J) and then redirect all logs from SLF4J to <a href="http://logback.qos.ch/" rel="nofollow">LogBack</a>. SLF4J has special support for handling all popular legacy APIs (log4j, commons-logging, java.util.logging, etc), see <a href="http://www.slf4j.org/legacy.html" rel="nofollow">here</a>.</p>

<p>Once you have your logs in LogBack you can use one of it's many appenders to <strong>aggregate</strong> logs over several machines, for details, see the manual <a href="http://logback.qos.ch/manual/appenders.html" rel="nofollow">section about appenders</a>. Socket, JMS and SMTP seem to be the most obvious candidates.</p>

<p>LogBack also has built-in support for monitoring for special conditions in log files and <strong>filtering events</strong> sent to particular appender. So you could set up SMTP appender to send you an e-mail every time there is an ERROR level event in logs.</p>

<p>Finally, to ease <strong>troubleshooting</strong>, be sure to add some sort of <strong>requestID</strong> to all your incoming "requests", see my answer to <a href="http://stackoverflow.com/q/618844/74694">this question</a> for details.</p>

<p><strong>EDIT</strong>: you could also implement your own <strong>custom LogBack appender</strong> and redirect all logs to <a href="http://github.com/facebook/scribe" rel="nofollow">Scribe</a>.</p>
