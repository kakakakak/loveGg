<p>I use Cython to wrap C++ code and expose it to Python for interactive work. My problem is that I need to read large graphs (several gigabytes) from file and they end up twice in the memory. Can anyone help me diagnose and solve this problem? </p>

<p>My Cython wrapper for the graph class looks like this:</p>

<pre><code>cdef extern from "../src/graph/Graph.h":
    cdef cppclass _Graph "Graph":
        _Graph() except +
        _Graph(count) except +
        count numberOfNodes() except +
        count numberOfEdges() except +


cdef class Graph:
    """An undirected, optionally weighted graph"""
    cdef _Graph _this

    def __cinit__(self, n=None):
        if n is not None:
            self._this = _Graph(n)

    # any _thisect which appears as a return type needs to implement setThis
    cdef setThis(self, _Graph other):
        #del self._this
        self._this = other
        return self

    def numberOfNodes(self):
        return self._this.numberOfNodes()

    def numberOfEdges(self):
        return self._this.numberOfEdges()
</code></pre>

<p>If a Python Graph needs to be returned, it needs to be created empty and then the <code>setThis</code> method is used to set the native <code>_Graph</code> instance. This happens, for example, when a <code>Graph</code> is read from file. This is the job of this class:</p>

<pre><code>cdef extern from "../src/io/METISGraphReader.h":
    cdef cppclass _METISGraphReader "METISGraphReader":
        _METISGraphReader() except +
        _Graph read(string path) except +

cdef class METISGraphReader:
    """ Reads the METIS adjacency file format [1]
        [1]: http://people.sc.fsu.edu/~jburkardt/data/metis_graph/metis_graph.html
    """
    cdef _METISGraphReader _this

    def read(self, path):
        pathbytes = path.encode("utf-8") # string needs to be converted to bytes, which are coerced to std::string
        return Graph(0).setThis(self._this.read(pathbytes))
</code></pre>

<p>Interactive usage looks like this:</p>

<pre><code> &gt;&gt;&gt; G = graphio.METISGraphReader().read("giant.metis.graph")
</code></pre>

<p>After the reading from file is done and X GB memory are used, there is a phase where obviously copying happens, and after that 2X GB memory are used. The entire memory is freed when <code>del G</code> is called.</p>

<p>Where is my error which leads to the graph being copied and existing twice in memory?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I don't have a definitive answer for you, but I have a theory.</p>

<p>The Cython wrappers that you wrote are unusual, in that they wrap the C++ object directly instead of a pointer to it.</p>

<p>The following code is particularly inefficient:</p>

<pre><code>cdef setThis(self, _Graph other):
    self._this = other
    return self 
</code></pre>

<p>The reason is that your <code>_Graph</code> class contains several STL vectors, and those will have to be copied over. So, when your <code>other</code> object is assigned to <code>self._this</code> the memory usage is effectively doubled (or worse, since the STL allocators can overallocate for performance reasons).</p>

<p>I wrote a simple test that matches yours and added logging everywhere to see how objects are created, copied or destroyed. I can't find any issues there. The copies do happen, but after the assignment is complete I see that only one object remains.</p>

<p>So my theory is that the extra memory that you see is related to STL allocator logic in the vectors. All that extra memory must be attached to the final object after the copies.</p>

<p>My recommendation is that you switch to the more standard pointer based wrapping. Your <code>_Graph</code> wrapper then should be defined more or less as follows:</p>

<pre><code>cdef class Graph:
    """An undirected, optionally weighted graph"""
    cdef _Graph* _this

    def __cinit__(self, n=None):
        if n is not None:
            self._this = new _Graph(n)
        else:
            self._this = 0

    cdef setThis(self, _Graph* other):
        del self._this
        self._this = other
        return self

    def __dealloc__(self):
        del self._this
</code></pre>

<p>Note that I need to delete <code>_this</code> because it is a pointer.</p>

<p>You will then need to modify your <code>METISGraphReader::read()</code> method to return a heap allocated <code>Graph</code>. The prototype of this method should be changed to:</p>

<pre><code>Graph* METISGraphReader::read(std::string path);
</code></pre>

<p>Then the Cython wrapper for it can be written as:</p>

<pre><code>    def read(self, path):
        pathbytes = path.encode("utf-8") # string needs to be converted to bytes, which are coerced to std::string
        return Graph().setThis(self._this.read(pathbytes))
</code></pre>

<p>If you do it this way there is only one object, the one that is created on the heap by <code>read()</code>. A pointer to that object is returned to the <code>read()</code> Cython wrapper, which then installs it in a brand new <code>Graph()</code> instance. The only thing that gets copied is the 4 or 8 bytes of the pointer.</p>

<p>I hope this helps!</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>You'll need to modify the C++ class to store it's data via a shared_ptr.
Make sure you have a proper copy constructor and assignment operator:</p>

<pre><code>#define  _CRT_SECURE_NO_WARNINGS
#include &lt;stdio.h&gt;
#include &lt;memory&gt;

struct Data { // your graph data
    Data(const char* _d = NULL) {
        if (_d)
            strncpy(d, _d, sizeof(d)-1);
        else
            memset(d, 0, sizeof(d));
    }
    Data(const Data&amp; rhs) {
        memcpy(d, rhs.d, sizeof(d));
    }
    ~Data() {
        memset(d, 0, sizeof(d));
    }
    void DoSomething() { /* do something */ } // a public method that was used in Python

    char d[1024];
};

class A { // the wrapper class
public:
    A() {}
    A(const char* name)  : pData(new Data(name)) {}
    A(const A&amp; rhs) : pData(rhs.pData) {}
    A&amp; operator=(const A&amp; rhs) {
        pData = rhs.pData;
        return *this;
    }
    ~A() {}
    // interface with Data
    void DoSomething() {
        if (pData.get() != NULL)
            pData-&gt;DoSomething();
    }

private:
    std::shared_ptr&lt;Data&gt; pData;
};

int main(int argc, char** argv)
{
    A o1("Hello!");
    A o2(o1);
    A o3;
    o3 = o2;
    return 0;
}
</code></pre>

<p>If your constraint/goal is <strong>“Compute on graphs with billions of edges, in a reasonable time, on a single PC.”</strong>, consider refactoring to leverage <a href="http://strata.oreilly.com/2012/12/graphchi-graph-analytics-over-billions-of-edges-using-your-laptop.html" rel="nofollow">GraphChi</a>.</p>

<p>If single-machine/in-memory is not a constraint, consider leveraging a graph database like <a href="http://www.neo4j.org/" rel="nofollow">Neo4j</a> instead of pulling all data into memory.  There are also graph APIs that overlay with Hadoop (e.g. <a href="https://www.youtube.com/watch?v=b5Qmz4zPj-M" rel="nofollow">Apache Giraph</a>).</p>
