<p>UPDATE: note there are many pending tasks, so this is not a case of a node being idle and therefore hosting tasks that read from other nodes.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p><strong>Hadoop's default (FIFO) scheduler works like this</strong>: When a node has spare capacity, it contacts the master and asks for more work. The master tries to assign a data-local task, or a rack-local task, but if it can't, it will assign any task in the queue (of waiting tasks) to that node. However, while this node was being assigned this non-local task (we'll call it task X), it is possible that another node also had spare capacity and contacted the master asking for work. Even if this node actually had a local copy of the data required by X, it will not be assigned that task because the other node was able to acquire the lock to the master slightly faster than the latter node. This results in <strong>poor data locality, but FAST task assignment</strong>.</p>

<p>In contrast, the <strong>Fair Scheduler</strong> uses a technique called <a href="http://www.cs.berkeley.edu/~matei/papers/2010/eurosys_delay_scheduling.pdf" rel="nofollow">delayed scheduling</a> that achieves higher locality by delaying non-local task assignment for a "little bit" (configurable). It <strong>achieves higher locality but at a small cost of delaying some tasks</strong>.</p>

<p>Other people are working on better schedulers, and this may likely be improved in the future. For now, you can choose to use the Fair Scheduler if you wish to achieve higher data locality.</p>

<p>I disagree with @donald-miner's conclusion that "With a default replication factor of 3, you don't see very many tasks that are not data local." He is correct in noting that <strong>more replicas will give improve your locality %, but the percentage of data-local tasks may still be very low</strong>. I've also ran experiments myself and saw very low data locality with the FIFO scheduler. You could achieve high locality if your job is large (has many tasks), but for the more common, smaller jobs, they suffer from a problem called "head-of-line scheduling". Quoting from <a href="http://www.cs.berkeley.edu/~matei/papers/2010/eurosys_delay_scheduling.pdf" rel="nofollow">this paper</a>:</p>

<blockquote>
  <p>The ﬁrst locality problem occurs in small jobs (jobs that
  have small input ﬁles and hence have a small number of data
  blocks to read). The problem is that whenever a job reaches
  the head of the sorted list [...] (i.e. has the fewest
  running tasks), one of its tasks is launched on the next slot
  that becomes free, no matter which node this slot is on. If
  the head-of-line job is small, it is unlikely to have data on
  the node that is given to it. For example, a job with data on
  10% of nodes will only achieve 10% locality.</p>
</blockquote>

<p>That paper goes on to cite numbers from a production cluster at Facebook, and they reported observing just 5% of data locality in a large, production environment.</p>

<p>Final note: Should you care if you have low data locality? Not too much. <strong>The running time of your jobs may be dominated by the <a href="http://www.cs.berkeley.edu/~matei/papers/2008/osdi_late.pdf" rel="nofollow">stragglers</a> (tasks that take longer to complete) and <a href="http://yanpeichen.com/professional/MapReduceWorkloadMASCOTSPublished.pdf" rel="nofollow">shuffle phase</a>, so improving data locality would only have a very modest improve in running time (if any at all)</strong>.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Unfortunately, the default scheduler isn't that smart. I'm not sure exactly what's going on, but I think it's using some sort of greedy-style scheduling where it tries to schedule what it can now for the next task, and then moves on. There could definitely be improvements made to the hadoop scheduler and <a href="http://scholar.google.com/scholar?hl=en&amp;q=hadoop%20scheduler&amp;btnG=&amp;as_sdt=1,21&amp;as_sdtp=" rel="nofollow">there have been a few academic attempts and making hadoop scheduling more optimal</a>. </p>

<p><a href="http://grids.ucs.indiana.edu/ptliupages/publications/InvestigationDataLocalityInMapReduce_CCGrid12_Submitted.pdf" rel="nofollow">This research paper</a> shows that the default hadoop scheduler is not optimal. In the results, they show that increasing the replication factor to three improves data locality significantly, with diminishing returns after that.</p>

<p>So, why hasn't the default scheduler been improved? Here is my opinion/theory: With a default replication factor of 3, you don't see very many tasks that are not data local. By having more replicas, you give the schedule more flexibility to fit tasks in the right spots. Basically, it's a coincidence that you have 3 replicas, and the default scheduler takes advantage of that by being implemented in a lazy manner. Since you typically have 3 replicas for redundancy sake already... there isn't much motivation to help scheduler performance for people with a replication of 1.</p>

<p>If you have the space, I suggest just upping the replication factor to two or three. There really isn't much downside.</p>
