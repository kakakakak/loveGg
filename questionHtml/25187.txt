    <div class="post-text" itemprop="text">

<p>At work, we're building a distributed application (possibly across several machines on a LAN, possibly later across several continents on a WAN+VPN). We don't want log files local to each machine (filling up its disk and impossible to view in aggregate) so we need to centralize logging over the network. Most logs won't be important, so UDP is fine for them, but some are money-losing important alerts and must be reliably delivered, implying TCP. We're worried about congesting the network if the logging protocol is too chatty, or dragging the apps to a crawl if it isn't responsive.</p>

<p>Some possibilities I've considered are:</p>

<ul>
<li><strong>syslog</strong> (it seems perfect, but my boss has an animus against this so I may not be able to choose it).</li>
<li><strong>scribe</strong> from facebook (but it seems a bit heavyweight with a server on every machine - not every log message needs ultra-reliability).</li>
<li>using a message queue like <strong>rabbitmq</strong> which can have multiple queues tuned to different levels of transaction safety.</li>
<li>worst case, I can write my own from scratch.</li>
</ul>

<p>Do you have other suggestions? What centralized logging solutions have you used, and how well did they work out?</p>

<p><strong>Edit:</strong> I was leaning towards scribe, because its store-and-forward design decouples the running app from network latency. But after struggling to install it, I found that (1) it's not available as a binary package - nowadays that's unforgivable - and (2) it depends intimately on a library (thrift) that isn't available as a binary package either! And worst of all, it wouldn't even compile properly. That is not release quality code, even in open source.</p>

<p>We have successfully used <a href="http://www.zeromq.org" rel="nofollow">ZeroMQ</a> for the logs of a distributed application scenario like the yours. It's very reliable and incredibly fast. We moved to ZeroMQ after a not-so-successful implementation with <a href="http://www.spread.org" rel="nofollow">Spread</a>. In our setup a single ZeroMQ server is able to handle more than 70 different logs from a medium to high busy distributed application. It receives data from LAN and via the Internet.</p>

<p>If you need a detailed queue server comparison, <a href="http://wiki.secondlife.com/wiki/Message_Queue_Evaluation_Notes" rel="nofollow">look at this page from the Second Life wiki</a>.</p>

<p>Hope it helps!</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>There're several alternatives recently. Notably, Scribe is not maintained any more. Facebook developed its successor called Caligraphus, and it's not open-sourced. Here's a list of alternatives.</p>

<ul>
<li>syslog: installed at all Linux distributions</li>
<li><a href="http://fluentd.org/" rel="nofollow">Fluentd</a>: C+Ruby-based lightweight logger, which handles logs as JSON stream</li>
<li><a href="http://github.com/cloudera/flume/" rel="nofollow">Flume</a>: developed at Cloudera, written in Java and works well with Hadoop ecosystems</li>
<li><a href="http://kafka.apache.org" rel="nofollow">Apache Kafka</a>: developed at LinkedIn, pull-based architecture</li>
<li>Scribe: open-sourced by Facebook, but not maintained anymore</li>
</ul>

<p>Disclaimer: I'm a committer of Fluentd project.</p>
