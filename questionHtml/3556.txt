<p>Apache Spark [<a href="http://spark.apache.org/]" rel="nofollow">http://spark.apache.org/]</a> claims to be 100x faster than Apache Hadoop in memory. How does it achieve this phenomenal speedup? It this speedup only applicable for iterative machine learning algorithms or also for ETL (extract-transform-load) tasks like JOINs and GROUPBYs? Can Spark's RDDs (Resilient Distributed Datasets) and DataFrames both provide this kind of a speedup? Are there any benchmark test results obtained by the Spark community for some of the scenarios described above?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<ol>
<li>Spark does data processing in-memory. </li>
<li>There will not be intermediary files as in Map Reduce, so there is no I/O or negligible. </li>
<li>It does not run 100x faster in all the scenarios, especially when it involves joins and sorting.</li>
<li>As it is memory intensive, it can saturate cluster quickly. You might be able to run one job 100x faster at a given point in time, but will not be able to run as many jobs/applications you can run using traditional hadoop approach.</li>
<li>RDDs and Data Frames are the internal data structures which come handy for processing the data. RDDs are in-memory data structures for the data and data frames are primarily metadata of those RDDs. They are more of representation of data in spark.</li>
</ol>

<p>Problem with most of these claims are not bench marked against true production use cases. There might be volume, but not quality in the data which can represent actual business applications. Spark can be very handy for streaming analytics, where you want to understand data in near real time. But for true batch processing Hadoop can be better solution, especially on commodity hardware.</p>

<p>Spark is faster than Hadoop due to in-memory processing. But there are some twisted facts about the numbers. </p>

<p>Still Spark has to rely on HDFS for some use cases. </p>

<p>Have a look at this <a href="http://www.slideshare.net/EdurekaIN/5-reasons-why-spark-is-in-demand?qid=df08873f-ae71-4d97-a9f4-5bc9601a40b4&amp;v=default&amp;b=&amp;from_search=2" rel="nofollow">slide</a> and especially <strong><em>slide no: 6</em></strong> and this <a href="https://databricks.com/blog/2014/10/10/spark-petabyte-sort.html" rel="nofollow">benchmarking article</a></p>

<p>Have a look at complete presentation.</p>

<p>Spark is faster for real time analytics due to in-memory processing. Hadoop is good for batch processing. If you are not worried about the latency of job, still you can use Hadoop.</p>

<p><strong><em>But one thing is sure. Spark and Hadoop have to co-exist. Neither of them can replace other.</em></strong></p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<ul>
<li><p><strong>Apache Spark</strong> processes data in-memory while Hadoop <strong>MapReduce</strong> persists back to the disk after a map or reduce action. But Spark needs a lot of memory</p></li>
<li><p><strong>Spark</strong> loads a process into memory and keeps it there until further notice, for the sake of caching.</p></li>
<li><p><strong>Resilient Distributed Dataset (RDD)</strong>, which allows you to transparently store data on memory and persist it to disc if it's needed.</p></li>
<li><p>Since <strong>Spark</strong> uses in-memory, there's no synchronisation barrier that's slowing you down. This is a major reason for Spark's performance.</p></li>
<li><p>Rather than just processing a batch of stored data, as is the case with MapReduce, Spark can also manipulate data in real time using <strong>Spark Streaming</strong>.</p></li>
<li><p>The <strong>DataFrames</strong> API was inspired by data frames in R and Python (Pandas), but designed from the ground-up to as an extension to the existing <strong>RDD API</strong>.</p></li>
<li><p>A <strong>DataFrame</strong> is a distributed collection of data organized into named columns, but with <strong>richer optimizations</strong> under the hood that supports to the speed of spark.</p></li>
<li><p>Using <strong>RDD</strong>s Spark simplifies complex operations like <strong>join</strong> and <strong>groupBy</strong> and in the backend, you’re dealing with fragmented data. That fragmentation is what enables <strong>Spark</strong> to execute in parallel.</p></li>
<li><p><strong>Spark</strong> allows to develop complex, multi-step data pipelines using directed acyclic graph (<strong>DAG</strong>) pattern. It supports in-memory data sharing across DAGs, so that different jobs can work with the same data. <strong>DAG</strong>s are a major part of <strong>Spark</strong>s speed.</p></li>
<li><p><strong>Spark</strong> code base is much smaller.
<a href="http://i.stack.imgur.com/R893h.jpg" rel="nofollow"><img src="http://i.stack.imgur.com/R893h.jpg" alt="enter image description here"></a></p></li>
</ul>

<p>Hope this helps.</p>
