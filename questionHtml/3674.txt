<p>My MR job ended at map 100% reduce 35% with lots of error messages similar to <code>running beyond physical memory limits. Current usage: 3.0 GB of 3 GB physical memory used; 3.7 GB of 15 GB virtual memory used. Killing container.</code></p>

<p>My input <code>*.bz2</code> file is about 4GB, if I uncompress it, the size of it will be about 38GB, it took about one hour to run this job with <code>one Master</code> and <code>two slavers</code> on the Amazon EMR.</p>

<p>My questions are<br>
- Why this job used so much memory?<br>
- Why this job took about one hour? <a href="http://blog.cloudera.com/blog/2009/12/7-tips-for-improving-mapreduce-performance/" rel="nofollow">Usually running a 40GB wordcount job on a small 4-node cluster takes about 10 mins</a>.<br>
- How to tune the MR parameters to solve this problem?<br>
- Which <a href="https://aws.amazon.com/ec2/instance-types/" rel="nofollow">Amazon EC2 Instance types</a> are the good fit to solve this problem?    </p>

<p>Please refer to the following log:<br>
- Physical memory (bytes) snapshot=43327889408  => <code>43.3GB</code><br>
- Virtual memory (bytes) snapshot=108950675456  => <code>108.95GB</code><br>
- Total committed heap usage (bytes)=34940649472 => <code>34.94GB</code> </p>

<p><strong>My proposed solutions are as follows, but I'm not sure if they are correct solutions or not</strong><br>
- use larger Amazon EC2 Instance which is at least 8GB in memory<br>
- tune the MR parameters using the following codes  </p>

<p>Version 1:</p>

<pre><code>Configuration conf = new Configuration();
Job job = Job.getInstance(conf, "jobtest1");

conf.setBoolean("yarn.nodemanager.vmem-check-enabled", false);
</code></pre>

Job job = Job.getInstance(conf, "jobtest2");

conf.set("mapreduce.reduce.memory.mb", "8192");                                         
conf.set("mapreduce.reduce.java.opts", "-Xmx6144m");                                             
</code></pre>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I am not sure of Amazon EMR. So few points to consider regarding map reduce:</p>

<ol>
<li><p>bzip2 is slower, although it compresses better than gzip. bzip2’s decompression speed is faster than its compression speed, but it is still slower than the other formats. So at a high level, you already have this compared to 40gb word count program which ran in ten minutes.(assuming that 40gb program don't have compression). Next question is, BUT HOW MUCH SLOWER</p></li>
<li><p>However, your job is still failing after one hour. Please confirm this. So only when the job runs successfully, can we thing of performance. For this reason, lets think of why is it failing. 
You were getting memory error. Also based on error, a container is failed during the reducer phase(as mapper phase is completed 100%). Mostly not even one reducer might have succeeded. Even though 32% might trick you to think that some reducers ran, that % could be due to preparing clean up work before first reducer runs. One way to confirm is, see if you have got any reducer output file generated.</p></li>
</ol>

<p>Once confirming that, none of the reducer ran, you can increase the memory for containers as per your version 2.</p>

<p>Your version 1 will help you to see if only a specific container is causing issue and allowing the job to complete. </p>

<p>Your input file size should conclude the number of reducers. Standard is 1 Reducer per 1 GB unless you are compressing the Mapper output data. So in this case ideal number should have been at least 38. Try passing the command line option as -D mapred.reduce.tasks=40 and see if there is any change.</p>
