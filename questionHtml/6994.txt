    <div class="post-text" itemprop="text">

<p>I am managing a Hadoop cluster that is shared between a number of users.  We frequently run jobs with extremely slow mappers.  For example, we might have a 32 GB file of sentences (one sentence per line) that we want to NLP parse (which takes say 100 ms per sentence).  If the block size is 128 MB, this is 250 mappers.  This fills our rather small cluster (9 nodes times 12 mappers per node is 108 mappers) but each mapper takes a very long time to complete (hours).</p>

<p>The problem is that if the cluster is empty and such a job is started, it uses all of the mappers on the cluster.  Then, if anyone else wants to run a short job, it is blocked for hours.  I know that newer versions of Hadoop support preemption in the Fair Scheduler (we are using the Capacity Scheduler), but newer versions also are not stable (I'm anxiously awaiting the next release).</p>

<p>There used to be the option of <a href="http://wiki.apache.org/hadoop/HowManyMapsAndReduces" rel="nofollow">specifying the number of mappers</a> but now <a href="http://hadoop.apache.org/common/docs/r0.20.203.0/api/org/apache/hadoop/mapred/JobConf.html" rel="nofollow">JobConf</a> is deprecated (strangely, it is <a href="http://hadoop.apache.org/common/docs/r0.20.205.0/api/org/apache/hadoop/mapred/JobConf.html#setNumReduceTasks%28int%29" rel="nofollow">not deprecated in 0.20.205</a>).  This would alleviate the problem because, with more mappers, each map task would work on a smaller data set and thus finish sooner.</p>

<p>Is there any way around this problem in 0.20.203?  Do I need to subclass my InputFormat (in this case TextInputFormat)?  If so, what exactly do I need to specify?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I believe that you should be able to increase the block size for these files : if you do that, then , naturally, your application will use far fewer mappers. </p>

<p>Remember also that there is the  map.input.length parameter in the job configuration.  This would increase the splits, so that you had, effectively, fewer mappers with larger inputs.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Not exactly sure if more mappers will solve your problem. <a href="http://hadoop.apache.org/mapreduce/docs/r0.21.0/api/org/apache/hadoop/mapred/JobConf.html#setNumMapTasks%28int%29" rel="nofollow">JobConf#setNumMapTasks</a> has no effect on the # on the map tasks spawned per job. Even the document says it's just a hint to the framework. The # of map tasks spawned equals to the # of input splits for the job. Here are the different options to decrease the size of the InputSplit and thereby to increase the # of InputSplits and increase the # of map tasks.</p>

<ul>
<li><p>Decrease the size of the HDFS block by changing <code>dfs.blocksize</code>. But, this will increase the load on the NameNode as it has to keep more of the file vs block mapping and the the size of the DataBlock report also increases. Also, <code>hadoop fs -D fs.local.block.size=134217728 -put local_name remote_location</code> will change the block size for the new files put in HDFS, old files will remain as-is. The old files have to be pulled out of HDFS and put back with the required block size.</p></li>
<li><p>Use the <a href="http://svn.apache.org/repos/asf/hadoop/common/branches/branch-0.20-security-203/src/mapred/org/apache/hadoop/mapred/lib/NLineInputFormat.java" rel="nofollow">NLineInputFormat</a> to control the # of input lines to each map. For this the Job has to be changed. <code>mapred.line.input.format.linespermap</code> which is defaulted to 1 has to be defined.</p></li>
<li><p>From 0.21 release <code>mapreduce.input.fileinputformat.split.minsize</code> and <code>mapreduce.input.fileinputformat.split.maxsize</code> have been defined, but it's with the new MR API. The InputSplit calculation is done on the client, so it can't be enforced to the Job clients.</p></li>
</ul>

<p>The logic for calculating the InputSplit size is below.</p>

<pre><code>protected long computeSplitSize(long blockSize, long minSize, long maxSize) {  
    return Math.max(minSize, Math.min(maxSize, blockSize));  
}
</code></pre>
