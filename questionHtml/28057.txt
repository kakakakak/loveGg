<td class="answercell">
    <div class="post-text" itemprop="text">
<p><code>mapred.min.split.size</code> generally form the lower limit for creating input splits, while the DFS block size is 128MB. So in your case, the lower limit is more than the higher limit, and it seems that hadoop doesn't care for that and goes with the higher limit and split the input data accordingly.</p>

<p>Quoting from wiki:</p>

<blockquote>
  <p>Actually controlling the number of maps is subtle. The
  mapred.map.tasks parameter is just a hint to the InputFormat for the
  number of maps. The default InputFormat behavior is to split the total
  number of bytes into the right number of fragments. However, in the
  default case the DFS block size of the input files is treated as an
  upper bound for input splits. A lower bound on the split size can be
  set via mapred.min.split.size. Thus, if you expect 10TB of input data
  and have 128MB DFS blocks, you'll end up with 82k maps, unless your
  mapred.map.tasks is even larger. Ultimately the InputFormat determines
  the number of maps.</p>
</blockquote>

<p>The hint for you is in the last sentence, so if you want to control the numer of mappers, you have to override <code>InputFormat</code>, generally we use <code>FileInputFormat</code> and it's <code>isSplittable()</code> method needs to be overridden to return <code>false</code>. That would ensure that there is one mapper per file.
Something like follows will siffice:</p>

<pre><code>Class NonSplittableFileInputFormat extends FileInputFormat{

    @Override
    public boolean isSplitable(FileSystem fs, Path filename){ 
        return false; 
    }
}
</code></pre>
