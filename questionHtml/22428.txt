    <div class="post-text" itemprop="text">

<p>I have a dataset of 1 minute data of 1000 stocks since 1998, that total around <code>(2012-1998)*(365*24*60)*1000 = 7.3 Billion</code> rows.</p>

<p>Most (99.9%) of the time I will perform only <strong>read</strong> requests.</p>

<p>What is the best way to store this data in a db?</p>

<ul>
<li>1 big table with 7.3B rows?</li>
<li>1000 tables (one for each stock symbol) with 7.3M rows each?</li>
<li>any recommendation of database engine? (I'm planning to use Amazon RDS' MySQL)</li>
</ul>

<p>I'm not used to deal with datasets this big, so this is an excellent opportunity for me to learn. I will appreciate a lot your help and advice.</p>

<p>Edit:</p>

<p>This is a sample row:</p>

<blockquote>
  <p>'XX', 20041208, 938, 43.7444, 43.7541, 43.735, 43.7444, 35116.7, 1, 0, 0</p>
</blockquote>

<p>Column 1 is the stock symbol, column 2 is the date, column 3 is the minute, the rest are open-high-low-close prices, volume, and 3 integer columns.</p>

<p>Most of the queries will be like "Give me the prices of AAPL between April 12 2012 12:15 and April 13 2012 12:52"</p>

<p>About the hardware: I plan to use Amazon RDS so I'm flexible on that </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Tell us about the queries, and your hardware environment.</p>

<p>I would be very very tempted to go <a href="https://www.google.com/webhp?sourceid=chrome-instant&amp;ix=seb&amp;ie=UTF-8&amp;ion=1#hl=en&amp;newwindow=1&amp;safe=off&amp;output=search&amp;sclient=psy-ab&amp;q=NoSQL&amp;oq=&amp;aq=&amp;aqi=&amp;aql=&amp;gs_l=&amp;pbx=1&amp;fp=d6dc31c87e654d3f&amp;ix=seb&amp;ion=1&amp;bav=on.2,or.r_gc.r_pw.r_cp.r_qf.,cf.osb&amp;biw=1704&amp;bih=1038">NoSQL</a>, using <a href="http://hadoop.apache.org/">Hadoop</a> or something similar, as long as you can take advantage of parallelism.</p>

<p><strong>Update</strong></p>

<p>Okay, why?</p>

<p>First of all, notice that I asked about the queries.  You can't -- and we certainly can't -- answer these questions without knowing what the workload is like.  (I'll co-incidentally have an article about this appearing soon, but I can't link it today.)  But the <em>scale</em> of the problem makes me think about moving away from a Big Old Database because</p>

<ul>
<li><p>My experience with similar systems suggests the access will either be big sequential (computing some kind of time series analysis) or very very flexible data mining (OLAP).  Sequential data can be handled better and faster sequentially; OLAP means computing lots and lots of indices, which either will take lots of time or lots of space.</p></li>
<li><p>If You're doing what are effectively big runs against many data in an OLAP world, however, a column-oriented approach might be best.</p></li>
<li><p>If you want to do random queries, especially making cross-comparisons, a Hadoop system might be effective.  Why? Because</p>

<ul>
<li>you can better exploit parallelism on relatively small commodity hardware.</li>
<li>you can also better implement high reliability and redundancy</li>
<li>many of those problems lend themselves naturally to the MapReduce paradigm.</li>
</ul></li>
</ul>

<p>But the fact is, until we know about your workload, it's impossible to say anything definitive.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>So databases are for situations where you have a large complicated schema that is constantly changing.  You only have one "table" with a hand-full of simple numeric fields.  I would do it this way:</p>

<p>Prepare a C/C++ struct to hold the record format:</p>

<pre><code>struct StockPrice
{
    char ticker_code[2];
    double stock_price;
    timespec when;
    etc
};
</code></pre>

<p>Then calculate sizeof(StockPrice[N]) where N is the number of records.  (On a 64-bit system)  It should only be a few hundred gig, and fit on a $50 HDD.</p>

<p>Then truncate a file to that size and mmap (on linux, or use CreateFileMapping on windows) it into memory:</p>

<pre><code>//pseduo-code
file = open("my.data", WRITE_ONLY);
truncate(file, sizeof(StockPrice[N]));
void* p = mmap(file, WRITE_ONLY);
</code></pre>

<p>Cast the mmaped pointer to StockPrice*, and make a pass of your data filling out the array.  Close the mmap, and now you will have your data in one big binary array in a file that can be mmaped again later.</p>

<pre><code>StockPrice* stocks = (StockPrice*) p;
for (size_t i = 0; i &lt; N; i++)
{
    stocks[i] = ParseNextStock(stock_indata_file);
}
close(file);
</code></pre>

<p>You can now mmap it again read-only from any program and your data will be readily available:</p>

<pre><code>file = open("my.data", READ_ONLY);
StockPrice* stocks = (StockPrice*) mmap(file, READ_ONLY);

// do stuff with stocks;
</code></pre>

<p>So now you can treat it just like an in-memory array of structs.  You can create various kinds of index data structures depending on what your "queries" are.  The kernel will deal with swapping the data to/from disk transparently so it will be insanely fast.</p>

<p>If you expect to have a certain access pattern (for example contiguous date) it is best to sort the array in that order so it will hit the disk sequentially.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Okay, so this is somewhat away from the other answers, but... it feels to me like if you have the data in a file system (one stock per file, perhaps) with a fixed record size, you can get at the data <em>really</em> easily: given a query for a particular stock and time range, you can seek to the right place, fetch all the data you need (you'll know exactly how many bytes), transform the data into the format you need (which could be very quick depending on your storage format) and you're away.</p>

<p>I don't know anything about Amazon storage, but if you don't have anything like direct file access, you could basically have blobs - you'd need to balance large blobs (fewer records, but probably reading more data than you need each time) with small blobs (more records giving more overhead and probably more requests to get at them, but less useless data returned each time).</p>

<p>Next you add caching - I'd suggest giving different servers different stocks to handle for example - and you can pretty much just serve from memory. If you can afford enough memory on enough servers, bypass the "load on demand" part and just load all the files on start-up. That would simplify things, at the cost of slower start-up (which obviously impacts failover, unless you can afford to always have <em>two</em> servers for any particular stock, which would be helpful).</p>

<p>Note that you don't need to <em>store</em> the stock symbol, date or minute for each record - because they're implicit in the file you're loading and the position within the file. You should also consider what accuracy you need for each value, and how to store that efficiently - you've given 6SF in your question, which you could store in 20 bits. Potentially store three 20-bit integers in 64 bits of storage: read it as a <code>long</code> (or whatever your 64-bit integer value will be) and use masking/shifting to get it back to three integers. You'll need to know what scale to use, of course - which you could probably encode in the spare 4 bits, if you can't make it constant.</p>

<p>You haven't said what the other three integer columns are like, but if you could get away with 64 bits for those three as well, you could store a whole record in 16 bytes. That's only ~110GB for the whole database, which isn't really very much...</p>

<p>EDIT: The other thing to consider is that presumably the stock doesn't change over the weekend - or indeed overnight. If the stock market is only open 8 hours per day, 5 days per week, then you only need 40 values per week instead of 168. At that point you could end up with only about 28GB of data in your files... which sounds a lot smaller than you were probably originally thinking. Having that much data in memory is <em>very</em> reasonable.</p>

<p>EDIT: I think I've missed out the explanation of <em>why</em> this approach is a good fit here: you've got a very predictable aspect for a large part of your data - the stock ticker, date and time. By expressing the ticker <em>once</em> (as the filename) and leaving the date/time entirely implicit in the <em>position</em> of the data, you're removing a whole bunch of work. It's a bit like the difference between a <code>String[]</code> and a <code>Map&lt;Integer, String&gt;</code> - knowing that your array index always starts at 0 and goes up in increments of 1 up to the length of the array allows for quick access and more efficient storage.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Let me recommend that you take a look at <a href="http://lucene.apache.org/solr/" rel="nofollow">apache solr</a>, which I think would be ideal for your particular problem.  Basically, you would first index your data (each row being a "document"). Solr is optimized for searching and natively supports range queries on dates. Your nominal query, </p>

<p><code>"Give me the prices of AAPL between April 12 2012 12:15 and April 13 2012 12:52"</code></p>

<p>would translate to something like:</p>

<pre><code>?q=stock:AAPL AND date:[2012-04-12T12:15:00Z TO 2012-04-13T12:52:00Z]
</code></pre>

<p>Assuming "stock" is the stock name and "date" is a "DateField" created from the "date" and "minute" columns of your input data on indexing.  Solr is incredibly flexible and I really can't say enough good things about it.  So, for example, if you needed to maintain the fields in the original data, you can probably find a way to dynamically create the "DateField" as part of the query (or filter).</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I think any major RDBMS would handle this. At the atomic level, a one table with correct partitioning seems reasonable (partition based on your data usage if fixed - this is ikely to be either symbol or date).</p>

<p>You can also look into building aggregated tables for faster access above the atomic level. For example if your data is at day, but you often get data back at the wekk or even month level, then this can be pre-calculated in an aggregate table. In some databases this can be done though a cached view (various names for different DB solutions - but basically its a view on the atomic data, but once run the view is cached/hardened intoa fixed temp table - that is queried for subsequant matching queries. This can be dropped at interval to free up memory/disk space).</p>

<p>I guess we could help you more with some idea as to the data usage.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>If you have the hardware, I recommend <a href="http://www.mysql.com/products/cluster/" rel="nofollow">MySQL Cluster</a>. You get the MySQL/RDBMS interface you are so familiar with, and you get fast and parallel writes. Reads will be slower than regular MySQL due to network latency, but you have the advantage of being able to parallelize queries and reads due to the way MySQL Cluster and the NDB storage engine works.</p>

<p>Make sure that you have enough MySQL Cluster machines and enough memory/RAM for each of those though - MySQL Cluster is a heavily memory-oriented database architecture.</p>

<p>Or <a href="http://redis.io/" rel="nofollow">Redis</a>, if you don't mind a key-value / NoSQL interface to your reads/writes. Make sure that Redis has enough memory - its super-fast for reads and writes, you can do basic queries with it (non-RDBMS though) but is also an in-memory database.</p>

<p>Like others have said, knowing more about the queries you will be running will help.</p>

<p>You should compare the slow solutions with a simple optimized in memory model. Uncompressed it fits in a 256 GB ram server. A snapshot fits in 32 K and you just index it positionally on datetime and stock. Then you can make specialized snapshots, as open of one often equals closing of the previous.</p>

<p>[edit] Why do you think it makes sense to use a database at all (rdbms or nosql)? This data doesn't change, and it fits in memory. That is not a use case where a dbms can add value.</p>

<p>First, there isn't 365 trading days in the year, with holidays 52 weekends (104) = say 250 x the actual hours of day market is opened like someone said, and to use the symbol as the primary key is not a good idea since symbols change, use a k_equity_id  (numeric) with a symbol (char) since symbols can be like this A , or GAC-DB-B.TO , then in your data tables of price info, you have, so your estimate of 7.3 billion is vastly over calculated since it's only about 1.7 million rows per symbol for 14 years.</p>

<p>k_equity_id
k_date
k_minute</p>

<p>and for the EOD table (that will be viewed 1000x over the other data)</p>

<p>k_equity_id
k_date</p>

<p>Second, don't store your OHLC by minute data in the same DB table as and EOD table (end of day) , since anyone wanting to look at a pnf, or line chart, over a year period , has zero interest in the by the minute information.</p>
