<p>Pardon my simple question but I'm relatively new to Spark/Hadoop.</p>

<p>I'm trying to load a bunch of small CSV files into Apache Spark. They're currently stored in S3, but I can download them locally if that simplifies things. My goal is to do this as efficiently as possible. It seems like it would be a shame to have some single-threaded master downloading and parsing a bunch of CSV files while my dozens of Spark workers sit idly. I'm hoping there's an idiomatic way to distribute this work.</p>

<p>The CSV files are arranged in a directory structure that looks like:</p>

<pre><code>2014/01-01/fileabcd.csv
2014/01-01/filedefg.csv
...
</code></pre>

<p>I have two years of data, with directories for each day, and a few hundred CSVs inside of each. All of those CSVs should have an identical schema, but it's of course possible that one CSV is awry and I'd hate for the whole job to crash if there are a couple problematic files. Those files can be skipped as long as I'm notified in a log somewhere that that happened.</p>

<p>It seems that every Spark project I have in mind is in this same form and I don't know how to solve it. (e.g. trying to read in a bunch of tab-delimited <a href="https://aws.amazon.com/datasets/2759" rel="nofollow">weather data</a>, or reading in a bunch of log files to look at those.)</p>

<h2>What I've Tried</h2>

<p>I've tried both SparkR and the Scala libraries. I don't really care which language I need to use; I'm more interested in the correct idioms/tools to use.</p>

<h3>Pure Scala</h3>

<p>My original thought was to enumerate and <code>parallelize</code> the list of all <code>year/mm-dd</code> combinations so that I could have my Spark workers all processing each day independently (download and parse all CSV files, then stack them on top of eachother (<code>unionAll()</code>) to reduce them). Unfortunately, downloading and parsing the CSV files using the <a href="https://github.com/databricks/spark-csv" rel="nofollow">spark-csv</a> library can only be done in the "parent"/master job, and not from each child as <a href="http://stackoverflow.com/questions/13770218/call-of-distinct-and-map-together-throws-npe-in-spark-library/14130534#14130534">Spark doesn't allow job nesting</a>. So that won't work as long as I want to use the Spark libraries to do the importing/parsing.</p>

<h3>Mixed-Language</h3>

<p>You can, of course, use the language's native CSV parsing to read in each file then "upload" them to Spark. In R, this is a combination of some package to get the file out of S3 followed by a <code>read.csv</code>, and finishing off with a <code>createDataFrame()</code> to get the data into Spark. Unfortunately, this is really slow and also seems backwards to the way I want Spark to work. If all my data is piping through R before it can get into Spark, why bother with Spark?</p>

<h3>Hive/Sqoop/Phoenix/Pig/Flume/Flume Ng/s3distcp</h3>

<p>I've started looking into these tailored tools and quickly got overwhelmed. My understanding is that many/all of these tools could be used to get my CSV files from S3 into HDFS.</p>

<p>Of course it would be faster to read my CSV files in from HDFS than S3, so that solves some portion of the problem. But I still have tens of thousands of CSVs that I need to parse and am unaware of a distributed way to do that in Spark.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>So right now (Spark 1.4) SparkR has support for <code>json</code> or <code>parquet</code> file structures. Csv files can be parsed, but then the spark context needs to be started with an extra jar (which needs to be downloaded and placed in the appropriate folder, never done this myself but my collegues have). </p>

<pre><code>sc &lt;- sparkR.init(sparkPackages="com.databricks:spark-csv_2.11:1.0.3")
sqlContext &lt;- sparkRSQL.init(sc)
</code></pre>

<p>There is more information in <a href="https://spark.apache.org/docs/latest/sparkr.html" rel="nofollow">the docs</a>. I expect that a newer spark release would have more support for this. </p>

<p>If you don't do this you'll need to either resort to a different file structure or use python to convert all your files from <code>.csv</code> into <code>.parquet</code>. Here is a snippet from a recent python talk that does this.</p>

<pre><code>data = sc.textFile(s3_paths, 1200).cache()

def caster(x):
    return Row(colname1 = x[0], colname2 = x[1])

df_rdd = data\
    .map(lambda x: x.split(','))\
    .map(caster)

ddf = sqlContext.inferSchema(df_rdd).cache()

ddf.write.save('s3n://&lt;bucket&gt;/&lt;filename&gt;.parquet')
</code></pre>

<p>Also, how big is your dataset? You may not even need spark for analysis. Note that also as of right now;</p>

<ul>
<li>SparkR has only DataFrame support.</li>
<li>no distributed machine learning yet. </li>
<li>for visualisation you will need to convert a distributed dataframe back into a normal one if you want to use libraries like <code>ggplot2</code>. </li>
<li>if your dataset is no larger than a few gigabytes, then the extra bother of learning spark might not be worthwhile yet </li>
<li>it's modest now, but you can expect more from the future</li>
</ul>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I've run into this problem before (but w/ reading a large qty of Parquet files) and my recommendation would be to avoid dataframes and to use RDDs.</p>

<p>The general idiom used was:</p>

<ol>
<li>Read in a list of the files w/ each file being a line (In the driver). The expected output here is a list of strings</li>
<li>Parallelize the list of strings and map over them with a customer csv reader. with the return being a list of case classes.</li>
</ol>

<p>You can also use flatMap if at the end of the day you want a data structure like List[weather_data] that could be rewritten to parquet or a database. </p>
