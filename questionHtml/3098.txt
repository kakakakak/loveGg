<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I'm quoting below some parts of the conclusion of <a href="http://bjclark.me/2009/08/04/nosql-if-only-it-was-that-easy/" rel="nofollow">NoSQL: If Only It Was That Easy</a> (the article is more about scalability but still contains interesting things that apply to your context): </p>

<blockquote>
  <p>[...]</p>
  
  <p>The real thing to point out is that if
  you are being held back from making
  something super awesome because you
  can’t choose a database, you are doing
  it wrong. <strong><em>If you know mysql, just use
  it. Optimize when you actually need
  to. Use it like a k/v store, use it
  like a rdbms, but for god sake, build
  your killer app! None of this will
  matter to most apps.</em></strong> Facebook still
  uses MySQL, a lot. Wikipedia uses
  MySQL, a lot. FriendFeed uses MySQL, a
  lot. NoSQL is a great tool, but it’s
  certainly not going to be your
  competitive edge, it’s not going to
  make your app hot, and most of all,
  your users won’t give a shit about any
  of this.</p>
  
  <p>What am I going to build my next app
  on? Probably Postgres. Will I use
  NoSQL? Maybe. I might also use Hadoop
  and Hive. I might keep everything in
  flat files. Maybe I’ll start hacking
  on Maglev. I’ll use whatever is best
  for the job. If I need reporting, I
  won’t be using any NoSQL. If I need
  caching, I’ll probably use Tokyo
  Tyrant. If I need ACIDity, I won’t use
  NoSQL. If I need a ton of counters,
  I’ll use Redis. If I need
  transactions, I’ll use Postgres. If I
  have a ton of a single type of
  documents, I’ll probably use Mongo. <strong><em>If
  I need to write 1 billion objects a
  day, I’d probably use Voldemort.</em></strong> If I
  need full text search, I’d probably
  use Solr. If I need full text search
  of volatile data, I’d probably use
  Sphinx.</p>
  
  <p>[...]</p>
</blockquote>

<p>So if a non ACID storage system is an option, I'd look at <a href="http://project-voldemort.com/" rel="nofollow">Voldemort</a>. If not, without more specific informations, I can't say if one DBMS is really better than another for write-intensive applications. Actually, I think it's more a matter of design/architecture/tuning and tend to agree with the author: 1) use the one you know the most 2) which one you choose won't matter to most apps.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Well I have seen the commercial DBs get up 2GB per minute on not particulary impressive hardware. The standard Open Source dbs (MySQL, Postgress even sqlite are not far behind).</p>

<p>For any volume of writes which will give a modern DB trouble there are three things which will affect performance (neither of which depends on the particular DB you choose).</p>

<p>One is basic design, particulary partitioning (spreading your db over several physical disks) and minimising the number of indexes on the tables (for write performance zero indexes is best!). </p>

<p>Two is log placement or if possible log avoidence. Logging is the bottleneck in most RDBMes. Making sure your are logging to dedicated fast disks is one way, turning of logging (varies according to the RDBMS but most support this) for the table, if you 
can afford to lose transactions. </p>

<p>Three is hardware -- lots of memory and lots of fast disks to spread out your I/O load.</p>

<p>There are some exotic options out there if this is still not fast enough.
Buy a z/OS mainframe and run the venerable IMS/DB with the DEDB (Data Entry database) feature. This is about four times faster than any other ACID DB. Buy Oracle's In Memory DB option (used to be HPs TimesTen).</p>

<p>Another possibility if you have some decent queing software avaiable is to capture the data and immediatly place it in a queue. You can then have one or more background processes pulling the data off the queue and doing the actual DB updates in the backgroud.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>This is a bit of a mystery question -- If you are doing lots and lots of writes and very few look-ups (reads) and (some) updates) ...</p>

<p>Use a fixed record random access file (Seek() and stuff on a posix platform), a flat file.  If you need indexing, simply index your keys to the flat-file to do reads and updates.  </p>

<p>The disadvantage is you need to keep the keys in sync. with the contents on writes and updates.  A pretty simple C++ or other OO class can handle that for you I feel.  Why write indexes if you aren't going to use them?  And, depending on you actual need, you can do away with the indexing all together -- And index at the end of the day or something-k!!</p>

<p>Cheers, w.</p>
