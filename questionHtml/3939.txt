    <div class="post-text" itemprop="text">

<p>I've recently seen some code (amazon hadoop code) that utilizes this type of syntax</p>

<pre><code>Foo bar = new Foo().setX(10).setY(11);
</code></pre>

<p>I thought that was sweet so I decided to give it a go. made my <code>setX()</code> type functions return <code>Foo</code> instead of <code>void</code> and put <code>return this;</code> in all of them. this worked well. until I tried it with inheritance, which produced some strage results.</p>

<p>I'll give a concrete example: I have two classes, <code>Location</code> class that has two fields, x and y. and another class <code>Location3D</code> that inherits from <code>Location</code> and adds a third field, z.
all the fields use the method described above for their setters.</p>

<p>now I want to create a new location3D instance and set its fields, what happens is</p>

<pre><code>new Location3D().setZ(7).setY(6).setX(5)
</code></pre>

<p>works while</p>

<pre><code>new Location3D().setX(7).setY(6).setZ(5)
</code></pre>

<p>doesn't.</p>

<p>by doesn't work I mean that what returns from <code>setY(6)</code> is a <code>Location</code> object and not a <code>location3D</code> object and therefore does not have a <code>setZ()</code> method!</p>

<p>after this long intro my question is: can this form of "setter stringing" be made to work with inheritance without forcing the caller to cast objects ? if so how ?</p>

<p>also I'm sure there's a term for this better then "setter stringing", what is it ?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>As you pointed out, the reason that <code>new Location3D().setX(7).setY(6).setZ(5)</code> doesn't work is because <code>setX()</code> and <code>setY()</code> return instances of <code>Location</code> and not <code>Location3D</code>.</p>

<p>You can get around this using generics (though the solution isn't particularly pretty) by adding a generic type parameter to your <code>Location</code> class:</p>

<pre><code>public class Location&lt;T extends Location&lt;?&gt;&gt; {
    protected int x, y;

    @SuppressWarnings("unchecked")
    public T setX(int x) {
        this.x = x;
        return (T) this;
    }

    @SuppressWarnings("unchecked")
    public T setY(int y) {
        this.y = y;
        return (T) this;
    }
}
</code></pre>

<p>Your subclass <code>Location3D</code> would then have set itself as the generic type parameter so that the superclass returns instances of <code>Location3D</code> instead of <code>Location</code>:</p>

<pre><code>public class Location3D extends Location&lt;Location3D&gt; {
    protected int z;

    public Location3D setZ(int z) {
        this.z = z;
        return this;
    }
}
</code></pre>

<p>Unfortunately, there's no way that I know of to avoid the warnings produced by the superclass, hence the <code>@SuppressWarnings("unchecked")</code> annotations.</p>

<p>It's also worth noting that if you define your subclass such that the generic type parameter is a different class type, then you could wind up with <code>ClassCastException</code>, hence you should document that restriction in your superclass for anyone who might want to create their own subclass.</p>

<p>Finally, chaining together method calls in the way you describe is usually referred to as <a href="http://en.wikipedia.org/wiki/Method_chaining"><em>method chaining</em></a>.  The style of setter methods that you're describing is closely related to the <a href="http://en.wikipedia.org/wiki/Builder_pattern">builder pattern</a>.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>The problem is that Location returns Location, as Location3D would return Location3D.  To solve this, override the methods you want to use in Location3D and change the return type:</p>

<pre><code>public class Location {
    private int x;
    private int y;

    public Location setY(int y){
        this.y = y;
        return this;
    }

    public Location setX(int x){
        this.x = x;
        return this;
    }
}
</code></pre>

<p>And Location3D:</p>

<pre><code>public class Location3D extends Location {

    private int z;

    public Location3D setY(int y){
        super.setY(y);
        return this;
    }

    public Location3D setX(int x){
        super.setX(x);
        return this;
    }

    public Location3D setZ(int z){
        this.z = z;
        return this;
    }
}
</code></pre>

<p><strong>Composition approach:</strong></p>

<pre><code>public class Location {
    private int x;
    private int y;

    public Location setY(int y){
        this.y = y;
        return this;
    }

    public Location setX(int x){
        this.x = x;
        return this;
    }
}
</code></pre>

<p>And Location3D:</p>

<pre><code>public class Location3D {

    private Location location = new Location();
    private int z;

    public Location3D setY(int y){
        location.setY(y);
        return this;
    }

    public Location3D setX(int x){
        location.setX(x);
        return this;
    }

    public Location3D setZ(int z){
        this.z = z;
        return this;
    }
}
</code></pre>
