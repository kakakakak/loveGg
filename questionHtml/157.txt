<p>I am doing some linguistic research that depends on being able to query a corpus of 100 million sentences. The information I need from that corpus is along the lines: how many sentences had "john" as first word, "went" as second word and "hospital" as the fifth word...etc So I just need the <b>count</b> and don't need to actually retrieve the sentences.</p>

<p>The idea I had was to split these sentences into words and store them into a database, where the columns would be the positions (word-1, word-2, word-3..etc) and the sentences would be the rows. So it looks like:</p>

<p>Word1     Word2    Word3   Word4   Word5 ....</p>

<p>Congress  approved a    new    bill</p>

<p>John      went     to   school</p>

<p>.....</p>

<p>And my purpose will then be fulfilled by calling something like COUNT(SELECT * where Word1=John and Word4=school). But I am wondering: Can this be better achieved using Lucene (or some other tool)?</p>

<p>The program I am writing (in Java) will be doing tens of thosands of such queries on that 100 million sentece corpus. So speed of look-up is important.</p>

<p>Thanks for any advice,</p>

<p>Anas</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<ul>
<li>I suggest you read <a href="http://www.lucidimagination.com/Community/Hear-from-the-Experts/Articles/Search-Engine-versus-DBMS" rel="nofollow">Search Engine versus DBMS</a>. From what I gather, you do need a database rather than a full text search library.</li>
<li>In any case, I suggest you preprocess your text and replace every word/token with a number using a dictionary. This replaces every sentence with an array of word codes. I would then store every word place in a separate database column, simplifying counts and making them quicker.
For example:</li>
</ul>

<blockquote>
  <p>A boy and a girl drank milk</p>
</blockquote>

<p>translates into:</p>

<blockquote>
  <p>120 530 14 120 619 447 253</p>
</blockquote>

<p>(I chose arbitrary word codes) leading to store a row</p>

<blockquote>
  <p>120 530 14 120 619 447 253 0 0 0 0 0 0 0 ....</p>
</blockquote>

<p>(until the number of words you allocate per a sentence is exhausted).</p>

<p>This is a somewhat <a href="http://en.wikipedia.org/wiki/Sparse_matrix" rel="nofollow">sparse matrix</a>, so maybe <a href="http://stackoverflow.com/questions/598089/how-to-store-a-sparse-queryable-matrix-on-disk-or-database">this question</a> will help.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I suppose you already have infrastructure to create tokens from a given sentence. You can create a lucene document with one field for each word in the sentence. You can name the fields as <code>field1, field2,</code> and so on. Since, lucene doesn't have a schema like DB, you can define as many fields, on the fly, as you wish. You can add an additional identifier field if you want to identify which sentences matched a query.</p>

<p>While searching, your typical lucene query will be </p>

<pre><code>+field1:John +field4:school
</code></pre>

<p>Since you are not bothered about the speed of retrieval, you can write a custom Collector which will ignore scores. (That will return results significantly faster as well.)</p>

<p>Since you don't plan to retrive back the matching sentences or words, you should only index these fields and not store. That should push performance up by a notch.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Lucene <a href="http://lucene.apache.org/java/3_0_0/api/core/org/apache/lucene/search/spans/SpanQuery.html" rel="nofollow">span queries</a> can implement positional search.  Use <strong>SpanFirst</strong> to find a word in the first N positions of a document, and combine it with <strong>SpanNot</strong> to rule out the first N-1.</p>

<p>Your example query would like this:</p>

<pre><code>&lt;BooleanQuery: +(+spanFirst(john, 1) +spanFirst(went, 2)) +spanNot(spanFirst(hospital, 5), spanFirst(hospital, 4))&gt;
</code></pre>

<p>Lucene also of course allows getting the total hit count of a search result without iterating all the docs.</p>
