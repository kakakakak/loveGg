<p>I am facing to a problem: database for process plants. There are up to 50,000 sensors at sampling rate of 50 ms. All measured values need to be stored at least 3 years and must support real-time queries (i.e. users can see historical data with delay less than 1 second). I recently read an article about <a href="http://www.erol.si/2015/01/the-complete-list-of-all-timeseries-databases-for-your-iot-project/" rel="nofollow">Time-series Database</a>, many options are on hand: OpenTSDB, KairosDB, InfluxDB, ...</p>

<p>I am confused which one would be proper for the purpose? Any one know about this please help me!</p>

<p><strong>UPDATE 15.06.25</strong></p>

<p>Today I run a test based on OpenTSDB. I used Virtual Box to create a cluster of 3 CentOS x64 VMs (1 master, 2 slaves). The host configuration is 8 GB RAM, core i5. The master VM configuration is 3 GB RAM, and the slaves configuration is 1.5 GB RAM. I write a python program to send data to OpenTSDB as below:</p>

<pre><code>s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.connect(("192.168.10.55", 4242))

start_time = time.time()
start_epoch = 1434192418;

for x in range(0, 1000000):
    curr_epoch = start_epoch + x

    tag1 = "put TAG_1 %d 12.9 stt=good\n" % (curr_epoch)
    tag2 = "put TAG_2 %d 12.9 stt=good\n" % (curr_epoch)
    tag3 = "put TAG_3 %d 12.9 stt=good\n" % (curr_epoch)
    tag4 = "put TAG_4 %d 12.9 stt=good\n" % (curr_epoch)
    tag5 = "put TAG_5 %d 12.9 stt=good\n" % (curr_epoch)
    tag6 = "put TAG_6 %d 12.9 stt=good\n" % (curr_epoch)
    tag7 = "put TAG_7 %d 12.9 stt=good\n" % (curr_epoch)
    tag8 = "put TAG_8 %d 12.9 stt=good\n" % (curr_epoch)
    tag9 = "put TAG_9 %d 12.9 stt=good\n" % (curr_epoch)
    tag10 = "put TAG_10 %d 12.9 stt=good\n" % (curr_epoch)
    str = tag1 + tag2 + tag3 + tag4 + tag5 + tag6 + tag7 + tag8 + tag9 + tag10

    s.send(str)

print("--- %s seconds ---" % (time.time() - start_time))
</code></pre>

<p>I run the python on host, and the work completes after ~220 seconds. So, I got an avg. speed of ~45000 records per second.</p>

<p><strong>UPDATE 15.06.29</strong></p>

<p>This time I used only 1 VM (5 GB RAM, 3 cores, CentOS x64, pseudo-distributed Hadoop). I run 2 python processes on Windows 7 host to send 2 halves of data to the OpenTSDB. The avg. speed of putting data was ~100,000 records per second.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>To handle the million writes per seconds, you will need to put some serious engineering in place.</p>

<p>Not all databases will be able to store that amount of data in a compact form.</p>

<p>For example ATSD uses 5 to 10 bytes per sample (float data type), depending on observed variance.</p>

<p>There is a type of distributed (clustered) databases built on HBase that will be able to handle this kind of load.</p>

<p>For example, you can try looking at <a href="http://opentsdb.net/" rel="nofollow">openTSDB</a> and <a href="https://axibase.com/products/axibase-time-series-database/" rel="nofollow">ATSD</a>.</p>

<p><strong>Update 1.</strong></p>

<p>We have run the following test for your particular use case:</p>

<p>30.000 analog sensors writing float type data, resulting in 540.000.000 records</p>

<p>20.000 digital sensors writing short type data (zeros and ones), resulting in 552.000.000 records</p>

<p>The data took up 3.68 gigabytes. The compression was lossless.</p>

<p>Resulting in an average 3.37 bytes per record.</p>

<p>This was a storage efficiency test.</p>

<p><em>Full disclosure, I work for the company that develops ATSD.</em></p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>InfluxDB won't handle a sustained million writes per second right now, but that's within the performance target for later this year. The bigger challenge I see is the sheer volume of data you want to store. If you need to keep three years' worth at full resolution with no downsampling, that's hundreds of terabytes of data. If that's not all on SSD, the query performance will not be good. If that is all on SSD, that's an extremely expensive database. Also, with that much raw data, it would be very easy to craft a query that explodes the RAM no matter how much is installed.</p>

<p>I would say check back with the InfluxDB team in 8-12 weeks and we might have a better idea how to handle your problem. My advice, though, is to find a way to split that up. If you really are sampling 50k machines at 50ms intervals, it's a huge amount of data, network traffic, etc.</p>

<td class="answercell">

<p>Very interresting use case.</p>

<p>I cannot speak for others since I only strongly evaluated and used KairosDB &amp; Cassandra but I can speak of my own experience.</p>

<p>KairosDB + Cassandra would stand the throughput but for 1M writes per second you would need a cluster with several front-ends (I'd recommend at least 10 nodes, but you have to test) and back-end servers (they can be colocated if needed).</p>

<p>Anyway with a throughput of 200 samples per second for each sensor... Retrieving historical data could be a problem for long queries (1s for historical data is interesting but you have to determine the amount of samples of the query duration).</p>

<p>As far as I can tell storage efficiency will not be as goot as ATSD (probably twice the size, which is already good).</p>

<p>What I personally like with KairosDB &amp; Cassandra and why we adopted it: it's simple, really distributed, KairosDB doesn't mess with the data, the design is elegant and doesn't involve complex systems, Cassandra manages the data and kairosDB offers the TSDB services, and performances and efficiency are not bad at all... Moreover Cassandra is much easier to manage as a cluster than its counterparts.</p>
