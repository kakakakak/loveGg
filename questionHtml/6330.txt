    <div class="post-text" itemprop="text">

<p>We are trying to run our spark cluster on yarn. We are having some performance issues especially when compared to the standalone mode. </p>

<p>We have a cluster of 5 nodes with each having 16GB RAM and 8 cores each. We have configured the minimum container size as 3GB and maximum as 14GB in yarn-site.xml. When submitting the job to yarn-cluster we supply number of executor = 10, memory of executor =14 GB. According to my understanding our job should be allocated 4 container of 14GB. But the spark UI shows only 3 container of 7.2GB each. </p>

<p>We are unable to ensure the container number and resources allocated to it. This causes detrimental performance when compared to the standalone mode. </p>

<p>Can you drop any pointer on how to optimize yarn performance?</p>

<p>This is the command I use for submitting the job: </p>

<pre><code>$SPARK_HOME/bin/spark-submit --class "MyApp" --master yarn-cluster --num-executors 10 --executor-memory 14g  target/scala-2.10/my-application_2.10-1.0.jar  
</code></pre>

<p>Following the discussion I changed my yarn-site.xml file and also the spark-submit command.</p>

<p>Here is the new yarn-site.xml code :</p>

<pre><code>&lt;property&gt;
&lt;name&gt;yarn.resourcemanager.hostname&lt;/name&gt;
&lt;value&gt;hm41&lt;/value&gt;
&lt;/property&gt;

&lt;property&gt;
&lt;name&gt;yarn.nodemanager.resource.memory-mb&lt;/name&gt;
&lt;value&gt;14336&lt;/value&gt;
&lt;/property&gt;

&lt;property&gt;
&lt;name&gt;yarn.scheduler.minimum-allocation-mb&lt;/name&gt;
&lt;value&gt;2560&lt;/value&gt;
&lt;/property&gt;

&lt;property&gt;
&lt;name&gt;yarn.scheduler.maximum-allocation-mb&lt;/name&gt;
&lt;value&gt;13312&lt;/value&gt;
&lt;/property&gt;
</code></pre>

<p>And the new command for spark submit is</p>

<pre><code>$SPARK_HOME/bin/spark-submit --class "MyApp" --master yarn-cluster --num-executors 4 --executor-memory  10g --executor-cores 6   target/scala-2.10/my-application_2.10-1.0.jar 
</code></pre>

<p>With this I am able to get 6 cores on each machine but the memory usage of each node is still around 5G. I have attached the screen shot of SPARKUI and htop.
<img src="http://i.stack.imgur.com/bqZwU.png" alt="enter image description here">
<img src="http://i.stack.imgur.com/CQM1a.png" alt="Spark UI Screenshot![][1]"></p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<ol>
<li>Withing yarn-site.xml check that <code>yarn.nodemanager.resource.memory-mb</code> is set the right way. In my understanding of your cluster it should be set to 14GB. This setting is responsible for giving the YARN know how much memory it can use on this specific node</li>
<li>If you have this set right and you have 5 servers running YARN NodeManager, then your job submission command is wrong. First, <code>--num-executors</code> is the number of YARN containers would be started for executing on the cluster. You specify 10 containers with 14GB RAM each, but you don't have this many resources on your cluster! Second, you specify <code>--master yarn-cluster</code>, which means that Spark Driver would run inside of the YARN Application Master that would require a separate container.</li>
<li>In my opinion it shows 3 containers because out of 5 nodes in the cluster you have only 4 of them running YARN NodeManager + you request to allocate 14GB for each of the containers, so YARN first starts Application Master and then polls the NM for available resources and see that it can start only 3 containers. Regarding heap size you see, after starting the Spark find its JVM containers and see the parameters of their start - you should have many -Xmx flags in a single line - one correct and one wrong, you should find its origin in config files (Hadoop or Spark)</li>
<li>Before submitting an application to the cluster, start the spark-shell with the same settings (replace <code>yarn-cluster</code> with <code>yarn-client</code>) and check how it is started, check WebUI and JVMs started</li>
</ol>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Just because YARN "thinks" it has 70GB (14GBx5), doesn't mean at run time there is 70GB available on the cluster. You could be running other Hadoop components (hive, HBase, flume, solr, or your own app, etc.) which consume memory. So the run-time decision YARN makes is based on what's currently available -- and it had only 52GB (3x14GB) available to you. By the way, the GB numbers are approximate because it is really computed as 1024MB per GB...so you will see decimals. </p>

<p>Use <a href="http://nmon.sourceforge.net/pmwiki.php" rel="nofollow">nmon</a> or top to see what else is using memory on each node. </p>
