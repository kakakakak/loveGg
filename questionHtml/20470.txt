<td class="answercell">
    <div class="post-text" itemprop="text">
<p>The <code>which</code> command returns the full path of an executable, e.g.:</p>

<pre><code>mureinik@comupter ~ $ which mkdir
/usr/bin/mkdir
</code></pre>

<p>and if you really need the directory, you could always apply <code>dirname</code> to it:</p>

<pre><code>mureinik@conmputer ~ $ dirname `which mkdir`
/usr/bin
</code></pre>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Reading the OP's question, I noticed the pointed exclusion of methods relying on <code>PATH</code>, and disregarded the obvious answer for <code>which</code> (because that is precisely what it does).  The <code>which</code> program, by the way was originally a C-shell script although there are other implementations (<a href="https://www.freebsd.org/cgi/man.cgi?query=which&amp;apropos=0&amp;sektion=0&amp;manpath=FreeBSD%2010.1-RELEASE&amp;arch=default&amp;format=html" rel="nofollow">FreeBSD</a>, and <a href="http://unixhelp.ed.ac.uk/CGI/man-cgi?which" rel="nofollow">bash</a> for instance).  <a href="http://www.manpages.info/sunos/which.1.html" rel="nofollow">Solaris</a>, for instance, has a version little changed from <a href="http://modman.unixdev.net/?sektion=1&amp;page=which&amp;manpath=4.2BSD" rel="nofollow">4.2BSD</a>, and its manual page states</p>

<blockquote>
  <p>Both aliases and path are taken from the user's .cshrc file.</p>
</blockquote>

<p>If you do not use C-shell, that original version of <code>which</code> has limited usefulness.  Because there is no standard version, the results (and capability of the tool) can vary from system to system.  Some shells (such as <code>ksh</code>) have similar features, but named differently, e.g., <code>whence</code> in the case of <code>ksh</code> (bash supplies the ksh alias <code>type</code> but not the `whence keyword).</p>

<p>I do not use <code>which</code> much, having better tools (<a href="http://invisible-island.net/conflict/conflict.html" rel="nofollow">conflict</a> and <a href="http://invisible-island.net/scripts/readme.html#item:path" rel="nofollow">path</a>).  But having ignored the obvious, the question as worded could easily pose a more interesting question: how can an application determine the directory from which another application was run.  Not the obvious <em>"where is the executable?"</em>, but <em>"what was the working directory when the program was run?"</em>.  There was a similar question recently dealing with relative pathnames.</p>

<p>On some of the systems which support the <code>/proc</code> pseudo-filesystem, there is a <code>cwd</code> node which gives the current working directory of a given process.  The question <em><a href="http://unix.stackexchange.com/questions/94357/find-out-current-working-directory-of-a-running-process">Find out current working directory of a running process?</a></em> goes into some detail regarding this.</p>

<p>xterm's optional feature "spawn-new-terminal" (added in <a href="http://invisible-island.net/xterm/xterm.log.html#xterm_225" rel="nofollow">patch #225</a> in 2007) uses this information to make a new terminal instance run in the same working directory as the user's current <em>shell</em> working directory.</p>

<p>A script could use this by starting with the process-id for the command in question, and inspecting the <code>/proc</code> tree for the corresponding <code>cwd</code> node.</p>
