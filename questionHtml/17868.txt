    <div class="post-text" itemprop="text">
<p>Here:</p>

<pre><code>open my $valid, '&lt;',....
</code></pre>

<p>you have opened <code>$valid</code> for <em>reading</em>. If you wish to write to the file, you must instead write:</p>

<pre><code>open my $valid, '&gt;',....
</code></pre>

<p>If you need to keep existing contents and write only to the end then instead use </p>

<pre><code>open my $valid, '&gt;&gt;',....
</code></pre>

<p>You're only opening the file under the $valid file handle with read capabilities, as you can see from this line:</p>

<pre><code>open my $valid, '&lt;', 'c:\perl\015_JiraGet\valid.txt' or die "Cannot read: $!\n";

<pre><code>open my $valid, '+&gt;', 'c:\perl\015_JiraGet\valid.txt' or die "Cannot read: $!\n";
</code></pre>

        <a class="vote-up-off" title="This answer is useful">up vote</a>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I am going to review this code as if it had been posted to <a href="http://codereview.stackexchange.com">Code Review</a>.</p>

<p>First off you are writing in <a href="http://programmers.stackexchange.com/questions/100839/should-we-hire-someone-who-writes-c-in-perl" title="Should we hire someone who writes C in Perl?">Perl as if it were C</a>. Which in general is not that bad, but it does mean that you are doing quite a bit more work than is necessary.</p>

<hr>

<p>Instead of using this overly verbose, and potentially memory intensive:</p>

<pre><code>my @totalReport
...
while(&lt;$input&gt;) { chomp; push @totalReport, $_; }
foreach(@totalReport)
{
  $eachAddr = $_;
  ...
}
</code></pre>



<pre><code>while( my $addr = &lt;$input&gt; ){
  chomp $addr;
  ...
}
</code></pre>

<p>Notice how I've eliminated a variable, and made it so that it loops on the input once, instead of twice. It also doesn't keep the values in memory over the entire length of the program.</p>

<hr>

<p>Instead of opening a file for writing, closing it, and opening it again:</p>

<pre><code>my $tempFile;

open $tempFile, '&gt;', $tempFilePath or die "Cannot open 1: $!\n";
print $tempFile get($eachAddr);
close $tempFile;

open $tempFile, '&lt;', $tempFilePath or die "Cannot open 2: $!\n";
</code></pre>



<pre><code>open my $tempFile, '+&gt;', $tempFilePath or die "Can't open '$tempFilePath' with mode'+&gt;': '$!'";
print $tempFile get($eachAddr);
seek $tempFile, 0, 0;
</code></pre>

<hr>

<p>Instead of getting the text at the given URL twice, and using a weird defined test:</p>

<pre><code>$undefCheck = get($eachAddr);
if(defined($undefCheck) == 0) { next; }
...
print $tempFile get($eachAddr);
</code></pre>



<pre><code>my $text = get( $addr );
next unless defined $text;
...
print $tempFile $text;
</code></pre>

<hr>

<p>Instead of a bunch of:</p>

<pre><code>open ... or die ...;
</code></pre>

<p>I would use <a href="http://p3rl.org/autodie" rel="nofollow">autodie</a>.</p>

<pre><code>use autodie;
...
# will now die on error and will tell you the file it fails on.
open my $fh, '&lt;', $filename;
</code></pre>

<p>Another thing I would like to point out is that <code>die "...\n"</code> prevents <a href="http://perldoc.perl.org/functions/die.html" rel="nofollow"><code>die</code></a> from appending the location of the error. The only time you should do that is if the default behaviour is unhelpful.</p>

<hr>

<p>If you closed <code>$tempFile</code> before checking <code>$dup</code> this could be simpler:</p>

<pre><code>if($dup == 1) { close $tempFile; next; }
close $tempFile;
</code></pre>



<pre><code>close $tempFile;
next if $dup;
</code></pre>

<hr>

<p>Instead of this repetitive block of code:</p>

<pre><code>while(&lt;$iReport&gt;)
{
    chomp;
    if   ($_ =~ /.*\&lt;title&gt;(.*)\&lt;\/title&gt;/) { if($1 ne 'ASF JIRA') { say $oReport "Title   : $1"; } }
    elsif($_ =~ /.*\&lt;assignee username="(.*)"&gt;.*\&lt;\/assignee&gt;/) { say $oReport "Assignee: $1"; }

    elsif($_ =~ /.*\&lt;priority[^&gt;]*&gt;(.*)\&lt;\/priority&gt;/) { say $oReport "Priority: $1"; }
    elsif($_ =~ /.*\&lt;created&gt;(.*)\&lt;\/created&gt;/)        { say $oReport "Created : $1"; }

<p>While this code isn't any shorter, or clearer, it would be <strong>very</strong> easy to add another tag to compare against. So it isn't really <em>better</em>, as less repetitive.<br>
I would like to point out that <code>$_ =~ /.../</code> is better written as <code>/.../</code>.</p>

<p>Perhaps the worst part of the code is that you define almost all of you variables up-front.</p>

<pre><code>my @totalReport;

<p>This means that you are practically using global variables. While some of these appear to need to be defined there, some of them don't, and therefore shouldn't be defined there. You should really be defining your variables at the point you need them, or at least at the beginning of the block where you need them.</p>
