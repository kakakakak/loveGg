<p>There are four high level APIs to access Cassandra and I do not have time to try them all. So I hoped to find somebody who could help me to choose the proper one.</p>

<p>I'll try to write down my findings about them:</p>

<h2>Datanucleus-Cassandra-Plugin</h2>

<p>pros:</p>

<ul>
<li>supports JPA1, JPA2, JDO1 - JDO3 - as I read in a review, JDO scales better than Hibernate with JPA </li>
<li>all the pros as mentioned in kundera?</li>
</ul>

<p>cons:</p>

<ul>
<li>no exeirience with JDO up to now (relevant only for me of course ;)</li>
<li>documentation not found!</li>
</ul>

<h2>kundera</h2>

<p>pros:</p>

<ul>
<li>JPA 1.0 annotations with all advantages (standard conform, no boilerplate code, ...)</li>
<li>promise for following features in near future: JPA listeners, @PrePersist @PostPersist etc. - Relationships, @OneToMany, @ManyToMany etc. - Transactional support, @Transactional</li>
</ul>

<p>cons:</p>

<ul>
<li>early development stage of the plugin?</li>
<li>bugs?</li>
<li>no possibillity to fix problems in the JDO / JPA framework?</li>
</ul>

<h2>s7 pelops</h2>

<p>pros:</p>

<ul>
<li>pure java api --> finer control over persistence?</li>
</ul>

<p>cons:</p>

<ul>
<li>pure java api --> boilerplate code</li>
</ul>

<h2>hector 0.7</h2>

<p>pros:</p>

<ul>
<li>mavenized</li>
<li>spring integration --> dependency injection</li>
<li>pure java api --> finer control over persistence?</li>
<li>jmx monitoring?</li>
<li>managing of nodes seems to be easy and flexible</li>
</ul>

<p>cons:</p>

<ul>
<li>pure java api (no annotations) --> boiler plate code</li>
</ul>

<h2>Conclusion so far</h2>

<p>As I am confident with RDMS, Hibernate, JPA, Spring and not so up to date anymore with EJB, my first impression was, to go for kundera would have been the right choice. But after reading some posts regarding JPO, DataNucleus, I am not sure anymore. As the learning curve should be steep (also for expirienced JPA developers?) for DataNucleus, I am not sure, whether I should go for it.</p>

<p>My major concern is the status of the plugin. Also the forum support/help for JDO and Datanucleus-Cassandra-Plugin, as it is not as wide spread, as far as I understood.</p>

<p>Is anybody out there, who has experience, with some of the framworks already and can give me a hint? Maybe a mixed strategy would make sense as well. In cases (if they exist) JDO is not flexible/sufficient/whatever enough for my needs, to fall back to one of the easier APIs of pelops or hector? Is this possible? Is there an approach like in JPA to get an sql connection and fetch/put data?</p>

<hr>

<p>After reading a bit on, I found following additional information:</p>

<blockquote>
  <p>Datanucleus-Cassandra-Plugin is based on the pelops, which also can be accessed for more flexibility, more performance (?), which should be used on the column families with a lot of data, JDO/JPA access should be only used on "administrative" data, where performance is not so important and data amount is not overwhelming.</p>
</blockquote>

<p>Which still leaves the question open to start with hector or pelops.</p>

<p>pelops for it's later Datanucleus-Cassandra-Plugin extensibility, or
hector for it's more sufficient support on node hanldling.</p>

<p>I tried most of these solutions and find hector the best. Even when you have some problem you can always reach people who wrote hector in #cassandra in freenode. and the code is more mature as far as I concern. In cassandra client the most critical part would be connection pooling management (since all the clients do mostly the same operations through thrift, but connection pooling is what makes high level client roll). In that case I would vote for hector since I am using it in production for over a year now with no visible problem (1 reconnect issue fixed as soon as I discovered and send an email about it). </p>

<p>I am still using cassandra 0.6 though.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>The Hector client was the API that we choose because of the following things that it had:</p>

<ul>
<li>Connection Pooling (huge performance gain when sharing a connection to a node) </li>
<li>Complete Custom Configuration using interfaces for most everything.</li>
<li>Auto Discovery Hosts</li>
<li>Custom Load Balancing Policy definitions (LeastActiveBalancingPolicy or RoundRobinBalancingPolicy or implement LoadBalancingPolicy)</li>
<li>Light-weight adapter on top of the Thrift API.</li>
<li>Great examples: See <a href="https://github.com/zznate/hector-examples" rel="nofollow">hector-examples</a></li>
<li>Built in JMX support.</li>
</ul>

<p>Downside of Hector:</p>

<ul>
<li>Documentation not bad, but the Java Docs are lacking a bit.  That could easily be a Git fork / pull request by the user community.</li>
<li>The ORM support was a bit limited, but not urgent for usage in our case.  I couldn't get some of the one-to-many associations to work easily, plus lack of describing what type of Cassandra model (super columns or column families for associated collections).  Also a lack of Java examples (maybe there are some, please post if you find some).</li>
</ul>

<p>Also, I tried using kundera with very little success.  Not many examples to use or try, very little forum support.  It appears to be maintained by one person, which makes it even hard to choose a tool like that.  It appears based on the SVN activity it was migrating to using Hadoop instead or support for it as well.</p>

<p>I would propose also Astyanax, I'm working with it and I'm quite happy. Only the documentation is not really good.</p>

<blockquote>
  <p><strong>Astyanax API</strong></p>
  
  <p>Astyanax implements a fluent API which guides the caller to narrow or
  customize the query via a set of well defined interfaces. We've also
  included some recipes that will be executed efficiently and as close
  to the low level RPC layer as possible. The client also makes heavy
  use of generics and overloading to almost eliminate the need to
  specify serializers. 
  Some key features of the API include: </p>
  
  <ul>
  <li>Key and column types are defined in a ColumnFamily class which
  eliminates the need to specify serializers. </li>
  <li>Multiple column family key types in the same keyspace. Annotation based composite column names. </li>
  <li>Automatic pagination. </li>
  <li>Parallelized queries that are token aware. </li>
  <li>Configurable consistency level per operation. </li>
  <li>Configurable retry policy per operation. </li>
  <li>Pin operations to specific node. </li>
  <li>Async operations with a single timeout using Futures. </li>
  <li>Simple annotation based object mapping.</li>
  <li>Operation result returns host, latency, attempt count. </li>
  <li>Tracer interfaces to log custom events for operation failure and success.</li>
  <li>Optimized batch mutation. </li>
  <li>Completely hide the clock for the caller, but provide hooks to customize it. </li>
  <li>Simple CQL support. </li>
  <li>RangeBuilders to simplify constructing simple as well as composite column ranges.</li>
  <li>Composite builders to simplify creating composite column names.</li>
  </ul>
  
  <p>Recipes Recipes for some common use cases:</p>
  
  <ul>
  <li>CSV importer.</li>
  <li>JSON exporter to convert any query result to JSON with a wide range of
  customizations. </li>
  <li>Parallel reverse index search. </li>
  <li>Key unique constraint validation.</li>
  </ul>
</blockquote>

<p><a href="http://techblog.netflix.com/2012/01/announcing-astyanax.html" rel="nofollow">http://techblog.netflix.com/2012/01/announcing-astyanax.html</a></p>

<p>You can try <a href="https://github.com/doanduyhai/Achilles/wiki" rel="nofollow">Achilles</a>, a new Entity Manager I've developed that supports all CQL3 features.</p>

<ul>
<li>Entity mapping</li>
<li>JPA style operations </li>
<li>Limited support for join</li>
<li>Mapping of clustered entities using compound primary key</li>
<li>Queries (native, typed, slice)</li>
<li>Support for counters</li>
<li>Support for Consistency level </li>
<li>TTL &amp; timestamp</li>
<li><p>JUnit 4 Rule to start embedded Cassandra server for testing</p>

<p>And so more ...</p>

<p>There are 2 implementations: <strong>Thrift</strong> &amp; <strong>CQL</strong></p>

<p>The <strong>Thrift</strong> version relies on <strong>Hector</strong> under the hood.</p>

<p>The <strong>CQL</strong> version pulls the brand new <strong>Java Driver Core</strong> from Datastax for all operations</p>

<p>Quick reference <a href="https://github.com/doanduyhai/Achilles/wiki/Quick-Reference" rel="nofollow">here</a></p></li>

<p>I would recommend Datastax java driver for Cassandra <a href="http://www.datastax.com" rel="nofollow">http://www.datastax.com</a>.</p>

<p>For JPA like support try my mapping tool.
<a href="https://github.com/valchkou/cassandra-driver-mapping" rel="nofollow">https://github.com/valchkou/cassandra-driver-mapping</a></p>

<p>Annotation driven.<br>
No mapping files, no scripts, no configuration files.<br>
No need for DDL scripts.<br>
Schema automatically synchronized with the entity definition.</p>

<p>Usage sample:</p>

<pre><code>   Entity entity = new Entity();
   mappingSession.save(entity);
   entity = mappingSession.get(Entity.class, id);
   mappingSession.delete(entity); 
</code></pre>

<p>available on maven central</p>

<pre><code>   &lt;dependency&gt;
      &lt;groupId&gt;com.valchkou.datastax&lt;/groupId&gt;
      &lt;artifactId&gt;cassandra-driver-mapping&lt;/artifactId&gt;          
    &lt;/dependency&gt;
