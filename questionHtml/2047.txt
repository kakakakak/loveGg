<td class="answercell">
    <div class="post-text" itemprop="text">
<p>In the most cases NOSQL databases are row oriented, so the are no primary keys. But if you need more flexible design, for example with the Hbase you can use the Apache Hive, which is currently used by the Facebook to extend the sql queries.</p>

<p><strong>You should make a choice in favor of a database that is suitable for you.</strong></p>

<p>My favorite and the fastest is <strong>REDIS</strong> - <a href="http://redis.io/" rel="nofollow">http://redis.io/</a>, written by a legendary antirez in ANSI C. Redis is an in-memory but persistent on disk database, so it represents a different trade off where very high write and read speed is achieved with the limitation of data sets that can't be larger than memory. Radis is very reliable with a proper configuration, but some people are still afraid to use it as the primary database.</p>

<p><strong>Hbase</strong> is good enough, fast, and reliable NOSQL databale, and it became some kind of industry-standard. In addition, there are many tutorials related to Hbase, also you can read mine <a href="http://www.alexbod.com/c-plus-plus-hbase-client/" rel="nofollow">C++ Hbase client</a>. Big disadvantage about the Hbase, that it is written in Java, so you must install java bulk environment first.</p>

<p><strong>Cassandra</strong> also is written in Java. To me the biggest difference is in their consistency models. Cassandra's consistency model is based on Amazon's Dynamo: it provides eventual consistency. This is very appealing for some applications where you want to guarantee the availability of writes. Similarly, Cassandra tends to provide very good write scaling. The tradeoff is that if you want consistent reads, you have to read from a majority of the copies of data, which is inherently slower and does not scale as well. For this reason many users employ a caching layer to accelerate reads.</p>

<p>HBase provides a strong consistency model, and range-baesd partitioning. Reads, including range-based reads, tend to scale much better on HBase, whereas writes do not scale as well as they do on Cassandra.</p>

<p>These are gross generalizations based on the architecture of the two systems. Any product makes tradeoffs. </p>

<p>To me HBase has more commercial companies supporting its development than does Cassandra, but Cassandra seems to be more popular and widely used. Hadoop users seem to prefer HBase over Cassandra.</p>
