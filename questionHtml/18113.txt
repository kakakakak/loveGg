    <div class="post-text" itemprop="text">

<p>I have a quad tree* which I would like to use across several different machines. Calling the constructor (ie building the tree) takes a long time. I don't want to build the tree each time I need to use it.</p>

<p>I am looking for a way to persist my tree in hard disk, send that to each node, and then quickly load it into memory so that I can perform lookups.</p>

<p>Can serialization help me with this? I know that I can serialize the tree, save it to disk then deserialize (that's about all I know about serialization). To the best of my knowledge, the deserilization step needs a default constructor to build the tree. Since building the tree is computationally expensive, this is of no use to me.</p>

<p>Is there some way I can import the persisted tree without having to rebuild it each time?</p>

<p>*quad tree base code that I used: <a href="https://bitbucket.org/utcompling/textgrounder/src/f38150c7f33a/src/main/java/ags/utils/KdTree.java" rel="nofollow">https://bitbucket.org/utcompling/textgrounder/src/f38150c7f33a/src/main/java/ags/utils/KdTree.java</a></p>

<p>Apologies if this is too basic, it's closely related to: <a href="http://stackoverflow.com/questions/14701245/how-do-i-perform-kd-tree-lookups-in-hadoop">How do I perform kd-tree lookups in Hadoop?</a></p>

<p><strong>edit</strong>: So, it looks like the no-arg constructor of the least-inherited class is the one that will be called. Here's my hirearchy:</p>

<p>abstract KDTree</p>

<p>QuadTree extends KdTree</p>

<p>SpecialQuadTree ( has an instance of QuadTree)</p>

<p>Now, I am only able to serialize SpecialQuadTree when I make everything Serializable (otherwise I'm getting NotSerizable or InvalidClassException no valid constructor):</p>

<p>abstract KDTree implements Serializable</p>

<p>QuadTree extends KdTree implements Serializable</p>

<p>SpecialQuadTree implements Serializable ( has an instance of QuadTree)</p>

<p>So, I suppose I can take the Serializable off the KdTree and then give it a do-nothing no-args constructor? That's not easy, I didn't write and don't understand the KdTree class. I suppose I could subclass it, give the subclass a do-nothing no-args constructor, and inherit the QuadTree from that...</p>

<p><strong>edit</strong>: Ok so, I have an abstract class 
abstract KDTree implements Serializable</p>

<p>and from that I extend:
QuadTree extends KdTree implements Serializable</p>

<p>From all my experiments, I need to implement Serializable on both. Now,</p>

<pre><code>public QuadTree() {
    super(2, 1000000);
    System.out.println("QuadTree no-args!");
}
</code></pre>

<p>and</p>

<pre><code>protected KdTree(int dimensions, Integer sizeLimit) {
    System.out.println("KdTree, constructor!");
</code></pre>

<p>Since I never see it print out (aside from the initial construction...), I guess it's all good.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>When an object is deserialized in Java, only some of the no-args constructors are called (see <a href="http://stackoverflow.com/questions/8141440/how-are-constructors-called-during-serialization-and-deserialization">this post</a>).</p>

<p>The easiest way is to make all involved classes <code>Serializable</code>. Then none of your constructors will be called. If you cannot do that, make sure that your <code>Serializable</code> classes <a href="http://docs.oracle.com/javase/6/docs/platform/serialization/spec/serial-arch.html#4539" rel="nofollow">have access to the no-args constructor of the first non-Serializable class</a>.</p>

<p>In case of a non-<code>Serializable</code> base class which is under your control, you could move your expensive logic to a different constructor (let's say, with a dummy <code>boolean</code> argument) and provide a simple no-args constructor that does nothing. Then all your fields will be there after deserialization without the computation overhead.</p>

<p>The ultimate source of information is the <a href="http://docs.oracle.com/javase/6/docs/platform/serialization/spec/serialTOC.html" rel="nofollow">the serialization spec</a>.</p>
