<p>I'm running a large (more than 100 nodes) series of mapreduce jobs on Amazon Elastic MapReduce.</p>

<p>In the reduce phase, already-completed map tasks keep failing with</p>

<p>The proportion of mappers for which this happens is few enough that I wouldn't mind except that when it does, the reducers all pause and wait for the 1 map-task to rerun so the entire job keeps pausing for 1-5 minutes each time.</p>

<p>I think this is related to this bug -> <a href="https://issues.apache.org/jira/browse/MAPREDUCE-2980" rel="nofollow">https://issues.apache.org/jira/browse/MAPREDUCE-2980</a>
Does anyone know how to run an EMR job without this happening?</p>

<p>EDIT:
Here's some more information if it's any help.  The input format is <code>SequenceFileInputFormat</code>.
The output format is a slightly modified version of <code>SequenceFileOutputFormat</code>.
The key-value pairs are user-defined (the value is large and implements <code>Configurable</code>).
There's no <code>Combiner</code>, just <code>Mapper</code> and <code>Reducer</code>.
I'm using  block compression for the input and output (and also there's record compression going on for intermediate kv pairs.  This is default for EMR).  The codec is the default which is <code>SnappyCodec</code> I believe.
Finally, it's actually a series of jobs which are run in sequence, each one using the output of the previous job as the input to the next.
The first couple jobs are small and run fine.  It's only when the jobs start to grow really big that this happens.</p>
