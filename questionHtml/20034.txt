    <div class="post-text" itemprop="text">

<p>I am testing MapReduce wordcount example on Amazon EC2 m1.small instance
I have followed Amazon command line getting started guide.</p>

<p><strong>bin/ec2hadoop launch-cluster test 2</strong></p>

<p>Using this command I have 2 slave nodes. (in total 3 instances running) Then I can login to master node to run the hadoop program (which is a bundled into jar file) It took 35 minutes.</p>

<p>For scalability experiment, then I ran the same program using 4 instances</p>

<p><strong>bin/ec2hadoop launch-cluster test 4</strong></p>

<p>To my surprise, I did not see any gain in performance. The MapReduce application took almost same amount of time (33 minutes)</p>

<p>Where can the problem lie ?
Is it acceptable behaviour ?</p>

<pre><code>In mapred-site.xml
mapred.tasktracker.map.tasks.maximum is set to 1
mapred.tasktracker.reduce.tasks.maximum is set to 1
</code></pre>

<p>Any suggestions are welcome.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>First of all, if properly configured and with growing number of reducers as cluster grows Hadoop should show linear scalability <br>
I think that the root cause of results you get is single reducer. When results of all mappers are passed to the single reducers it limit any performance gains from the cluster size. If you will set number of reducers to 4 (by number of nodes in cluster) you should see the gain. <br>
In addition - I have some doubt about hadoop efficient operation on Small instances. The memory is near limit and swapping can start and kill any performance. In addition - such instance will get very little fraction of the 1GB ethernet and it also can be limiting factor. <br></p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Based on your configuration you have a maximum of 1 map and 1 reduce tasks per node respectively. Depending on the type of job you're running, it might be useful to set these parameters to the number of cores on the node. Especially when the map/reduce tasks are computationally expensive.</p>

<p>In case, as your said in your comment, the reduce phase is dominating the overall performance of the job, you should focus on this part. In Hadoop the number of reduce tasks for a job can be specified in the job's configuration, because it directly affects the number of produced output files.</p>

<p>Having a single reduce task would give you a single output file. Having N reduce tasks would result in N output files. Each of this files contains data sorted by key. Additionally reduce task is ensured to get all data for a given key from the map tasks.</p>

<p>In short words: increasing the number of reduce tasks would improve the overall performance of the job, but produce multiple output files. Depending on your needs, these files would have to be merged and sorted by key either in a second map reduce job (or outside Hadoop) in order to ensure, that you get the same single output file, just like with a single reduce task.</p>
