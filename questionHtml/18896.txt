    <div class="post-text" itemprop="text">

<p>What's a nice, readable way of getting the byte representation (i.e. a <code>byte[]</code>) of an <code>int</code>, but only using 3 bytes (instead of 4)? I'm using Hadoop/Hbase and their <code>Bytes</code> utility class has a <code>toBytes</code> function but that will always use 4 bytes.</p>

<p>Ideally, I'd also like a nice, readable way of encoding to as few bytes as possible, i.e. if the number fits in one byte then only use one.</p>

<p>Please note that I'm storing this in a <code>byte[]</code>, so I know the length of the array and thus variable length encoding is not necessary. This is about finding an elegant way to do the cast.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>A general solution for this is impossible.</p>

<p>If it were possible, you could apply the function iteratively to obtain unlimited compression of data.</p>

<p>Your domain might have some constraints on the integers that allow them to be compressed to 24-bits. If there are such constraints, please explain them in the question.</p>

<p>A common variable size encoding is to use 7 bits of each byte for data, and the high bit as a flag to indicate when the current byte is the last.</p>

<hr>

<p>You can predict the number of bytes needed to encode an <code>int</code> with <a href="http://docs.oracle.com/javase/6/docs/api/java/lang/Integer.html#numberOfLeadingZeros%28int%29" rel="nofollow">a utility method</a> on <code>Integer</code>:</p>

<pre><code>int n = 4 - Integer.numberOfLeadingZeros(x) / 8;
byte[] enc = new byte[n];
while (n-- &gt; 0) 
  enc[n] = (byte) ((x &gt;&gt;&gt; (n * 8)) &amp; 0xFF);
</code></pre>

<p>Note that this will encode 0 as an empty array, and other values in <em>little-endian</em> format. These aspects are easily modified with a few more operations.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>If you need to represent the whole 2^32 existing 4-byte integers, you need to chose between:</p>

<ul>
<li>fixed-size representation, using 4 bytes always; or</li>
<li>variable-size representation, using at least 5 bytes for some numbers.</li>
</ul>

<p>Take a look on how UTF-8 encodes the Unicode charactes, you might get some insights. (you use some short prefix to describe how many bytes must be read for that unicode character, then you read that many bytes and interpret them).</p>

<p>Try using ByteBuffer. You can even set little endian mode if required:</p>

<pre><code>int exampleInt = 0x11FFFFFF;
ByteBuffer buf = ByteBuffer.allocate(Integer.SIZE / Byte.SIZE);
final byte[] threeByteBuffer = new byte[3];
buf.putInt(exampleInt);
buf.position(1);
buf.get(threeByteBuffer);
</code></pre>

<p>Or the shortest signed, Big Endian:</p>

<pre><code>BigInteger bi = BigInteger.valueOf(exampleInt);
final byte[] shortestSigned = bi.toByteArray();
</code></pre>

<p>Convert your <code>int</code> to a 4 <code>byte</code>s array, and iterate it, if every high order byte is zero then remove it from array.</p>

<p>Something like:</p>

<pre><code>byte[] bytes = toBytes(myInt);
int neededBytes = 4;
for (;neededBytes &gt; 1; i--) {
    if (bytes[neededBytes - 1] != 0) {
       break;
    }
}

byte[] result = new byte[neededBytes];
// then just use array copy to copy first neededBytes to result.
</code></pre>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>You can start with something like this:</p>

<pre><code>byte[] Convert(int i)
{  // warning: untested
  if (i == 0)
    return new byte[0];
  if (i &gt; 0 &amp;&amp; i &lt; 256)
    return new byte[]{(byte)i};
  if (i &gt; 0 &amp;&amp; i &lt; 256 * 256)
    return new byte[]{(byte)i, (byte)(i &gt;&gt; 8)};
  if (i &gt; 0 &amp;&amp; i &lt; 256 * 256 * 256)
    return new byte[]{(byte)i, (byte)(i &gt;&gt; 8), (byte)(i &gt;&gt; 16)};
  return new byte[]{(byte)i, (byte)(i &gt;&gt; 8), (byte)(i &gt;&gt; 16), (byte)(i &gt;&gt; 24)};
}
</code></pre>

<p>You'll need to decide if you want to be little-endian or big-endian.  Note that negative numbers are encoded in 4 bytes.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>If i understand right that you really, desperately want to save space, even at expense of arcane bit shuffling: any array type is an unecessary luxury because you cannot use less than one whole byte for the length = addressing space 256 while you know that at most 4 will be needed. So i would reserve 4 bits for the length and sign flag and cram the rest aligned to that number of bytes. You might even save one more byte if your MSB is less than 128. The sign flag i see useful for ability to represent negative numbers in less than 4 bytes too. Better have the bit there every time (even for positive numbers) than overhead of 4 bytes for representing -1.</p>

<p>Anyway, this all is a thin water until you make some statistics on your data set, how many integers are actually compressible and whether the compression overhead is worth the effort.</p>
