    <div class="post-text" itemprop="text">

<p>We have a single-tenant database architecture at the moment with MySQL running upward of 100 databases. We switch database connection on subdomain using the Apartment gem and all is dandy!</p>

<p>However, we have a requirement now to create so-called "Umbrella" clients which can access all the data from a group of our existing clients. I don't see this as immediately feasible with our single-tenant database architecture (I looked into it and querying multiple MySQL databases just seemed hellish), so I'm beginning to look at different implementations with Postgres schemas.</p>

<p>I'm looking for some advice:</p>

<ul>
<li><p>is it possible to query multiple schemas in Postgres and collate the results somehow (looking for Rails implementation)? I can foresee problems with conflicting primary keys?</p></li>
<li><p>Would it be better to have a new schema that is somehow
represents/duplicates all the data in the group of schemas that need
to be accessed? It would need to be realtime.</p></li>
<li><p>If so, can something similar be achieved in my current multiple DB
set-up with MySQL? (to minimise the pain)</p></li>
</ul>

<p>I'm wary of using a database field to achieve multitenancy in MySQL as data security/privacy is a huge thing for this product, and there's so much potential for developer error that way.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>just thinking through this on a high level approach to this problem. </p>

<p>You could create a pg <a href="http://www.postgresql.org/docs/devel/static/sql-createview.html" rel="nofollow">view</a> table to access this data (although it <em>will</em> be slower than accessing the databases themselves). </p>

<p>Then you hopefully have enough unique fields in your table to create a compound or composite key. (Then you wouldn't have to create a new key column, just an index). Because Rails 3 is ORM agnostic, you could then use DataMapper (or maybe the new ROM gem) to establish the connection for this one model.</p>

<p>If you do compound keys, realize that you might have to explicitly define the *to_param* method in your model to build the key as a string. This is for unwrapping the :id when you send it in a url. </p>

<p>You can setup access to this view through a different Postgres user and then use Rails' ability for multiple connections to create a model for it. We did this previously to aggregate data from multiple tables with some restrictions on what was being shown, but I don't see why that wouldn't apply in your use case. </p>

<p>Another option is that perhaps you could use Mongo as the "transient query database". BSON would give you unique keys automatically. And you could create Objects that are essentially SQL Scalar Objects. Not sure you'd want to do a write back to the original database in this case though... but you feasibly could do it.  </p>

<p>Bottom line IMO is I think the best solution lies on the database side of the house because you're using multiple databases. Dealing with these items at the database layer seems the best solution. </p>

<p>Having said all of this... this also seems like a process smell. If I read the problem as stated correctly, I think what you're really trying to do in this case is what <a href="http://hadoop.apache.org/" rel="nofollow">Hadoop</a> is designed for... essentially map/reduce of relevant data (aka Big Data Analysis)</p>

<p>Good luck!</p>

<p>Despite finding many examples of multitenancy in Rails applications when I first set out to do it, I couldn't find one that made me felt completely comfortable either.  But I finally have a solution I'm happy with.</p>

<p>I started with the 'multitenancy with scopes' railscast</p>

<p><a href="http://railscasts.com/episodes/388-multitenancy-with-scopes" rel="nofollow">http://railscasts.com/episodes/388-multitenancy-with-scopes</a></p>

<p>then looked at making multi-tenancy work with devise subdomains using this guide:</p>

<p><a href="https://github.com/plataformatec/devise/wiki/How-To:--Isolate-users-to-log-into-a-single-subdomain" rel="nofollow">https://github.com/plataformatec/devise/wiki/How-To:--Isolate-users-to-log-into-a-single-subdomain</a></p>

<p>But I didn't take that at face value; I dove in to really understand how devise worked that way.</p>

<p>Once I had all that in place, I was ready for the multitenant gem:</p>

<p><a href="https://github.com/wireframe/multitenant" rel="nofollow">https://github.com/wireframe/multitenant</a></p>

<p>But I didn't stop there.  the multitenant gem requires that you say Multitenant.with_tenant whenever you want things scoped appropriatey, so I created a TenantController that looks like this:</p>

<pre><code>  around_filter :scope_current_tenant

  def scope_current_tenant
    begin
      Firm.current = Firm.find_by_subdomain!(request.subdomain)
    rescue
      raise ActionController::RoutingError.new('Not Found')
    end

    Multitenant.with_tenant Firm.current do
      yield
    end

    ensure
      Firm.current = nil
    end
  end
</code></pre>

<p>and then any controller I want to be scoped by tenant inherits from TenantController rather than ApplicationController.  That way I didn't have to remember anything in the details of the controller, it 'just worked'.  the only thing developers had to think about was 'is this a controller that is handling tenant data?'</p>

<p>While this still depends on developers doing a few things correctly (inheriting from the right controller, saying 'acts_as_multitenant' in the model, it works really well in practice.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>OBS: I'm not a Ruby guy, so I can only give you the PG side of the idea.</p>

<p>With PostgreSQL schemas you can easily manage it. Just create a schema for each tenant you have, so in you application, when you need to change the tenant you just do:</p>

<pre><code>SET search_path TO client1;
</code></pre>

<p>With that, when you query for a table, let's say <code>customer</code>, you only need to do on the same connection:</p>

<pre><code>SELECT ... FROM customer ...;
</code></pre>

<p>And, when you need to query for another tenant (not in <code>search_path</code>) you can query the table with the schema:</p>

<pre><code>SELECT ... FROM client2.customer ...;
</code></pre>

<p>And you can use another schema to store public information, like the <code>public</code> schema, just adding it to the end of <code>search_path</code>:</p>

<pre><code>SET search_path TO client1, public;
</code></pre>

<p>Hate to say it since it sounds like you're already quite invested in your current system but it sounds like a job that would be better suited for a NoSQL solution.</p>
