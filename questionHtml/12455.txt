<p>Elasticsearch/Spark serialization does not appear to play well with nested types.</p>

<p>For example:</p>

<pre><code>public class Foo implements Serializable {
   private List&lt;Bar&gt; bars = new ArrayList&lt;Bar&gt;();
   // getters and setters

   public static class Bar implements Serializable {
   }
}

List&lt;Foo&gt; foos = new ArrayList&lt;Foo&gt;();
foos.add( new Foo());
// Note: Foo object does not contain nested Bar instances

SparkConf sc = new SparkConf(); //
sc.setMaster("local");
sc.setAppName("spark.app.name");
sc.set("spark.serializer", KryoSerializer.class.getName()); 
JavaSparkContext jsc = new JavaSparkContext(sc);
JavaRDD javaRDD = jsc.parallelize(ImmutableList.copyOf(foos));
JavaEsSpark.saveToEs(javaRDD, INDEX_NAME+"/"+TYPE_NAME);  
</code></pre>

<p>The above code above works, and documents of type <code>Foo</code> will be indexed within Elasticsearch.</p>

<p>The issue arises when the <code>bars</code> list in a <code>Foo</code> object is not empty, for instance:</p>

<pre><code>Foo = new Foo();
Bar = new Foo.Bar();
foo.getBars().add(bar);
</code></pre>

<p>Then, when indexing to Elasticsearch, the following exception is thrown:</p>

<pre><code>org.elasticsearch.hadoop.serialization.EsHadoopSerializationException: 
Cannot handle type [Bar] within type [class Foo], instance [Bar ...]] 
within instance [Foo@1cf628a] 
using writer [org.elasticsearch.spark.serialization.ScalaValueWriter@4e635d]
at org.elasticsearch.hadoop.serialization.builder.ContentBuilder.value(ContentBuilder.java:63)
at org.elasticsearch.hadoop.serialization.bulk.TemplatedBulk.doWriteObject(TemplatedBulk.java:71)
at org.elasticsearch.hadoop.serialization.bulk.TemplatedBulk.write(TemplatedBulk.java:58)
at org.elasticsearch.hadoop.rest.RestRepository.writeToIndex(RestRepository.java:148)
at org.elasticsearch.spark.rdd.EsRDDWriter.write(EsRDDWriter.scala:47)
at org.elasticsearch.spark.rdd.EsSpark$$anonfun$saveToEs$1.apply(EsSpark.scala:68)
at org.elasticsearch.spark.rdd.EsSpark$$anonfun$saveToEs$1.apply(EsSpark.scala:68)
at org.apache.spark.scheduler.ResultTask.runTask(ResultTask.scala:61)
at org.apache.spark.scheduler.Task.run(Task.scala:64)
at org.apache.spark.executor.Executor$TaskRunner.run(Executor.scala:203)
at java.util.concurrent.ThreadPoolExecutor.runWorker(Unknown Source)
at java.util.concurrent.ThreadPoolExecutor$Worker.run(Unknown Source)
at java.lang.Thread.run(Unknown Source)
</code></pre>

<pre><code>&lt;dependency&gt;
   &lt;groupId&gt;com.sksamuel.elastic4s&lt;/groupId&gt;

&lt;/dependency&gt;

&lt;dependency&gt;

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>One day ScalaValueWriter &amp; JdkValueWriter will possibly support user defined types (like <code>Bar</code> in our example), other than just Java types like String, int, etc.</p>

<p>In the meantime, there is the following workaround. Instead of having <code>Foo</code> expose a List of <code>Bar</code> objects, internally transform the List to a <code>Map&lt;String, Object&gt;</code> and expose that.</p>

<p>Something like this:</p>

<pre><code>private List&lt;Map&lt;String, Object&gt;&gt; bars= new ArrayList&lt;Map&lt;String, Object&gt;&gt;();

public List&lt;Map&lt;String, Object&gt;&gt; getBars() {
   return bars;
}

public void setBars(List&lt;Bar&gt; bars) {
   for (Bar bar: bars){
      this.bars.add(bar.getAsMap());
   }
}
</code></pre>

<p>A solution could be to build a json from the object you're trying to save, using for example <a href="http://json4s.org/" rel="nofollow">Json4s</a>.
In this case your "JavaEsSpark" RDD would be a RDD of strings.
Then you simply have to call</p>

<p><code>JavaEsSpark.saveJsonToEs...</code></p>

<p>instead of</p>

<p><code>JavaEsSpark.saveToEs...</code></p>

<p>This workaround helped me save countless hours trying to figure out a way to Serialize nested maps.</p>
