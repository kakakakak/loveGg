    <div class="post-text" itemprop="text">

<p>I'm about to start a new project which should have a rather large database. </p>

<p>The number of tables will not be large (&lt;15), majority of data (99%) will be contained in one big table, which is almost insert/read only (no updates).</p>

<p>The estimated amount of data in that one table is going to grow at <strong>500.000 records a day</strong>, and we should keep at least <strong>1 year</strong> of them to be able to do various reports.</p>

<p>There needs to be (read-only) <strong>replicated</strong> database as a backup/failover, and maybe for offloading reports in peak time.</p>

<p>I don't have first hand experience with that large databases, so I'm asking the ones that have which DB is the best choice in this situation. I know that <strong>Oracle</strong> is the safe bet, but am more interested if anyone have experience with <strong>Postgresql</strong> or <strong>Mysql</strong> with similar setup.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I've used PostgreSQL in an environment where we're seeing 100K-2M new rows per day, most added to a single table.  However, those rows tend to be reduced to samples and then deleted within a few days, so I can't speak about long-term performance with more than ~100M rows.</p>

<p>I've found that insert performance is quite reasonable, especially if you use the bulk COPY.  Query performance is fine, although the choices the planner makes sometimes puzzle me; particularly when doing JOINs / EXISTS.  Our database requires pretty regular maintenance (VACUUM/ANALYZE) to keep it running smoothly.  I could avoid some of this by more carefully optimizing autovacuum and other settings, and it's not so much of an issue if you're not doing many DELETEs.  Overall, there are some areas where I feel it's more difficult to configure and maintain than it should be.</p>

<p>I have not used Oracle, and MySQL only for small datasets, so I can't compare performance.  But PostgreSQL does <em>work</em> fine for large datasets.</p>

<p>Do you have a copy of "<a href="http://books.google.com/books?id=2OCbq8Azdm8C&amp;dq=data%2Bwarehouse%2Btoolkit&amp;printsec=frontcover&amp;source=bn&amp;hl=en&amp;ei=Xjq2SavALcyatwe-nYm3CQ&amp;sa=X&amp;oi=book%5Fresult&amp;resnum=4&amp;ct=result">The Data Warehouse Toolkit</a>"?  </p>

<p>The suggestion there is to do the following.</p>

<ol>
<li><p>Separate facts (measurable, numeric) values from the dimensions which qualify or organize those facts.  One big table isn't really the best idea.  It's a fact table that dominates the design, plus a number of small dimension tables to allow "slicing and dicing" the facts.</p></li>
<li><p>Keep the facts in simple flat files until you want to do SQL-style reporting.  Don't create and back up a database.  Create and back up files; load a data base only for the reports you must do from SQL.</p></li>
<li><p>Where possible create summary or extra datamarts for analysis.  In some cases, you may need to load the whole thing to a database.  If your files reflect your table design, all databases have bulk loader tools that can populate and index SQL tables from the files.</p></li>
</ol>

<p>Some interesting points regarding Google BigTable in there are...</p>

<p>Bigtable Vs DBMS</p>

<ul>
<li>Fast Query rate</li>
<li><strong>No Joins, No SQL support</strong>, column-oriented database</li>
<li>Uses one Bigtable instead of having many normalized tables</li>
<li>Is not even in 1NF in a traditional view</li>
<li>Designed to support historical queries timestamp field => what did this webpage look like yesterday ?</li>
<li>Data compression is easier –rows are sparse</li>
</ul>

<p>I highlighted the Joins and No SQL Support as you mentioned you will need to run a series of reports.  I dont know how much (if any) not having the abililty to do this will have on you running reports if you where to use this.</p>
