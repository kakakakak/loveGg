    <div class="post-text" itemprop="text">

<p>I have a project that requires the storage of several dozens of <code>TBs</code> of data. At first, few <code>TBs</code> of data will be stored. Then, the data is expected to grow at a pace of <code>5 GB</code> per day. </p>

<p>Based on business logic, I am able to split this across <code>10,000s</code> of files / tables (each file / table will consist of data of size ~ <code>1 [GB]</code>). </p>

<p>(<strong>Note:</strong> at any given point in time, only <code>10-20 [GB]</code> (say, 20 tables) are considered <code>hot data</code> that I'll actively query against.)</p>

<p>Can <a href="http://aws.amazon.com/" rel="nofollow">Amazon Web Services (AWS)</a> be used to store &amp; query such huge amount of data?</p>

<p>If instead I'd like to store this data in-house, what is the best way to approach this 
(for instance, it is clear that I cannot have all of the data on one machine etc...)? </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Lacking information about data format, schema, etc, here's how I'd handle it:</p>

<ul>
<li>store the data in S3 in some Map/Reduce friendly format</li>
<li>use hadoop (or Elastic Map Reduce) to query the entire dataset when needed. Typical pattern is to use M/R to get all the answers you need ahead of time and store the much smaller results in some other database.</li>
<li>If there's really a need to have 20GB of 'hot data' in SQL, I'd probably have some ETL for loading into Amazon RDS (MySQL), but I'd try to avoid it if I could use M/R to answer all my questions and have a small and optimized data set.</li>
</ul>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>No, I do not think. Not for hugh databases.</p>

<p>Those are the areas where you buy special hardware - for example Oracle ExaData (my last data warehouse had 20.000gb and that was small in their terms).</p>

<p>The problem is:</p>

<ul>
<li>Backups. Yes, Amazon sometimes looses data - happened.</li>
<li>Loading hugh amounts of data.</li>
<li>Resources. You likely need a LOT more than even their best virtual machines. Tuning that is a pain if you do not control the hardware.</li>
<li>Cost. Really. Anything up 24/7 is expensive at amazon.</li>
</ul>

<p>If it can be map/recude enabled or something like that - yes. Buta 96 core Oracle RAC install (which is small in Exadata terms) is not something I would like to try on Amazon.</p>

<p>And then they dont ahve the special hardware. Like Oracle ExaData RAID controllers that already filter out unneeded rows (by use of their storage index).</p>

<p>So, I would say: no.</p>
