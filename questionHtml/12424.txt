<p>To add to the answer on Hadoop: there are at least two Scala wrappers that make working with Hadoop more palatable.</p>

<p>Scala Map Reduce (SMR): <a href="http://scala-blogs.org/2008/09/scalable-language-and-scalable.html">http://scala-blogs.org/2008/09/scalable-language-and-scalable.html</a></p>

<p>SHadoop: <a href="http://jonhnny-weslley.blogspot.com/2008/05/shadoop.html">http://jonhnny-weslley.blogspot.com/2008/05/shadoop.html</a></p>

<p><strong>UPD 5 oct. 11</strong></p>

<p>There is also <a href="https://github.com/NICTA/scoobi">Scoobi</a> framework, that has awesome expressiveness.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>For a scala API on top of hadoop check out <a href="http://github.com/NICTA/scoobi" rel="nofollow">Scoobi</a>, it is still in heavy development but shows a lot of promise. There is also some effort to implement distributed collections on top of hadoop in the <a href="http://github.com/scala-incubator/distributed-collections" rel="nofollow">Scala incubator</a>, but that effort is not usable yet.</p>

<p>There is also a new scala wrapper for cascading from Twitter, called <a href="https://dev.twitter.com/blog/scalding" rel="nofollow">Scalding</a>.
After looking very briefly over the documentation for Scalding it seems
that while it makes the integration with cascading smoother it still does
not solve what I see as the main problem with cascading: type safety.
Every operation in cascading operates on cascading's tuples (basically a
list of field values with or without a separate schema), which means that
type errors, I.e. Joining a key as a String and key as a Long leads
to run-time failures.</p>
