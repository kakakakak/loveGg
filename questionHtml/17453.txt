<p><strong>background:</strong></p>

<p>I have 'inherited' a php webapp in my small company and
after years of nagging have finally gotten the
go to throw away the spaghetti code and start again.</p>

<p>We want to log every action that is made in the system for example:</p>

<ul>
<li>user X viewed item Y</li>
<li>user X updated item Y</li>
<li>new item Y on city Z</li>
</ul>

<p>and later provide graphs on different resolutions ( day, month , year )
of the actions done in the system.</p>

<p>On the previous version we have a table with 20,000,000 of records since
2005 so this would give you an idea of the ammount of data we already
have and this is only for one of many statistics.</p>

<p><strong>actual question:</strong></p>

<p>What recomendations you have on building a near to realtime
system to create this statistics?</p>

<p><strong>notes:</strong></p>

<ol>
<li>Graphing is already covered via google's visualization api</li>
<li>Im not adversed of using any NoSql databases or
messaging servers, crons or whatever that gets
the job done, but would prefer mysql/php
solution </li>
<li>My current train of thought is
automatically create a table for each
statistic I want to save, and create
several aggregation tables ( by month , by day , by year )
to cache the results.</li>
<li>I know is a broad question but any suggestions are welcomed</li>
</ol>

    <div class="post-text" itemprop="text">
<p>If all users must be registered I would go with a complete normalized solution.</p>

<pre><code>USERS TABLE            OBJECTS TABLE
---------------        -----------------     
user_id (primary)      object_id (primary)


USERS_TO_OBJECTS TABLE
--------------------
user_id (index)
object_id (index)
time (index)
action (index)
? object_type (index) // could be useful to speed things up
</code></pre>

<p>This setup will probably give you the best flexibility while charting and will also be pretty fast as you can leave out the user or the object if you don't need them.</p>

<p><strong>Edit:</strong></p>

<p>Say city X (id 9876) was updated by user 123 (id 1234)...</p>

<pre><code>1234    - user_id (the user that did the action)
9876    - object_id (the object where the action was done)
xyz     - time
updated - action type (so that you select only specific actions)
city    - object type (so that you select only specific objects)
</code></pre>

<p>I've feed up this table with 40M rows and the results are pretty acceptable.</p>

<p>0.002 secs for a simple COUNT on the number of UPDATED cities in the last WEEK. Data was inserted randomly.</p>

<p><strong>Edit 2</strong></p>

<p>If you find yourself with a really huge table you can resort to MySQL partitions and your schema is perfect. I really don't know how you're gonna use the tables but you could:</p>

<p>PARTITION BY RANGE. Organize the partition on dates. Every new month or so you'd have a new partition.   </p>

<p>PARTITION BY KEY. Organize the partition by actions. Every action goes to it's proper partition.</p>

<p>You can check out <a href="http://dev.mysql.com/tech-resources/articles/mysql_5.1_partitions.html" rel="nofollow">more on partitions on MySQL's site</a> and <a href="http://de.planet.mysql.com/entry/?id=24853" rel="nofollow">this article gives you some detail</a> into fine-tunning partitions.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>You can consider a "database" like Redis. Redis uses linked lists to store list types, so you could easily just LPUSH to a list like so:</p>

<pre><code>$action = array(
    "type"=&gt;"page_view",
    "url"=&gt;"/example/path",
    "user"=&gt;"user1",
    "timestamp"=&gt;$_SERVER["REQUEST_TIME"]
);
$r = new Redis();
// Redis connection info //
$r-&gt;lPush("global_history", json_encode($action));
</code></pre>

<p>The LPUSH operation runs at <code>O(1)</code>, so that shouldn't be an issue. Retrieving results might be a little trickier (<code>O(n)</code> time for lists). Redis is also largely memory-based (though dev builds include "virtual memory" support), so it's fast, but you could run out of room rather easily.</p>

<p>I use this technique to record history and statistics for a music site I run. It's very successful and provides very fast results with very little effort.</p>

<p>If you want to make it more solid, you could use Hadoop or another technology to pull from the end of the list (RPOP) and archive the results into a more permanent format (XML pushed to Amazon S3, for instance). You could then have your Google Visualizations look for their data from the archive (which would be static, compiled data) when viewing older data and the Redis front-end for more recent data.</p>

<p>Hope this helps!</p>
