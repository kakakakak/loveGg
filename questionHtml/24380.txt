    <div class="post-text" itemprop="text">

<p>I have a database with a large table containing more that a hundred million rows. I want to export this data (after some transformation, like joining this table with a few others, cleaning some fields, etc.) and store it int a big text file, for later processing with Hadoop.</p>

<p>So far, I tried two things:</p>

<ol>
<li><p>Using Python, I browse the table by chunks (typically 10'000 records at a time) using <a href="http://explainextended.com/2009/10/23/mysql-order-by-limit-performance-late-row-lookups/" rel="nofollow">this subquery trick</a>, perform the transformation on each row and write directly to a text file. The trick helps, but the <code>LIMIT</code> becomes slower and slower as the export progresses. I have not been able to export the full table with this.</p></li>
<li><p>Using the <code>mysql</code> command-line tool, I tried to output the result of my query in <code>CSV</code> form to a text file directly. Because of the size, it ran out of memory and crashed.</p></li>
</ol>

<p>I am currently investigating <a href="http://sqoop.apache.org/" rel="nofollow">Sqoop</a> as a tool to import the data directly to <code>HDFS</code>, but I was wondering how other people handle such large-scale exports?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Memory issues point towards using the wrong database query machanism.</p>

<p>Normally, it is advisable to use <code>mysql_store_result()</code> on C level, which corresponds to having a <code>Cursor</code> or <code>DictCursor</code> on Python level. This ensures that the database is free again as soon as possible and the client can do with thedata whatever he wants.</p>

<p>But it is not suitable for large amounts of data, as the data is cached in the client process. This can be very memory consuming.</p>

<p>In this case, it may be better to use <code>mysql_use_result()</code> (C) resp. <code>SSCursor</code> / <code>SSDictCursor</code> (Python). This limits you to have to take the whole result set and doing nothing else with the database connection in the meanwhile. But it saves your client process a lot of memory. With the <code>mysql</code> CLI, you would achieve this with the <code>-q</code> argument.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I don't know what query exactly you have used because you have not given it here, but I suppose you're specifying the limit and offset. This are quite quick queries at begin of data, but are going very slow.</p>

<p>If you have unique column such as ID, you can fetch only the first N row, but modify the query clause:</p>

<pre><code>WHERE ID &gt; (last_id)
</code></pre>

<p>This would use index and would be acceptably fast.</p>

<p>However, it should be generally faster to do simply</p>

<pre><code>SELECT * FROM table
</code></pre>

<p>and open cursor for such query, with reasonable big fetch size.</p>
