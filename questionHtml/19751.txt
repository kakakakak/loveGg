<td class="answercell">
    <div class="post-text" itemprop="text">
<p>If you're using SQL Sever 2005, why not use the <code>VARCHAR(MAX)</code> datatype? </p>

<p><code>VARCHAR(MAX)</code> and <code>NVARCHAR(MAX)</code> can hold the same amount of data that the old <code>TEXT</code> and <code>NTEXT</code> could hold (2 GB) but they are much more flexible than <code>TEXT</code>/<code>NTEXT</code>. They are stored in the same type of data pages used for other data types. </p>

<p>As far as what to do with images, my recommendation would be to store them in the filesystem and simply use normal <code>img</code> tags that point to them. </p>

<p>This is fairly typical for CMSes, to store article content in a table which would later be embedded in a template somewhere. Although document databases like MongoDB are increasing in popularity for this type of application. </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Use blob fields in your database (e.g. BIGTEXT in MySQL). You can then stream your HTML content into the database field. Alternatively, if you want to take a documentum type approach, save the HTML content as a file and hold the location of the content in your database. The latter approach is good if you want to use hadoop and the like for searching huge amounts of content. If you are going for something simpler initially I would go with the blob approach as the searching of content in blobs has come on a long way in the last few years. However, be aware that clustered version of MySQL (for example) doesn't have great support for some things you might want such as xpath.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>You should not save images, HTML or anything that won't be used in SQL WHERE statements. Instead, I would save those HTML code fragments and images on filesystem, something named like [PARENT_TABLE_NAME].[ROW_PRIMARY_KEY].html. This way you free up database time when querying since you probably won't use WHERE filters with those HTML fragments, and your HTML updates / deletes / inserts can't possibly be faster AND can be properly cached by OS.</p>
