<p>I have a very large unsorted file, 1000GB, of ID pairs</p>

<ol>
<li>ID:ABC123 ID:ABC124 </li>
<li>ID:ABC123 ID:ABC124 </li>
<li>ID:ABC123 ID:ABA122</li>
<li>ID:ABC124 ID:ABC123 </li>
<li>ID:ABC124 ID:ABC126</li>
</ol>

<p>I would like to filter the file for </p>

<p>1) duplicates</p>

<pre><code>example
ABC123 ABC124
ABC123 ABC124
</code></pre>

<p>2) reverse pairs (discard the second occurrence)</p>

<pre><code>example
ABC123 ABC124
ABC124 ABC123
</code></pre>

<p>After filtering, the example file above would look like</p>

<ol>
<li>ID:ABC123 ID:ABC124 </li>
<li>ID:ABC123 ID:ABA122</li>
<li>ID:ABC124 ID:ABC126</li>
</ol>

<p>Currently, my solution is this</p>

<pre><code>my %hash;

while(my $line = &lt;FH&gt;){
     chomp $line; #remove \n
     my ($id1,$id2) = split / /, $line;
     if(exists $hash{$id1$1d2} || exists $hash{$id2$id1}){
            next;
     }
     else{
         $hash{$id1$id2} = undef ; ## store it in a hash
         print "$line\n";
      }
}
</code></pre>

<p>which gives me the desired results for smaller lists, but takes up too much memory for larger lists, as I am storing the hash in memory.</p>

<p>I am looking for a solution that will take less memory to implement.
Some thoughts I have are</p>

<p>1) save the hash to a file, instead of memory</p>

<p>2) multiple passes over the file</p>

<p>3) sorting and uniquing the file with unix <code>sort -u -k1,2</code></p>

<p>After posting on stack exchange cs, they suggested an external sort algorithm </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>You could use <a href="http://en.wikipedia.org/wiki/MapReduce" rel="nofollow"><strong>map reduce</strong></a> for the tasks.</p>

<p>Map-Reduce is a framework for batch-processing that allows you to easily distribute your work among several machines, and use parallel processing without taking care of synchronization and failure tolerance.</p>

<pre><code>map(id1,id2):
    if id1&lt;id2:
        yield(id1,id2)
   else:
        yield(id2,id1)

reduce(id1,list&lt;ids&gt;):
   ids = hashset(ids) //fairly small per id
   for each id2 in ids:
       yield(id1,id2)
</code></pre>

<p>The map-reduce implementation will allow you to distribute your work on several machines with really little extra programming work required.
<br>This algorithm also requires linear (and fairly small) number of traversals over the data, with fairly small amount of extra memory needed, assuming each ID is associated with a small number of other IDs.</p>

<p>Note that this will alter the order of pairs (make first id second in some cases)
<br>If the order of original ids does matter, you can pretty easily solve it with an extra field.
<br>Also note that the order of data is altered, and there is no way to overcome it when using map-reduce.</p>

<p>For better efficiency, you might want to add a combiner, which will do the same job as the reducer in this case, but if it will actually help depends a lot on the data.</p>

<p><a href="http://hadoop.apache.org/" rel="nofollow">Hadoop</a> is an open source library that implements Map-Reduce, and is widely used in the community.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Depending on the details of your data (see my comment on the question) a <a href="http://en.wikipedia.org/wiki/Bloom_filter" rel="nofollow">Bloom filter</a> may be a simple way to get away with two passes. In the first pass insert every pair into the filter after ordering the first and the second value and generate a set of possible duplicates. In the second pass filter the file using the set of possible duplicates. This obviously requires that the set of (possible) duplicates is not itself large.</p>

<p>Given the characteristics of the data set - up to around 25 billion unique pairs and roughly 64 bit per pair - the result will be on the order of 200 GB. So you either need a lot of memory, many passes or many machines. Even a Bloom filter will have to be huge to yield a acceptable error rate.</p>

<p><a href="http://sortbenchmark.org/" rel="nofollow">sortbenchmark.org</a> can provide some hints on what is required because the task is not to different from sorting. The 2011 winner used 66 nodes with 2 quadcore processors, 24 GiB memory and 16 500 GB disks each and sorted 1,353 GB in 59.2 seconds.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Your approach is almost fine, you just need to move you hashes to disk instead of keeping them in memory. But let's go step by step. </p>

<p><strong>Reorder IDs</strong></p>

<p>It's inconvenient to work with records with different order of IDs in them. So, if possible, reorder IDs, or, if not, create additional keys for each record that holds correct order. I will assume you can reorder IDs (I'm not very good in Bash, so my code will be in Python):</p>

<pre><code>with open('input.txt') as file_in, open('reordered.txt', 'w') as file_out:
    for line in file_in:
        reordered = ' '.join(sorted(line.split(' ')))  # reorder IDs
        file_out.write(reordered + '\n')
</code></pre>

<p><strong>Group records by hash</strong></p>

<p>You cannot filter all records at once, but you can split them into reasonable number of parts. Each part may be uniquely identified by hash of records in it, e.g.: </p>

<pre><code>N_PARTS = 1000
with open('reordered.txt') as file_in:
    for line in file_in: 
        part_id = hash(line) % N_PARTS # part_id will be between 0 and (N_PARTS-1)
        with open('part-%8d.txt' % part_id, 'a') as part_file:
            part_file.write(line + '\n')
</code></pre>

<p>Choice of has function is important here. I used standard Python's <code>hash()</code> (module N_PARTS), but you may need to use another function, that gives distribution of number of records with each hash close to <a href="http://en.wikipedia.org/wiki/Uniform_distribution_%28discrete%29" rel="nofollow">uniform</a>. If hash function work more or less ok, instead of 1 large file of 1Tb you will get 1000 small files of ~100Mb. And the most important thing is that you have a guarantee that there are no 2 same records in different parts. </p>

<p>Note, that opening and closing part files for each line isn't really a good idea, since it generates countless system calls. In fact, better approach would be to keep files open (you may need to increase your <code>ulimit -f</code>), use batching or even write to database - this is up to implementation, while I will keep the code simple for the purpose of demonstration. </p>

<p><strong>Filter each group</strong></p>

<p>100Mb file are much easier to work with, aren't they? You can load them into memory and easily remove duplicates with hash set:</p>

<pre><code>unique = set([])
for i in range(N_PARTS):                          # for each part
    with open('part-%8d.txt') as part_file: 
        file line in part_file:                   # for each line
            unique.add(line)
with open('output.txt', 'w') as file_out:
    for record in unique:
        file_out.write(record + '\n')
</code></pre>

<p>This approach uses some heavy I/O operations and 3 passes, but it is linear in time and uses configurable amount of memory (if your parts are still too large for a single machine, just increase <code>N_PARTS</code>). </p>

<p>So if this were me I'd take the database route as described by @Tom in another answer. I'm using Transact SQL here, but it seems that most of the major SQL databases have similar windowing/ranking <code>row_number()</code> implementations (except MySQL).</p>

<p>I would probably run a two sweep approach, first rewriting the <code>id1</code> and <code>id2</code> columns into a new table so that the "lowest" value is in <code>id1</code> and the highest in <code>id2</code>. </p>

<p>This means that the subsequent task is to find the dupes in this rewritten table.</p>

<p>Initially, you would need to bulk-copy your source data into the database, or generate a whole bunch of <code>insert</code> statements. I've gone for the insert here, but would favour a bulk insert for big data. Different databases have different means of doing the same thing.</p>

<pre><code>CREATE TABLE #TestTable
(
    id int,
    id1 char(6) NOT NULL,
    id2 char(6) NOT NULL
)

insert into 
#TestTable (id, id1, id2) 
values 
    (1, 'ABC123', 'ABC124'),
    (2, 'ABC123', 'ABC124'),
    (3, 'ABC123', 'ABA122'),
    (4, 'ABC124', 'ABC123'),
    (5, 'ABC124', 'ABC126');

select 
    id, 
    (case when id1 &lt;= id2 
        then id1 
        else id2 
    end) id1,
    (case when id1 &lt;= id2 
        then id2 
        else id1 
    end) id2
    into #correctedTable 
from #TestTable

create index idx_id1_id2 on #correctedTable (id1, id2, id)

;with ranked as
(select 
    ROW_NUMBER() over (partition by id1, id2 order by id) dupeRank, 
    id,
    id1,
    id2
 from #correctedTable)

select id, id1, id2 
  from ranked where dupeRank = 1

drop table #correctedTable
drop table #TestTable
</code></pre>

<p>Which gives us the result:</p>

<pre>
3           ABA122 ABC123
1           ABC123 ABC124
5           ABC124 ABC126
</pre>

<p>I'm no trying to answer the question, merely adding my 0.02€ to other answers.</p>

<p>A must-to-do to me is to split the task into multiple smaller tasks as was already suggested. Both the control flow and the data structures.</p>

<p>The way that <a href="http://en.wikipedia.org/wiki/Merge_sort#Use_with_tape_drives" rel="nofollow">Merge Sort was used with Tape Drives</a> to sort big data volumes (larger than memory and larger then random-access-disk). In nowaday terms it would mean that the storage is distributed accross multiple (networked) disks or networked disk-sectors.</p>

<p>There are already languages and even operating systems that support this kind of distribution with different granularity. Some 10 years ago I had my hot candidates for this kind of tasks but I don't remember the names and things had changed since then.</p>

<p>On of the first was the distributed <a href="http://en.wikipedia.org/wiki/Linda_%28coordination_language%29" rel="nofollow">Linda Operating System</a> with parallel processors attached/disconnected as needed. Basic coordination structure was huge distributed <a href="http://en.wikipedia.org/wiki/Tuple_space" rel="nofollow">Tuple Space</a> data structure where processors read/wrote tasks and wrote results.</p>

<p>More recent approach with similar distribution of work are the <a href="http://en.wikipedia.org/wiki/Multi-agent_system" rel="nofollow">Multi agent systems</a> (<a href="http://cs.wikipedia.org/wiki/Multiagentn%C3%AD_syst%C3%A9m" rel="nofollow">Czech Wikipedia article</a> perhaps contains more links)</p>

<p>Related wikipedia article are <a href="http://en.wikipedia.org/wiki/Parallel_computing#Parallel_programming_languages" rel="nofollow">Parallel Computing</a>, <a href="http://en.wikipedia.org/wiki/Supercomputer_operating_systems#Modern_approaches" rel="nofollow">Supercomputer Operating Systems</a> and <a href="http://en.wikipedia.org/wiki/List_of_concurrent_and_parallel_programming_languages" rel="nofollow">List of concurrent and parallel programming languages</a></p>

<p>I don't mean to say that you should buy some processor time on a supercomputer and run the computation there. I'm listing them as algorithmic concepts to study.</p>

<p>As there will be many times some free or open source software solutions available that will allow you to do the same in the small. Starting with cheap software and available hardware. e.g. back at university in 1990 we used the night time in computer lab to calculate <a href="http://en.wikipedia.org/wiki/Ray_trace" rel="nofollow">ray-traced 3D images</a>. It was very computationally expensive process as for every pixel you must cast a "ray" and calculate its collisions with the scene model. On 1 machine with scene with some glasses and mirrors it ran like 1 pixel per second (C++ and optimized assembly language code). At the lab we had some ~15 PCs available. So the final time might be reduced ~15 times (I386, I486 and image of 320x200 256 colors). The image was split into standalone tasks, computed in parallel and them merged into one. The approach scaled well at that time and similar approach would help you also today.</p>

<p>There always was and always will be something like "big data", that big that it does not fit into RAM and it does not fit on disk and it can't be computed on 1 computer in finite time.</p>

<p>Such tasks were solved successfully since the very first days of computing. Terms like B-Tree, Tape drive, Seek time, Fortran, Cobol, IBM AS/400 come from that era. If you're like engineers of those times than you'll for sure come out with something smart :)</p>

<p><strong>EDIT: actually, you are probably looking for <a href="http://en.wikipedia.org/wiki/External_sorting" rel="nofollow">External Sorting</a></strong></p>
