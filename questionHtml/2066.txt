<p>I am trying to create an inverted index for Wikipedia pages however I keep running out of memory. I am not sure what else I can to do ensure it doesn't run out of memory. However we are talking about 3.9Mil words.</p>

<p><code>indexer.java</code></p>

<pre><code>public void index() {
    ArrayList&lt;Page&gt; pages = parse(); // Parse XML pages
    HashMap&lt;String, ArrayList&lt;Integer&gt;&gt; postings = getPostings(pages);
}

public HashMap&lt;String, ArrayList&lt;Integer&gt;&gt; getPostings(ArrayList&lt;Page&gt; pages) {
    assert pages != null;

    englishStemmer stemmer = new englishStemmer();
    HashSet&lt;String&gt; stopWords = getStopWords();
    HashMap&lt;String, ArrayList&lt;Integer&gt;&gt; postings = new HashMap&lt;&gt;();
    int count = 0;
    int artCount = 0;

    for (Page page : pages) {

        if (!page.isRedirect()) { // Skip pages that are redirects.

            StringBuilder sb = new StringBuilder();
            artCount = count; // All the words until now
            boolean ignore = false;

            for (char c : page.getText().toCharArray()) {

                if (c == '&lt;') // Ignore words inside &lt;&gt; tags.
                    ignore = true;

                if (!ignore) {
                    if (c != 39) {
                        if (c &gt; 47 &amp;&amp; c &lt; 58 || c &gt; 96 &amp;&amp; c &lt; 123) // Character c is a number 0-9 or a lower case letter a-z.
                            sb.append(c);

                        else if (c &gt; 64 &amp;&amp; c &lt; 91) // Character c is an uppercase letter A-Z.
                            sb.append(Character.toLowerCase(c));

                        else if (sb.length() &gt; 0) { // Check if there is a word up until now.

                            if (sb.length() &gt; 1) { // Ignore single character "words"

                                if (!stopWords.contains(sb.toString())) { // Check if the word is not a stop word.

                                    stemmer.setCurrent(sb.toString());
                                    stemmer.stem(); // Stem word s

                                    String s = sb.toString(); // Retrieve the stemmed word

                                    if (!postings.containsKey(s)) // Check if the word already exists in the words map.
                                        postings.put(s, new ArrayList&lt;&gt;()); // If the word is not in the map then create an array list for that word.
                                    postings.get(s).add(page.getId()); // Place the id of the page in the word array list.
                                    count++; // Increase the overall word count for the pages
                                }
                            }
                            sb = new StringBuilder();
                        }
                    }
                }

                if (c == '&gt;')
                    ignore = false;
            }
        }
        page.setCount(count - artCount);
    }
    System.out.println("Word count:" + count);
    return postings;
}
</code></pre>

<h1>Advantages</h1>

<p>Some advantages for this approach are:</p>

<ul>
<li>You can get the number of occurrences of a given word simply by getting the size of the associated ArrayList.</li>
<li>Looking up the number of times a given word occurs in a page is relatively easy.</li>
</ul>

<h1>Optimizations</h1>

<p>Current optimizations:</p>

<ul>
<li>Ignoring common words (stop words).</li>
<li>Stemming words to their roots and storing those.</li>
<li>Ignoring common Wikipedia tags that aren't English words (included in stop word list such as: lt, gt, ref .. etc).</li>
<li>Ignoring text within <code>&lt; &gt;</code> tags such as: <code>&lt;pre&gt;, &lt;div&gt;</code></li>
</ul>

<h1>Limitations</h1>

<p>Array lists become incredibly large with number of occurrences for words, the major disadvantage of this approach comes when an array list has to grow. A new array list is created and the items from the previous array list need to be copied into the new array list. This could be a possible performance bottleneck. Would a Linked list make more sense here? As we are simply adding more occurrences and not reading the occurrences. This would also mean that since linked lists do not rely on an array as their underlying data structure they can grow without bounds and do not need to be replaced when they are too large.</p>

<h1>Alternative approaches</h1>

<p>I have considered dumping the counts for each word into a database like MongoDB after each page has been processed and then append the new occurrences. It would be: <code>{word : [occurrences]}</code> and then let the GC clean <code>postings</code> HashMap after each page has been processed.</p>

<p>I've also considered moving the pages loop into the <code>index()</code> method such that GC can clean up <code>getPostings()</code> before a new page. Then merging the new <code>postings</code> after each page but I don't think that will alleviate the memory burden.</p>

<p>As for the hash maps would a tree map be a better fit for this situation?</p>

<h1>Execution</h1>

<p>On my machine this program runs on all 4 cores using 90 - 100% and takes about 2-2.5GB RAM. It runs for over an hour and a half then: <code>GC Out of memory</code>.</p>

<p>I have also considered increasing the available memory for this program but it needs to run on my instructors machine as well. So it needs to operate as standard without any "hacks".</p>

<p>I need help making considerable optimizations, I'm not sure what else would help.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p><strong>TL;DR</strong> Most likely your data structure won't fit in memory, no matter what you do.</p>

<p>Side note: you should actually explain what your task is and what your approach is. You don't do that and expect us to read and poke in your code.</p>

<p>What you're basically doing is building a multimap of word -> ids of Wikipedia articles. For this, you parse each non-redirect page, divide it into single words and build a multimap by adding a word -> page id mapping.</p>

<p>Let's roughly estimate how big that structure would be. Your assumption is around 4 millions of words. There's around 5 millions of articles in EN Wikipedia. Average word length in English is around 5 characters, so let's assume 10 bytes per word, 4 bytes per article id. We're getting around 40 MB for words (keys in the map), 20 MB for article ids (values in the map).<br>
Assuming a multihashmap-like structure you could estimate the hashmap size at around 32*size + 4*capacity.  </p>

<p>So far this seems to be manageable, a few dozen MBs.</p>

<p>But there will be around 4 millions collections to store ids of articles, each will be around 8*size (if you'll take array lists), where the size is a number of articles a word will be encountered in. According to <a href="http://www.wordfrequency.info/" rel="nofollow">http://www.wordfrequency.info/</a>, the top 5000 words are mentioned in COCAE over 300 million times, so I'd expect Wikipedia to be in this range.<br>
That would be around 2.5 GB just for article ids just for 5k top words. This is a good hint that your inverted index structure will probably take too much memory to fit on a single machine.</p>

<p>However I don't think that the you've got problems with the size of the resulting structure. Your code indicates that you load pages in memory first and process them later on. And that definitely won't work.</p>

<p>You'll most probably need to process pages in a stream-like fashion and use some kind of a database to store results. There's basically a thousand ways to do that, I'd personally go with a Hadoop job on AWS with PostgreSQL as the database, utilizing the UPSERT feature.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>ArrayList is a candidate for replacement by a class Index you'll have to write. It should use int[] for storing index values and a reallocation strategy that uses an increment based on the overall growth rate of the word it belongs to. (ArrayList increments by 50% of the old value, and this may not be optimal for rare words.) Also, it should leave room for optimizing the storage of ranges by storing the first index and the negative count of following numbers, e.g.,</p>

<pre><code>..., 100, -3,...   is index values for 100, 101, 102, 103
</code></pre>

<p>This may result in saving entries for frequently occurring words at the cost of a few cycles.</p>

<p>Consider a dump of the postings HashMap after entering a certain number of index values and a continuation with an empty map. If the file is sorted by key, it'll permit a relatively simple merge of two or more files.</p>
