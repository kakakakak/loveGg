<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I've used it for a number of internal and commercial projects with great success.  My warnings:</p>

<ul>
<li>Write lots of unit tests and make sure they ALL pass under Mono -- this will save you a lot of trouble.</li>
<li>Unless you absolutely have to, do NOT use their embedding API.  It's damn easy to use, but it's ungodly easy to garbage collect valid memory or leak all of your memory.</li>
<li>Don't ever, ever, ever even come close to SVN and unless there's no choice, do not compile your own.  Things change so often in SVN that it's highly likely you'll end up implementing something that doesn't work on a release version if your project is significantly large.</li>
<li>Don't try and figure out problems on your own for long, use the IRC channel.  The people there are helpful and you'll save yourself days upon days -- don't make the same mistake I did.</li>
</ul>

<p>Good luck!</p>

<p>Edit:  The reason I say not to compile your own from source (release or SVN) is that it's easy to configure it differently than release binaries and hide bugs, for instance in the garbage collection.</p>

<p>Edit 2:  Forgot to answer the second part to your question.  In my case, I had no issues with porting code, but I wasn't using any MS-specific libraries (WinForms, ASP.NET, etc).  If you're only using System.* stuff, you'll be fine;  beyond that, you may run into issues.  Mono 2.0 is quite solid, though.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I had some experience with Mono. </p>

<p><strong>Pure .NET stuff (like business logic, controllers or algorithms) can be ported without any problems.</strong> Yet, weird things start showing up in the components that interact with operating system, UI, services or persistence. So be prepared for some debugging and hacking.</p>

<p>Things that might help:</p>

<ul>
<li><a href="http://abdullin.com/wiki/component-driven-development.html">Component-Driven Development</a> - so that code is reused by Windows .NET and Mono, while differences are isolated and tested)</li>
<li><a href="http://abdullin.com/wiki/continuous-integration.html">Continuous Integration</a> running and checking everything against Mono and MS.NET, so that possible issues could be discovered as fast as possible (automated deployment and sanity checks are also recommended)</li>
<li>There are not a lot of UI component suites for shell development in Mono.</li>
<li>When a component vendor says that his code is "compatible with Mono", it is not same as "runs on Mono and is supported".</li>
</ul>

<p>Although in the present, there are some <a href="http://mono-project.com/Companies%5FUsing%5FMono">companies going into production with Mono</a>, I'd still wait before rushing in there due to:</p>

<ul>
<li>Lack of decent and commercially supported UI component suites</li>
<li>Issues with efficient garbage collection</li>
<li>Not the best debugging experience (compare with the historical debugger in VS 2010)</li>
</ul>

<p>PS: if there is a company offering fully managed cloud computing solution (not just a VM, but more like Hadoop equivalent for .NET), then I'll be forced to jump in despite these issues.</p>
