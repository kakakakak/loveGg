    <div class="post-text" itemprop="text">
<ol>
<li>You don't have to change anything.</li>
</ol>

<p>The default configuration is done for small environment. You may change it if you grow the environment. Ant there is a lot of params and a lot of time for fine tuning.</p>

<p>But I admit your configuration is smaller than the usual ones for tests.</p>

<ol start="2">
<li>The log you have to read isn't the services ones but the job ones. Find them in /var/log/hadoop-yarn/containers/</li>
</ol>

<p>If you want a better view of your MR, use the web interface on <code>http://127.0.0.1:8088/</code>. You will see your job's progression in real time.</p>

<ol start="3">
<li><p>IMO, Basic tuning = use hadoop web interfaces. There are plenty available natively.</p></li>
<li><p>I think you find your problem. This can be nomal, or not.</p></li>
</ol>

<p>But quickly, YARN launch MR to use all the available memory :</p>

<ul>
<li>Available memory is set in your yarn-site.xml : yarn.nodemanager.resource.memory-mb (default to 8 Gio).</li>
<li>Memory for a task is defined in mapred-site.xml or in the task itself by the property : mapreduce.map.memory.mb (default to 1536 Mio)</li>
</ul>

<p>So :</p>

<ol>
<li>Change the available memory for your nodemanager (to 3Gio, in order to let 1 Gio for the system)</li>
<li>Change the memory available for hadoop services (-Xmx in hadoop-env.sh, yarn-env.sh) (system + each hadoop services (namenode / datanode / ressourcemanager / nodemanager) &lt; 1 Gio.</li>
<li>Change the memory for your map tasks (512 Mio ?). The lesser it is, more task can be executed in the same time.</li>
<li>Change yarn.scheduler.minimum-allocation-mb to 512 in yarn-site.xml to allow mappers with less than 1 Gio of memory.</li>
</ol>

<p>I hope this will help you.</p>
