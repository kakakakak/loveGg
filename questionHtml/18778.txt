    <div class="post-text" itemprop="text">

<p>Can anyone help me understand following observation that is opposite to my understand of Hadoop data locality. <P>
A Hadoop cluster with 3 nodes:<P>
master: 10.28.75.146<P>
slave1: 10.157.6.202<P>
slave2: 10.31.130.224<P>
run a task successfully. From job console:</p>

<pre><code>Task Attempts:attempt_201304030122_0003_m_000000_0
Machine: /default-rack/10.31.130.224&lt;p&gt;
Task log: INFO: consuming hdfs://10.28.75.146:9000/input/22.seq
</code></pre>

<p>We know 224 node is processing /input/22.seq data. By command:</p>

<pre><code>$hadoop fsck /input -files -blocks  -locations |grep -A 1 "22.seq"
/input/22.seq 61731242 bytes, 1 block(s):  OK
0. blk_-8703092405392537739_1175 len=61731242 repl=1 [10.157.6.202:9200]
</code></pre>

<p>22.seq fits in one block which is smaller than default HDFS block size (64MB) and not replicated to other node.  <p></p>

<p>Question: since 22.seq is not local to 224 node, why Hadoop assigns 224 node processing data remotely on 202? </p>

<p>Note: this is not an exception. I notice many data files are fetched remotely, and observe huge network traffic on eth0 at both nodes. I am expecting near-zero traffic between two nodes, since all my data files are &lt;64MB, and data should processed locally. </p>

<p>FYI: This is observed on Amazon's AWS EMR.</p>

    <div class="post-text" itemprop="text">
<p>I am not sure if this will answer your question fully, but I will attempt to shine some light. </p>

<p>The network traffic you encountered above may have been influenced by the process by which the mapreduce framework submits a job; part of which transfers by default 10 copies of your job jar and all libraries contained therein across the cluster (in cases like yours where there are not 10 nodes I am not sure how it would behave): there are heatbeats and getting input split info and reporting progress which seem like small bandwidth operation although I am ignorant about the specifics on their network resource consumption.</p>

<p>Regarding the job you are running: If it is a map only job then Hadoop tries (tries because there may be resource limiting factors running on the data-local node) for data locality optimization and runs the job where the input split is located. It sounds like in your case, the file is less than the default 64MB so 1 split should equal your data which in turn should result in one map since the number is maps is directly proportional to the number of splits you have, but if your job is a Map and Reduce job then the network traffic may be picking up some of the reduce copy and sort phase HTTP network traffic which can end up on separate nodes. </p>

<p><strong>N Input Splits = N Maps --output--> M partitions = M Reducers</strong></p>

<p>Of course the network traffic and data locality optimizations are dependent on the availability of the nodes resources so your test assumptions should take this into consideration.</p>

<p>Hope I was a tiny bit helpful.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Short answer - because Hadoop scheduler sucks. It has no up-front global plan on which file split should go where. As nodes ask for work - it looks at the available splits - and gives out the best match. There are parameters that tune how aggressive Hadoop is in finding a best match (ie. - when a request for work arrives - does it give the best match available at that time? or does it wait for sometime to see if other, better matching nodes also send requests?)</p>

<p>By default (and I am pretty sure this is the case with EMR) - the scheduler would always give back some work to a requesting node - if there was <em>any</em> work available. You can see that if your input is small (spans only a few blocks/nodes), but the number of nodes are larger (in comparison) - then you will get very poor locality. On the other hand - if the size of input is large - then your odds of getting good locality goes up a lot.</p>

<p>The FairScheduler has parameters to delay scheduling - so as to get better locality. However i don't think that is the default scheduler with EMR.</p>
