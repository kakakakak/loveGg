    <div class="post-text" itemprop="text">

<p>I am looking at CouchDB, which has a number of appealing features over relational databases including:</p>

<ul>
<li>intuitive REST/HTTP interface</li>
<li>easy replication</li>
<li>data stored as documents, rather than normalised tables</li>
</ul>

<p>I appreciate that this is not a mature product so should be adopted with caution, but am wondering whether it is actually a viable replacement for an RDBMS (in spite of the intro page saying otherwise - <a href="http://couchdb.apache.org/docs/intro.html">http://couchdb.apache.org/docs/intro.html</a>). </p>

<ol>
<li>Under what circumstances would CouchDB be a better choice of database than an RDBMS (e.g. MySQL), e.g. in terms of scalability, design + development time, reliability and maintenance.</li>
<li>Are there still cases where an RDBMS is still clearly the right choice?</li>
<li>Is this an either-or choice, or is a hybrid solution more likely to emerge as best practice?</li>
</ol>

    <div class="post-text" itemprop="text">
<p>I recently attended the NoSQL conference in London and think I have a better idea now how to answer the original question. I also wrote a <a href="http://andrewwhitehouse.wordpress.com/2010/04/24/nosql-is-it-time-to-ditch-your-relational-database/">blog post</a>, and there are a couple of other <a href="http://blogs.the451group.com/information_management/2010/04/21/user-perspectives-on-nosql/">good</a> <a href="http://www.redmonk.com/jgovernor/2010/04/20/the-guardian-nosql-eu-dont-melt-the-database/">ones</a>.</p>

<p>Key points:</p>

<ul>
<li>We have accumulated probably 30 years knowledge of adminstering relational databases, so shouldn't replace them without careful consideration; non-relational data stores are less mature than relational ones, and so are inherently more risky to adopt</li>
<li>There are different types of non-relational data store; some are key-value stores, some are document stores, some are graph databases</li>
<li>You could use a hybrid approach, e.g. a combination of RDBMS and graph data store for a social software site</li>
<li>Document data stores (e.g. CouchDB and MongoDB) are probably the closest to relational databases and provide a JSON data structure with all the fields presented hierarchically which avoids having to do table joins, and (some might argue) is an improvement on the traditional object-relational mapping that most applications currently use</li>
<li>Non-relational databases support replication (including master-master); relational databases support replication too but it may not be as comprehensive as the non-relational option</li>
<li>Very large sites such as Twitter, Digg and Facebook use Cassandra, which is built from the ground up to support clustering</li>
<li>Relational databases are probably suitable for 90% of cases</li>
</ul>

<p>In summary, consensus seems to be "proceed with caution".</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Until someone gives a more in-depth answer, here are some pros and cons for CouchDB</p>

<p>Pros:</p>

<ul>
<li>you don't need to fit your data into one of those pesky higher-order normal forms</li>
<li>you can change the "schema" of your data at any time</li>
<li>your data will be indexed exactly for your queries, so you will get results in constant time.</li>
</ul>

<p>Cons:</p>

<ul>
<li>you need to create views for each and every query, i.e. ad-hoc like queries (such as concatenating dynamic WHERE's and SORT's in an SQL) queries are not available.</li>
<li>you will either have redundant data, or you will end up implementing join and sort logic yourself on "client-side" (e.g. sorting a many-to-many relationship on multiple fields)</li>
</ul>

<p>Pros or Cons:</p>

<ul>
<li>creating your views are not as straightforward as in SQL, it's more like solving a puzzle. Depends on your type if this is a pro or a con :)</li>
</ul>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>CouchDB is one of several available 'key/value stores', others include oldies like <a href="http://www.oracle.com/technology/products/berkeley-db/index.html" rel="nofollow">BDB</a>, web-oriented ones like <a href="http://www.persvr.org/" rel="nofollow">Persevere</a>, <a href="http://www.mongodb.org/" rel="nofollow">MongoDB</a> and CouchDB, new super-fast like <a href="http://www.danga.com/memcached/" rel="nofollow">memcached</a> (RAM-only) and <a href="http://tokyocabinet.sourceforge.net/" rel="nofollow">Tokyo Cabinet</a>, and huge stores like Hadoop and Google's BigTable (MongoDB also claims to be on this space).</p>

<p>There's certainly space for both key/value stores and relational DBs.  Traditionally, most RDBs are considered a layer above key/value.  For example, MySQL used to use BDB as an optional backend for tables.  In short, key/values know nothing about fields and relationships, which are the foundations of SQL.</p>

<p>Key/value stores typically are easier to scale, which makes them an attractive choice when growing explosively, like Twitter did.  Of course, that means that any relationships between the stored values have to be managed on your code, instead of just declared in SQL.  CouchDB's approach is to store big 'documents' in the value part, making them (mostly) self contained, so you can get most of the needed data in a single query.  Many use cases fit on this idea, others don't.</p>

<p>The current theme I see is that after the "Rails doesn't scale!!" scare, now many people is realizing that it's not about your web framework; but about intelligent cacheing, to avoid hitting the database, and even the webapp when possible.  The rising star there is memcached.</p>

<p>As always, it all depends on your needs.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>This one is a hard question to answer. So I'll try to highlight the areas where CouchDB might work against you.</p>

<p>The two greatest sources of difficulty on the Couch Users and Dev mailing lists that people have are:</p>

<ul>
<li>Complex Joins of Data.</li>
<li>Multi-Step Map/Reduce.</li>
</ul>

<p>Couch Views are pretty much islands unto themselves. If you need to aggregate/merge/intersect a set of views you pretty much have to do so in the application layer for now. There are some tricks you can do with view collation and complex keys to help with joins but these only go so far for some types of data. This may or may not be livable for different applications. That being said many times this problem can reduced or eliminated by structuring your data differently.</p>

<p>The comments of the other folks on this question demonstrate some of the different types of data that are well suited to CouchDB.</p>

<p>One other thing to keep in mind is that a lot of times the data you might need to combine/merge/intersect would be data that you would do offline in an RDBMS database anyway so you might not lose anything by doing the same in CouchDB.</p>

<p>Short Answer: I think eventually CouchDB will be able to handle any kind of problem you want to throw at it. But the comfort level you have using it may differ from developer to developer. It's somewhat subjective I think. I happen to like using a turing complete language to query my data with and keeping more logic in the application layer. Your mileage may vary.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>If you are working with tabular data where there is only a shallow data hierarchy, than an RDBMS system is probably your best choice. This is the main use for RDBMS systems, and the documentation and tool support is very good.</p>

<p>For more nested data like xml, a document database should provide faster access to your data. Also, the storage model more closely resembles that of the data, so retrieval should be more straight forward.</p>
