    <div class="post-text" itemprop="text">

<p>I have a data transformation product, which lets one select tables in a database and transform row data in the source database to a destination database. </p>

<p>This is handled in the current product (java based workbench and engine) by processing 1000 rows at a time and doing it 10 threads parallely. This approach works fine on smaller data sets. But, when I have to transform huge data sets (say about X million records) at one time - this approach still works, but </p>

<ul>
<li>The host machine's CPU on which my product runs, is under heavy load.</li>
<li>The source database and the target database are punched with too many transactions that they start slowing down. (Now, this could be attributed to the fact that the database server is probably running on a slower piece of hardware.)</li>
</ul>

<p>I started looking for solutions and I was quick to address this by requesting a  hardware "beef up" on the source/destination database server machines. This involved, say, buying a new multi-core CPU and some extra RAM. Turns out, upgrading hardware wasn't just the only concern: multiple software licenses for the database were required to be procured - thanks to multi-core processors (per core license).</p>

<p>So, the ball is in my court now, I will have to come up with ways to address this issue, by making changes to my product. And, here is where I need your help. At this moment, I could think of one possible approach to handling huge loads: </p>

<p><strong>Approach1</strong></p>

<ol>
<li>Reading data from source database, persisting it to a temporary medium (file).</li>
<li>Transform data in persisted file, by running it in a distributed environment (cheaper single core machines), there by handling the "trade-off move" of switching to file persistence. (Using something like Apache Hadoop to handle the distributed computing part)</li>
<li>Writing data to a destination database. </li>
</ol>

<p>This is all I could come up with for now, from an architectural perspective. 
Have you handled this situation before? If yes, how did you handle it? 
Appreciate your suggestions and help. </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>There are a couple of things that you could do without increasing your database license cost:</p>

<ul>
<li>Your tool is putting the CPU under a heavy load, assuming that your tool is running on a machine that is not running a database, increase the CPU power on that machine, or if your tool allows it run it on several machines.</li>
<li>One of the reasons that the number of active transactions goes up is that each individual transaction takes time to complete. You can speed this up by optimising your disks or putting in faster disks.</li>
</ul>

<p>Also if you are using insert instead of bulk insert there is a massive improvement potential. The problem with normal insert is that it writes information to logs so that it is possible to rollback the transaction.</p>

<p>In <a href="http://stackoverflow.com/questions/1456086/what-is-the-fastest-way-to-load-an-xml-file-into-mysql-using-c/1473167#1473167">this</a> case I was able to help someone reduce the load time from <strong>10 hours</strong> to <strong>6 minutes</strong> :)</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Divide and conquer!  </p>

<p>If the source DB can't handle both jobs at once (the ETL and "regular" transactions) then don't make it suffer: </p>

<ul>
<li>Copy the source data to a "mirror". </li>
<li>Perform the ETL on the "mirror".</li>
</ul>

<p>NB - when I say "mirror" I simply mean a copy that allows fast and efficient copying of the data (a bit like a "staging" DB) - <strong>not</strong> another big/slow/nasty ETL process.  The idea here is to optimize the process to benefit the source DB.</p>

<p>Then, you can optimize the ETL to the target DB to benefit the target DB; because you've teased the source and target apart it will be easier to optimize the read / insert portions of the overal process.</p>

<p>You could probably do a similar thing at the target end as well (using another "mirror" / staging DB).</p>

<p>This approach isn't that different from what you've suggested but I'm assuming that at straight copy of data between two identical db's of the <strong>same type</strong> will be both the easiest to manage and the most efficient.</p>

<p>After that you can start applying some of the other suggestions that others can put forward.</p>

<p>One last thing - you could experiment with the ETL tool - if you're running </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>The first thing to think about here is if you really need transactions for this amount of data. If the answer is no, your database product likely has a bulk insert option which is made for this kind of large database insertion.</p>

<p>Edit (further to the comments): I think the most bang for your buck (in SQL Server anyway) would be to set the target database to simple recovery mode for the duration of the operation. In fact, if you did that, it is likely that you would not have to make any other code changes.</p>

<p>However, that is only appropriate if the target database is not being used for other things at the same time. I would say that is a basic requirement. It is a fundamental database mistake to try to insert 25 million records into a database while it is live with OLAP transactions. If that is strictly necessary, then I think the solution is to make the process very slow (with significant pauses) so as to free up resources so that the databases can keep running.</p>
