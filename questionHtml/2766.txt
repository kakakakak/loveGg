<td class="answercell">
    <div class="post-text" itemprop="text">
<p>The Problem is the Hostname configuration. If you are using custom hostnames (which are defined only in you /etc/hosts file and not in the dns) then the hadoop can behave in strange ways sometime.</p>

<p>You are using names enode1, enode2 etc as the name of your nodes.</p>

<p>But in the error that you have posted it is showing:</p>

<pre><code>15/04/28 05:45:54 INFO mapreduce.Job: The url to track the job: http://ubuntu:8088/proxy/application_1430180067597_0001/
</code></pre>

<p>Here it is saying that to track the job, the url is <a href="http://ubuntu">http://ubuntu</a>...
It means hadoop is picking up the hostname of the system to do its operations.</p>

<p>Now one of the obvious solution will be go in the /etc/hosts file and add the entry( in each of the node, including master). For example on enode1</p>

<pre><code>192.168.1.231 ubuntu 
</code></pre>

<p>This will work fine when you try to format the namenode and start the cluster.</p>

<p>But if you try to run some job, you wil get into trouble, because the slave will try to connect the resource manager using the address</p>

<pre><code>ubuntu/192.168.1.231
</code></pre>

<p>which means, if you are not able to resolve the ubuntu hostname, use the IP. But the slave is able to resolve the IP which is mapped to its own IP.</p>

<p>For example, when the slave running on the machine enode2 tries to connect to the Resource Manager, it uses <strong>ubuntu/192.168.1.231</strong>. The ubuntu hostname gets resolved to <strong>192.168.1.232</strong> because you just defined this in your /etc/hosts file.</p>

<p>During the execution of the job, in logs you should be able to see the erros:</p>

<pre><code>org.apache.hadoop.ipc.Client: Retrying connect to server
</code></pre>

<p>It really tries to connect to Resource Manager for a long time and <strong>that is why you teragen job is taking so long to execute</strong>. Because the MAP task that are scheduled on the slaves, tries a long time to connect to Resource Manager and eventually fails. Only those MAP tasks, which are scheduled on the master node (as you are using your master also as a slave) will succeed (because the  ubuntu correctly resolved the Resource Manager IP on the master node only).</p>

<p>The solution to this problem is.</p>

<ol>
<li><p>Stop the Hadoop Cluster</p></li>
<li><p>Edit the file /etc/hostname on each of the machines
For example on the machine enode1</p></li>
</ol>

<p>From:</p>

<pre><code>ubuntu
</code></pre>

<p>To:</p>

<pre><code>enode1 
</code></pre>

<p>enode2, enode3 on the corresponding machines.</p>

<ol start="3">
<li>Remove the entry for ubuntu in the /etc/hosts file. </li>
<li>Reboot</li>
<li><p>Make sure the hostname is changed by the command</p>

<p>hostname</p></li>
<li><p>Format the Namdenode </p></li>
<li>Start the cluster and run that teragen again. It should run fine.</li>
</ol>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>try to remove these lines from <code>/etc/hosts</code> and disable you ipv6 if you are not using it :</p>

<pre><code>127.0.0.1       localhost
127.0.0.1       ubuntu
# The following lines are desirable for IPv6 capable hosts
::1     ip6-localhost ip6-loopback
fe00::0 ip6-localnet
ff00::0 ip6-mcastprefix
ff02::1 ip6-allnodes
ff02::2 ip6-allrouters
~   
</code></pre>

<p>One problem with IPv6 is that using 0.0.0.0 for the various networking-related Hadoop configuration options will result in Hadoop binding to the IPv6 addresses. </p>

<p>So if you are not using IPv6, it’s better to disable it because it can cause problems while running Hadoop.</p>

<p>To disable IPv6, open <code>/etc/sysctl.conf</code> in the editor of your choice and add the following lines to the end of the file:</p>

<pre><code># disable ipv6
net.ipv6.conf.all.disable_ipv6 = 1
net.ipv6.conf.default.disable_ipv6 = 1
net.ipv6.conf.lo.disable_ipv6 = 1
</code></pre>

<p>hope this will help you</p>
