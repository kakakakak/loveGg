    <div class="post-text" itemprop="text">
<p>To compress the intermediate output (your map output), you need to set the following properties in your mapred-site.xml:</p>

<pre><code>&lt;property&gt;
    &lt;name&gt;mapred.compress.map.output&lt;/name&gt;
    &lt;value&gt;true&lt;/value&gt;
&lt;/property&gt;

&lt;property&gt;
    &lt;name&gt;mapred.map.output.compression.codec&lt;/name&gt;
    &lt;value&gt;org.apache.hadoop.io.compress.LzoCodec&lt;/value&gt;
&lt;/property&gt;
</code></pre>

<p>If you want to do it on a job per job basis, you could also directly implement that in your code in 1 of the following ways:</p>

<pre><code>conf.set("mapred.compress.map.output", "true")
conf.set("mapred.map.output.compression.codec", "org.apache.hadoop.io.compress.LzoCodec");
</code></pre>

<p>or</p>

<pre><code>jobConf.setMapOutputCompressorClass(LzoCodec.class);
</code></pre>

<p>Also it's worth mentioning that the property <code>mapred.output.compression.type</code> should be left to the default of <code>RECORD</code>, because <code>BLOCK</code> compression for intermediate output causes bad performance.</p>

<p>When choosing what type of compression to use, I think you need to consider 2 aspects:</p>

<ul>
<li><strong>Compression ratio</strong>: how much compression actually occurs. The higher the %, the better the compression.</li>
<li><strong>IO performance</strong>: since compression is an IO intensive operation, different methods of compression have different performance implication.</li>
</ul>

<p>The goal is to balance compression ratio and IO performance, you can have a compression codec with very high compression ratio but poor IO performance.</p>

<p>It's really hard to tell you which one you should use and which one you should not, it also depends on your data, so you should try a few ones and see what makes more sense. In my experience, Snappy and LZO are the most efficient ones. Recently I heard about LZF which sounds like a good candidate too. I found a post proposing a benchmark of compressions <a href="http://blog.erdemagaoglu.com/post/4605524309/lzo-vs-snappy-vs-lzf-vs-zlib-a-comparison-of" rel="nofollow">here</a>, but I would definitely advise to not take that for ground truth and do your own benchmark.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>If you are using Hadoop 0.21 or later, you have to set these properties in your mapred-site.xml:</p>

<pre><code>&lt;property&gt;
  &lt;name&gt;mapreduce.map.output.compress&lt;/name&gt;
  &lt;value&gt;true&lt;/value&gt;
&lt;/property&gt;
&lt;property&gt;
  &lt;name&gt;mapreduce.map.output.compress.codec&lt;/name&gt;
  &lt;value&gt;com.hadoop.compression.lzo.LzoCodec&lt;/value&gt;
&lt;/property&gt;
</code></pre>

<p>And do not forget to restart hadoop after the change. Also make sure that you have both 32-bit and 64-bit liblzo2 installed. For detailed help on how to set this you can refer the following links :</p>

<p><a href="https://github.com/toddlipcon/hadoop-lzo" rel="nofollow">https://github.com/toddlipcon/hadoop-lzo</a></p>

<p><a href="https://code.google.com/a/apache-extras.org/p/hadoop-gpl-compression/wiki/FAQ?redir=1" rel="nofollow">https://code.google.com/a/apache-extras.org/p/hadoop-gpl-compression/wiki/FAQ?redir=1</a></p>

<p>And along with the points made by Charles sir, you should keep 1 more aspect in mind :</p>

<ul>
<li><strong>CPU cycles</strong> : The compression algorithm which you are going to use should consume lesser number of CPU cycles. Otherwise, the cost of compression can void or reverse the speed advantage.</li>
</ul>

<p>Snappy is another option, but it is primarily optimized for <code>64-bit</code> machines. If you are on 32-bit machines better be careful. </p>

<p>Based on the recent progress <code>LZ4</code> also seems good and has been recently integrated into Hadoop. It's fast but has higher memory requirements. You can go <a href="http://fastcompression.blogspot.in/p/lz4.html" rel="nofollow">here</a> to find more on LZ4.</p>

<p>But as Charles sir has said a fair decision can be made only after some experiments.</p>
