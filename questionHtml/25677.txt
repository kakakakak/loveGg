<td class="answercell">
    <div class="post-text" itemprop="text">
<p>You just need to specify number of reducers in your MapReduce
job config. The default partitioner will distribute data to reducers based on hash of key modulus number of specified reducers.</p>

<p>To override behavior of default partitioner, you can implement your own custom partitioner specifying how your data should get across to the reducers.</p>

<p>---Edit to answer questions in the comments section---</p>

<p><strong><em>How can i specify more than one reducer class in the Map-reduce driver</em></strong></p>

<p>To set number of reducers, in job conf you can set it like below -</p>

<pre><code>int numReducers = /*number of reducers you want*/;
job.setNumReduceTasks(numReducers);
</code></pre>

<p><strong><em>Whether I should write four different Jobs for this. Or can I do this with a single Job</em></strong></p>

<p>Hadoop MR jobs are I/O intensive, in your MR job design you should work on minimizing the I/O and parallel processing as much possible.</p>

<p>If your reducers need same input for generating all 4 outputs, it will be better to keep single job, but another consideration can be skewness of data for either output.
For example output1 has more processing time + most of incoming data is likely to be processed for output1.
If you have scenerio like time taken to process output1 is much higher then total time taken to process output2 + output3 + output4, then you should considering splitting processing of output1 in multiple steps.</p>

<p>However if we consider all 4 outouts have more or less equal processing times and consumes same data throughout,
It will be better to have some conditional processing logic in the reducer and let your custom partioner decide which data goes to which reducer.</p>

<p>Your custom partioner can have some check like this incoming data qualifies to be contributing to "GC content" so let it got to Reducer 3.
But if your incoming data needs to be processed for more then one output/distribution use conditional processing and to write multiple output files from same reducer use "MultipleOutputs".
You can google it up and find usage examples, it lets you write output to multiple folders/files at the same time from within a Mapper or Reducer.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p><strong>First Approach</strong></p>

<p>I think you should implement the code in a unique reduce method, and emit n keys depending the process performed. For example: You implement A,B,C and D techiniques, then, in your mapper you could implement this (pseudo-code):</p>

<pre><code>dataA = ProcessA(key,value)
context.write("A", dataA)
dataB = ProcessB(key,value)
context.write("B", dataB)
dataC = ProcessC(key,value)
context.write("C", dataC)
dataD = ProcessD(key,value)
context.write("D", dataD)
</code></pre>

<p>You should be careful about data types of output. Also, the output key could be more complex.</p>

<p><strong>Second Approach</strong></p>

<p>You could generate N MapReduce applications in the same java project, and then you re-use the Map, and develop N reducers.</p>

<p>In <code>job.setReducerClass</code> in each main class you set each Reducer. The Map will be the same.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Hadoop let's you specify the number of reducer tasks from the job driver <strong>job.setNumReduceTasks(num_reducers);</strong>. Since you want four outputs, you would specify <code>int num_reducers = 4;</code> Here's an example driver class.</p>

<pre><code>public class run {

    public static void main(String[] args) throws Exception {

        Configuration conf = new Configuration();
        Job job = new Job(conf, "Run NB Count");

        job.setJarByClass(NB_train_hadoop.class);
        // set mappers, reducers, other stuff
        job.setNumReduceTasks(num_reducers);

        FileInputFormat.addInputPath(job, new Path(args[0]));
        FileOutputFormat.setOutputPath(job, new Path(args[1]));
        System.exit(job.waitForCompletion(true) ? 0 : 1);
    }
}
</code></pre>

<p>While this is handy, you have to understand that there is an optimal number of reducers you can choose which is dependent on the number of nodes in your cluster.</p>

<p>For example, running 4 Amazon m3.xlarge instances (1 master, 3 slaves, and 4 cores an instance), has the following relationship between wall time and number of reducer tasks used in the MapReduce job. You can see that more isn't necessarily better and if you use too many, well then you might as well crunch your data with your mother's hair curler because it would be faster that way.</p>

<p><img src="http://i.stack.imgur.com/O5uxR.png" alt="enter image description here"></p>
