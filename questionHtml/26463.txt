    <div class="post-text" itemprop="text">

<ol>
<li>Assume a key-value table with at least 10s of millions of rows.  </li>
<li>Define an operation that takes a large number of IDs (again, 10s of millions) finds the corresponding values and sums them. </li>
</ol>

<p>Using a database, this operation seems like it can approach <code>(disk seek time) * (number of lookups)</code>. </p>

<p>Using a flat file, and reading through the entire contents, this operation will approach <code>(file size)/(drive transfer rate)</code>.</p>

<p>Plugging in some (rough) values (from wikipedia and/or experimentation):<br>
<code>seek time = 0.5ms</code><br>
<code>transfer rate = 64MByte/s</code><br>
<code>file size = 800M</code>  (for 70 million int/double key/values)<br>
<code>65 million value lookups</code></p>

<p>DB time = <code>0.5ms * 65000000</code> = <code>32500s</code> = <code>9 hours</code><br>
Flat file = <code>800M/(64MB/s)</code> = <code>12s</code></p>

<p>Experimental results are not as bad for MySQL, but the flat file still wins.</p>

<hr/>

<p><strong>Experiments:</strong><br>
Create InnoDB and MyISAM id/value pair tables. e.g.   </p>

<pre><code>CREATE TABLE `ivi` (
  `id` int(11) NOT NULL AUTO_INCREMENT,
  `val` double DEFAULT NULL,
  PRIMARY KEY (`id`)
) ENGINE=InnoDB
</code></pre>

<p>Fill with 32 million rows of data of your choice. Query with:  </p>

<pre><code>select sum(val) from ivm where id not in (1,12,121,1121);  //be sure to change the numbers each time or clear the query cache
</code></pre>

<p>Use the following code to create &amp; read key/value flat file from java.  </p>

<pre><code> private static void writeData() throws IOException {
        long t = -System.currentTimeMillis();
        File dat = new File("/home/mark/dat2");
        if (dat.exists()){
            dat.delete();
        }
        FileOutputStream fos = new FileOutputStream(dat);
        ObjectOutputStream os = new ObjectOutputStream(new BufferedOutputStream(fos));
        for (int i=0; i&lt; 32000000; i++){
            os.writeInt(i);
            os.writeDouble(i / 2.0);
        }
        os.flush();
        os.close();        
        t += System.currentTimeMillis();
        System.out.println("time ms = " + t);
    }
    private static void performSummationQuery() throws IOException{
        long t = -System.currentTimeMillis();

        File dat = new File("/home/mark/dat2");
        FileInputStream fin = new FileInputStream(dat);
        ObjectInputStream in = new ObjectInputStream(new BufferedInputStream(fin));
        HashSet&lt;Integer&gt; set = new HashSet&lt;Integer&gt;(Arrays.asList(11, 101, 1001, 10001, 100001));
        int i;
        double d;
        double sum = 0;
        try {
            while (true){
                i = in.readInt();
                d = in.readDouble();
                if (!set.contains(i)){
                    sum += d;
                }
            }
        } catch (EOFException e) {
        }

        System.out.println("sum = " + sum);
        t += System.currentTimeMillis();
        System.out.println("time ms = " + t);
    }
</code></pre>

<p><strong>RESULTS:</strong></p>

<pre>

InnoDB        8.0-8.1s            
MyISAM        3.1-16.5s
Stored proc   80-90s
FlatFile      1.6-2.4s (even after: echo 3 > /proc/sys/vm/drop_caches) 

</pre>

<p>My experiments have shown that a flat file wins against the database here.   Unfortunately, I sill need to do "standard" CRUD operations on this table. But this is the use pattern that's killing me.</p>

<p><strong>So what's the best way I can have MySQL behave like itself most of the time, yet win over a flat file in the above scenario?</strong></p>

<hr>

<p><strong>EDIT:</strong><br>
To clarify some points:<br>
1. I have dozens such tables, some will have hundreds of millions of rows and I and cannot store them all in RAM.<br>
2. The case I have described is what I need to support. The values associated to an ID might change, and the selection of IDs is ad-hoc. Therefor there is no way to pre-generate &amp; cache any sums. I need to do the work of "find each value and sum them all" every time.</p>

<p>Thanks.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Your numbers assume that MySQL will perform disk I/O 100% of the time while in practice that is rarely the case.  If your MySQL server has enough RAM and your table is indexed appropriately your cache hit rate will rapidly approach 100% and MySQL will perform very little disk I/O as a direct result of your sum operation.  If you are frequently having to deal with calculations across 10,000,000 rows you may also consider adjusting your schema to reflect real-world usage (keeping a "cached" sum on hand isn't always a bad idea depending on your specific needs).</p>

<p>I highly recommend you put together a test database, throw in 10s millions of test rows, and run some real queries in MySQL to determine how the system will perform.  Spending 15 minutes doing this would give you far more accurate information.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Telling MySQL to ignore the primary (and only) index speeds both queries up.</p>

<p>For InnoDB it saves a second the queries. On MyISAM it keeps the query time consistently at the minimum time seen.</p>

<p>The cange is to add  </p>

<pre><code>ignore index(`PRIMARY`)   
</code></pre>

<p>after the tablename in the query.</p>

<p><strong>EDIT:</strong><br>
I appreciate all the input but much of it was of the form "you shouldn't do this", "do something completely different", etc. None of it addressed the question at hand:</p>

<blockquote>
  <p>"So what's the best way I can have
  MySQL behave like itself most of the
  time, yet win over a flat file in the
  above scenario?"</p>
</blockquote>

<p>So far, the solution I have posted: use MyISAM and ignore the index, seems to be closest to flat file performance for this use case, while still giving me a database when I need the database.</p>

<p>I'd use a summary table maintained by triggers which gives sub 1 second performance - something like as follows:</p>

<pre><code>select
 st.tot - v.val 
from
 ivi_sum_total st
join
(
 select sum(val) as val from ivi where id in (1,12,121,1121)
) v;

+---------------------+
| st.tot - v.val      |
+---------------------+
| 1048317638720.78064 |
+---------------------+
1 row in set (0.07 sec)
</code></pre>

<p><strong>Full schema</strong></p>

<pre><code>drop table if exists ivi_sum_total;
create table ivi_sum_total
(
tot decimal(65,5) default 0
) 
engine=innodb;

drop table if exists ivi;
create table ivi 
(
id int unsigned not null auto_increment,
val decimal(65,5) default 0,
primary key (id, val)
) 
engine=innodb;

delimiter #

create trigger ivi_before_ins_trig before insert on ivi
for each row
begin
  update ivi_sum_total set tot = tot + new.val;
end#

create trigger ivi_before_upd_trig before update on ivi
for each row
begin
  update ivi_sum_total set tot = (tot - old.val) + new.val;
end#

-- etc...
</code></pre>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>You are comparing apples and oranges as far as I see. MySQL (or any other relational databases) doesn't suppose work with data which does I/O all the time. then you are destroying the meaning of index. Even worse index would become a burden since it doesn't fit to RAM at all. Thats why people use sharding / summary tables. In you example the size of database (so the disk io) would be much more than flat file since there is a primary index on top of data itself. as z5h stated ignoring primary index can save you some time but it would never be as fast as plain text file.</p>

<p>I would suggest you to use summary tables like having a bg job doing a summary and you UNION this summary table with the rest of the "live" table. But even mysql would not handle rapidly growing data well after some 100s of millions it would start to fail. Thats why people are working for distributed systems like hdfs and map/reduce frameworks like hadoop.</p>

<p>P.S: My technical examples are not 100% right, I just want to go through the concepts.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>There is one option nobody has consider as of yet...</p>

<p>Since the aforementioned JAVA code uses a HashSet, why not use a Hash Index ?</p>

<p>By default, indexes in MyISAM tables use BTREE indexing.<BR>
By default, indexes in MEMORY tables use HASH indexing.<BR></p>

<p>Simply force the MyISAM table to use a HASH index instead of a BTREE</p>

<p><strong>CREATE TABLE `ivi`<BR>
(<BR>
&nbsp;&nbsp;&nbsp;&nbsp;`id` int(11) NOT NULL AUTO_INCREMENT,<BR>
&nbsp;&nbsp;&nbsp;&nbsp;`val` double DEFAULT NULL,<BR>
&nbsp;&nbsp;&nbsp;&nbsp;PRIMARY KEY (`id`) USING HASH<BR>
) ENGINE=MyISAM;<BR></strong></p>

<p>Now that should level the playing field a litte. However, index range searching has poor performance when using a hash index. If you retrieve one id at a time, it should be faster than your previous testing n MyISAM.</p>

<p>If you want to load the data much faster</p>

<ol>
<li>Get rid of the AUTO_INCREMENT property</li>
<li>Get rid of the primary key</li>
<li>Use a regular index</li>
</ol>

<p><strong>CREATE TABLE `ivi`<BR>
(<BR>
&nbsp;&nbsp;&nbsp;&nbsp;`id` int(11) NOT NULL,<BR>
&nbsp;&nbsp;&nbsp;&nbsp;`val` double DEFAULT NULL,<BR>
&nbsp;&nbsp;&nbsp;&nbsp;KEY id (`id`) USING HASH<BR>
) ENGINE=MyISAM;<BR></strong></p>

<p>Then do something like this:</p>

<p>ALTER TABLE ivi DISABLE KEYS;<BR>
...<BR>
... (Load data and manually generate id)<BR>
...<BR>
ALTER TABLE ivi ENABLE KEYS;<BR></p>

<p>This will build the index after it is done being load</p>

<p>You shoudld also consider sizing the key_buffer_size in /etc/my.cnf to handle large numbers of MyISAM keys.</p>

<p>Give it a Try and let us know if this helped and what you found !!!</p>
