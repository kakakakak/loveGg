<p>I'm evaluating Apache Spark to see if it's good platform for the following <strong>requirements</strong>:</p>

<ul>
<li> Cloud computing environment.
<li> Commodity hardware.
<li> Distributed DB (e.g. HBase) with possibly a few petabytes of data.
<li> Lots of simultaneous small computations that need to complete fast (within seconds). Small means 1-100 MBs of data.
<li> A few large computations that don't need to complete fast (hours is fine). Large means 10-1000 GBs of data.
<li> Very rarely, very large computations that don't need to complete fast (days is fine). Very large means 10-100 TBs of data.
<li> All computations are mutually independent.
<li> Real-time data stream incoming for some of the computations.
<li> Machine learning involved.
</ul>

<p>Having read a bit about Spark, I see the following <strong>advantages</strong>:</p>

<ul>
<li> Runs well on commodity hardware and with HBase/Cassandra.
<li> MLlib for machine learning.
<li> Spark Streaming for real-time data.
<li> While MapReduce doesn't seem strictly necessary, maybe it could speed things up, and would let us adapt if the requirements became tighter in the future.
</ul>

<p>These are the main <strong>questions</strong> I still have:</p>

<ul>
<li> Can it do small computations very fast?
<li> Will it load-balance a large number of simultaneous small computations?
</ul>

<p>I also wonder if I'm generally not trying to use Spark for a purpose it wasn't designed for, not using the main advantages: MapReduce and in-memory RDDs. If so, I'd also welcome a suggestion for an alternative. Many thanks!</p>

<h1>Small computations fast</h1>

<p>We do use Spark in an interactive setting, as the backend of a web interface. Sub-second latencies are possible, but not easy. Some tips:</p>

<ul>
<li>Create <code>SparkContext</code> on start up. It takes a few seconds to get connected and get the executors started on the workers.</li>
<li>You mention many simultaneous computations. Instead of each user having their own <code>SparkContext</code> and own set of executors, have just one that everyone can share. In our case multiple users can use the web interface concurrently, but there's only one web server.</li>
<li>Operate on memory cached RDDs. Serialization is probably too slow, so use the default caching, not Tachyon. If you cannot avoid serialization, use Kryo. It is way faster than stock Java serialization.</li>
<li>Use <code>RDD.sample</code> liberally. An unbiased sample is often good enough for interactive exploration.</li>
</ul>

<h1>Load balancing</h1>

<p>Load balancing of operations is a good question. We will have to tackle this as well, but have not done it yet. In the default setup everything is processed in a first-in-first-out manner. Each operation gets the full resources of the cluster and the next operation has to wait. This is fine if each operation is fast, but what if one isn't?</p>

<p>The alternative <a href="http://spark.apache.org/docs/latest/job-scheduling.html#scheduling-within-an-application">fair scheduler</a> likely solves this issue, but I have not tried it yet.</p>

<p>Spark can also off-load scheduling to YARN or Mesos, but I have no experience with this. I doubt they are compatible with your latency requirements.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I think the short answer is "yes". Spark advertises itself as "near real-time". The one or two papers I've read describe throughput latency as either one second or several seconds. For best performance, look at combining it with Tachyon, an in-memory distributed file system.</p>

<p>As for load-balancing, later releases of Spark can use a round-robin scheduler so that both large and small jobs can co-exist in the same cluster:</p>

<p><em>Starting in Spark 0.8, it is also possible to configure fair sharing between jobs. Under fair sharing, Spark assigns tasks between jobs in a “round robin” fashion, so that all jobs get a roughly equal share of cluster resources. This means that short jobs submitted while a long job is running can start receiving resources right away and still get good response times, without waiting for the long job to finish. This mode is best for multi-user settings.</em></p>

<p>However, I am not clear on what you mean by "<em>...also wonder if I'm generally not trying to use Spark for a purpose it wasn't designed for, not using the main advantages: MapReduce and in-memory RDDs.</em>"</p>

<p>I might be wrong, but I don't think you can use Spark without RDDs. Spark <em>is</em> distributed computing with RDDs. What kind of jobs are you trying to run if not MapReduce style jobs? If your use cases are not a good fit for the example use cases that provided with the Spark documentation or tutorials, then what do your use cases fit? Hadoop/Spark shine when there are tons of data, and very little iterative computation on the data. For example, solving systems of equations is not a traditional use case for these technologies. </p>

<p>Is there a need to distribute jobs that only involve 1-100 MB? Such small amounts of data are often processed most quickly on a single powerful node. If there is a reason to distribute the computation, look at running MPI under Mesos. A lot jobs that fall under the name "scientific computing" continue to use MPI as the distributed computing model. </p>

<p>If the jobs are about crunching numbers (e.g. matrix multiplication), then small-medium jobs can be handled quickly with GPU computing on a single node. I've used Nvidia's CUDA programming environment. It rocks for computationally intensive tasks like computer vision.</p>

<p>What is the nature of the jobs that will run in your environment?</p>

<p>If I get it right from the comments you provided, IF you are into statistics and prediction , then MLlib with distributed machine learning algorithms will be very helpful. When your concern comes to non distributed task, it uses breeze library for linear algebra calculations and it is very fast. So you can have both kind of implementation but not in same file. <p>As other answers also pointed out, creating a common spark context is a nice idea.
Something similar we use for testing algorithms using scalaTest</p>

<p><div class="snippet" data-lang="js" data-hide="false">
<div class="snippet-code">
<pre class="snippet-code-html lang-html prettyprint-override"><code>import org.scalatest.Suite
import org.scalatest.BeforeAndAfterAll

import org.apache.spark.{SparkConf, SparkContext}

trait LocalSparkContext extends BeforeAndAfterAll { self: Suite =&gt;
  @transient var sc: SparkContext = _

  override def beforeAll() {
    val conf = new SparkConf()
      .setMaster("local")
      .setAppName("test")
    sc = new SparkContext(conf)
    super.beforeAll()
  }

  override def afterAll() {
    if (sc != null) {
      sc.stop()
    }
    super.afterAll()
  }
}</code></pre>
</div>
</div>
</p>

<p>Something on this line will help you create spark context for many users. </p>
