<p>I use it in python all the time :-).</p>

<p>I'm starting to see that the heterogeneous approach is often the best and it looks like other folks are doing the same.</p>

<p>If you look at projects like protocol-buffers or facebook's thrift you see that sometimes it's just best to use an app written in another language and build the glue in the language of your preference.</p>

<p>I answered your question in my question <a href="http://stackoverflow.com/questions/8257324/net-and-hadoop">here</a> </p>

<p>To say it here in the source:</p>

<p>Microsoft <a href="http://www.zdnet.com/blog/microsoft/microsoft-drops-dryad-puts-its-big-data-bets-on-hadoop/11226">dropped</a> its alternative (<a href="http://research.microsoft.com/en-us/projects/dryad/">Dryad</a>) in favor of Hadoop.
Next year they will release MS SQL Server 2012 with Hadoop integration. Azure and Windows Sever support is being developed even as we speak.</p>

<p>It will be available in the first half in 2012.</p>

<p>Hadoop is the #1 <a href="http://en.wikipedia.org/wiki/Big_data">BigData</a> platform and is going to be supported by opensource and proprietary source (Java, .Net, Python, ...) even Oracle is adopting it.</p>

<p>If you were developing something, you should wait if you're on the .Net platform.</p>

<p>More information about what is possible will be available <a href="http://stackoverflow.com/questions/8257324/net-and-hadoop">here</a></p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I would say that DryadLinq is the closest thing that us .NET folk have to Hadoop.  But it depends what you want to use hadoop for.  If you are looking for the optimized self maintaining distributed file (DFS) system then <a href="http://research.microsoft.com/en-us/projects/dryadlinq/default.aspx" rel="nofollow">DryadLINQ</a> isn't what you are looking for.  It has an analog to the DFS but you have to manually build the partitions and distribute each partition.</p>

<p>That being said, if its the distributed execution aspect of Hadoop that you are looking for than DryadLINQ is truly wonderful (and no, i'm not affiliated with MS).  As long as you have a <a href="http://www.microsoft.com/hpc/" rel="nofollow">Microsoft HPC</a> cluster setup than getting going with DryadLINQ is really easy.  </p>

<p>The code you write is really just straight LINQ code, except instead of executing the LINQ on <code>IEnumerable&lt;T&gt;</code> you have to execute it on <code>PartitionedTable&lt;T&gt;</code> (the self build distributed data structure).</p>

<p>What has really been cool about DryadLINQ is the fast turn around time (try, test, adjust, repeat) when developing algorithms.  You just write LINQ code to do your calculations and DryadLINQ will take care of the whole distributed execution part.  It's the most natural analog I've come across that makes writing code for distributed processing just like writing code for single process processing.</p>
