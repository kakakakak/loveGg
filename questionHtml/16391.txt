    <div class="post-text" itemprop="text">

<p>I have roughly ~600GB of dictionaries I've accumulated over the years, and I decided I want to clean them up and sort them</p>

<p>First of all, each file on average is very large, anywhere from 500MB to 9GB in size. A prerequisite for what I want to do is that I sort each dictionary. My end goal is to entirely remove duplicate words <em>within</em> and <em>throughout</em> <strong>all</strong> dictionary files.</p>

<p>The reason for this is that most of my dictionaries are sorted and organized by categories, but duplicates still often exist.</p>

<pre><code>Load file
     Read each line and put into data structure
     Sort and remove any and all duplicate
Load next file and repeat

Once all files are individually unique, compare against eachother and remove duplicates
</code></pre>

<p><strong>For Dictionaries D{1} to D{N}:</strong></p>

<p>1) Sort <em>D{1}</em> through <em>D{N}</em> individually. </p>

<p>2) Check uniqueness of each word in <em>D{i}</em></p>

<p>3) For each word in <em>D{i}</em>, check ALL words across <em>D{i+1}</em> to <em>D{N}</em>. Delete each word if unique in <em>D{i}</em> first.</p>

<ul>
<li>I am considering using a sort of "hash" to improve this algorithm. Possibly by only checking the first one or two characters, since the list will be sorted (e.g. hash beginning line location for words starting with a, b, etc.).</li>
</ul>

<p>4) Save and exit. </p>

<p>Example before (but far smaller):</p>

<pre><code>    Dictionary 1            Dictionary 2            Dictionary 3

    ]a                      0u3TGNdB                2 KLOCK
    all                     avisskriveri            4BZ32nKEMiqEaT7z
    ast                     chorion                 4BZ5
    astn                    chowders                bebotch
    apiala                  chroma                  bebotch
    apiales                 louts                   bebotch
    avisskriveri            lowlander               chorion
    avisskriverier          namely                  PC-Based
    avisskriverierne        silking                 PC-Based
    avisskriving            underwater              PC-Based
</code></pre>

<p>So it would see avisskriveri, chorion, bebotch and PC-Based are words that repeate both within and among each of the three dictionaries. So I see avisskriveri in <em>D{1}</em> first, so remove it in all other instances that I have seen it in. Then I see chorion in <em>D{2}</em> first, and remove that in all other instances first, and so forth. In <em>D{3}</em> bebotch and PC-Based are replicated, so I want to delete all but one entry of it (unless I've seen it before). Then save all files and close.</p>

<p>Example after:</p>

<pre><code>     Dictionary 1           Dictionary 2            Dictionary 3

     ]a                     0u3TGNdB                2 KLOCK
     all                    chorion                 4BZ32nKEMiqEaT7z
     ast                    chowders                4BZ5
     astn                   chroma                  bebotch
     apiala                 louts                   PC-Based
     apiales                lowlander                   
     avisskriveri           namely              
     avisskriverier         silking                 
     avisskriverierne       underwater                          
     avisskriving 
</code></pre>

<p><strong>Remember:</strong> I do NOT want to create any new dictionaries, only remove duplicates across all dictionaries.</p>

<p><strong>Options:</strong></p>

<ul>
<li><p>"Hash" the amount of unique words for each file, allowing the program to estimate the computation time.</p></li>
<li><p>Specify a way give the location of the first word beginning with the desired first letter. So that the search may "jump" to a line and skip unecessary computational time.</p></li>
<li><p>Run on GPU for high performance parallel computing. (This is an issue because getting the data off of the GPU is tricky)</p></li>
</ul>

<p><strong>Goal:</strong> Reduce computational time and space consumption so that the method is affordable on a standard machine or server with limited abilities. Or device a method for running it remotely on a GPU cluster.</p>

<p>tl;dr - Sorting unique words across hundreds of files, where each file is 1-9GB in size.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Assuming the dictionaries are in alphabetical order and line by line, one word per line (as are most dictionaries), you could do something like this:</p>

<pre><code>Open a file stream to each file.
Open a file stream to the compiled list file.
Read 1 entry from each file and put it onto a heap, priority queue, or other sorted data structure.
while you still have entries
    find &amp; remove the first entry, storing the word (it is not necessary to store the file)
    read in the next entry from that file, if one exists
    find &amp; remove any duplicates of the stored entry
    read in the next entry for each of those files, if one exists
    write the stored word to your compiled list file
Close all of the streams
</code></pre>

<p>The efficiency of this is something like O(n*m*log(n)) and the space efficiency is O(n), where n is the number of files and m is the average number of entries.</p>

<p>Note that you'll want to create a data type that pairs entries (strings) with file pointers/references, and sorts by string storing. You'll also need a data structure that allows you to peek before you pop.</p>

<p>If you have questions in implementation, ask me.</p>

<p>A more thorough analysis of the efficiency:</p>

<p>Space efficiency is pretty easy. You fill the data structure, and for every item you put on, you take one off, so it stays at O(n).</p>

<p>Computational efficiency is more complex. The looping itself is O(n*m), because you will consider each entry, and there are n*m entries. Some c percent of those will be valid, but that's a constant, so we don't care.</p>

<p>Next, adding and removing from a priority queue is log(n) both ways, so to find &amp; remove is 2*log(n).</p>

<p>Because we add and remove each entry, we get n*m add and removes, so O(n*m*log(n)). I think it might actually be a theta in this case, but meh.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>As far as I understand, there is no pattern to exploit in a clever way. So we want to do raw sorting.</p>

<p>Let us assume that no cluster farm is available (we could do other things then)</p>

<p>Then I would start with the easiest approach possible, the command line tool <code>sort</code>:</p>

<blockquote>
  <p>sort -u inp1 inp2 -o sorted</p>
</blockquote>

<p>This will sort <code>inp1</code> and <code>inp2</code> together in output file <code>sorted</code> without duplicates (u = unique). Sort typically uses a customized mergesort algorithm, which can handle a limited amount of memory. So you should not run in memory problems.<br>
You should have at least 600 gb (double the size) of free disk space.<br>
You should test with only 2 input files how long it takes and what happens. My tests did not show any problems, but they had used different data and an afs server (which is rather slow, but is a better emulation as some HPC filesystem provider):</p>

<pre><code>$ ll
2147483646 big1
2147483646 big2

$ time sort -u big1 big2 -o bigsorted
1009.674u 6.290s 28:01.63 60.4% 0+0k 0+0io 0pf+0w

$ ll
2147483646 big1
2147483646 big2
 117440512 bigsorted
</code></pre>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I'd start with something like:</p>

<pre><code>#include &lt;string&gt;
#include &lt;set&gt;

int main()
{
    typedef std::set&lt;string&gt; Words;
    Words words;
    std::string word;
    while (std::cin &gt;&gt; word)
        words.insert(word);  // will only work if not seen before
    for (Words::const_iterator i = words.begin(); i != words.end(); ++i)
        std::cout &lt;&lt; *i;
}
</code></pre>

<p>Then just:</p>

<pre><code>cat file1 file2... | ./this_wonderful_program &gt; greatest_dictionary.txt
</code></pre>

<p>Should be fine assuming the number of non-duplicate words fits in memory (likely on any modern PC, especially if you've 64 bits and > 4GB), this will probably be I/O bound anyway so no point fussing over unordered map vs (binary-tree) map etc..  You may want to convert to lower-case, strip spurious characters etc. before inserting to the map.</p>

<p>EDIT:</p>

<p>If the unique words don't fit in memory, or you're just stubbornly determined to sort each individual input then merge them, you can use the unix <code>sort</code> command on each file, then <code>sort -m</code> to efficiently merge the pre-sorted files.  If you're not on UNIX/Linux, you can probably still find a port of <code>sort</code> (e.g. from Cygwin for Windows), your OS may have an equivalent program, or you could try compiling the <code>sort</code> source code.  Note that this approach is a little different from tb-'s suggestion of asking one invocation of <code>sort</code> to sort everything (presumably in memory) - I'm not sure how well that would work, so best to try/compare.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>On that that scale of 300GB+, you may want to consider using <a href="http://hadoop.apache.org/" rel="nofollow">Hadoop</a> or some other scalable store - otherwise, you will have to deal with memory issues through your own coding.  You can try other, more direct methods (UNIX scripting, small C/C++ programs, etc...), but you will likely run out of memory unless you have a ton of duplicate words in your data.</p>

<p><strong>Addendum</strong></p>

<p>Just came across <a href="http://en.wikipedia.org/wiki/Memcached" rel="nofollow">memcached</a> which seems very close to what you are trying to accomplish: but you may have to tweak it <strong>not</strong> to throw away the oldest values.  I don't have time to check right now, but you should do a search on <strong><a href="http://en.wikipedia.org/wiki/Distributed_hash_tables" rel="nofollow">Distributed Hash Tables</a></strong>.</p>
