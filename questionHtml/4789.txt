<p>I went through all the resources (almost) on google and I did not get something regarding Hadoop and NoSQL.</p>

<p>Let's imagine that I have lot of data to store. I use Hadoop and it's native HD File System BUT I also want to get real-time informations, so I need NoSQL.
Where do my database going to be installed ? On the datanode itself ? On the namenode ? Both ?</p>

<p>Let's imagine (once again) that I have data coming in my system, the namenode going to cut it into several pieces and replicate them on different datanode. With NoSQL, will it work the same way ? Does HDFS take a part in this process ?</p>

<p>The metadatas stored in the NameNode give the adress, so how is it going to be queried ? </p>

<p>I think I understand the concept of Hadoop and HBase basically but I misunderstand when I go further..</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p><strong>Overlapping capabilities of Hadoop and NoSQL</strong></p>

<p>Both hadoop and NoSQL are great for managing large and rapidly growing data sets.
They both can handle a variety of data formats including log files, documents and rich media. Also both can leverage commodity hardware and support horizontal scaling. if you have structured data in which the structure differs between records, or if the structure likely will change in the future, then both NoSQL and Hadoop are appropriate technologies for your usecase.</p>

<p><strong>Both the technologies are intended for different type of workloads</strong></p>

<p>NoSQL is meant for real time access and includes both read/write processes. While hadoop is about large scale data processing.</p>

<p><strong>Both can co-exist in an EDW</strong></p>

<p>NoSQL and Hadoop can exist/work/deployed together in an enterprise data architecture. In a typical architecture, NoSQL can be used for real time &amp; interactive data, and Hadoop cluster can be used for large-scale data processing(Batch Mode) and analytics.</p>

<p><strong>Role of Hbase</strong></p>

<p>Now HBase is a column based NoSQL database that runs on top of Hadoop. It combines the scalability of Hadoop by running on HDFS, with real-time data access as a key/value store and deep analytic capabilities of Map-Reduce.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>HDFS is a distributed file system (DFS). It allows you to view disks of multiple machines (data nodes) as a huge single disk. In order to read and write files from HDFS you need a client application that talks over network to name node (for metadata about files and directories) and to data nodes (for the actual file data).</p>

<p>HBase is a distributed key-value store that uses HDFS to store its data. To achieve the best performance out of HBase you need to run HBase nodes on the same servers as you run HDFS data nodes to avoid unnecessary network calls.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Regarding your queries</p>

<p>1) Data will be stored in datanodes (RegionServer). HBase need to be installed on data nodes</p>

<p>2) HBase is not required on namenode.</p>

<p>Have a look at : <a href="http://stackoverflow.com/questions/13741946/role-of-datanode-regionserver-in-hbase-hadoop-integration">Role of datanode, regionserver in Hbase-hadoop integration</a> and <a href="http://stackoverflow.com/questions/27796591/should-the-hbase-region-server-and-hadoop-data-node-on-the-same-machine">Should the HBase region server and Hadoop data node on the same machine?</a> quesitons too.</p>

<p><strong><em>HOW HBASE WORKS?</em></strong></p>

<p>Since you know Hadoop &amp; HBase individually, I am not explaining both of them in details. I will explain how HBase works with HDFS/Hadoop eco system ( Formatted and edited for readability of <a href="http://blog.cloudera.com/blog/2013/04/how-scaling-really-works-in-apache-hbase/" rel="nofollow">Edureka article</a>)</p>

<p><strong>HBase</strong> is the Hadoop storage manager that provides low-latency random reads and writes on top of HDFS. It can handle peta bytes of data.</p>

<p><code>HBase</code> uses auto-sharding feature, which implies large tables are dynamically distributed by the system.</p>

<p>The basic unit of horizontal scalability in <code>HBase</code> is called a <strong>Region</strong>. Regions are a subset of the table’s data and they are essentially a contiguous, sorted range of rows.</p>

<p>Initially, there is only one region for a table. When regions become too large after adding more rows, the region is split into two at the middle key, creating two roughly equal halves.</p>

<p><a href="http://i.stack.imgur.com/NLfTL.png" rel="nofollow"><img src="http://i.stack.imgur.com/NLfTL.png" alt="enter image description here"></a></p>

<p>In <code>HBase</code> the slaves are called <strong>Region Servers</strong>. Each Region Server is responsible to serve a set of regions, and one Region (i.e. range of rows) can be served only by one Region Server.</p>

<p>The <code>HBase</code> architecture has two main services: <strong>HMaster</strong> : which is responsible to coordinate the cluster and execute administrative operations, AND the <strong>HRegionServer</strong> responsible for handling a subset of the table’s data.</p>

<p><strong>HBase Master</strong> coordinates the HBase Cluster and is responsible for administrative operations.</p>

<p>A <strong>Region Server</strong> can serve one or more Regions. Each Region is assigned to a Region Server on startup. The master can decide to move a Region from one Region Server to another when there is a need of load balancing. The Master also handles Region Server failures by assigning the region to another Region Server.</p>

<p>The mapping of <strong>Regions and Region Servers</strong> is kept in a system table called <strong>META</strong>. By reading META, you can identify which region is responsible for your key. This means that for read and write operations, the master is not involved at all and clients can go directly to the Region Server responsible to serve the requested data.</p>

<p>To identify the <strong>Region Server</strong>, the client does a query on the <strong>META</strong> table.</p>

<p><strong>META</strong> is a system table used to keep track of regions. It contains the server name and a region identifier comprising a table name and the start row-key. By looking at the start-key and the next region start-key, clients are able to identify the range of rows contained in a a particular region.</p>

<p>If you need low latency for real time data access (less than 10 TB) use <strong>SPARK</strong>. Hadoop is good for batch processing of large data. <strong>SPARK + HBASE</strong> is good combination for your requirements.</p>

<p>Have a look at this <a href="http://blog.cloudera.com/blog/2015/08/apache-spark-comes-to-apache-hbase-with-hbase-spark-module/" rel="nofollow">cloudera</a> article regarding <strong>HBase-Spark</strong> Module</p>
