    <div class="post-text" itemprop="text">

<p>I have around 14000 small .gz files (~from 90kb to 4mb) which are loaded into HDFS all in the same directory.</p>

<p>So the size of each of them is far away from the standard 64mb or 128mb block size of HDFS, which can lead to serious trouble (the "small files problem", see <a href="http://blog.cloudera.com/blog/2009/02/the-small-files-problem/" rel="nofollow">this</a> blog post by cloudera) when running MR jobs which process these files.</p>

<p>The aforementioned blog post contains a number of solutions to this problem, which mostly involve writing a MapReduce Job or using Hadoop Archives (HAR).</p>

<p>However, I would like to tackle the problem at the source and merge the small files into 64mb or 128mb .gz files which will then be fed directly into HDFS.</p>

<p>What's the simplest way of doing this? </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>After experimenting a bit further, the following two steps do what I want:</p>

<pre><code>zcat small-*.gz | split -d -l2000000 -a 3 - large_
</code></pre>

<p>This works in my case, because there is very little variance in the length of a line.
2000000 lines correspond to almost exactly 300Mb files. 
Unfortunately, for some reason, gzip cannot be piped like this, so I have to do another step:</p>

<pre><code>gzip *
</code></pre>

<p>This will then also compress the generated large files.
Gzip compresses each of these files by a factor of ~5, leading to 60mb files and thus satisfying my initial constraint of receiving .gz files &lt; 64mb.  </p>
