    <div class="post-text" itemprop="text">

<p>there are many files need to process with two computers real-timely,I want to distribute them to the two computers and these tasks need to be completed as soon as possibile(means real-time processing),I am thinking about the below plan:</p>

<p>(1) distributed queue like Gearman</p>

<p>(2）distributed computing platform like hadoop/spark/storm/s4 and so on</p>

<p>I have two questions</p>

<p>（1）what is the advantage and disadvantage between (1) and (2)?</p>

<p>（2) How to choose in (2),hadoop?spark?storm?s4?or other?</p>

<p>thanks!</p>

<p>Maybe I have not described the question clearly. In most case,there are 1000-3000 files with the same format , these files are independent,you do not need to care their order,the size of one file maybe tens to hundreds of KB and in the future, the number of files and size of single file will rise. I have wrote a program , it can process the file and pick up the data and then store the data in mongodb. Now there are only two computers, I just want a solution that can process these files with the program quickly（as soon as possibile） and is easy to extend and maintain</p>

<p>distributed queue is easy to use in my case bur maybe hard to extend and maintain , hadoop/spark is to "big" in the two computers but easy to extend and maintain, which is better, i am confused.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>It depends a lot on the nature of your "processing". Some dimensions that apply here are: </p>

<ol>
<li>Are records independent from each other or you need some form of aggregation? i.e: do you need some pieces of data to go together? Say, all transactions from a single user account.</li>
<li>Is you processing CPU bound? Memory bound? FileSystem bound?</li>
<li>What will be persisted? How will you persist it?</li>
<li>Whenever you see new data, do you need to recompute any of the old?</li>
<li>Can you discard data?</li>
<li>Is the data somewhat ordered?</li>
<li>What is the expected load?</li>
</ol>

<p>A good solution will depend on answers to these (and possibly others I'm forgetting). For instance:</p>

<ol>
<li>If computation is simple but storage and retrieval is the main concern, you should maybe look into a distributed DB rather than either of your choices. </li>
<li>It could be that you are best served by just logging things into a distributed filesystem like HDFS and then run batch computations with Spark (should be generally better than plain hadoop). </li>
<li>Maybe not, and you can use Spark Streaming to process as you receive the data.</li>
<li>If order and consistency are important, you might be better served by a publish/subscribe architecture, especially if your load could be more than what your two servers can handle, but there are peak and slow hours where your workers can catch up.</li>
</ol>

<p>etc. So the answer to "how you choose?" is "by carefully looking at the constraints of your particular problem, estimate the load demands to your system and picking the solution that better matches those". All of these solutions and frameworks dominate the others, that's why they are all alive and kicking. The choice is all in the tradeoffs you are willing/able to make.</p>

<p>Hope it helps.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>First of all, dannyhow is right - this is not what real-time processing is about. There is a great book <a href="http://www.manning.com/marz/" rel="nofollow">http://www.manning.com/marz/</a> which says a lot about lambda archtecture. </p>

<p>The two ways you mentioned serves completly different purposes and are connected to the definition of word "task". For example, Spark will take a whole job you got for him and divide it into "tasks", but the outcome of one task is useless for you, you still need to wait for whole job to finish. You can create small jobs working on the same dataset and use spark's caching to speed it up. But then you won't get much advantage from distribution (if they have to be run one after another).</p>

<p>Are the files big? Are there connected somehow to each other? If yes, I'd go with Spark. If no, distributed queue.</p>
