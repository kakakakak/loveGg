<p>Hi recently I need to handle market data by importing it from different kind of exchange and analysis it</p>

<p>The market table is pretty simple it consist of below column</p>

<pre><code>market_name varchar(45) =&gt; this will be exchange name
market_type varchar(5) =&gt; buy sell bids of asks 
currency varchar(8) =&gt; cur1_cur2 e.g usd_eur
volume decimal(30,10) 
price decimal(30,10)
import_time int =&gt; unix time
</code></pre>

<p>Due to some limitation in protocol of getting the data from different exchange, I can only at best update the data every 5seconds. </p>

<p>Now the problem is this :</p>

<p>Every 5 second for one exchange I will import two market type buy and sell.
Every market type will have 100 record.</p>

<p>So for one day for one exchange I will be importing</p>

<pre><code>( 86400 / 5 ) * 2 * 100 = 3,456,000 row
</code></pre>

<p>For one month I'll be importing</p>

<pre><code>3,456,000 * 30 = 10,368,000 row
</code></pre>

<p>currently we have 5 exchange with 2 currency each which means we will import around 100,368,000 row in one month.</p>

<p>Now for the analysis currently we will output below data :</p>

<ol>
<li>Last imported data for selected exchange ( 2 - 4 ) and one currency , which is simple.</li>
<li>One day data for selected exchange ( 2 - 4 ) and selected one currency.</li>
<li>same as above but with data more than one month less than a year.</li>
</ol>

<p>You will see that some data will not be use for now, but we still need to kept it for future usage.</p>

<p>Any advice of best way to handling this ?</p>

<p>Currently I always using MySQL as my database, but I'm not sure it's the correct DB to handling this.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Step 1.  Normalize.  Change market_name and market_type and currency from a bulky VARCHAR into an ENUM or a TINYINT.  This will cut the datasize in half.  DECIMAL(30,10) takes 14 bytes and has far more precision than is needed.  Find out what the largest value is.  Are the decimal places limited to 2 digits as is currently common in the US?  Or do you need more decimal places.  Will FLOAT (4 bytes, about 7 significant digits) do?  That change will also save a lot.</p>

<p>Step 2.  Decide whether you need the data "forever".  If not, what is the "purge" policy?  It would probably be good to PARTITION by MONTH.  This would make purging easy and very fast, plus it might help in some of the queries.</p>

<p>Step 3.  Please show us the actual SELECTs; we need to further tune them, plus see what other shrinkage/optimizations/etc can be done.</p>

<p>Step 4.  Would "summary tables" help?  That is, rather than keeping 5-sec data for last month, would 1-minute, or even 1-hour, data suffice?  That would save a huge amount of space, and tremendously speed up the queries.</p>

<p>Step 5.  After deciding on PARTITIONing and the SELECTs, let's discuss the INDEXes.</p>

<p>I can elaborate on any of these; which ones would you like more details on?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>You slipped a decimal ... 30 * 3,456,000 = 103,680,000 rows per month per exchange, not easily indexable.  With 5 exchanges and 2 currencies, that is 1,036,800,000 rows per month.</p>

<p>This looks to me like a definite start for a bigdata project.  It is already structured perfectly for any hadoop based solution.  You will need to look a a wide but shallow cluster architecture with good sharding.</p>

<p>Depending on the need for transactional integrity, a hive (sql against raw files dumped into the HDFS partition), hBase (aka Amazon technology) or Cassandra  (aka Facebook Technology derived from Amazon technology) abstraction layer might make sense.  I would need to know more about the product before suggesting which way to jump.</p>

<p>On the other hand, Instagram runs just fine on postgresql, and their CTO has a blog detailing what they have done to scale it our and the headaches they have had to deal with.</p>
