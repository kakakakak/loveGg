<p>I am reading about MapReduce and the following thing is confusing me.</p>

<p>Suppose we have a file with 1 million entries(integers) and we want to sort them using MapReduce. The way i understood to go about it is as follows:</p>

<p>Write a mapper function that sorts integers. So the framework will divide the input file into multiple chunks and would give them to different mappers. Each mapper will sort their chunk of data independent of each other. Once all the mappers are done, we will pass each of their results to Reducer and it will combine the result and give me the final output.</p>

<p>My doubt is, if we have one reducer, then how does it leverage the distributed framework, if, eventually, we have to combine the result at one place?. The problem drills down to merging 1 million entries at one place. Is that so or am i missing something?</p>

<p>Thanks,
Chander</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Check out merge-sort.</p>

<p>It turns out that sorting partially sorted lists is much more efficient in terms of operations and memory consumption than sorting the complete list.</p>

<p>If the reducer gets 4 sorted lists it only needs to look for the smallest element of the 4 lists and pick that one. If the number of lists is constant this reducing is an O(N) operation.</p>

<p>Also typically the reducers are also "distributed" in something like a tree, so the work can be parrallelized too.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>As others have mentioned, merging is much simpler than sorting, so there's a big win there.</p>

<p>However, doing an O(N) serial operation on a giant dataset can be prohibitive, too. As you correctly point out, it's better to find a way to do the merge in parallel, as well.</p>

<p>One way to do this is to replace the partitioning function from the random partitioner (which is what's normally used) to something a bit smarter. What Pig does for this, for example, is sample your dataset to come up with a rough approximation of the distribution of your values, and then assign ranges of values to different reducers. Reducer 0 gets all elements &lt; 1000, reducer 1 gets all elements >= 1000 and &lt; 5000, and so on. Then you can do the merge in parallel, and the end result is sorted as you know the number of each reducer task.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Sorry for being late but for future readers,yes chandler, you are getting it wrong</p>

<p>Logic is that Reducer can handle shuffled and then sorted data of its node only on which it is running. I mean reducer that run at one node cant look at other node's data, it aplies the reduce algo on its data only. SO merging procedure of merger sort cant be applied.</p>

<p>So for big data we use Tera sort, which is nothing but identity mapper and reducer with custom partitioner. Read more about it from here You should read more about it from here <a href="http://sortbenchmark.org/YahooHadoop.pdf%3a%3a" rel="nofollow">Hadoop's implementation for Terasort</a>. It states:</p>

<p>"TeraSort is a standard map/reduce sort, except for a custom partitioner that uses a sorted list of N − 1 sampled keys that define the key range for each reduce. In particular, all keys such that sample[i − 1] &lt;= key &lt; sample[i] are sent to reduce i. This guarantees that the output of reduce i are all less than the output of reduce i+1."</p>
