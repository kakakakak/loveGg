    <div class="post-text" itemprop="text">

<p>I have to run something inside a Hadoop cluster which cannot be expressed in terms of Map/Reduce. I thought of writing a YARN application for it. I discovered <a href="https://spring.io/guides/gs/yarn-basic/" rel="nofollow">Spring Yarn</a> for spring-boot and followed the Getting Started (see link). This works so far, but there are some flaws:</p>

<ul>
<li>In the tutorial three JARs are produced (one for the client, one for the appmaster and one for the container), which have to be in a specific folder structure when submitting the app</li>
<li>I have to hard-code HDFS URI and Resource manager host/ports in an application.yml or supply them as command-line parameters</li>
<li>Since it is based on Spring Boot, and the application is started with <code>java -jar</code>, the JAR files created are very large with basically a whole Hadoop stack in them</li>
<li>The exact names of the JAR files have to be mentioned in application.yml</li>
</ul>

<p>What I want:</p>

<ul>
<li>Single JAR with the JARs for appmaster and container packaged in it</li>
<li>Runnable from the command line with <code>hadoop jar</code></li>
<li>Using the configuration which is available when running with <code>hadoop jar</code> (for MR2, this is possible by launching a class extending <code>Configured</code> and implementing <code>Tool</code> with <code>ToolRunner.run()</code>, this makes a <code>Configuration</code> available in the Tool's <code>run</code> method)</li>
</ul>

<p>The approach I think of is:</p>

<ol>
<li>Write the Container and AppMaster, set the YARN and Hadoop dependencies to provided in their POMs, have them packaged with the maven-shade-plugin as I do for MR jobs</li>
<li>Write the client, add the AppMaster and Container as dependencies, package it with maven-assembly-plugin to prevent the JARs from being extracted</li>
</ol>

<p>I tried Twill, but to no avail. I get</p>

<pre><code>java.lang.NoSuchMethodError: com.google.common.collect.Sets.newCopyOnWriteArraySet()Ljava/util/concurrent/CopyOnWriteArraySet;
</code></pre>

<p>because my Hadoop installation uses Guava 11 and Twill needs 13. Even though Guava 13 is shaded into the Jar, it is simply ignored.</p>

<p>I found something I call a "workaround" which works sufficiently enough for my use case:</p>

<ul>
<li>I build my application with Spring YARN, resulting in separate JARs for Client, Container and AppMaster</li>
<li>I add them as modules to a Master POM which controls the version number (whenever I change anything in one of the former three projects, I increment the Master POM's version)</li>
<li>This Master POM is a module itself, with my big whole-project-wide Parent POM</li>
<li>the Master POM's parent is not the big project-wide POM, but <code>spring-boot-starter-parent</code></li>
</ul>

<p>When being built by Jenkins this creates the mentioned three JARs, I currently manually pack them into a folder with a start script beside them. This is just a temporary solution, as this application contains a long-running task which is later to be started by the user from a web application (also based on Spring). I still have to figure out how to submit the application from there.</p>

<p>My idea is the following, this is similar to how I currently do it for MR jobs:</p>

<ul>
<li>Add the JARs as dependencies to the web application's pom.xml</li>
<li>Include a basic <code>application.yml</code> with no YARN and JAR information in the three JARs</li>
<li>Use the same technique as <code>Job.setJarByClass()</code> uses to locate the AppMaster and Container JARs</li>
<li>Call the client's main class with <code>SpringApplication.run()</code> passing in connection properties and the resolved locations of JAR files via the command line (<code>args</code> variable)</li>
</ul>

<p>If anyone could give me a hint if this were a feasible situation, please let me know.</p>

    <div class="post-text" itemprop="text">
<p>I dug around this some more, and found out that it is not easy to package and spawn Spring Boot apps from inside other Spring Boot apps. For my use case, calling Spring YARN applications from a Spring Boot application not using YARN, the following approach works:</p>

<ol>
<li>Create the Spring YARN application in "single file mode" like in <a href="https://spring.io/guides/gs/yarn-basic-single/" rel="nofollow">this tutorial</a></li>
<li>Package the resulting JAR into the application from where you want to deploy it
<ul>
<li>e.g. when using Maven, you can add it as a dependency</li>
</ul></li>
<li>Make sure the <strong>deploying</strong> application excludes the <code>YarnClientAutoConfiguration</code> like so: <code>@EnableAutoConfiguration(exclude = YarnClientAutoConfiguration.class)</code></li>
<li>Make sure the packaging plugin packages the JARs as whole archives, like the <code>maven-assembly-plugin</code> or <code>spring-boot-maven-plugin</code> does (NOT the <code>maven-shade-plugin</code>)</li>
<li>Unpack the big JAR of your deploying application to a temporary directory</li>
<li>Use a <code>ProcessBuilder</code> to run <code>java -jar</code> on the Spring-YARN application to run, passing the correct configuration options at the command line</li>
</ol>

<p>This is all kind of hacky, Hadoop definitely needs something which is similar to the <code>Job</code> for MR jobs which just runs things on YARN.</p>
