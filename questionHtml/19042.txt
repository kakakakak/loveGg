<p>I am working on a project that is logging a lot of information about viewers from an online streaming platform. The problem today with the MySQL solution is that is too slow to query, and such.
Even with scaling and better performance tuning, that will now work because there are just to much data real time thats write/reads. </p>

<p>What will be a good(the best) NoSQL solution for me?</p>

<p>Extra:</p>

<ul>
<li>We are currently also using Amazon Web services, where we store our data.</li>
<li>With Java API, and a open source solution is preferred.</li>
<li>Object orientated.</li>
</ul>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>The first challenge you may have is how to collect huge amount of data reliably with ease of management. There're some open-source log collector implementation such as syslog, <a href="http://fluentd.org/" rel="nofollow">Fluentd</a>, <a href="http://github.com/facebook/scribe/" rel="nofollow">Scribe</a>, and <a href="http://flume.apache.org/" rel="nofollow">Flume</a> :)</p>

<p>The big problem is how to store and process data. As you pointed out, using NoSQL solution works really well, but you need to choose among them depending on your data volume.</p>

<p>At first, you can use <a href="http://mongodb.org/" rel="nofollow">MongoDB</a> to store all of your data, but at some moment you end up using <a href="http://hadoop.apache.org/" rel="nofollow">Apache Hadoop</a> to architect a massively scalable architecture.</p>

<p>The poing here is you should have a distributed logging layer which abstracts away the storage backend, and choosing the right NoSQL solution for data volume.</p>

<p>Here're some links to put the Apache Logs into MongoDB, or Hadoop HDFS by Fluentd.</p>

<ul>
<li><a href="http://docs.fluentd.org/articles/apache-to-mongodb" rel="nofollow">Store Apache Logs into MongoDB</a></li>
<li><a href="http://docs.fluentd.org/articles/http-to-hdfs" rel="nofollow">Fluentd + HDFS: Instant Big Data Collection</a></li>
</ul>
