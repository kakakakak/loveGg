<p>i was hoping to get feedback on an example mysql structure for a web application in a real world environment from people who have used complex mysql in real world situations before.</p>

<p>example ~</p>

<p>education management app.
80,000 users.
each user gets <strong>his own database</strong> containing tables for
-messages
-uploads
-grades
-info</p>

<p>and more tables for other features</p>

<p>what I'm wondering is, and any info would be appreciated, in a situation like this</p>

<p>-is this databasing model efficient? (basically like 80,000 databases) or is there (and I have this <em>itching</em> feeling there is) a better way to do it?
-what kind of dedicated server would this require? 80,000 databases each with 10-15 tables containing TONS of tables, with all 80,000 people accessing the site 20-30 times a day for 10-20 sessions</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Start running. </p>

<p>Now! </p>

<p>Jokes aside, don't do that. Don't create one database per user. That's a hell to administer, maintain and to query. What if you need to know which users logged in yesterday? Will you query each database??</p>

<p>The structure you need is the same, only the amount of data changes. Just have one database, see how it goes and then optimize/fine tune.</p>

<p>I hate to bring this quote up, but in your case it totally applies:</p>

<blockquote>
  <p><a href="http://en.wikipedia.org/wiki/Program_optimization">Premature optimization</a> is the root of all evil (Donald Knuth)</p>
</blockquote>

<p>Don't try to optimize your solution <strong>before</strong> you know where your bottlenecks will be.</p>

<p>Just model your database the best you can. Worry about your constraints, PKs, FKs, Indexes. Do your <a href="/questions/tagged/database-design" class="post-tag" title="show questions tagged 'database-design'" rel="tag">database-design</a> homework. Then have your data and software going. Only then you will see where it works and where it hurts. At this moment, you optimize.</p>

<p>Only attack your enemy when you know who it is.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>it can be efficient with the appropiate store engine to support that model. Most of the recent no-sql data stores (hadoop, bigtable, mongodb, etc.) work great for this sort of scenario.</p>

<p>If you think about it, a user data is an excellent way to partition data stores in isolated islands (most of interactions between different user databases does not need to be transactional, and there is very little if any exchange of write requests)</p>

<p>Basically i would think that no-sql doesn't give you any benefits for data relationships inside the user data island itself, so relational and no-relational stores performance differences <em>should</em> not matter that much</p>

<p>Given said that, traditional relational databases as mysql are not designed to be managed in-masse, so in this respect you might want to consider a different data store (or hiring a dba rock star)</p>

<p>Both Adrian and Lurscher have already given lot of details.  </p>

<p>Without knowing how much data and the schema it is hard to get into design.  From outset with 80K active users the load doesn't seem to be huge even if they access the data hundreds of times a day.  I have a feeling you could be able to create  normalized schema for an OLTP environment with users table and work from there with other tables. And again it is the requirement that will drive the design. For example, what should be the response time for user query - sub-second, a second?</p>
