    <div class="post-text" itemprop="text">

<p>I've set up a distributed Hadoop environment within VirtualBox: 4 virtual Ubuntu 11.10 installations, one acting as the master node, the other three as slaves. I followed <a href="http://www.michael-noll.com/tutorials/running-hadoop-on-ubuntu-linux-multi-node-cluster/">this tutorial</a> to get the single-node version up and running and then converted to the fully-distributed version. It was working just fine when I was running 11.04; however, when I upgraded to 11.10, it broke. Now all my slaves' logs show the following error message, repeated ad nauseum:</p>

<pre><code>INFO org.apache.hadoop.ipc.Client: Retrying connect to server: master/192.168.1.10:54310. Already tried 0 time(s).
INFO org.apache.hadoop.ipc.Client: Retrying connect to server: master/192.168.1.10:54310. Already tried 1 time(s).
INFO org.apache.hadoop.ipc.Client: Retrying connect to server: master/192.168.1.10:54310. Already tried 2 time(s).
</code></pre>

<p>And so on. I've found other instances of this error message on the Internet (and <a href="http://stackoverflow.com/questions/8699753/hadoop-hdfs-cannot-connect-to-port-on-master">StackOverflow</a>) but none of the solutions have worked (tried changing the core-site.xml and mapred-site.xml entries to be the IP address rather than hostname; quadruple-checked <code>/etc/hosts</code> on all slaves and master; master can SSH password-less into all slaves). I even tried reverting each slave back to a single-node setup, and they would all work fine in this case (on that note, the master always works fine as both a Datanode and the Namenode).</p>

<p>The only symptom I've found that would seem to give a lead is that from any of the slaves, when I attempt a <code>telnet 192.168.1.10 54310</code>, I get <code>Connection refused</code>, suggesting there is some rule blocking access (which must have gone into effect when I upgraded to 11.10).</p>

<p>My <code>/etc/hosts.allow</code> has not changed, however. I tried the rule <code>ALL: 192.168.1.</code>, but it did not change the behavior.</p>

<p>Oh yes, and <code>netstat</code> on the master clearly shows tcp ports 54310 and 54311 are listening.</p>

<p><strong>Anyone have any suggestions to get the slave Datanodes to recognize the Namenode?</strong></p>

<p><strong>EDIT #1</strong>: In doing some poking around with nmap (see comments on this post), I'm thinking the issue is in my <code>/etc/hosts</code> files. This is what is listed for the master VM:</p>

<pre><code>127.0.0.1    localhost
127.0.1.1    master
192.168.1.10 master
192.168.1.11 slave1
192.168.1.12 slave2
192.168.1.13 slave3
</code></pre>

<p>For each slave VM:</p>

<pre><code>127.0.0.1    localhost
127.0.1.1    slaveX
192.168.1.10 master
192.168.1.1X slaveX
</code></pre>

<p>Unfortunately, I'm not sure what I changed, but the NameNode is now always dying with the exception of trying to bind a port "that's already in use" (127.0.1.1:54310). I'm clearly doing something wrong with the hostnames and IP addresses, but I'm really not sure what it is. Thoughts?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>This solution worked for me. i.e make sure that the name you used in property in core-site.xml and mapred-site.xml :</p>

<pre><code>`&lt;property&gt;
   &lt;name&gt;fs.default.name&lt;/name&gt;
   &lt;value&gt;hdfs://master:54310&lt;/value&gt;
   &lt;final&gt;true&lt;/final&gt;
 &lt;/property&gt;`
</code></pre>

<p>i.e. master is defined in /etc/hosts as xyz.xyz.xyz.xyz   master on BOTH master and slave nodes.
Then restart the namenode and check using
<code>netstat -tuplen</code>
and to see that it is bound to the "external" IP address</p>

<pre><code>`tcp        0      xyz.xyz.xyz.xyz:54310         0.0.0.0:*                   LISTEN      102        107203     - `
</code></pre>

<p>and NOT local IP 192.168.x.y or 127.0.x.y</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I am running a 2-nodes cluster. </p>

<p>192.168.0.24   master <br>
192.168.0.26   worker2 <br></p>

<p>I was facing the same problem of Retrying connect to server: master/192.168.0.24:54310 in my worker2 machine logs. But the people mentioned above encountered errors running this command - telnet 192.168.0.24 54310. However, in my case the telnet command worked fine. Then I checked my /etc/hosts file</p>

<p>master /etc/hosts <br>
127.0.0.1 localhost<br>
192.168.0.24 ubuntu <br>
192.168.0.24 master <br>
192.168.0.26 worker2 <br></p>

<p>worker2 /etc/hosts <br>
127.0.0.1 localhost <br>
192.168.0.26 ubuntu <br>
192.168.0.24 master <br>
192.168.0.26 worker2 <br></p>

<p>When I hit <a href="http://localhost:50070" rel="nofollow">http://localhost:50070</a> on master, I saw Live nodes : 2. But when I clicked on it, I saw only one datanode which was of master's. I checked jps both on master and worker2. Datanode process was running on both the machines.</p>

<p>Then after several trial and errors, I realized that my master and worker2 machines had the same host name "ubuntu". I changed the worker2's hostname from "ubuntu" to "worker2" and removed the "ubuntu" entry from the worker2 machine. </p>

<p>Note: To change the hostname edit the /etc/hostname with sudo. </p>

<p>Bingo! It worked :) I was able to see two datanodes on the dfshealth UI page ( locahost:50070)</p>
