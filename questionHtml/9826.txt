    <div class="post-text" itemprop="text">

<p>Here is one of my interview question. Given an array of N elements and where an element appears <strong>exactly N/2</strong> times and the rest N/2 elements are <strong>unique</strong>. How would you find the element with a better run time?</p>

<p>Remember the elements are not sorted and you can assume N is even. For example,</p>

<pre><code>input array [] = { 10, 2, 3, 10, 1, 4, 10, 5, 10, 10 }
</code></pre>

<p>So here 10 appears extactly 5 times which is N/2.</p>

<p>I know a solution with O(n) run time. But still looking forward to know a better solution with O(log n).</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>There is a constant time solution if you are ready to accept a small probability of error. Randomly samples two values from the array, if they are the same, you found the value you were looking for. At each step, you have a 0.75 probability of not finishing. And because for every epsilon, there exists one n such that (3/4)^n &lt; eps, we can sample at most n time and return an error if we did not found a matching pair. </p>

<p>Also remark that, if we keep sampling until we found a pair, the expected running time is constant, but the worst case running time is  not bounded.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Here is my attempt at a proof of why this cannot be done in less than O(n) array accesses (for worst case, which surely is the only interesting case in this example):</p>

<p>Assume a worst case log(n) algorithm exists.  This algorithm accesses the array at most log(n) times.  Since it can make no assumptions about which elements are where, let me choose which log(n) elements it sees.  I will choose to give it the first log(n) unique elements.  It has not found the duplicate yet, and there still exist n/2 - log(n) unique elements for me to feed it if need be.  In fact, I cannot be forced to feed it a duplicated number until it has read n/2 elements.  Therefore such an algorithm cannot exist.</p>

<p>From a purely intuitive standpoint, this just seems impossible.  Log(4 billion) is 32.  So with an array of 4 billion numbers, 2 billion of which are unique, in no particular order, there is a way to find the duplicated element by only checking 32 elements?</p>

<p>I think you simply need to parse through the array keeping a backlog of two elements. As N/2 are equal and the rest is guaranteed to be distinct there must be one place i in your array where </p>

<p><code>a[i] == a[i-1] OR a[i] == a[i-2]</code></p>

<p>iterate once through your array and you have complexity of roughly 2*N which should be well inside O(N).</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>For worst-case <em>deterministic</em> behavior, O(N) is correct (I've already seen more than one proof in the previous answers).</p>

<p>However, modern algorithmic theory isn't concerned just with worst-case behavior (that's why there are so many other big-somethings besides big-O, even though lazy programmers in a hurry often use big-O even when what they have in mind is closer to big-theta OR big-omega;-), nor just with determinism (withness the Miller-Rabin primality test...;).</p>

<p>Any random sample of K &lt; N items will show no duplicates with a probabllity that &lt; 2**K -- easily and rapidly reduced to essentially as low as you wish no matter what N is (e.g. you could reduce it to less than the probability that a random cosmic ray will accidentally and undetectably flip a bit in your memory;-) -- this observation hardly requires the creativity Rabin and Miller needed to find <em>their</em> probabilistic prime testing approach;-).</p>

<p>This would make a pretty lousy interview question. Similar less-lousy questions are often posed, often mis-answered, <strong>and</strong> often mis-remembered by unsuccessful candidates. For example, a typical question might be, given an array of N items, <em>not knowing</em> whether there is a majority item, to determine whether there is one, and which one it is, in O(N) time <em>and</em> O(1) auxiliary space (so you can't just set up a hash table or something to count occurrences of different values). "Moore's Voting Approach" is a good solution (probably the best one) to <em>that</em> worthy interviewing question.</p>

<p>Another interesting variation: what if you have <code>10**18</code> 64-bit numbers (8 Terabytes' worth of data overall, say on a bigtable or clone thereof), and as many machines as you want, each with about 4GB of RAM on a pretty fast LAN, say one that's substantially better than GB ethernet -- how do you shard the problem under <strong>those</strong> conditions? What if you have to use mapreduce / hadoop? What if you're free to design your own dedicated framework just for this one problem -- could you get better performance than with mapreduce? How much better, at the granularity of back-of-envelope estimation?  I know of no published algorithm for THIS variant, so it may be a great test if you want to check general facility of a candidate with highly-distributed approaches to tera-scale computation...</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>You can't do this in sublinear time because you need to read the array. To process an array of a million records in logarithmic time would require only reading ~20 (log2) elements--clearly impossible. After all if you assume the first duplicate found is repeated N/2 times it's still O(n) because you may need to look at 500,001 elements to find a duplicate.</p>

<p>You can do this in O(n) if you assume the integers are nonnegative. It goes like this (pseudo-Java):</p>

<pre><code>int repeatedNumber = -1; // sentinel value
int count = 0;
BitSet bits = new BigSet(); // this bitset needs to have 2^31 bits, roughly 2.1 billion
boolean duplicate = false;
for (int i : elements) {
  if (bits[i].isSet()) {
    if (repeatedNumber == -1) {
      repeatedNumber = i;
      count = 1;
    } else if (i == repeatedNumber) {
      count++;
    } else {
      System.out.println("Array has more than one repeated element");
      duplicate = true;
      break;
    }
  } else {
    bits[i].set();
  }
}
if (!duplicate &amp;&amp; repeatedNumber != -1 &amp;&amp; count == elements.length/2) {
  System.out.println(repeatedNumber + " occurred " + count + " times. The rest of the elements are unique");
} else {
  System.out.println("Not true");
}
</code></pre>

<p>A similar method is used to sort an array of unique integers in O(n) (radix sort).</p>

<p>My Answer was,</p>

<ol>
<li>Divide N elements into [N/3] parts (i.e) each part will have 3 elements.</li>
<li>Now compare these 3 elements among each other. - <strong>3 comparisions</strong></li>
<li>Atleast one of the part will have two copies of the same element. Hence the number.</li>
</ol>

<p>Runtime - O(N)</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Peter is exactly right. Here is a more formal way of restating his proof:</p>

<p>Let set S be a set containing N elements. It is the union of two sets: p, which contains a symbol &alpha; repeated N/2 times, and q, which contains N/2 unique symbols &omega;<sub>1</sub>..&omega;<sub>n/2</sub>. S = p &cup; q. </p>

<p>Assume there is an algorithm that can detect your duplicated number in log(n) comparisons <i>in the worst case</i> for all N > 2. In the worst case means that <b>there does not exist any subset r &sub; S such that |r| = log<sub>2</sub> N where &alpha; &notin; r</b>. </p>

<p>However because S = p &cup; q, there are |p| many elements &ne; &alpha; in S. |p| = N/2, so &forall; N/2 such that N/2 &ge; log<sub>2</sub>N, there must exist at least one set r &sub; S such that |r| = log<sub>2</sub>N and &alpha; &notin; r. This is the case for any N &ge; 3. This contradicts the assumption above, so there cannot be any such algorithm. </p>

<p>QED.</p>

<p>The answer is straightforward.. and can be achieved in worst case (n/2 + 1) comparisons</p>

<ol>
<li><p>Compare pairwise first (n-2) numbers, that is, compare nos. at 0 and 1, then 2 and 3 and so on... total n/2 -1 comparisons.
If we find identical numbers in any of the above comparisons.. we have the repeated number... else:</p></li>
<li><p>Take any one of the last two remaining numbers (say second last one I took)  and compare it with the numbers in the second last pair.. if match occurs..second last no. is the repated one, else last one is the repeated one... in all 2 comparisons.</p></li>
</ol>

<p>Total comparisons = n/2 - 1 + 2 =n/2 + 1 (worst case)
I dont think there is any O(log n) method to achieve this</p>

    <div class="post-text" itemprop="text">
<p>It's fairly simple to see that no O(log n) algorithm exists. Clearly you have to look at the array elements to figure out which is the repeated element, but no matter what order you choose to look at the elements, the first floor(n/2) elements you look at might all be unique. You could simply be unlucky. If that happened, you would have no way of knowing which was the repeated element. Since no algorithm that uses less than floor(n/2) array references or fewer on every run will work, there is definitely no sub-linear algorithm.</p>

<p>Restating my solution from a comment to Ganesh's version so I can format it:</p>

<pre><code>for (i=0; i&lt;N-2; i+=3) { 
   if a[i] == a[1+1] || a[i] == a[i+2] return a[i];
   if a[i+1] == a[i+2] return a[i+1]; 
} 
return a[N-1]; // for very small N
</code></pre>

<p>Probability of winning after 1 iteration:  50%</p>

<p>Probability of winning after 2 iterations: 75%</p>

<p>Etc.</p>

<p>Worst case, O(n) time O(1) space.</p>

<p>Note that after N/4 iterations you've used up all the N/2 unique numbers, so this loop will never iterate through more than 3/4 of the array if it is as specified.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Suppose you have a python algorithm like this:</p>

<pre><code>import math
import random

def find_duplicate(arr, gap):
    cost, reps = 0, 0
    while True:
        indexes = sorted((random.randint(0,len(arr)-i-1) for i in xrange(gap)), reverse=True)
        selection = [arr.pop(i) for i in indexes]
        selection_set = set(selection)
        cost += len(selection)
        reps += 1
        if len(selection) &gt; len(selection_set):
            return cost, reps
</code></pre>

<p>The idea is that <strong>arr</strong> is your set of values and <strong>gap</strong> is the log base-2 of the size. Each time you select <strong>gap</strong> elements and see if there are duplicated values. If so, return your cost (in count of elements examined) and the number of iterations (where you examine log2(size) elements per iteration). Otherwise, look at another <strong>gap</strong>-sized set.</p>

<p>The problem with benchmarking this algorithm is that the creation of the data each time through the loop and alteration of the data is expensive, assuming a large amount of data. (Initially, I was doing 1 000 000 elements with 10 000 000 iterations.)</p>

<p>So let's reduce to an equivalent problem. The data is passed in as n/2 unique elements and n/2 repeated elements. The algorithm picks the random indexes of log2(n) elements and checks for duplicates. Now we don't even have to create the data and to remove elements examined: <strong>we can just check if we have two or more indexes over the halfway point</strong>. Select <strong>gap</strong> indexes, check for 2 or more over the halfway point: return if found, otherwise repeat.</p>

<pre><code>import math
import random

def find_duplicate(total, half, gap):
    cost, reps = 0, 0
    while True:
        indexes = [random.randint(0,total-i-1) for i in range(gap)]
        cost += gap
        reps += 1
        above_half = [i for i in indexes if i &gt;= half]
        if len(above_half) &gt;= 2:
            return cost, reps
        else:
            total -= len(indexes)
            half -= (len(indexes) - len(above_half))
</code></pre>

<p>Now drive the code like this:</p>

<pre><code>if __name__ == '__main__':
    import sys
    import collections
    import datetime
    for total in [2**i for i in range(5, 21)]:
        half = total // 2
        gap = int(math.ceil(math.log10(total) / math.log10(2)))
        d = collections.defaultdict(int)
        total_cost, total_reps = 0, 1000*1000*10
        s = datetime.datetime.now()
        for _ in xrange(total_reps):
            cost, reps = find_duplicate(total, half, gap)
            d[reps] += 1
            total_cost += cost
        e = datetime.datetime.now()
        print "Elapsed: ", (e - s)
        print "%d elements" % total
        print "block size %d (log of # elements)" % gap
        for k in sorted(d.keys()):
            print k, d[k]
        average_cost = float(total_cost) / float(total_reps)
        average_logs = average_cost / gap
        print "Total cost: ", total_cost
        print "Average cost in accesses: %f" % average_cost
        print "Average cost in logs: %f" % average_logs
        print
</code></pre>

<p>If you try this test, you'll find that the number of times the algorithm has to do multiple selections declines with the number of elements in the data. That is, <strong>your average cost in logs asymptotically approaches 1</strong>.</p>

<pre><code>elements    accesses    log-accesses

<p>Now is this an argument for the ideal algorithm being log2(n) <strong>in the average case</strong>? Perhaps. It certainly is not so in the worst case.</p>

<p>Also, you don't have to pick log2(n) elements at once. You can pick 2 and check for equality (but in the degenerate case, you will not find the duplication at all), or check any other number greater for duplication. At this point, all the algorithms that select elements and check for duplication are identical, varying only in how many they pick and how they identify duplication.</p>

    <div class="post-text" itemprop="text">
<p>Here is <a href="http://stackoverflow.com/questions/1191740/find-a-number-where-it-appears-exactly-n-2-times/1192967#1192967">Don Johe's answer</a> in Ruby:</p>

<pre><code>#!/usr/bin/ruby1.8

def find_repeated_number(a)
  return nil unless a.size &gt;= 3
  (0..a.size - 3).each do |i|
    [
      [0, 1],
      [0, 2],
      [1, 2],
    ].each do |j1, j2|
      return a[i + j1] if a[i + j1] == a[i + j2]
    end
  end
end

p find_repeated_number([1, 1, 2])   # =&gt; 1
p find_repeated_number([2, 3, 2])   # =&gt; 1
p find_repeated_number([4, 3, 3])   # =&gt; 1
</code></pre>

<p>O(n)</p>

<p>Similar to <a href="http://stackoverflow.com/a/1191881/199556">http://stackoverflow.com/a/1191881/199556</a> explanation.</p>

<p>Let's compare 3 elements(3 comparison operation) in worse case "same" element will appear once.
So we reduce the tail by 3 and reduce count of "same" elements by one.</p>

<p>In final step(after k iterations) our tail will contain (n/2) - k "same" elements. Let's compare the length of the tail.</p>

<p>On the one hand it will n-3k
on the other hand (n/2) - k + 1. Last unsame elements may exist.</p>

<p>n-3k = (n/2) - k + 1</p>

<p>k = 1/4*(n-2)</p>

<p>After k iterations we'll surely get result.</p>

<p>Number of comparisons 3/4*(n-2) </p>

<p>First off, it's past my bed time and I should know better than to post code in public without trying it first, yada, yada. I hope the criticism I'll get will at least be educational. :-)</p>

<p>I believe the problem can be restated as: "Find the number that occurs more than once."</p>

<p>In the absolute worst case, we would need to iterate through a little more than half the list (1 + N/2) before we found the 2nd instance of a non-unique number.</p>

<p>Worst case example: array [] = { 1, 2, 3, 4, 5, 10, 10, 10, 10, 10 }</p>

<p>On <em>average</em> though, we'd only need to iterate though 3 or 4 elements since half of the elements will contain the non-unique number i.e roughly every other number.</p>

<p>Perfectly even distribution examples: </p>

<ul>
<li>array [] = { 1, 10, 2, 10, 3, 10, 4, 10, 5, 10 }</li>
<li>array [] = { 10, 1, 10, 2, 10, 3, 10, 4, 10, 5 }</li>
</ul>

<p><strong>In other words even if N = 1 million you would still only need to search; on average, the first 3 or 4 elements before you discovered a duplicate.</strong></p>

<p><strong>What's the big O notation for a fixed/constant runtime that doesn't increase with N?</strong></p>

<p>Code:</p>

<pre><code>int foundAt = -1;

for (int i=0; (i&lt;N) &amp;&amp; (foundAt==-1); i++)
{
    for (int j=i+1; j&lt;N; j++)
    {
        if (array[i] == array[j])
        {
             foundAt = i;
             break;
        }
     }
}

int uniqueNumber = array[foundAt];
</code></pre>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Contrary to answers above, there is a solution with <strong>worst case</strong> behavior as requested, O(log n) <strong>RUN TIME</strong>.
The problem is not to find a solution with O(log N) comparisons worst case (which is impossible), but to do it O(log N) time.</p>

<p>If you can do N comparisons in parallel, the solution is a trivial divide-and-conquer.
Not very practical in the real world, but it's an interview question, not a real-world problem.</p>

<p>Update: I think you can do it in constant time with O(N) processors</p>
