<p>Hadoop: (ver - 1.2.1), (1 + 8 node cluster)</p>

<p>My use case is, I am trying to get the time taken to execute a specific pig script and how that time is being spent from a mapreduce point of view. I need to run the pig script multiple times(say 100) to get the average time. I've enabled <code>pig.udf.profile</code> which gives me time spent on each udf function as mapreduce counters. I am also interested in other latency,memory metrics reported for each job (cpu time,heap useage). I can see these counters from the jobtracker web UI( <code>host:50030/jobdetails.jsp?jobid=blah</code>). Now, my question is, is there a way to aggregate these counters across jobs. Or, how will I build a table that looks like</p>

<pre><code>                        Run1   Run2   Run3 ...
Cpu time              |      |      |
Redcr wait            |      |      |
UdfCntr1(approx_us)   |      |      |
UdfCntr2(approx_invc) |      |      |
CounterY(approx_us)   |      |      |
</code></pre>

<p>Each run is a different job as far as Hadoop is concerned. After grep'ing through the log folder, I figured out the counters are in the <code>history/done/..</code> folders. Is there an existing technique to combine the results, or am I doomed to write my own parser that goes through each log file. I thought the use case is common enough for an existing solution - any pointers would be helpful.</p>

<p>Thanks.</p>

    <div class="post-text" itemprop="text">
<p>You have a couple of options, and I apologize in advance that none of them are particularlly appealing.  </p>

<p><em><strong>Implement a PPNL</em></strong></p>

<p>The <em>PigProgressNotificationListener</em> is a <a href="http://pig.apache.org/docs/r0.9.1/api/org/apache/pig/tools/pigstats/PigProgressNotificationListener.html" rel="nofollow">java interface</a> for exposing events that occur over the course of a Pig job to arbitrary clients.  If you implement this interface and attach an instance of your class you can then grab hadoop counters (and many other M/R related metrics) and store them away for later usage. Note that this requires a reasonable understand of Pig internals, though not necessarily an expert level of understanding.</p>

<p><br>
<strong><em>Use Another System that Gathers Metrics</em></strong></p>

<p>Right now your options are limited <a href="https://github.com/twitter/ambrose" rel="nofollow">Ambrose</a> which was open sourced by Twitter, and <a href="https://github.com/Netflix/Lipstick" rel="nofollow">Lipstick</a> which was open sourced by Netflix.   I do not know for sure if Ambrose collects hadoop counters, but it could be extended to do.  Lipstick does collect hadoop counters as is.  With either of these you could then analyze the counters with a varying level of difficulty, depending on how you configured them to store data.</p>

<p><br>
<strong><em>Parse Your Log Files</em></strong></p>

<p>Sounds like you've already thought about going down this route for a couple of reasons:</p>

<ol>
<li>All the other options require you to know quite about the internals of another project, be it Pig, Lipstick, or Ambrose.  If you were planning to dive into one of these for other reasons as well then go for it, but otherwise it's a big investment for a single usecase.</li>
<li>All the other options are limited to Pig jobs.  You aren't going to get this data if you run a vanilla Map/Reduce job or start working with other tooling like hive, cascading, etc.  If you build up some log parsing tooling that converts the data to a well known format that then gets processed you should be able to reuse all your code (except for the actual parsing) with hadoop tools.</li>
<li>If your data is based on your logs, and you archive all your logs, you can always backfill if six months down the road you're interested in some data point you hadn't collected before.  This isn't possible with the other approaches.</li>
<li>It's an established pattern that works.  I can't point to a lot of online resources discussing people who have taken this approach, but I have met quite a few people who independently took this approach and were happy with it.</li>
</ol>

* Install Graphite 
* Have all your Jobs run through a central Main class which at the end of the job, takes the counters, converts into Key Value, where key can be .., and value is the counter value which is a long number.
