<p>(Don't suggest Hadoop or map reduce solution even they sounds logically the same)</p>

<p>I have a big file - 70GB of raw html files and I need to do the parsing to get the information I need. </p>

<p>I have delt with 10GB file successfully before using standardI/O: </p>

<pre><code>cat input_file | python parse.py &gt; output_file
</code></pre>

<p>And in my python script, it reads every html (each html per line) from standard input and writes the result back to standard output. </p>

<pre><code>from bs4 import BeautifulSoup
import sys
for line in sys.stdin:
    print ....
</code></pre>

<p>The code is very simple but right now, I am dealing with a big file which is horribly slow on one node. I have a cluster of about 20 nodes. And I am wondering how could I easily distribute this work. </p>

<p>What I have done so far:</p>

<pre><code>split -l 5000 input_file.all input_file_   # I have 60K lines in total in that 70G file
</code></pre>

<p>Now the big file have been splitted into several small files:  </p>

<pre><code>input_file_aa
input_file_ab
input_file_ac
...
</code></pre>

<p>Then I have no problem working with each one of them:</p>

<pre><code>cat input_file_aa | python parser.py &gt; output_file_aa 
</code></pre>

<p>What I gonna do is probably scp the input_file to each node and do the parsing and then scp the result back, but there are 10+ nodes! I it is so tedious to do that manually.</p>

<p>I am wondering how could I easily distribute these files to other nodes and do the parsing and move the result back? </p>

<p>I am open to basic SHELL, JAVA, Python solution. Thanks a lot in advance and let me know if you need more explanation. </p>

<p>Note, I do have a folder called /bigShare/ that could be assessible on every node and the contents are synchronized and stay the same.  I don't know how the architect implemented that (NFS..? I don't know how to check) but I could put my input_file and python script there so the rest is how to easily log into those nodes and execute the command. 
Btw, I am on Red Hat.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Execute the command remotely with remote piping to stdout.  Then make the local command pipe to a local file.</p>

<p>Example:</p>

<pre><code>ssh yourUserName@node1 "cat input_file_node1 | python parser.py" &gt;output_file_node1
</code></pre>

<p>If the files have not been copied to the different nodes, then:</p>

<pre><code>ssh yourUserName@node1 "python parser.py" &lt;input_file_node1 &gt;output_file_node1
</code></pre>

<p>This assumes that yourUserName has been configured with key-based authentication.  Otherwise, you will need to enter your password manually (20 times! :-( ).  To avoid this, you can use <code>expect</code>, but I will strongly suggest to setup key-based authentication.  You can do the later using <code>expect</code> too.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Assuming you want to process a piece of each file on a host of its own: first copy the python script to the remote hosts. Then loop over the remote hosts:</p>

<pre><code>for x in aa ab ac ...; do
   ssh user@remote-$x python yourscript.py &lt;input_file_$x &gt;output_file_$x &amp;
done;
</code></pre>

<p>If the processing nodes don't have names that are easy to generate you can create aliases for them in your <code>.ssh/config</code>, for example:</p>

<pre><code>Host remote-aa
    Hostname alpha.int.youcompany

Host remote-ab
    Hostname beta.int.yourcompany

Host remote-ac
    Hostname gamma.int.yourcompany
</code></pre>

<p>This particular use case could be more easily solved by editing /etc/hosts though.</p>
