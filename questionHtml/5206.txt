<p>I have 2 separate applications one in Java and the other is C++. I am using Murmurhash3 for both. However, in C++ I get a different result as compared to Java for the same string</p>

<p>Here is the one from C++: <a href="https://code.google.com/p/smhasher/source/browse/trunk/MurmurHash3.cpp?r=144" rel="nofollow">https://code.google.com/p/smhasher/source/browse/trunk/MurmurHash3.cpp?r=144</a></p>

<p>I am using the following function:</p>

<pre><code>void MurmurHash3_x86_32 ( const void * key, int len,
                      uint32_t seed, void * out )
</code></pre>

<p>Here is the one for Java: <a href="http://search-hadoop.com/c/HBase%3ahbase-common/src/main/java/org/apache/hadoop/hbase/util/MurmurHash3.java||server+void+%2522hash" rel="nofollow">http://search-hadoop.com/c/HBase:hbase-common/src/main/java/org/apache/hadoop/hbase/util/MurmurHash3.java||server+void+%2522hash</a></p>

<p>There are many versions of the same Java code above. </p>

<p>This is how I am making a call for Java:</p>

<pre><code>String s = new String("b2622f5e1310a0aa14b7f957fe4246fa");
System.out.println(MurmurHash3.murmurhash3_x86_32(s.getBytes(), 0, s.length(), 2147368987));
</code></pre>

<p>The output I get from Java:
-1868221715</p>

<p>The output I get from C++
3297211900</p>

<p>When I tested for some other sample strings like 
"7c6c5be91430a56187060e06fd64dcb8" and "7e7e5f2613d0a2a8c591f101fe8c7351" they match in Java and C++.</p>

<p>Any pointers are appreciated</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>There are two problems I can see. First, C++ is using <code>uint32_t</code>, and giving you a value of 3,297,211,900. This number is larger than can fit in a signed 32-bit int, and Java uses only signed integers. However, -1,868,221,715 is not equal to 3,297,211,900, even accounting for the difference between signed and unsigned ints.</p>

<p>(In Java 8 they have added <code>Integer.toUnsignedString(int)</code>, which will convert a signed 32-bit int to its unsigned string representation. In earlier versions of Java, you can cast the <code>int</code> to a <code>long</code> and then mask off the high bits: <code>((long) i) &amp; 0xffffffffL</code>.)</p>

<p>The second problem is that you are using the wrong version of <code>getBytes()</code>. The one that takes no argument converts a Unicode <code>String</code> to a <code>byte[]</code> using the default platform encoding, which may vary depending on how your system is set up. It could be giving you UTF-8, Latin1, Windows-1252, KOI8-R, Shift-JIS, EBCDIC, etc.</p>

<p><strong>Never, ever, ever call the no arguments version of <code>String.getBytes()</code>, under any circumstances. It should be deprecated, decimated, defenestrated, destroyed, and deleted.</strong></p>

<p>Use <code>s.getBytes("UTF-8")</code> (or whatever encoding you're expecting to get) instead.</p>

<p>As the <em>Zen of Python</em> says, "Explicit is better than implicit."</p>

<p>I can't tell if there may be any other problems beyond these two.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I had the same problem with you. But the Java version of my Murmurhash3 is different from yours. After making some changes to the C++ version of Murmurhash3, I made the hash values generated from the two versions the same. I give you my solution, which you can use to check if it also works for you.<br/><br/>Maybe the biggest difference between the Java and C++ versions lies in the <strong>right shift operation(in Java you can see >> and >>>, while in C++ you can only see >>)</strong>. The integers in Java are all signed, while in C++ you can use signed or unsigned integers. In Java version, the <strong>>></strong> means arithmetic right shift and the <strong>>>></strong> means logical right shift. And in C++, the <strong>>></strong> means arithmetic right shift. The original C++ version of Murmurhash3 uses unsigned integer, and in order to generate the negative hash value like in Java, <strong>first</strong> in C++ you should change all the unsigned type <strong>uint32_t</strong> to signed type <strong>int32_t</strong>. <strong>Then</strong> you should locate the <strong>>>></strong> in Java and make changes around the corresponding <strong>>></strong> in C++. For me, <strong>I change from：</strong>    </p>

<pre><code>inline uint32_t rotl32 ( uint32_t x, int8_t r )
{
  return (x &lt;&lt; r) | (x &gt;&gt; (32 - r));
}
</code></pre>

<p><strong>to:</strong></p>

<pre><code>inline int32_t rotl32 ( int32_t x, int8_t r )
{
  return (x &lt;&lt; r) | (int32_t)((uint32_t)x &gt;&gt; (32 - r)); //similar to &gt;&gt;&gt; in Java
}
</code></pre>

<p><strong>and from:</strong></p>

<pre><code>FORCE_INLINE uint32_t fmix32 ( uint32_t h )
{
  h ^= h &gt;&gt; 16;
  h *= 0x85ebca6b;
  h ^= h &gt;&gt; 13;
  h *= 0xc2b2ae35;
  h ^= h &gt;&gt; 16;

  return h;
}
</code></pre>

<p><strong>to:</strong></p>

<pre><code>FORCE_INLINE int32_t fmix32 ( int32_t h )
{
  h ^= (int32_t)((uint32_t)h &gt;&gt; 16); // similar to &gt;&gt;&gt; in Java
  h *= 0x85ebca6b;
  h ^= (int32_t)((uint32_t)h &gt;&gt; 13);
  h *= 0xc2b2ae35;
  h ^= (int32_t)((uint32_t)h &gt;&gt; 16);

  return h;
}
</code></pre>

<p>In this way, my two versions of Murmurhash3 in Java and C++ generate the same hash value.</p>
