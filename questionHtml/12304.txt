<td class="answercell">
    <div class="post-text" itemprop="text">
<p>The problem could be that the namenode was formatted after the cluster was set up and the datanodes were not, so the slaves are still referring to the old namenode. </p>

<p>We have to delete and recreate the folder /home/hadoop/dfs/data on the local filesystem for the datanode. </p>

<ul>
<li>Check your hdfs-site.xml file to see where dfs.data.dir is pointing to </li>
<li>and delete that folder </li>
<li>and then restart the datanode daemon on the machine </li>
</ul>

<p>The steps above should recreate the folder and resolve the problem. </p>

<p>Please share your config info if the instructions above do not work.</p>

<p>DataNode dies because of incompatible Clusterids. To fix this problem
If you are using hadoop 2.X, then you have to delete everything in the folder that you have specified in hdfs-site.xml - "dfs.datanode.data.dir" (but NOT the folder itself).</p>

<p>The ClusterID will be maintained in that folder. Delete and restart dfs.sh. This should work!!!</p>

<ol>
<li>You need to delete both 
<br><strong>C:\hadoop\data\dfs\datanode</strong> and 
<br><strong>C:\hadoop\data\dfs\namenode</strong> folders.<br></li>
</ol>

<p>If you don't have this folders - open your <strong>C:\hadoop\etc\hadoop\hdfs-site.xml</strong> file and get pathes for this folders for next deletion. For me it says:</p>

<pre><code> &lt;property&gt;
        &lt;name&gt;dfs.namenode.name.dir&lt;/name&gt;
        &lt;value&gt;file:/hadoop/data/dfs/namenode&lt;/value&gt;
    &lt;/property&gt;
    &lt;property&gt;
        &lt;name&gt;dfs.datanode.data.dir&lt;/name&gt;
        &lt;value&gt;file:/hadoop/data/dfs/datanode&lt;/value&gt;
    &lt;/property&gt;
</code></pre>

<ol start="2">
<li><strong>Run command for Format namenode</strong><br><code>c:\hadoop\bin&gt;hdfs namenode -format</code><br><br>
Now it should works!</li>
</ol>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I think the recommended way of doing this without deleting the data directory is to simply change the clusterID variable in the datanode's VERSION file. </p>

<p>If you look in your daemons directory, you will see the datanode directory exmaple</p>

<pre><code>data/hadoop/daemons/datanode
</code></pre>

<p>The VERSION file should look like this.</p>

<pre><code>cat current/VERSION 
#Tue Oct 14 17:31:58 CDT 2014
storageID=DS-23bf7f3a-085c-4531-808f-801ff6d52d14
clusterID=CID-bb3547b0-03e4-4588-ac25-f0299ff81e4f
cTime=0
datanodeUuid=63154929-ae68-4149-9f75-9a6558545041
storageType=DATA_NODE
layoutVersion=-55
</code></pre>

<p>You need to change the clusterId to the first value in the output of the message so in your case that would be CID-1745a89c-fb08-40f0-a14d-d37d01f199c3 instead of CID-bb3547b0-03e4-4588-ac25-f0299ff81e4f</p>

<p>The updated version should appear like this with the altered clusterId</p>

<pre><code>  cat current/VERSION 
    #Tue Oct 14 17:31:58 CDT 2014
    storageID=DS-23bf7f3a-085c-4531-808f-801ff6d52d14
    clusterID=CID-1745a89c-fb08-40f0-a14d-d37d01f199c3
    cTime=0
    datanodeUuid=63154929-ae68-4149-9f75-9a6558545041
    storageType=DATA_NODE
    layoutVersion=-55
</code></pre>

<p>Restart hadoop and the datanode should start just fine.</p>
