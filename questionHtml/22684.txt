    <div class="post-text" itemprop="text">

<p>I am implementing a JSON RecordReader in Hadoop with Jackson.
By now I am testing locally with JUnit + MRUnit.
The JSON files contain one object each, that after some headers, it has a field whose value is an array of entries, each of which I want to be understood as a Record (so I need to skip those headers).</p>

<p>I am able to do this by advancing the FSDataInputStream up to the point of reading.
In my local testing, I do the following:</p>

<pre><code>fs = FileSystem.get(new Configuration());
in = fs.open(new Path(filename));
long offset = getOffset(in, "HEADER_START_HERE");       
in.seek(offset);
</code></pre>

<p>where getOffset is a function where points the InputStream where the field value starts - which works OK, if we look at <code>in.getPos()</code> value.</p>

<p>I am reading the first record by: </p>

<pre><code>ObjectMapper mapper = new ObjectMapper();
JsonNode actualObj = mapper.readValue (in, JsonNode.class);
</code></pre>

<p>The first record comes back fine. I can use <code>mapper.writeValueAsString(actualObj)</code> and it has read it fine, and it was valid.</p>

<p>Fine till here.</p>

<p>So I try to iterate the objects, by doing: </p>

<pre><code>ObjectMapper mapper = new ObjectMapper();
JsonNode actualObj = null;
do {
    actualObj = mapper.readValue (in, JsonNode.class);
    if( actualObj != null) {
        LOG.info("ELEMENT:\n" + mapper.writeValueAsString(actualObj) );
    }
} while (actualObj != null) ;
</code></pre>

<p>And it reads the first one, but then it breaks: </p>

<pre><code>java.lang.NullPointerException: null
    at org.apache.hadoop.fs.BufferedFSInputStream.getPos(BufferedFSInputStream.java:54)
    at org.apache.hadoop.fs.FSDataInputStream.getPos(FSDataInputStream.java:57)
    at org.apache.hadoop.fs.ChecksumFileSystem$ChecksumFSInputChecker.readChunk(ChecksumFileSystem.java:243)
    at org.apache.hadoop.fs.FSInputChecker.readChecksumChunk(FSInputChecker.java:273)
    at org.apache.hadoop.fs.FSInputChecker.read1(FSInputChecker.java:225)
    at org.apache.hadoop.fs.FSInputChecker.read(FSInputChecker.java:193)
    at java.io.DataInputStream.read(DataInputStream.java:132)
    at org.codehaus.jackson.impl.ByteSourceBootstrapper.ensureLoaded(ByteSourceBootstrapper.java:340)
    at org.codehaus.jackson.impl.ByteSourceBootstrapper.detectEncoding(ByteSourceBootstrapper.java:116)
    at org.codehaus.jackson.impl.ByteSourceBootstrapper.constructParser(ByteSourceBootstrapper.java:197)
    at org.codehaus.jackson.JsonFactory._createJsonParser(JsonFactory.java:503)
    at org.codehaus.jackson.JsonFactory.createJsonParser(JsonFactory.java:365)
    at org.codehaus.jackson.map.ObjectMapper.readValue(ObjectMapper.java:1158)
</code></pre>

<p>Why is this exception happening? </p>

<p>Does it have to do with being reading locally?</p>

<p>Is it needed some kind of reset or something when reusing an <code>ObjectMapper</code> or its underlying stream?</p>

    <div class="post-text" itemprop="text">
<p>I managed to work it around. In case it helps: </p>

<p>First of all, I'm using Jackson 1.x latest version.
It seems that once <code>JsonParser</code> is instantiated with an <code>InputStream</code>, it takes control over it.
So, when using <code>readValue()</code>, once it is read (internally it calls <code>_readMapAndClose()</code> which automatically closes the stream.
There is a setting that you can set to tell the <code>JsonParser</code> not to close the underlying stream. You can pass it to your <code>JsonFactory</code> like this before your create your <code>JsonParser</code>:</p>

<pre><code>JsonFactory f = new MappingJsonFactory();
f.configure(JsonParser.Feature.AUTO_CLOSE_SOURCE, false);
</code></pre>

<p>Beware you are responsible for closing the stream (FSDataInputStream in my case). 
So, answers:</p>

<ul>
<li>Why is this exception happening?</li>
</ul>

<p>Because the parser manages the stream, and closes it after readValue().</p>

<ul>
<li>Does it have to do with being reading locally?</li>
</ul>

<p>No</p>

<ul>
<li>Is it needed some kind of reset or something when reusing an ObjectMapper or its underlying stream?</li>
</ul>

<p>No. What you need to be aware of when using Streaming API mixed with ObjectMapper-like methods, is that sometimes the mapper/parser may take control of the underlying stream. Refer to the Javadoc of <a href="http://jackson.codehaus.org/1.9.9/javadoc/org/codehaus/jackson/JsonParser.html" rel="nofollow">JsonParser</a> and check the documentation on each of the reading methods to meet your needs.</p>
