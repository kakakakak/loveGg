    <div class="post-text" itemprop="text">

<p>We have a Java project. We enable <code>-Xlint</code> (enable warnings) and <code>-Werror</code> (treat warning as error) flags for <code>javac</code>, to make sure our code is warning-free. Recently we decide to deprecate a class. The problem is in some cases <code>@SuppressWarnings("deprecation")</code> will not suppress the deprecation warning at all, resulting in build failure. Below is a list of use cases that I ran into:</p>

<ol>
<li>Imported in other non-deprecated classes.</li>
<li>Imported in other deprecated classes.</li>
<li>Parent class.</li>
<li><p>Type parameter. For example</p>

<pre><code>@SuppressWarnings("deprecation")
public class Foo extends Bar&lt;DeprecatedClass&gt;
{ ... }
</code></pre>

<p>However, this one has no warning even without suppress:</p>

<pre><code>@Deprecated
public class DeprecatedClass extends Bar&lt;DeprecatedClass&gt;
{ ... }
</code></pre></li>
</ol>

<p>AFAIK, there is no syntax for annotating imports, so for case 1 and 2 our solution is to either import * or avoid importing. For case 3 and 4, both Java 6 and 7 do not suppress the warning. Java 8 will correctly suppress it (maybe a bug is fixed). So far no solution for this.</p>

<p>Unfortunately, we have to support Java 6, 7 and 8 at this point. Is there way to deal with the problem? It is a road block for our Java API evolution.</p>

<p><strong>ADDENDUM</strong></p>

<p>Many people ask why do we still use the deprecated class in our own codebase. The reason is that the project is a library, supporting many different clients. When introducing new replacement API, we have to first deprecate our old API, keep it in our codebase, wait for all clients to migrate then remove it. There are three common use cases:</p>

<ul>
<li>We deprecate class <code>Foo</code> and <code>Bar</code>, where <code>Foo</code> extends <code>Bar</code>. This is the case 2 and 3 in my question.</li>
<li>We deprecate class <code>Foo</code> and <code>Bar</code>, where <code>Foo</code> extends <code>Collection&lt;Bar&gt;</code>. This is the case 2 and 4.</li>
<li>We must keep all test code for class <code>Foo</code> and <code>Bar</code>. The test code imports these classes. This is the case 1.</li>
</ul>

<p>Why keep the test? Don't forget that if a serious bug (e.g. memory leak, security issue) is discovered, and the clients can't easily migrate to the new version, we still need to provide bug fix to the old API. And all changes must be tested.</p>

<p>I feel our situation should be fairly common in software library development and API evolution. Surprisingly it took Java such long time (until Java 8) to fix the bug.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I'm sorry to say that I don't have a solution to the problem you're facing, though as you've observed, there has been some progress. We've been trying to get rid of all the Java compilation warnings in the JDK itself, and this has been a long, difficult process. During JDK 8 development in 2011 I helped <a href="http://stuartmarks.wordpress.com/2011/11/30/jdk8-warnings-cleanup-day/" rel="nofollow">kick off the warnings cleanup effort</a> and I later <a href="http://stuartmarks.wordpress.com/2012/10/09/javaone-2012-dare-to-compile-your-code-with-xlintall-werror/" rel="nofollow">co-presented a JavaOne talk</a> (<a href="https://oracleus.activeevents.com/connect/sessionDetail.ww?SESSION_ID=7147" rel="nofollow">slides and audio</a>) on the subject.</p>

<p>More recently, my colleage Joe Darcy has continued the warnings cleanup work and has worked through the different warnings categories and has <a href="http://mail.openjdk.java.net/pipermail/jdk9-dev/2014-August/001115.html" rel="nofollow">finally reached deprecation warnings</a>. As you noted, there have been some bugs in the compiler's handling of suppression of deprecation warnings, such as <a href="https://bugs.openjdk.java.net/browse/JDK-6480588" rel="nofollow">JDK-6480588</a> which was fixed in JDK 8. Unfortunately, it is still not possible in JDK 8 to suppress warnings on imports of deprecated items. This bug, <a href="https://bugs.openjdk.java.net/browse/JDK-8032211" rel="nofollow">JDK-8032211</a>, was fixed quite recently in our JDK 9 development line. In fact, we're still tuning up the handling of the <code>@Deprecated</code> annotation. For example, bug <a href="https://bugs.openjdk.java.net/browse/JDK-6481080" rel="nofollow">JDK-6481080</a> clarifies that attempting to use <code>@Deprecated</code> in a <code>package-info.java</code> file does not in fact deprecate the package; this bug was fixed just last week. And there is <a href="https://oracleus.activeevents.com/2014/connect/sessionDetail.ww?SESSION_ID=6377" rel="nofollow">more work to be done</a> but it's somewhat speculative at this point.</p>

<p>The JDK is facing similar problems to yours, in that we have to maintain deprecated APIs for clients that are still using them. But since we use and implement such APIs internally, we have a lot of deprecation warnings to suppress. As of this writing, in our JDK 9 development line, we <em>still</em> have not been able to compile the system without deprecation warnings. As a result, the <code>javac</code> options for lint warnings are still: </p>

<pre><code>-Xlint:all,-deprecation
</code></pre>

<p>You will probably have to disable deprecation warnings in your compilation as well, especially if you are still building on JDK 6. I don't see a way around it at this point.</p>

<p>One final note on one of your deprecation cases:</p>

<pre><code>@Deprecated
public class DeprecatedClass extends Bar&lt;DeprecatedClass&gt; { ... }
</code></pre>

<p>This does not issue a deprecation warning, nor should it. The <a href="http://docs.oracle.com/javase/specs/jls/se8/html/jls-9.html#jls-9.6.4.6" rel="nofollow">Java Language Specification, section 9.6.4.6</a>, specifies that deprecation warnings are not issued if the use of a deprecated entity is within an entity that is itself deprecated.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Normally, when you deprecate class, you don't want anybody to use it in later versions. Also, YOUR codebase should stop using deprecated class too. It looks strange, when you say everybody not to use MySuperDeprecatedUtil class but continue using it in your codebase.</p>

<p>If you need to use your MySuperDeprecatedUtil class in some other class - you should mark class where you use it as @Deprecated - every class that used deprecated code should be either deprecated, or would produce compilation warnings, or should be removed, or should stop using deprecated code.</p>

<p>If you can't stop using your class - maybe it's too early to deprecate it?</p>

<p>In my practice, when I want to deprecate some class, I create replacement class e.g. MySuperFreshUtil. Switch all classess using MySuperDeprecatedUtil to MySuperFresh util preserving interfaces where possible(if not possible - use FQCN and mark method as deprecated). Mark MySuperDeprecatedUtil as @Deprecated and add comment which class and how should be used instead. Then I commit this changes in single changelist. </p>
