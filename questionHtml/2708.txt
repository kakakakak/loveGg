    <div class="post-text" itemprop="text">

<p>So in a program I'm writing I use a bi-directional breadth first search to search a graph. I do this by running 1 breadth first search in 1 thread, and one in another. Now the search is said to have found the optimal solution when it either an element from the other search is hit, or when the goal is found (which never really happens, but just in case it does for some reason..).</p>

<p>The problem I am running into, is that I need to save this optimal solution to a field, because I need to continue to find all of the solutions, but the field value is getting messed up because both threads hit it at the same time (I think).</p>

<p>Is there a way to block access to the thread who gets there last? I've tried using an AtomicReference and its compareAndSet method but that didn't do the trick. The value still gets messed up....</p>

<p>Btw I'm using java, and for the threads I'm using Callable objects.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Looks like you have a possible <code>Livelock</code> or <a href="http://en.wikipedia.org/wiki/Race_condition" rel="nofollow"><code>Race condition</code></a> that is occuring because of the random order of threads. I'd recommending taking a different approach. (otherwise at some point you will run into <a href="http://en.wikipedia.org/wiki/NP-complete" rel="nofollow"><code>NP-Complete</code></a>).</p>

<blockquote>
  <p>The problem I am running into, is that I need to save this <strong>optimal solution</strong> to a field. <strong>because I need to continue to find all of the solutions</strong>, but the field value is getting messed up because both threads hit it at the same time (I think).</p>
</blockquote>

<p>There is a way to greatly increase your current technique. You shouldn't need to look at every solution, take a <a href="http://en.wikipedia.org/wiki/Greedy_algorithm" rel="nofollow"><code>Greedy Approach</code></a> and use a <a href="http://www.cslab.ece.ntua.gr/~anastop/files/papers/mtaap09dijkstra.pdf" rel="nofollow"><code>Paralleled version</code></a> of <a href="http://en.wikipedia.org/wiki/Dijkstra%27s_algorithm" rel="nofollow"><code>Dijkstra's Shortest Path Algorithm</code></a>.</p>

<p><strong>Shortest Path (or in your case <em>Optimal Solution</em>)</strong></p>

<blockquote>
  <p>Dijkstra's Algorithm is a graph search algorithm that solves the single-source shortest path problem for a graph with nonnegative edge path costs, producing a shortest path tree. This algorithm is often used in routing and as a subroutine in other graph algorithms.</p>
</blockquote>

<p><strong>Linear Implementation</strong></p>

<p>The original algorithm Figure 1. (<a href="http://www.cslab.ece.ntua.gr/~anastop/files/papers/mtaap09dijkstra.pdf" rel="nofollow">Source</a>)</p>

<p><img src="http://iforce.co.nz/i/w4b43s1r.34o.png" alt="Parallel PDF Algorithm Linear"></p>

<p>Java Implementations can be found <a href="http://renaud.waldura.com/doc/java/dijkstra/" rel="nofollow">Here</a>, and <a href="http://www.vogella.com/articles/JavaAlgorithmsDijkstra/article.htm" rel="nofollow">Here</a> </p>

<ul>
<li><em>I think these implementations are based on BSP Trees (but you should get the idea)</em></li>
</ul>

<p><strong>Parallel Implementation</strong></p>

<p>The parallel algorithm Figure 2. (<a href="http://www.cslab.ece.ntua.gr/~anastop/files/papers/mtaap09dijkstra.pdf" rel="nofollow">Source</a>)</p>

<ul>
<li>There are additional ways to speed up the algorithm with <code>recursion</code>, if you want to get classy.</li>
</ul>

<p><img src="http://iforce.co.nz/i/15ndcts5.p3a.png" alt="Parallel PDF Algorithm Atomic"></p>

<p>Another idea if you are still having problems, might be to use a different concurrent data structure like <code>Map Reduce</code> or <code>Hadoop</code> instead of using <code>Threads</code> to search through a <code>Binary Tree</code>, to fix your search.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p><strong>Note: not an answer, just checking the validity of the algorithm</strong></p>

<p>Does your algorithm actually yield the shortest path?  Consider this graph:</p>

<pre><code> 1A - 2A - 3A - 4A - 5A
  \             /     
   -- 2B -------  
</code></pre>

<p>And assume perfect interweaving (the threads each visit nodes at the exact same rate).  Thread <code>X</code> starts at <code>1A</code> and thread <code>Y</code> starts at <code>5A</code>.  The order would look like this:</p>

<pre><code>X visits 1A +{2A, 2B}
Y visits 5A +{4A}
X visits 2A +{3A}
Y visits 4A +{3A, 2B}
X visits 2B +{4A}
Y visits 3A +{2A}
X visits 3A (overlap; shortest path is computed to be 4)
</code></pre>

<p>But we know from inspection that the shortest path is 3: <code>1A - 2B - 4A - 5A</code>.  </p>

<p>How does your approach prevent this case?  No matter which approach you take for checking for duplicates, I always see <code>3A</code> overlapping before <code>2B</code>.  Do you always finish out the "level" before making a decision on what the optimal path length is?</p>
