    <div class="post-text" itemprop="text">

<p>I currently have CDH 5 installed on CentOS 6.5 with java jdk1.7 and I am trying to get CDH to use jdk1.8. </p>

<p>I do know that Java 1.8 is not a recommended version of CDH, but it is only a test cluster, so it isn't a big deal. </p>

<p>I have installed both Java 1.7 and Java 1.8 from Oracle's website using the RPM installation, so both versions of Java are currently under <code>/usr/java</code>. Using <code>ls -ld</code> my Java directory looks like:</p>

<pre><code>/usr/java/default -&gt; /usr/java/latest
/usr/java/jdk1.7.0_75
/usr/java/jdk1.8.0_31
/usr/java/latest -&gt; /usr/java/jdk1.8.0_31
</code></pre>

<p>I also have script set up in <code>/etc/profile.d</code> to set <code>$JAVA_HOME</code> to <code>/usr/java/default</code>. The contents of my profile.d script: </p>

<pre><code>export JAVA_HOME=/usr/java/default
export PATH=${JAVA_HOME}/bin:${PATH}
</code></pre>

<p>So when felt that I have this right, I run:</p>

<pre><code>$ which java
/usr/java/default/bin/java
</code></pre>

<p>Telling me that it is pointing to the version of Java symlinked in default. And to determine which version of java is running, I run:</p>

<pre><code>$ java -version
java version "1.8.0_31"
Java(TM) SE Runtime Environment (build 1.8.0_31-b13)
Java HotSpot(TM) 64-Bit Server VM (build 25.31-b07, mixed mode)
</code></pre>

<p>And I can see that I have Java 1.8 currently running.</p>

<p>Everything seems great, except when I try to start a Hadoop service. The easiest to start is ZooKeeper, because it only has one service. HDFS has multiple servers so it is more work that to just start and stop ZooKeeper.</p>

<p>Starting ZooKeeper is done with the following command:</p>

<pre><code>$ sudo service zookeeper-server start
</code></pre>

<p>Then to check which version of java it is running, I search the running processes list for <code>java</code>:</p>

<pre><code>$ ps -ef | grep java
495       7170     1  7 12:27 ?        00:00:00 /usr/java/jdk1.7.0_75/bin/java -Dzookeeper.datadir.autocreate=false -Dzookeeper.log.dir=/var/log/zookeeper -Dzookeeper.root.logger=INFO,ROLLINGFILE -cp /usr/lib/zookeeper/bin/../build/classes:/usr/lib/zookeeper/bin/../build/lib/*.jar:/usr/lib/zookeeper/bin/../lib/slf4j-log4j12.jar:/usr/lib/zookeeper/bin/../lib/slf4j-log4j12-1.7.5.jar:/usr/lib/zookeeper/bin/../lib/slf4j-api-1.7.5.jar:/usr/lib/zookeeper/bin/../lib/netty-3.2.2.Final.jar:/usr/lib/zookeeper/bin/../lib/log4j-1.2.16.jar:/usr/lib/zookeeper/bin/../lib/jline-0.9.94.jar:/usr/lib/zookeeper/bin/../zookeeper-3.4.5-cdh5.3.0.jar:/usr/lib/zookeeper/bin/../src/java/lib/*.jar:/etc/zookeeper/conf::/etc/zookeeper/conf:/usr/lib/zookeeper/*:/usr/lib/zookeeper/lib/* -Dzookeeper.log.threshold=INFO -Dcom.sun.management.jmxremote -Dcom.sun.management.jmxremote.local.only=false org.apache.zookeeper.server.quorum.QuorumPeerMain /etc/zookeeper/conf/zoo.cfg
</code></pre>

<p>I know that runs off the screen, bu the important part is that Zookeeper is being started by <code>/usr/java/jdk1.7.0_75/bin/java</code>.</p>

<p>To fix this, I have tried a few things: </p>

<ol>
<li><p>Looking at the conf files for Hadoop and ZooKeeper under <code>/etc/hadoop/conf</code> and <code>/etc/zookeeper/conf</code>, respectively.</p>

<p>I found nothing setting <code>JAVA_HOME</code>.</p></li>
<li><p>Looking at the <code>/usr/bin/zookeeper</code> script to see if <code>JAVA_HOME</code> was set elsewhere, </p>

<p>I did find the script <code>/usr/lib/bigtop-utils/bigtop-detect-javahome</code> has the ability to set <code>JAVA_HOME</code>, but my profile.d script overrides that.</p></li>
<li><p>Manually moving <code>/usr/java/jdk1.7</code> to <code>/tmp</code>.</p>

<p>Sadly, this is the only thing that works. When I move the jdk1.7 dir to another directory, and start ZooKeeper, it will use Java 1.8. Moving the jdk1.7 dir back, reverts to ZooKeeper using Java 1.7.</p></li>
</ol>

<p>Has anyone dealt with this problem and does anyone know how to deal with this? I feel that I have Java set up correctly, but something is telling ZooKeeper/Hadoop to use an old version of Java?</p>

    <div class="post-text" itemprop="text">
<p>I hate to answer my own questions, but here is the answer:</p>

<p>The wrong version of <code>$JAVA_HOME</code> is getting set for 2 reasons:</p>

<ol>
<li><p>Using the command <code>service</code> removes most environmental variables. From <code>man service</code>:</p>

<pre><code>service  runs a System V init script in as predictable environment as 
possible, removing most environment variables and with current work-
ing directory set to /.
</code></pre></li>
<li><p>The <code>/usr/lib/bigtop-utils/bigtop-detect-javahome</code> script has the ability to be configured with the environment variable <code>BIGTOP_JAVA_MAJOR</code> to manually set which version of Java to use. I tried setting this as an environment variable, but <code>service</code> removes it :(. It is also important to note that this script will find all versions of java installed, but the order of preference is Java 6, Java 7, Java 8, Open Java. So if you have Java 6, and 8 installed, it will prefer 6 over 8. </p></li>
</ol>

<p>In short, to fix my problem, I added the following to the top of <code>/usr/lib/bigtop-utils/bigtop-detect-javahome</code>:</p>

<pre><code>BIGTOP_JAVA_MAJOR=8
</code></pre>

<p>You can also set <code>JAVA_HOME</code> in this file to specify a specific version or path.</p>
