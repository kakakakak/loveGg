<p>While working on <strong>Hadoop</strong> Implementation in Pseudo-Distributed Operation, I found following exception of <strong>JAVA_HOME</strong> variable not setting, but When I tried to echo it, it was set. </p>

<p>Variable is preset in <code>conf/hadoop-env.sh</code> (edited <code>export JAVA_HOME=/usr/lib/jvm/java-6-sun</code>), <code>bash.bashrc</code>.</p>

<pre><code>vardan@vardan-HP-G62-Notebook-PC:~/hadoop-0.20.203.0$ echo $JAVA_HOME
/usr/lib/jvm/java-6-sun
vardan@vardan-HP-G62-Notebook-PC:~/hadoop-0.20.203.0$ bin/start-all.sh
starting namenode, logging to /home/vardan/hadoop-0.20.203.0/bin/../logs/hadoop-vardan-namenode-vardan-HP-G62-Notebook-PC.out
localhost: starting datanode, logging to /home/vardan/hadoop-0.20.203.0/bin/../logs/hadoop-vardan-datanode-vardan-HP-G62-Notebook-PC.out
localhost: Error: JAVA_HOME is not set. 
localhost: starting secondarynamenode, logging to /home/vardan/hadoop-0.20.203.0/bin/../logs/hadoop-vardan-secondarynamenode-vardan-HP-G62-Notebook-PC.out
localhost: Error: JAVA_HOME is not set. 
starting jobtracker, logging to /home/vardan/hadoop-0.20.203.0/bin/../logs/hadoop-vardan-jobtracker-vardan-HP-G62-Notebook-PC.out 
localhost: starting tasktracker, logging to /home/vardan/hadoop-0.20.203.0/bin/../logs/hadoop-vardan-tasktracker-vardan-HP-G62-Notebook-PC.out
localhost: Error: JAVA_HOME is not set.
</code></pre>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>The JAVA_HOME variable you set with <code>set JAVA_HOME=</code> is relevant only for the current shell. Given that you are starting a new shell when executing <code>bin/start-all.sh</code> you need to "export" the environmental variable to make it available in the global scope:</p>

<pre><code>export JAVA_HOME=/usr/lib/jvm/java-6-sun
</code></pre>

<h2>Installing java 1.6.x</h2>

<ol>
<li>Download "jdk-6u32-linux-i586.bin"</li>
<li><p>sh jdk-6u32-linux-i586.bin</p></li>
<li><p>mv /etc/alternatives/java /etc/alternatives/java_bak
mv /etc/alternatives/javac /etc/alternatives/javac_bak</p></li>
<li><p>create link 
ln -s /opt/jdk1.6.0_32/bin/java /etc/alternatives/java
ln -s /opt/jdk1.6.0_32/bin/javac /etc/alternatives/javac</p></li>
</ol>

<p>5.
  java -version</p>

<p>-----------you must see this --------------------------------
java version "1.6.0_32"
Java(TM) SE Runtime Environment (build 1.6.0_32-b05)</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>The problem is that the scripts that initializes dfs and mapreduce daemons do not have JAVA_HOME defined in its environment.<br><br>
Posibly, when executing HDFS and MapReduce, its execution environment is the one specified by the script $HADOOP_HOME/conf/hadoop-env.sh. Consequently, it would be enough to define JAVA_HOME at $HADOOP_HOME/conf/hadoop-env.sh :<br></p>

<pre><code>export JAVA_HOME=jdk-home-path
</code></pre>

<p>Otherwise, when this is not enough, the problem may be that the configuration environment used (hadoop-env.sh) is not the one we are expecting; hadoop is choosing another one or the default value if none found. The fastest solution will be defining the conf directory where the hadoop-env.sh script is placed. Setting the HADOOP_CONF_DIR environment variable will be enough:</p>

<pre><code>export HADOOP_CONF_DIR=hadoop-home-path/conf
</code></pre>
