<p>HDFS (or any distributed Filesystems) makes distributing your data much simpler. Using a local filesystem you would have to partition/copy the data by hand to the individual nodes and be aware of the data distribution when running your jobs. In addition HDFS also handles failing nodes failures.
From an integration between Spark and HDFS, you can imagine spark knowing about the data distribution so it will try to schedule tasks to the same nodes where the required data resides.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I doubt effectiveness of HDFS for Spark work loads for faster query execution. Have a look at this <a href="http://www.forbes.com/sites/johnwebster/2014/12/08/is-it-time-for-hadoop-alternatives/" rel="nofollow">Forbes article</a></p>

<p>At present, the majority of Spark implementations are tethered to Hadoop, either by using Hadoop’s HDFS as a data store or by running Spark concurrently with other applications within a Hadoop cluster. </p>

<p>However, Spark can use a number of different data stores as sources and repositories including Apache Cassandra and Amazon S3. Since it’s not bound to HDFS, it doesn’t inherit the shortcomings of HDFS. Spark can take data sets from other sources/tools with HIVE/PIG/HBASE etc.</p>

<p>Furthermore, it can use these data sources as an independent analytics tool in stand-alone mode—no dependence on Hadoop at all.</p>

<p><strong>EDIT:</strong></p>

<p><strong>Hadoop &amp; Spark</strong> : Both can co-exist and Spark cannot replace Hadoop.</p>

<p><strong>Spark</strong> is faster due to in memory processing. Can be used for interactive queries unlike Hadoop, which can be used for batch processing. If data volume is huge in multiple of Terrabytes, Spark cannot be quicker since you can't load that huge data in memory across VMs. So Spark uses HDFS as storage. When Spark created record of teragen sorting, it used HDFS for sorting instead of in-memory.</p>

<p><strong>Hadoop</strong> (with HDFS storage/ YARN architecture) will stay for batch processing of Terabytes/Petabytes of data. Batch processing does not care much about finish time. It is OK for batch job to complete in 45 minutes instead of 15 minutes is fine. </p>

<p>Have a look at <a href="http://www.slideshare.net/EdurekaIN/5-reasons-why-spark-is-in-demand?qid=df08873f-ae71-4d97-a9f4-5bc9601a40b4&amp;v=default&amp;b=&amp;from_search=2" rel="nofollow">Spark &amp; Hadoop </a> article.</p>
