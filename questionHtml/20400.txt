    <div class="post-text" itemprop="text">

<p>I have a table with friends <code>id, u1,</code> <code>u2</code> and about <code>&lt; 500,000</code> entries on a single mysql server</p>

<p>and i want to take <code>userA</code> and <code>userB</code> and check whether they have any mutual friends.</p>

<p>Is it faster to do </p>

<pre><code>select u2 from friends where u1 = userA and u2 IN (select u2 from friends where u1 = userB)
</code></pre>

<p>than to run a shortest path algorithm on a graph (on one server)?</p>

<p>What is the standard way, big networks like LinkedIn and Facebook use to handle this?</p>

<p>Thanks!</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>If the table friends is indexed by both u1 and u2, then SQL query is to take intersection of 2 subsets and is pretty fast. It is because indexing is already done. If you do computations in memory, time depends on whether you have prebuilt indexes: if you have, you'll be faster because of no database connection overhead. If indexing is included in computational time, and database is warmed (all data in memory), you can lost.</p>

<p>I'm talking on indexing, not shortest path algorithm, because shortest path algorithm computes more data than you need.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>In MySQL, the query that you wrote will be slower than any other way of finding this information.  Perhaps slower than asking each person individually.  Your query:</p>

<pre><code>select u2
from friends
where u1 = userA and
      u2 IN (select u2 from friends where u1 = userB)
</code></pre>

<p>Has a subquery in the IN clause.  MySQL evaluates the query for every row encountered.  A better way to write this is:</p>

<pre><code>select u2
from friends
where u1 = userA and
      exists (select 1 from friends where u1 = userB limit 1)
</code></pre>

<p>If your data all fits on one server and fits into memory, the performance of an optimized MySQL query should be fine.  Sites such as LinkedIn and FaceBook are dealing with a myriad of issues -- constant updates to the network, vastly larger amounts of data, different types of links, and so on.  Your simple example is not representative of what they are doing.  But, many of their analyses use Hadoop or Hadoop in conjunction with relational databases.</p>

<p>In a graph database you could write your query in <a href="https://github.com/tinkerpop/gremlin/wiki" rel="nofollow">gremlin</a> as:</p>

<pre><code>g.V('username','userB').out('friend').retain(g.V('username','userA').out('friend').gather)
</code></pre>

<p>Most graph databases should execute this quickly.</p>

<p>If you use Titan, you can additionally exploit that Titan maintains adjacent vertices in sort order, which mean that you can compute the intersection of the two friend lists using only one iteration over the data and without creating additional data structures. This will likely be faster than MySQL and a lot faster if the average number of friends is large.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Here's another take for second degree connections using a simple <code>inner join</code>:</p>

<pre><code>select fA.u2 
from friends fA 
inner join friends fB on
           fA.u2 = fB.u2 
where fA.u1 = userA and
      fB.u1 = userB
</code></pre>

<p>This is the same approach as a many to many type query. 
You don't need to use a shortest path for that level of relationship.</p>

<p>If you wish to look for larger degree relationships, then you should look into adjacency lists, but it's not easy to implement it using MySQL. There are some issues to really look after in that setup: </p>

<ul>
<li>disjoint graphs (could be handled by maintaining transitive closures on subgraphs, and merge them when required),</li>
<li>directed vs. undirected graph,</li>
<li>data distribution (another answer mentioned hadoop as a way to accelerate processing, but it requires a good partition scheme)</li>
</ul>

<p>to name a few.</p>
