    <div class="post-text" itemprop="text">

<p>Assume a client application that uses a <code>FileSplit</code> object in order to read the actual bytes from the corresponding file.</p>

<p>To do so, an <code>InputStream</code> object has to be created from the <code>FileSplit</code>, via code like:</p>

<pre><code>    FileSplit split = ... // The FileSplit reference
    FileSystem fs   = ... // The HDFS reference

    FSDataInputStream fsin = fs.open(split.getPath());

    long start = split.getStart()-1; // Byte before the first

    if (start &gt;= 0)
    {
        fsin.seek(start);
    }
</code></pre>

<p>The adjustment of the stream by -1 is present in some scenarios like the <code>Hadoop</code> <code>MapReduce</code> <code>LineRecordReader</code> class. However, the documentation of the <code>FSDataInputStream</code> <code>seek()</code> method says explicitly that, after seeking to a location, the next read will be from that location, meaning (?) that the code above will be 1 byte off (?).</p>

<p>So, the question is, would that "-1" adjustment be necessary for all InputSplit reading cases?</p>

<p>By the way, if one wants to read a <code>FileSplit</code> correctly, seeking to its start is not enough, because every split also has an end that may not be identical to the end of the actual HDFS file. So, the corresponding <code>InputStream</code> should be "bounded", i.e. have a maximum length, like the following:</p>

<pre><code>    InputStream is = new BoundedInputStream(fsin, split.getLength());
</code></pre>

<p>In this case, after the "native" <code>fsin</code> steam has been created above, the <code>org.apache.commons.io.input.BoundedInputStream</code> class is used, to implement the "bounding".</p>

<p><strong>UPDATE</strong></p>

<p>Apparently the adjustment is necessary only for use cases line the one of the <code>LineRecordReader</code> class, which exceeds the boundaries of a split to make sure that it reads the full last line.</p>

<p>A good discussion with more details on this can be found in an <a href="http://stackoverflow.com/questions/14291170/how-does-hadoop-process-records-records-split-across-block-boundaries">earlier question</a> and in the comments for <a href="https://issues.apache.org/jira/browse/MAPREDUCE-772" rel="nofollow">MAPREDUCE-772</a>. </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Seeking to position 0 will mean the next call to InputStream.read() will read byte 0. Seeking to position -1 will most probably throw an exception.</p>

<p>Where specifically are you referring to when you talk about the standard pattern in examples and source code?</p>

<p>Splits are not neccessarily bounded as you note - take TextInputFormat for example and files that can be split. The record reader that processes the split will:</p>

<ul>
<li>Seek to the start index, then find the next newline character</li>
<li>Find the next newline character (or EOF) and return that 'line' as the next record</li>
</ul>

<p>This repeats until either the next newline found is past the end of the split, or the EOF is found. So you see that i this case the actual bounds of a split might be right shifted from that given by the Input split</p>

<p><strong>Update</strong></p>

<p>Referencing this code block from LineRecordReader:</p>

<pre><code>if (codec != null) {
  in = new LineReader(codec.createInputStream(fileIn), job);
  end = Long.MAX_VALUE;
} else {
  if (start != 0) {
    skipFirstLine = true;
    --start;
    fileIn.seek(start);
  }
  in = new LineReader(fileIn, job);
}
if (skipFirstLine) {  // skip first line and re-establish "start".
  start += in.readLine(new Text(), 0,
                       (int)Math.min((long)Integer.MAX_VALUE, end - start));
}
</code></pre>

<p>The <code>--start</code> statement is most probably to deal with avoiding the split starting on a newline character and returning an empty line as the first record. You can see that if the seek occurs, the first line is skipped to ensure the file splits don't return overlapping records</p>
