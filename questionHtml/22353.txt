<p>I'm trying to maximize cluster usage for a simple task.</p>

<p>Cluster is 1+2 x m3.xlarge, runnning Spark 1.3.1, Hadoop 2.4, Amazon AMI 3.7</p>

<p>The task reads all lines of a text file and parse them as csv.</p>

<p>When I spark-submit a task as a yarn-cluster mode, I get one of the following result:</p>

<ul>
<li>0 executor: job waits infinitely until I manually kill it</li>
<li>1 executor: job under utilize resources with only 1 machine working</li>
<li>OOM when I do not assign enough memory on the driver</li>
</ul>

<p>What I would have expected:</p>

<ul>
<li>Spark driver run on cluster master with all memory available, plus 2 executors with 9404MB each (as defined by install-spark script).</li>
</ul>

<p>Sometimes, when I get a "successful" execution with 1 executor, cloning and restarting the step ends up with 0 executor.</p>

<p>I created my cluster using this command:</p>

<pre><code>aws emr --region us-east-1 create-cluster --name "Spark Test"
--ec2-attributes KeyName=mykey 
--ami-version 3.7.0 
--use-default-roles 
--instance-type m3.xlarge 
--instance-count 3 
--log-uri s3://mybucket/logs/ 
--bootstrap-actions Path=s3://support.elasticmapreduce/spark/install-spark,Args=["-x"] 
--steps Name=Sample,Jar=s3://elasticmapreduce/libs/script-runner/script-runner.jar,Args=[/home/hadoop/spark/bin/spark-submit,--master,yarn,--deploy-mode,cluster,--class,my.sample.spark.Sample,s3://mybucket/test/sample_2.10-1.0.0-SNAPSHOT-shaded.jar,s3://mybucket/data/],ActionOnFailure=CONTINUE
</code></pre>

<p>With some step variations including:</p>

<p>--driver-memory 8G --driver-cores 4 --num-executors 2</p>

<hr>

<p>install-spark script with -x produces the following spark-defaults.conf:</p>

<pre><code>$ cat spark-defaults.conf
spark.eventLog.enabled  false
spark.executor.extraJavaOptions         -verbose:gc -XX:+PrintGCDetails -XX:+PrintGCDateStamps -XX:+UseConcMarkSweepGC -XX:CMSInitiatingOccupancyFraction=70 -XX:MaxHeapFreeRatio=70
spark.driver.extraJavaOptions         -Dspark.driver.log.level=INFO
spark.executor.instances        2
spark.executor.cores    4
spark.executor.memory   9404M
spark.default.parallelism       8
</code></pre>

<p><strong>Update 1</strong></p>

<p>I get the same behavior with a generic JavaWordCount example:</p>

<pre><code>/home/hadoop/spark/bin/spark-submit --verbose --master yarn --deploy-mode cluster --driver-memory 8G --class org.apache.spark.examples.JavaWordCount /home/hadoop/spark/lib/spark-examples-1.3.1-hadoop2.4.0.jar s3://mybucket/data/
</code></pre>

<p>However, if I remove the '--driver-memory 8G', the task gets assigned 2 executors and finishes correctly.  </p>

<p>So, what's the matter with driver-memory preventing my task to get executors?</p>

<p>Should the driver be executed on the cluster's master node alongside with Yarn master container as explained <a href="http://badrit.com:8000/blog/2015/2/29/running-spark-on-yarn#.VXXXgvnBzRZ" rel="nofollow">here</a>?</p>

<p>How do I give more memory to my spark job driver? (Where collects and some other useful operations arise)</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>The solution to maximize cluster usage is to forget about the '-x' parameter when installing spark on EMR and to adjust executors memory and cores by hand.</p>

<p>This <a href="http://blog.cloudera.com/blog/2015/03/how-to-tune-your-apache-spark-jobs-part-2/" rel="nofollow">post</a> gives a pretty good explanation of how resources allocation is done when running Spark on YARN.</p>

<p>One important thing to remember is that all executors must have the same resources allocated!  As we speak, Spark does not support heterogeneous executors.  (Some work is currently being made to support GPUs but it's another topic)</p>

<p>So in order to get maximum memory allocated to the driver while maximizing memory to the executors, I should split my nodes like this (this <a href="http://www.slideshare.net/SandyRyza/spark-job-failures-talk" rel="nofollow">slideshare</a> gives good screenshots at page 25):</p>

<ul>
<li>Node 0 - Master (Yarn resource manager)</li>
<li>Node 1 - NodeManager(Container(Driver) + Container(Executor))</li>
<li>Node 2 - NodeManager(Container(Executor) + Container(Executor))</li>
</ul>

<p>NOTE: Another option would be to <code>spark-submit</code> with <code>--master yarn --deploy-mode client</code> from the master node 0.  Are there any counter example this is a bad idea?</p>

<p>In my example, I can have at most have 3 executors of 2 vcores with 4736 MB each + a driver with same specs.</p>

<p>4736 memory is derived from the value of <code>yarn.nodemanager.resource.memory-mb</code> defined in <code>/home/hadoop/conf/yarn-site.xml</code>.  On a m3.xlarge, it is set to 11520 mb (see <a href="http://docs.aws.amazon.com/ElasticMapReduce/latest/DeveloperGuide/TaskConfiguration_H2.html" rel="nofollow">here</a> for all values associated to each instance types)</p>

<p>Then, we get:</p>

<p>(11520 - 1024) / 2 (executors per nodes) = 5248 => 5120 (rounded down to 256 mb increment as defined in yarn.scheduler.minimum-allocation-mb)</p>

<p><strong>7%</strong> * 5120 = 367 rounded up to 384 (memory overhead) <strong>will become 10% in spark 1.4</strong></p>

<p>5120 - 384 = 4736</p>

<ul>
<li><a href="http://stackoverflow.com/questions/29940711/apache-spark-setting-executor-instances-does-not-change-the-executors">Apache Spark: setting executor instances does not change the executors</a></li>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>The issue is around the expectations for how Spark works on YARN.  When Spark is run with a deployment mode of cluster or master set to yarn-cluster the driver is <em>not</em> executed on the master node but in the Application Master container on one of the slave nodes. For more details see <a href="https://spark.apache.org/docs/latest/running-on-yarn.html" rel="nofollow">https://spark.apache.org/docs/latest/running-on-yarn.html</a></p>

<p>I expect what is happening is that the cluster cannot fulfill the memory requirements for the driver (remember that memory actually requested of the cluster is what you ask for plus an overhead) and thus waiting forever to allocate the Application Master where driver will run or for executors.</p>

<p>To give the driver the amount of memory you are requesting you would need to use additional slaves in order to provide resources for the cluster-based driver and executors at the same time.  With the overhead on the driver, I suspect you may need to utilize an instance type with more memory.  When you request 8G for the driver take a look at the resource manager log and verify the real amount requested.</p>

<p>To run the driver on the master node the deployment mode would need to be client.  This can still be done with EMR steps if you utilize one step to call a script to localize the driver jars onto the master node and then the next step can call spark-submit set for deployment mode client and referencing the JAR on the local master file system.</p>
