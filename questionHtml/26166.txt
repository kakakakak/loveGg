<pre><code>import java.io.InputStream;

{
    static

            URL.setURLStreamHandlerFactory(new FsUrlStreamHandlerFactory());
    }

    public static void main(String[] args) throws Exception

        InputStream in = null;
        try 
        {
            in = new URL(args[0]).openStream();
            IOUtils.copyBytes(in, System.out, 4096, false);

        {
            IOUtils.closeStream(in);

}

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>1) This is because of the bug in the FSURLStreamHandlerFactory class provided by hadoop. Please note that the bug is fixed in the latest jar which contains this class.</p>

<p>2) This file is located in hadoop-common-2.0.0-cdh4.2.1.jar. To understand the problem completely we have to understand how the java.net.URL class works.</p>

<h2>Working of URL object</h2>

<p>When we create a new URL using any one of its constructor without passing "URLStreamHandler" (either through passing null for its value or calling constructor which does not take URLStreamHandler object as its parameter) then internally it calls a method called getURLStreamHandler(). This method returns the URLStreamHandler object and sets a member </p>

<p>variable in URL class. </p>

<p>This object knows how to construct a connection of a particular scheme like "http", "file"... and so on. This URLStreamHandler is constructed by the factory called </p>

<p>URLStreamHandlerFactory.</p>

<p>3) In the problem example given above the URLStreamHandlerFactory was set to "FsUrlStreamHandlerFactory" by calling the following static method.</p>

<pre><code>    URL.setURLStreamHandlerFactory(new FsUrlStreamHandlerFactory());
</code></pre>

<p>So when we create a new URL then this "FSUrlStreamHandlerFactory" is used to create the URLStreamHandler object for this new URL by calling its createURLStreamHandler(protocol) method.</p>

<p>This method inturn calls a method called loadFileSystems() of FileSystem class. The loadFileSystems() method invokes the ServiceLoader.load("FileSystem.class") so it tries to read the binary names of the FileSystem implementation classes by searching all META-INF/services/*.FileSystem files of all jar files in classpath and reading its entries. </p>

<p>4) Remember that the each jar is handled as URL object meaning for each jar an URL object is created by the ClassLoader internally. The class loader supplies the URLStreamHandler object </p>

<p>when constructing the URL for these jars so these URLs will not be affected by the "FSUrlStreamHandlerFactory" we set because the URL has already having the "URLStreamHandler". Since we are </p>

<p>dealing with jar files the class loader sets the "URLStreamHandler" as of type "sun.net.www.protocol.jar.Handler". </p>

<p>5) Now inorder to read the entries inside the jar files for the FileSystem implementation classes the  "sun.net.www.protocol.jar.Handler" needs to construct the URL object for each entry by </p>

<p>calling the URL constructor without the URLStreamHandler object. Since we already defined the URLStreamHandlerFactory as "FSUrlStreamHandlerFactory" it calls the createURLStreamHandler</p>

<p>(protocol) method which causes to recurse indefinetly and lead to the "StackOverflowException". </p>

<p>This bug is known as the "HADOOP-9041" by the Hadoop committters. The link is <a href="https://issues.apache.org/jira/browse/HADOOP-9041" rel="nofollow">https://issues.apache.org/jira/browse/HADOOP-9041</a>.</p>

<p>I know this is somewhat complicated.</p>

<p>So in short the solution to this problem is given below.</p>

<p>1) Use the latest jar hadoop-common-2.0.0-cdh4.2.1.jar which has the fix for this bug</p>

<p>or</p>

<p>2) Put the following statement in the static block before setting the URLStreamHandlerFactory.</p>

<pre><code>      static {
               FileSystem.getFileSystemClass("file",new Configuration()); 
               URL.setURLStreamHandlerFactory(new FsUrlStreamHandlerFactory());
             } 
</code></pre>

<p>Note that the first statement inside the static block doesn't depend on FsUrlStreamHandlerFactory now and uses the default handler for file:// to read the file entires in META-INF/services/*.FileSystem files.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I have a <strong><em>workaround.</em></strong> </p>

<p>It would be great if someone more familiar with the current state of the Hadoop world (Jan 2014) would enlighten us and/or explain the behavior. </p>

<p>I encountered the same of StackOverflowError when trying to run URLCat from Haddop The Definitive Guide Third Edition Tom White</p>

<p>I have the problem with Cloudera QuickStart 4.4.0 and 4.3.0</p>

<p>Using both jdk1.6.0_32 and jdk1.6.0_45</p>

<p>The problem occurs during initializion/class loading of org.apache.hadoop.fs.FileSystem underneath java.net.URL
There is some kind of recursive exception handling that is kicking in. 
I did the best I could to trace it down. 
The path leads to java.util.ServiceLoader which then invokes sun.misc.CompoundEnumeration.nextElement()
Unfortunately, the source for sun.misc.CompoundEnumeration is <strong>not</strong> included in the jdk src.zip ... perhaps an oversight because it is in java package sun.misc</p>

<p>In an attempt to trigger the error through another execution path I came up with a workaround ...</p>

<p>You can avoid the conditions that lead to StackOverflowError by invoking org.apache.hadoop.fs.FileSystem.getFileSystemClass(String, Configuration) prior to registering the StreamHandlerFactory. </p>

<p>This can be done by modifying the static initialization block (see original listing above):</p>

<pre><code>   static {
        Configuration conf = new Configuration();
        try {
            FileSystem.getFileSystemClass("file", conf);
        } catch (Exception e) {
            throw new RuntimeException(e.getMessage());
        };
        URL.setURLStreamHandlerFactory(new FsUrlStreamHandlerFactory());
    }
</code></pre>

<p>This can also be accomplished by moving the contents of this static block to your main(). </p>

<p>I found another reference to this error from Aug 2011 at <a href="http://stackoverflow.com/questions/7038292/stackoverflow-with-fsurlstreamhandlerfactory">stackoverflow with FsUrlStreamHandlerFactory</a></p>

<p>I am quite puzzled that more hadoop newbies have not stumbled onto this problem ... buy the Hadoop book ... download Cloudera QuickStart ... try a very simple example ... <strong>FAIL!?</strong></p>

<p>Any insight from more experienced folks would be appreciated. </p>
