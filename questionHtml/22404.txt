<p>My cluster: 1 master, 11 slaves, each node has 6 GB memory.</p>

<p>My settings:</p>

<pre><code>spark.executor.memory=4g, Dspark.akka.frameSize=512
</code></pre>

<p><strong>Here is the problem:</strong></p>

<p><strong>First</strong>, I read some data (2.19 GB) from HDFS to RDD:</p>

<pre class="lang-scala prettyprint-override"><code>val imageBundleRDD = sc.newAPIHadoopFile(...)
</code></pre>

<p><strong>Second</strong>, do something on this RDD:</p>

<pre class="lang-scala prettyprint-override"><code>val res = imageBundleRDD.map(data =&gt; {
                               val desPoints = threeDReconstruction(data._2, bg)
                                 (data._1, desPoints)
                             })
</code></pre>

<p><strong>Last</strong>, output to HDFS:</p>

<pre class="lang-scala prettyprint-override"><code>res.saveAsNewAPIHadoopFile(...)
</code></pre>

<p>When I run my program it shows:</p>

<pre><code>.....
14/01/15 21:42:27 INFO cluster.ClusterTaskSetManager: Starting task 1.0:24 as TID 33 on executor 9: Salve7.Hadoop (NODE_LOCAL)
14/01/15 21:42:27 INFO cluster.ClusterTaskSetManager: Serialized task 1.0:24 as 30618515 bytes in 210 ms
14/01/15 21:42:27 INFO cluster.ClusterTaskSetManager: Starting task 1.0:36 as TID 34 on executor 2: Salve11.Hadoop (NODE_LOCAL)
14/01/15 21:42:28 INFO cluster.ClusterTaskSetManager: Serialized task 1.0:36 as 30618515 bytes in 449 ms
14/01/15 21:42:28 INFO cluster.ClusterTaskSetManager: Starting task 1.0:32 as TID 35 on executor 7: Salve4.Hadoop (NODE_LOCAL)
Uncaught error from thread [spark-akka.actor.default-dispatcher-3] shutting down JVM since 'akka.jvm-exit-on-fatal-error' is enabled for ActorSystem[spark]
java.lang.OutOfMemoryError: Java heap space
</code></pre>

<p>There are too many tasks?</p>

<p><strong>PS</strong>: Every thing is ok when the input data is about 225 MB. </p>

<p>How can I solve this problem?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I have a few suggestions:</p>

<ul>
<li>If your nodes have 6g, then use 6g rather than 4g, <code>spark.executor.memory=6g</code>. Make sure <strong>you're using all the memory</strong> by checking the UI (it will say how much mem you're using)</li>
<li>Try using more partitions, you should have 2 - 4 per CPU.  IME increasing the number of partitions is often the easiest way to make a program more stable (and often faster). For huge amounts of data you may need way more than 4 per CPU, I've had to use 8000 partitions in some cases!</li>
<li>Decrease the <strong>fraction of memory reserved for caching</strong>, using <code>spark.storage.memoryFraction</code>. If you don't use <code>cache()</code> or <code>persist</code> in your code, this might as well be 0.  It's default is 0.6, which means you only get 0.4 * 4g memory for your heap.  IME reducing the mem frac often makes OOMs go away. <strong>UPDATE:</strong> From spark 1.6 apparently we will no longer need to play with these values, spark will determine them automatically.</li>
<li>Similar to above but <strong>shuffle memory fraction</strong>.  If your job need much shuffle memory then set it to a lower value (this might cause your shuffles to spill to disk which can have catastrophic impact on speed).  Sometimes when it's a shuffle operation that's OOMing you need to do the opposite i.e. set it to something large, like 0.8, or make sure you allow your shuffles to spill to disk (it's the default since 1.0.0).</li>
<li>Watch out for <strong>memory leaks</strong>, these are often caused by accidentally closing over objects you don't need in your lambdas.  The way to diagnose is to look out for the "task serialized as XXX bytes" in the logs, if XXX is larger than a few k or more than an MB, you may have a memory leak.</li>
<li>Related to above; use <strong>broadcast variables</strong> if you really do need large objects.</li>
</ul>

<p><a href="http://spark.apache.org/docs/1.2.1/configuration.html">http://spark.apache.org/docs/1.2.1/configuration.html</a></p>

<p>EDIT: (So I can google myself easier) The following is also indicative of this problem:</p>

<pre><code>java.lang.OutOfMemoryError : GC overhead limit exceeded
</code></pre>

<p>Have a look at <a href="https://github.com/apache/spark/blob/master/bin/spark-class#L93" rel="nofollow">the start up scripts</a> a Java heap size is set there, it looks like you're not setting this before running Spark worker.</p>

<pre><code># Set SPARK_MEM if it isn't already set since we also use it for this process
SPARK_MEM=${SPARK_MEM:-512m}
export SPARK_MEM

# Set JAVA_OPTS to be able to load native libraries and to set heap size
JAVA_OPTS="$OUR_JAVA_OPTS"
JAVA_OPTS="$JAVA_OPTS -Djava.library.path=$SPARK_LIBRARY_PATH"
JAVA_OPTS="$JAVA_OPTS -Xms$SPARK_MEM -Xmx$SPARK_MEM"
</code></pre>

<p>You can find the documentation to deploy scripts <a href="http://spark.apache.org/docs/latest/spark-standalone.html" rel="nofollow">here</a>.</p>

<p>The location to set the memory heap size (at least in spark-1.0.0) is in conf/spark-env.
The relevant variables are <code>SPARK_EXECUTOR_MEMORY</code> &amp; <code>SPARK_DRIVER_MEMORY</code>.
More docs are in the <a href="http://spark.apache.org/docs/latest/cluster-overview.html" rel="nofollow">deployment guide</a></p>

<p>Also, don't forget to copy the configuration file to all the slave nodes.</p>
