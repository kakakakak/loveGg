<p>Suppose I have the the following event data scheme:</p>

<pre><code>event_record_unique_id: long
event_timestamp: long
session_id: long
event_id: int
event_data: data  # concrete type depends on event_id
</code></pre>

<p>... so, the contents of the data may depend on, let's say 500, event_ids, leading to 200 different concrete data types for "data". For example:</p>

<pre><code>{
   event_record_unique_id: 17126721
   event_timestamp: 1234
   session_id: 3452
   event_id: 50
   event_data: {
       user_id: 123
       page_id: 789
   }
}
{
   event_record_unique_id: 1712672123
   event_record_unique_id: 17126723
   event_timestamp: 1234
   session_id: 3454
   event_id: 51
   event_data: {
       user_id: 124
       button_id: 789
   }
}
{
   event_timestamp: 1234
   session_id: 3454
   event_id: 51
   event_data: {
       crash_report: "text"
       device_id: "12312"
   }
}
</code></pre>

<p>Also:</p>

<ul>
<li>many of the event_data attributes appear in many of the concrete event_data objects</li>
<li>I need to perform indexed searches on some of the event_data attributes (e.g. find me all the records where user_id=X )</li>
<li>there's a continuing need to keep on adding event types and new attributes</li>
<li>the above data structure is always trivially flattened so that a single record can be represented equivalently as a row with N columns where (and attributes name/type collision
 are solved by renaming attributes).</li>
</ul>

<p>The naive RDBMS approach would involved making ~500 tables (one per concrete type of "data"). I've discounted this approach (= excessive waste of human effort in modelling). Plus, I cannot easily search all records over user_id (since user_id appears in very many tables).</p>

<p>Flattening the structure in an RDBMS is also quite costly (N-8 of the elements are NULL and contain no information).</p>

<p>Mongodb-type document database solutions appear to be a good, however, space costs seems quite high if attribute names are held with each record, not much better than an RDBMS. However, this does allow me to index by fields in the data object.</p>

<p>For me, an ideal data representation of this would be a table that is optimized to allow rows with many null elements (e.g. by keeping an active column bitmask per row). Or a document DB in which a document collection maintains a library of document schemas used enable compacting the data (and each document having reference to its schema).</p>

<p>What kind of database would people recommend for the above example case?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>MS SQL Server 2008 and up have <a href="http://technet.microsoft.com/en-us/library/cc280604.aspx" rel="nofollow">Sparse Columns</a>. Up to 30,000 can be added in a table, and they can be indexed (filtered indexes are recommended). Or so says BOL, I have not used them myself. This would results in a single very large table that might support what you need.</p>

<p>With that said, I don't know it would be particularly efficient. Some math:</p>

<ul>
<li>Assume 10 rows a second</li>
<li>becomes 10*60*60*24 = 864,000 rows a day</li>
<li>or 315,360,000 rows a year</li>
<li>with a <em>very</em> rough over-estimate of 50 bytes a row</li>
<li>is about 14GB a year</li>
<li>for how many years do you have to keep the data?</li>
<li>and double that if it's more like 20 rows per second</li>
</ul>

<p>So storage doesn't seem too way out of line... but I don't know, you want to work up some serious size projection factors. And that's just storage, what do you want or need to <em>do</em> with the data? Is retrieval time for specified rows important? What about analysis and data mining? I'm a SQL guy through and through, and I think it could be done, but this pretty much is the kind of problem that Hadoop and NoSQL solutions were devised for, and it could well be worth your time to thoroughly investigating those options.</p>
