    <div class="post-text" itemprop="text">

<p>I've got a tree of 125,000 nodes (at most 2 children). I'm trying to determine the number of children (direct and indirect) of each node. Because the tree is a DAG <em>but</em> the number of links to each child is unbounded, many nodes will have virtually all of the other nodes as children. The total complexity of the tree, just for reference, is way over 10^30 if expressed without memoization. This means that, even storing a simple pointer to each child (and memoizing the output) produces a 15.625GB lump of data, even ignoring hash table, memory allocator, and other overhead.</p>

<p>While this is the desired output, it's taking a little too long and a little too much memory to achieve. I've only got a single workstation, of fair but not top-of-the-line power (i7 930, 6GB RAM).</p>

<p>Is there any way that I can memoize or otherwise cache the tables so that the data is still accessible in a reasonable period of time (I will be doing potentially hundreds of thousands of accesses on the data)? I've considered lazily evaluating the queries, but I'm concerned about how long it would take to access them.</p>

<p>In addition, I'm not specifically interested in which nodes are children, but I <em>do</em> need to know the <em>number</em> of them- which basically amounts to the same thing I believe, as I can't count the same child twice.</p>

<p>Edit: The tree is immutable. All I will do is read the number of children.</p>

<p>If you want to traverse a direct acyclic graph without going over a node twice (for example to count each node once), you can add a <code>mutable</code> boolean to each node indicating whether you have traversed the node before.  You can look at all descendants of a nodes by marking the node, looking at the node, and recursively traversing at the unmarked children of the node.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>It looks like you've already found an answer, but just for kicks <a href="http://en.wikipedia.org/wiki/Transitive_closure#Graph_theory" rel="nofollow">the transitive closure of a DAG</a> might be useful for others thinking about this kind of problem.</p>

<p><a href="http://www.cs.uwaterloo.ca/~tmchan/" rel="nofollow">Timothy Chan</a> published <a href="http://www.cs.uwaterloo.ca/~tmchan/apsp.ps" rel="nofollow">a paper in 2005 with a footnote about efficiently computing the transitive closure of a DAG</a>.  Quoting from the paper:</p>

<blockquote>
  <p>...for the simpler problem of computing the transitive closure of an
  unweighted directed graph, Yuster and Zwick in a recent paper asked
  for an O(mn)-time algorithm, but an O(mn/log n + n<sup>2</sup>) time
  bound is actually easy to get on the word RAM.<sup>2</sup></p>
  
  <p>...</p>
  
  <p><sup>2</sup> Proof: Assume that the graph is acyclic, since we can precompute the
  strongly connected components in linear time and contract each
  component.  We want to find the set <em>S<sub>u</sub></em> over all vertices
  reachable from each vertex <em>u</em>.  For each vertex <em>u</em> in reverse
  topological order, we can compute <em>S<sub>u</sub></em> by taking the union of
  <em>S<sub>v</sub></em> over all vertices <em>v</em> incident from <em>u</em>.  Each of these O(m)
  set-union operations can be carried out in O(n/log n) time by
  representing a set as an (n/log n)-word vector and by using the
  bitwise-or operation.</p>
</blockquote>

<p>Clearly there's still a bit to figure out there -- you need to precompute the "strongly connected components" and have to be able to visit nodes in reverse topological order -- but the process he describes of efficiently taking a repeated union sounds like a reasonable way to count the number of children for a given node in a DAG.</p>

<ul>
<li>Timothy M. Chan, <strong><a href="http://www.cs.uwaterloo.ca/~tmchan/apsp.ps" rel="nofollow">All-Pairs Shortest Paths with Real Weights in O(n<sup>3</sup>/log n) Time</a></strong>.  <em>Lecture Notes in Computer Science, 2005, Volume 3608, Algorithms and Data Structures, Pages 318-324.</em></li>
</ul>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I would cache the number of descendants of a node in the node itself.  Since the nodes are immutable, you can calculate and cache the descendant count and not worry about the cached values becoming stale.</p>

<p>The number of descendants of a given node is 1 plus the sum of the number of descendants of for each of its children (direct children, not indirect descendants).  Since each child has cached the count of its descendants, this is a very fast calculation.</p>

<p>Trying to break some algorithm? :naughty: The simplest way is to use n/nlogn( work in general ) n ( reference in general ) cases, in which you would store key references to that table, and look up whenever you need particular mapped hash stack. So for instance, n root node has n1 and n2 child nodes, n1 child node is not processed, but referenced and kept somewhere on your disk, but n1 root node( which is basically the same as n1 child node, but without the privelege to be counted ) will be further processed and have n11 and n12 child nodes, then n11 root node will be mapped as a reference to that tree and so on and on. So for instance you would have only 125,000 reference keys with the pointers to other reference keys and if needed something whatever you want, which will your PC truly process, and the rest is only kept to be processed and frittered in the mean time. </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>This could be done via a mapreduce approach, since part of the problem is large datasets. It's a very different way to do it, as it's in the realm of text files and clusters rather than C++, but it will at least scale with machine count without impacting time to completion too much.</p>

<p>The method would be to start with key:value pairs denoting each directed edge, and build up the sets of nodes descending from each until each is marked done. When collecting from two child nodes, use the intersection of the sets of children, and maintain the set of incomplete nodes separately. This should take something like 2 maps &amp; collects per level of depth, but obviously with reducing workload as you ascend the graph.</p>

<p>It is obviously more work computationally, but it can leverage the existing hadoop etc systems for ease of scaling.</p>

<p>I suppose there are two levels to this answer: a) consider a multi-pass approach, and b) consider hadoop et al to distribute the work.</p>
