    <div class="post-text" itemprop="text">

<p>There appears to be a myriad of NoSQL databases available these days:</p>

<ul>
<li>CouchDB</li>
<li>MongoDB</li>
<li>Cassandra</li>
<li>Hadoop</li>
</ul>

<p>There's also a boundary between these tools and tools such as Redis that work as a memcached replacement.</p>

<p>Without hand waving and throwing too many buzz words - my question is the following:</p>

<p>How does one intelligently decide which tool here makes the most sense for their project? Are the projects similar enough to where the answer to this is subjective, eg: Ruby is better than Python or Python is better than Ruby? Or are we talking Apples and oranges here in that they each of them solve different problems?</p>

<p>What's the best way to educate myself on this new trend?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Perhaps one way to think of it is, programming has recently evolved from using one general-purpose language for everything to using the general-purpose language for most things, plus domain-specific languages for the more appropriate parts. For example, you might use Lua to script artificial intelligence of a character in a game.</p>

<p>NoSQL databases might be similar. SQL is the general purpose database with the longest and broadest adoption. While it could be shoehorned to serve many tasks, programmers are beginning to use NoSQL as a domain-specific database when it is more appropriate.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I would argue, that the 4 major players you named do have quite different featuresets and try to solve different problems with different priority.</p>

<p>For instance, as far as i know Cassandra (and i assume Hadoop) central focus is on large scale installations.</p>

<p>MongoDb tries to be a better scaling alternative to classic SQL servers in providing comparably powerful query functions.</p>

<p>CouchDB's focus is comparably small scale (will not shard at all, "only" replicate), high durability and easy synchronization of data.</p>

<p>You might want to check out <a href="http://nosql-database.org/" rel="nofollow">http://nosql-database.org/</a> for some more information.</p>

<p>I am facing pretty much the same problem as you, and i would say there is no real alternative to look at all solutions in detail.</p>

<p>It's easy. NoSQL databases are ACID compliant databases minus some guarantees. So just decide which guarantees you can do without and find the database that fits. If you don't need durability for example, maybe redis is best. Or if you don't need multi-record transactions, then perhaps look into mongodb.</p>
