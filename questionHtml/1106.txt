<p>in additional to other answers there is one more factor: <br>
You have 30 files to process - so there are 30 tasks to execute. Hadoop MR overhead of 1 task execution is between 1 to 3 seconds. If you will merge data into one file - performance will improve seriously, while you will still have job start overhead.  <br>
I think local native program will always outperform hadoop MR. Hadoop is built with scalability and fault tolerance in mind - in many cases scarifying performance.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>For improving the Hadoop performance :</p>

<ul>
<li><p>Set the number of <strong>Mappers and Reducers</strong>.</p>

<p>[ By seeing the output of your program, I think you've used a number of reducers and mappers. Use it according to your need . Using too many mappers or reducers will not boost up the performance] </p></li>
<li><p>Use <strong>Larger chunk</strong> of data. (in Terabytes or at least in GBs)</p>

<p>[ In Hadoop, there is some funda of <strong>block size</strong> 64 mb.] </p></li>
<li><p>Setup Hadoop to some other terminals and try to run in <strong>multi node clusters</strong>. It will boost up the performance.</p></li>
</ul>

<p><strong>Hadoop</strong> is the next big thing.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>Hmm.. there is a confusion here, or let me create a confusion here. </p>

<p>Suppose you have a problem which could be solved in say <code>O(n) complexity</code> what hadoop will do if you apply lets suppose <strong>K machines</strong> then it will reduce the complexity by K times. So in your case the task should have performed faster(hadoop task).</p>

<p>WHAT WENT WRONG ?????</p>

<blockquote>
  <p>Assuming you have standard hadoop installation and all the standard hadoop configuration and also assuming you are by default running hadoop in local mode.</p>
</blockquote>

<p>1) You are running the program on a single node so dont expect a running time of anything less than a standard program. (The case would have been different if you used a multi node cluster)</p>

<p>Now the question arises since single machine is used the running time should have been same ???</p>

<p>The answer is no, in hadoop the data is first read by the record reader which emits key value pairs which are passed to the mapper which then processes and emits key value pairs(supposing no combiner is used) then the data sorted and shuffled and then it is passed to the reducer phase then writing to hdfs is done. So see there are much more overheads here. And you perceiving a degraded performance because of these reasons. </p>

<p>You want to see what hadoop can do. Run the same task on a K node cluster and take peta bytes of data and also run a single threaded application. I promise you will be startled.</p>

<p>Even though Hadoop is not meant for this small file, we still can tune it to some extent. The file size is 180 kb. But the number of blocks are 30. You must have reduced "dfs.block.size" in hdfs-site.xml. As the input splits are more, the number of mappers are also more which is unnecessary in this case. Hadoop has to be tuned as per the number of nodes and input data. So you have to increase the "dfs.block.size" upto 64MB, to perform this word count with one mapper which will significantly improve the performance. </p>
