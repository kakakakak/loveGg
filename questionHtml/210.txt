<p>I noticed that doxygen uses the graphviz library for creating diagrams. Have you ever used graphviz for generating documentation? Is it worth learning the graphviz for documentation purposes outside the scope of doxygen? Or am I better off to sticking with a standard data modeling package like Visio?</p>

<p>I understand the merits of it as a graphing library, but for trying to represent more complex UML (or similar) is it still worth looking into?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>If you're just talking about creating inheritance/collaboration diagrams like Doxygen does, it's worth investigating IDEs that will do that for you automatically. For from-scratch or hand-tuned documentation, I use <a href="http://www.omnigroup.com/applications/OmniGraffle/">OmniGraffle</a> (since I'm on a Mac) which I highly recommend.</p>

<p>However, <a href="http://en.wikipedia.org/wiki/Graphviz">GraphViz</a> and <a href="http://en.wikipedia.org/wiki/DOT%5Flanguage">DOT</a> can be really handy, not only for documentation, but for debugging and code comprehension as well, particularly for data structures. I generally don't write DOT by hand, but automatically-generated DOT can be well worth the minimal effort.</p>

<p>One of the places I've found GraphViz extremely useful is for understanding and debugging binary search tree algorithms. I develop <a href="http://cocoaheads.byu.edu/code/CHDataStructures">CHDataStructures.framework</a>, an open-source Objective-C framework, which includes several varieties of BSTs. I implemented two methods: <code>-(NSString*)dotGraphString</code> on the parent class and <code>-(NSString*)dotGraphStringForNode:</code> on each child class. In about 30-40 lines of code (most of it at the bottom of <code>CHAbstractBinarySearchTree.m</code>), I added the ability to iteratively traverse a binary tree and create a DOT representation of it, including balancing information, coloring nodes red or black, etc. (With a little care, you can easily represent null sentinel nodes and display the tree in proper sorted order.)</p>

<p>In my test code, after each modification of the tree, I called -<code>dotGraphString</code> and saved the result to a .dot file, stopped there with a breakpoint, then opened that file with GraphViz, which is smart enough to re-render the DOT graph when the file is updated. This approach made it vastly easier to see what was happening in the tree and spot bugs in my implementation of a given algorithm. This approach can be adapted fairly easily for various kinds of data structures, and is generally much faster and easier than creating a UI just for visualizing the structure.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I use GraphViz extensively for documentation and often sketch out relationships or architecture diagrams using GraphViz externally then add them to extra pages in my Doxygen code using the @dot/@enddot. I've also recently started using @dotfile which has the dual benefit of keeping large dot statements out of the code docs and allowing me to continue to preview them with the GraphViz GUI.</p>

<p>The other big benefit with GraphViz is that the simple textual format works very well with version control. You can see changes to diagrams in your git diff which would be impossible with any binary documentation format. As I've used it more an more over the years, this is becoming a more important feature to me.</p>

<p>However, for UML I use a true UML tool (<a href="http://www.sparxsystems.com.au/" rel="nofollow">Enterprise Architect</a>) rather than stuffing around in Visio.</p>

<p>I had a go with Graphviz quite a few times, though found it quite limiting.</p>

<p>What I found best for manual text-entry diagrams is <a href="http://www.texample.net/tikz/" rel="nofollow">Tikz</a>
I used it for <a href="http://www.texample.net/tikz/examples/feature/automata-and-petri-nets/" rel="nofollow">FSM's</a> </p>

<p>If Graphviz output satisfies your particular needs, it's probably fine.
If you are not quite happy with what it does and have some TeX experience -
have a look at Tikz. There many different packages out there, you don't
need to learn pure Tikz/PGF (which may seem quite heavy).</p>
