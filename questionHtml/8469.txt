    <div class="post-text" itemprop="text">

<p>I am working on translating an existing time series database system to MapReduce model using Hadoop. 
The database system has both historical and real-time processing capabilities. 
So far, I was able to translate the batch processing functionality into Hadoop. </p>

<p>Unfortunately, when it comes to real-time processing, I see there are some conceptual inconsistencies with the MapReduce model.</p>

<p>I can write my own implementation of Hadoop's <em>InputFormat</em> interface which will <em>continuously</em> feed mappers with new data so that mappers can process and <em>continuously</em> emit data. 
However, because no <em>reduce()</em> method is being called until all mappers have completed their execution,  my computation is bound to be stuck at the mapping stage. </p>

<p>I've seen some posts mentioning <code>mapred.reduce.slowstart.completed.maps</code>, but, as I understand, this only controls when reducers will start pulling data to their local destinations (shuffling) -- the actual reduce method is called only after all mappers have completed their execution. </p>

<p>Of course, there is an option of mimicking a continuous execution by processing small batches of data over small time-intervals using a continuous stream of separate MR-jobs, but this would introduce additional latencies which is not acceptable in my case. </p>

<p>I've also considered using <a href="http://storm.incubator.apache.org/" rel="nofollow">Storm</a> or <a href="http://incubator.apache.org/s4/" rel="nofollow">S4</a>, but before moving any further I need to make sure that this falls out of the scope of Hadoop. </p>

<p>In summary, it looks like people have been able to develop real-time Hadoop applications (such as Impala) or real-time processing solutions built on top of Hadoop. The question is how?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>You are right that the <em>reduce</em> method will never get called if the InputFormat/mappers emit the data continuously. The reason for that is that the reduce method has to iterate over all values for the key, and the full set of values is unknown until the map phase is done, since the value to be given to that <em>reduce</em> method may come from any mapper at any time.</p>

<p>The reduce method accesses values through an iterator, so it is theoretically possible from an API standpoint to call <em>reduce()</em> upfront and make it run perpetually blocking on the iterator until values become available. The reason why this feature is absent from Hadoop is that it would require keeping the context for each key in memory, which does not make sense for batch processing of large data sets.</p>

<p>One way to achieve continuous analysis of the stream of data within the Hadoop MapReduce programming model is to submit a continuous stream of small MR-jobs, with each analyzing a chunk of data. The way to deal with additional latency in that case is to use one of the many available Hadoop accelerators (<em>Disclaimer: I work for a company, ScaleOut Software, who provides such an accelerator: ScaleOut hServer — available in a free community edition</em>). <a href="http://www.scaleoutsoftware.com/hserver" rel="nofollow">ScaleOut hServer</a> is in-memory MapReduce engine that can run MR jobs in milliseconds. It reuses JVMs between jobs, so the job start up latency is minimal compared to Hadoop. This is a good fit for continuously running MapReduce jobs on chunks of data because it is optimized for real-time performance on the data sets that fit in memory.</p>

<p>One exception to all of the above is if an analysis doesn't require a reduce phase (i.e., the number of reducers is set 0): if the algorithm can be expressed as map-only, it can be done continuously with one Hadoop batch job. </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I've personally gone down the path to implement a streaming solution with MapReduce and it is not pretty. Whenever you try to do something out of the box with MapReduce, it'll fail one way or another.</p>

<p>MapReduce is specifically for batch and the streaming processing capabilities you are looking for is one of the main reasons technologies such as Flume and Storm came to be. Both of these are considered defacto standards in the hadoop ecosystem and have good integration with the rest of Hadoop.</p>

<hr>

<p>You mention that you are dealing with time series data. Have you taken a look at <a href="http://opentsdb.net/" rel="nofollow">OpenTSDB</a>? It's a time series database built on top of HBase and HDFS.</p>
